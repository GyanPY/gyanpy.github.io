Phase 1
LLM foundations
Build intuition for how LLMs work, how to call them, and how to ground them in real data. You'll ship your first RAG system with a proper evaluation layer.
⏱ weeks 1–3 🎯 1 project skip python basics
1.1
LLM API mastery (raw, no frameworks)
Start with raw API calls — frameworks hide crucial details about tokens, pricing, and failure modes.
What to learn
  • OpenAI / Anthropic / Gemini API — chat completions, system prompts, message history
  • Tokens, context windows, temperature, top-p — how they affect output
  • Streaming responses, async calls, error handling, retries
  • Cost estimation — tokens per request, budget planning
Tools
  • OpenAI SDK, Anthropic SDK, litellm (multi-provider switching)
  • FastAPI for wrapping LLM calls as API endpoints
1.2
Prompt engineering (systematic, not ad-hoc)
The difference between a mediocre and excellent AI product is often just prompt design.
What to learn
  • Zero-shot, few-shot, chain-of-thought prompting
  • Role-based prompts, output formatting (JSON mode, structured outputs)
  • Prompt templating, versioning, and testing
  • Prompt injection attacks and defenses
1.3
Embeddings + vector databases
This is the foundation of RAG — you must understand how text becomes searchable vectors.
What to learn
  • What embeddings are — semantic similarity, cosine distance
  • OpenAI ada-002 / text-embedding-3-large, Cohere embed, open-source (BAAI/bge)
  • Vector DB operations — indexing, querying, filtering, metadata
  • Chunking strategies — fixed-size, semantic, recursive character splitting
Tools
  • ChromaDB (local dev), Pinecone (managed), Weaviate, pgvector (PostgreSQL — leverage your DB skills)
1.4
RAG pipeline — build to eval
RAG is the single most in-demand AI skill in 2026. But building it is easy — evaluating it is what gets you hired.
What to learn
  • Ingestion → chunking → embedding → storage → retrieval → generation
  • Retrieval precision vs recall — are you getting the right chunks?
  • Answer faithfulness — is the answer grounded in retrieved context?
  • Hallucination detection — does the model make things up?
🏗️ Project 1: RAG document Q&A with eval harness
Build a RAG pipeline over technical docs (e.g., Kubernetes docs, your company's internal wiki). Include a comprehensive eval layer — compare 2 chunking strategies, 2 embedding models. Measure retrieval precision, answer faithfulness, and hallucination rate using RAGAS.
stack: Python · FastAPI · pgvector or Chroma · OpenAI/Anthropic API · RAGAS · LlamaIndex or LangChain
hiring signal
"This candidate understands the difference between building an AI system and knowing whether it works."
Phase 2
Advanced RAG + MCP servers
Level up RAG to production-grade, and learn MCP — the protocol that connects agents to the real world. This is your biggest backend-engineer advantage.
⏱ weeks 4–7 🎯 2 projects core skill
2.1
Advanced RAG patterns
Basic RAG fails in production. You need hybrid search, re-ranking, and query transformation.
What to learn
  • Hybrid search — combining vector similarity + BM25 keyword search
  • Re-ranking — Cohere reranker, cross-encoder models
  • Query transformation — HyDE (hypothetical document embeddings), multi-query
  • Parent-child chunking, contextual compression
  • Multi-modal RAG — images, tables, PDFs with layout understanding
Tools
  • LlamaIndex (query pipelines, retrievers), LangChain (LCEL chains)
  • Cohere reranker, ColBERT, sentence-transformers
2.2
Function calling + structured outputs
The bridge between LLMs and code — how models trigger real actions.
What to learn
  • OpenAI/Anthropic tool-use / function calling — JSON schema definitions
  • Structured output with Pydantic — type-safe LLM responses
  • Parallel tool calls, tool choice forcing
  • Instructor library — structured extraction from LLM outputs
2.3
MCP — Model Context Protocol
MCP is the TCP/IP of AI agents. Building MCP servers is the highest-leverage skill for backend engineers.
What to learn
  • MCP architecture — clients, servers, tools, resources, prompts
  • Transport protocols — stdio (local), SSE, Streamable HTTP
  • Building an MCP server — exposing tools, handling requests
  • Progressive discovery — how agents find and use MCP servers
  • Security — auth, input validation, rate limiting
Tools
  • @modelcontextprotocol/sdk (official Python SDK)
  • Claude Desktop, VS Code as MCP clients for testing
🏗️ Project 2: Production MCP server
Build an MCP server that exposes a real backend system — Kafka topic browser, PostgreSQL query interface, Kubernetes cluster inspector, or a Jira/Confluence connector. Include auth, error handling, and rate limiting. Test it with Claude Desktop.
stack: Python · MCP SDK · FastAPI · your existing infra tools
your edge
Most AI engineers can build toy MCP servers. You can build one that wraps real production infrastructure — Kafka, K8s, Postgres. This is rare and immediately hireable.
2.4
Production RAG system
🏗️ Project 3: Enterprise document assistant
Build a multi-source RAG system that handles PDFs, Confluence pages, and code repos. Implement hybrid search (vector + BM25), re-ranking, and caching. Deploy with FastAPI, add auth, and track costs per query. Write a thorough README with architecture diagram.
stack: Python · LlamaIndex · pgvector · Redis · FastAPI · Docker
Phase 3
AI agents + multi-agent systems
The frontier of 2026 AI. Learn to build agents that plan, use tools, manage state, and collaborate with other agents.
⏱ weeks 8–12 🎯 2 projects frontier skill
3.1
Agent fundamentals
Understand the building blocks before touching frameworks.
What to learn
  • ReAct pattern — Reason + Act loop (the core agent loop)
  • Planning — chain-of-thought, tree of thoughts, reflexion
  • Tool use — function calling as the action mechanism
  • Memory — short-term (context window), long-term (vector store), episodic (session)
  • Failure modes — infinite loops, hallucinated tool calls, context overflow
3.2
LangGraph — the production standard
LangGraph models agents as state machines with checkpointing and crash recovery — exactly how you think about distributed systems.
What to learn
  • Graph-based workflows — nodes (functions), edges (transitions), conditional routing
  • State management — TypedDict state, reducer functions for concurrent updates
  • Checkpointing — MemorySaver, SqliteSaver, PostgresSaver
  • Human-in-the-loop — interrupt nodes, approval gates
  • Time-travel debugging — rollback to any state and replay
  • Sub-graphs — composing complex workflows from reusable sub-agents
Tools
  • LangGraph, LangSmith (tracing), LangServe (deployment)
🏗️ Project 4: Stateful LangGraph agent with recovery
Build a code review agent that clones a repo, analyzes PRs, identifies issues, and suggests fixes. Implement checkpointing so it survives crashes mid-review. Add human-in-the-loop approval before auto-commenting. Include branching logic (security issues → escalate, style issues → auto-fix).
stack: Python · LangGraph · GitHub API · PostgresSaver · LangSmith
3.3
Multi-agent systems + protocols
The next wave — multiple specialized agents collaborating on complex tasks.
What to learn
  • CrewAI — role-based agent teams, task delegation
  • AutoGen / AG2 — conversational agent patterns, GroupChat
  • A2A protocol — Agent Cards, cross-framework agent discovery and delegation
  • AG-UI protocol — how agents communicate with user interfaces
  • Orchestration patterns — sequential, parallel, hierarchical, debate
  • Token cost management — each agent turn = full LLM call
🏗️ Project 5: Multi-agent research system
Build a system with 3 agents: Researcher (web search + document reading), Analyst (data processing + comparison), and Writer (synthesis + report generation). Use CrewAI or LangGraph sub-graphs. Agents should coordinate via structured outputs, not free-form chat. Add an MCP server for data access.
stack: Python · CrewAI or LangGraph · MCP · Tavily API · FastAPI
3.4
Agentic RAG
When basic RAG isn't enough — agents that decide when, what, and how to retrieve.
What to learn
  • Query routing — agent decides which data source to query
  • Adaptive retrieval — agent reformulates queries based on initial results
  • Self-correcting RAG — agent evaluates retrieval quality and retries
  • Multi-step retrieval — complex questions decomposed into sub-queries
Phase 4
Evaluation + observability + guardrails
The rarest skill in AI engineering. Most people ship without evals. The ones who get hired at Staff+ level know how to measure whether their AI system actually works.
⏱ weeks 13–16 🎯 1 project staff-level signal
4.1
LLM evaluation frameworks
You can't improve what you can't measure. Eval design is the #1 gap in AI hiring.
What to learn
  • RAGAS metrics — context precision, faithfulness, answer relevancy
  • LLM-as-judge — using one model to evaluate another
  • DeepEval — pytest-style testing for LLM outputs
  • Custom eval metrics — domain-specific quality measures
  • Regression testing for prompts — detecting when prompt changes break things
  • A/B testing for AI — comparing model versions in production
Tools
  • RAGAS, DeepEval, Promptfoo, Braintrust
4.2
Observability + tracing
When your agent does something wrong at 2 AM, you need to know exactly which step failed and why.
What to learn
  • OpenTelemetry for LLM calls — spans, traces, attributes
  • LangSmith — trace visualization, prompt playground, datasets
  • Arize Phoenix — open-source LLM observability
  • Cost tracking — per-request token usage and spend
  • Latency monitoring — p50, p95, p99 for LLM calls
4.3
Guardrails + safety
What to learn
  • Input validation — prompt injection detection, content filtering
  • Output validation — PII detection, hallucination flags, format enforcement
  • Guardrails AI, NeMo Guardrails — declarative guardrail frameworks
  • Human-in-the-loop escalation patterns
🏗️ Project 6: Eval + observability platform
Build an evaluation harness for your RAG/agent projects from phases 1-3. Create a benchmark dataset (50+ test cases), implement RAGAS + LLM-as-judge evals, add OpenTelemetry tracing, build a dashboard showing accuracy, latency, cost, and hallucination rate over time. Add prompt regression tests that run in CI.
stack: Python · RAGAS · DeepEval · OpenTelemetry · Grafana or Streamlit · GitHub Actions
staff-level signal
This single project — an eval framework with CI integration — is the highest-signal artifact you can show in a Google L6+ interview. It proves you think about systems, not just features.
Phase 5
MLOps + fine-tuning
Understand model internals just enough to fine-tune, serve, and operate models in production. You don't need to train from scratch — but you need to know how to adapt and deploy.
⏱ weeks 17–20 🎯 1 project differentiator
5.1
Transformer internals (just enough)
You don't need to implement a transformer — but you need to understand why attention matters, what KV-cache is, and why context windows have limits.
What to learn
  • Attention mechanism — Q/K/V, self-attention, multi-head attention
  • Tokenization — BPE, SentencePiece, why token counts ≠ word counts
  • KV-cache — why inference gets slower with longer contexts
  • Model sizes — parameter count, quantization (4-bit, 8-bit), GGUF
5.2
Fine-tuning with LoRA / QLoRA
When prompting isn't enough — adapt a model to your domain without training from scratch.
What to learn
  • When to fine-tune vs when to use RAG — decision framework
  • LoRA — low-rank adaptation, freezing base weights
  • QLoRA — quantized fine-tuning for consumer GPUs
  • Dataset preparation — JSONL format, instruction tuning format
  • Training with Hugging Face — Trainer API, PEFT library
  • Evaluation — perplexity, task-specific metrics
Tools
  • Hugging Face Transformers, PEFT, TRL, Axolotl
  • Llama 3, Mistral, Phi-3 (open-source base models)
5.3
Model serving + deployment
Your K8s and Docker skills shine here — this is where backend engineers dominate.
What to learn
  • vLLM — high-throughput model serving with PagedAttention
  • Ollama — local model serving for development
  • Model registries — Hugging Face Hub, MLflow model registry
  • Containerized deployment — Docker + K8s for model serving
  • GPU allocation, scaling, batch inference
🏗️ Project 7: Fine-tuned model deployment pipeline
Fine-tune Llama 3 or Mistral on a domain-specific dataset (e.g., code review comments, support tickets). Deploy with vLLM behind FastAPI. Containerize with Docker. Add an A/B test comparing fine-tuned vs base model using your eval framework from Phase 4. Document the cost analysis (training cost vs API cost savings).
stack: Python · HuggingFace · PEFT · vLLM · Docker · FastAPI · your eval framework
your edge
Most AI engineers can fine-tune a model in a notebook. You can deploy it in a container, serve it behind an API, scale it on K8s, and prove it works with evals. That's a Staff Engineer's portfolio.
Phase 6
Capstone + interview prep
Combine everything into one impressive end-to-end system, polish your portfolio, and practice AI system design interviews.
⏱ weeks 21–24 🎯 1 capstone interview-ready
6.1
AI system design
Google L6/L7 interviews test system design for AI — not just traditional distributed systems.
What to learn
  • Design a RAG system at scale — sharding vectors, caching, invalidation
  • Design an AI agent platform — multi-tenant, rate limiting, cost controls
  • Design a real-time recommendation system with LLM re-ranking
  • Design a document processing pipeline — chunking, OCR, entity extraction at scale
  • Latency budgets — LLM call latency vs total system SLA
  • Cost modeling — per-query cost, caching strategies to reduce API spend
6.2
AI governance + responsible AI
What to learn
  • Bias detection in AI outputs — fairness metrics
  • Data privacy — PII handling in RAG systems, GDPR compliance
  • AI transparency — explainability, audit trails for agent decisions
  • EU AI Act basics — risk categories, compliance requirements
6.3
Capstone project
🏗️ Project 8: Full-stack AI operations assistant
Build an end-to-end AI-powered DevOps/SRE assistant that combines everything:

• MCP servers for K8s, Postgres, Grafana, and PagerDuty access
• LangGraph agent with state management and human-in-the-loop
• Agentic RAG over runbooks and incident history
• Multi-agent — triage agent → diagnosis agent → remediation agent
• Eval framework — accuracy benchmarks on historical incidents
• Observability — full OpenTelemetry tracing
• Guardrails — never auto-execute destructive commands without approval

Deploy on K8s. Write a detailed architecture doc. Record a 5-minute demo video.
stack: Python · LangGraph · MCP · pgvector · vLLM or API · K8s · OpenTelemetry · RAGAS · FastAPI · Docker
portfolio signal
This single project demonstrates: agent orchestration, production infrastructure, evaluation discipline, safety thinking, and distributed systems expertise. It's a Staff Engineer's thesis project.
