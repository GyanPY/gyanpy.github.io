PILLAR 01 Core Concepts & Mental Models
The Problem, The Abstraction, The Guarantees

Understanding Memcached demands we start with the world as it existed before it — a world where every user request translated into expensive, blocking database queries at a time when databases couldn't scale horizontally.

What Problem Does Memcached Solve?

Before 2003, the dominant web architecture was a thin application layer sitting directly atop a relational database. Every page view, every API call, every user profile lookup translated into one or more SQL queries against a single database server. As sites like LiveJournal — where Memcached was born — began accumulating tens of thousands of concurrent users, this architecture cracked under load. The database became the universal bottleneck. It wasn't compute-bound on the application tier; it was I/O and CPU bound deep inside the RDBMS that was doing B-tree traversals for every HTTP request.

The insight that birthed Memcached is simple but profound: most data read from a database is read many times, but changes rarely. A user's profile, a product's details, a rendered HTML fragment — these are fetched thousands of times per second but updated at most a few times per minute. The database is doing enormous redundant work. What if you could remember the result of a query in RAM, where access is ~100 nanoseconds instead of ~10 milliseconds? That's a 100,000x improvement in access latency. Even a 60–80% cache hit rate transforms a system that's collapsing under load into one with headroom to spare.

The world before Memcached dealt with this by adding read replicas (expensive, high operational overhead, lag issues), by denormalizing schemas aggressively (painful to maintain, often incorrect), by caching inside the application process itself (not shared across application servers, so each server has a cold cache), or simply by buying bigger, faster database hardware (vertical scaling has hard limits and costs grow super-linearly). None of these solutions were composable, shareable across a fleet of application servers, or independently scalable.

The Core Abstraction

Memcached provides a distributed, in-memory key-value store with TTL-based expiry and a slab allocator for O(1) memory management. The abstraction it offers is deceptively simple: a giant hash map that lives in RAM, accessible over the network, shared by all application servers, where entries automatically expire. That simplicity is the source of its power — and the source of many misuse mistakes.

The abstraction is powerful for three reasons. First, network locality is irrelevant — any application server can reach any cached value in sub-millisecond time, so the cache is effectively shared global state for the entire application tier without any process-level coordination. Second, TTL-based expiry is a built-in cache invalidation strategy — you don't have to track every write to know when to evict; you can declare "this data is valid for 60 seconds" and let the system enforce it, accepting eventual consistency. Third, the interface is intentionally minimal — get, set, delete, add, replace, incr, decr — which means the learning curve is near-zero and the protocol overhead is minimal (binary protocol headers are 24 bytes).

Fundamental Guarantees (and Explicit Non-Guarantees)

✓ Guarantees
What Memcached Promises

O(1) get/set: Hash table lookups are constant time. With a well-distributed key space, there are no B-tree traversals, no index scans, no disk seeks.

Sub-millisecond latency: Typical P50 is 100–300 μs, P99 under 1 ms over LAN. This is fundamental to its value proposition.

Atomic single-key operations: get, set, delete on a single key are atomic. incr/decr are atomic counters with wrap-around semantics.

TTL-based expiry: Items will not be served after their TTL expires. The actual deletion may be lazy (on next access) or active (background sweeper), but the semantic guarantee holds — expired items are never returned.

Memory boundedness: Memcached will never consume more RAM than its configured limit. It enforces this through LRU eviction within slab classes.

✗ Non-Guarantees
What Memcached Explicitly Does NOT Promise

No durability: A process restart, OOM kill, or server crash means all cached data is gone. You must treat Memcached as a warm-up cache, never as a system of record.

No replication: Vanilla Memcached has no built-in replication. Data lives on exactly one node (as determined by client-side consistent hashing). Node loss = cache miss for all keys owned by that node.

No cross-key atomicity: There is no transaction primitive. You cannot atomically update keys A and B together. Race conditions between gets and sets across keys are your responsibility to handle.

No ordered access: You cannot range-scan, iterate, or enumerate keys. There's no secondary index. It's a pure hash map.

No strong consistency: With multiple app servers doing concurrent writes, there is no mechanism to serialize updates or prevent lost writes. Memcached offers last-write-wins semantics (via CAS) at best.

Invariants That Must Always Hold

For Memcached to function correctly in your system, certain invariants must be maintained — not by Memcached itself (it has no way to enforce them), but by your application code and operational discipline:

  • The database is always the source of truth. Memcached is a read-through cache. Any invalidation race or eviction must fall back to the database, not to stale cache data.
  • Cache values must be logically consistent with database state within their TTL window. If you cache a user's account status for 60 seconds, your product must accept up to 60 seconds of stale data for that field.
  • Client-side hashing configuration must be consistent across all application servers. If server A and server B disagree on which Memcached node owns key "user:1234", you get split-brain caching — double the misses, and writes to different nodes for the same key.
  • Memory must never be allowed to be the bottleneck for Memcached itself. If the OS starts swapping Memcached to disk, you have fundamentally broken the latency guarantee. Memcached must always have its full allocation in physical RAM.
  • Slab class distribution must approximately match your value size distribution. A poorly configured slab layout causes massive internal fragmentation, wasting the very memory Memcached was given.

Key Takeaway — Pillar 1

Memcached's power lies not in technical sophistication but in the radical simplicity of its abstraction: a shared, TTL-aware, network-accessible hash map that converts expensive database round-trips into cheap in-memory lookups — but it provides none of the durability, consistency, or replication guarantees that engineers often unconsciously assume.

PILLAR 02 Architecture Deep Dive
Inside the Engine

Memcached's internal architecture is a masterclass in doing one thing exceptionally well. Every component is optimized for the single goal: serve cached values at sub-millisecond latency under high concurrency.

Internal Component Breakdown

1. Network Layer — libevent-based I/O Multiplexer

Memcached uses libevent, an event notification library that wraps epoll (Linux), kqueue (BSD/macOS), or poll as an abstraction. A single main thread handles all new connection accepts on the listening socket. When a new connection arrives, the main thread dispatches it to one of the worker threads (configured via -t, default 4). Each worker thread runs its own libevent loop, managing its own set of connections. This means Memcached's I/O model is non-blocking, event-driven, and multi-threaded — but critically, each connection is pinned to a single worker thread for its lifetime, avoiding the need to lock around per-connection state. The number of simultaneous connections Memcached can handle is bounded only by file descriptor limits and memory for connection state, typically tens of thousands.

2. Slab Allocator — The Heart of Memory Management

This is the most important internal component to understand deeply. The slab allocator is Memcached's answer to the malloc fragmentation problem. Traditional malloc would, over time, fragment the heap as objects of varying sizes are allocated and freed, eventually making large allocations fail even when technically enough free bytes exist. The slab allocator eliminates this entirely by pre-dividing memory into size classes and never mixing objects of different sizes within a slab.

Memory is organized as follows: The entire Memcached memory budget (e.g., 64 GB) is divided into 1MB slabs. Each slab is assigned to a slab class, which represents a fixed item size. Slab classes range from 48 bytes (the minimum chunk size) up to 1MB, with each class being approximately 1.25× the previous (the growth factor, configurable via -f, default 1.25). An item stored in Memcached occupies the smallest slab class chunk that can fit its key + value + metadata. If your value is 100 bytes, it goes into the ~128-byte slab class, wasting up to 27% of space — that's internal fragmentation, and it's a known trade-off for elimination of external fragmentation.

Within each slab, chunks are tracked in a free list. Allocation is O(1): pop the head of the free list. Deallocation is O(1): push onto the free list. No coalescing, no compaction, no garbage collection. This is what allows Memcached to sustain millions of operations per second without GC pauses.

3. Hash Table — Main Key-Value Index

Memcached maintains a global hash table for key lookup. In recent versions (1.5+), this is a lock-striped hash table — the key space is divided into many buckets, each protected by its own lock, so concurrent lookups for different keys don't block each other. The hash function used is Jenkins hash (older versions) or MurmurHash (newer versions for better distribution). When a key-value item is stored, its key is hashed, and a pointer to the item's location in the slab allocator is stored in the hash table bucket. Lookup is a hash computation + pointer dereference — O(1) and typically two cache line misses in the L3 cache.

The hash table grows dynamically: when the load factor (items/buckets) exceeds a threshold (1.5 by default), the table doubles in size. This doubling happens in the background via a dedicated hash table maintenance thread to avoid blocking request processing, but during doubling, some operations may need to check two locations.

4. LRU Eviction — Per-Slab-Class LRU Lists

Each slab class maintains an LRU doubly-linked list. When a slab class runs out of free chunks (it's full), it must evict the least-recently-used item to make space for a new one. This is important: LRU eviction is scoped to the slab class, not global. This means a slab class full of large items won't steal memory from a slab class holding small items, even if the small-item class has many old, stale entries. This design choice prevents thrashing between workloads of different value sizes but can lead to underutilization if your value size distribution changes significantly from what was configured at startup.

Memcached 1.5+ introduced a significant improvement: segmented LRU (SLRU) with three sub-queues per slab class: HOT (recently accessed), WARM (accessed more than once), and COLD (candidates for eviction). Items graduate from COLD to WARM when accessed again. This dramatically reduces the "cache pollution" problem where a sequential scan of rarely-accessed items could evict hot items from a pure LRU.

5. Worker Threads — Request Processing Pipeline

Each worker thread maintains its own read buffer, write buffer, and per-connection state machine. When a client sends a GET user:1234, the worker reads the command off the socket, parses it (ASCII protocol) or reads the 24-byte header (binary protocol), computes the hash, acquires the relevant hash table lock for that bucket, looks up the item pointer, checks TTL expiry, releases the lock, reads the value from the slab, and writes the response to the output buffer. The entire operation happens without blocking any other worker thread.

6. Expiry — Lazy + Active

Memcached uses two-tier expiry. Lazy expiry: when a get request finds an item, it checks the timestamp stored in the item header against current time. If expired, the item is treated as a miss and the space reclaimed. This means expired items can sit in memory indefinitely if never accessed again. Active expiry: a background LRU crawler thread (introduced in 1.4.27) walks the LRU tails periodically and removes expired items, reclaiming their slab chunks for reuse. The crawler's aggressiveness is configurable.

Full Request Data Flow — READ PATH (GET)

  1. Client sends GET command
    Application server serializes the cache key (e.g., user:profile:8823411), computes which Memcached node owns it via client-side consistent hashing (Ketama algorithm), opens or reuses a connection from the connection pool to that node, and sends the ASCII or binary protocol request over TCP.
  2. Network layer receives data
    libevent on the Memcached node detects the socket is readable, invokes the read callback for the worker thread handling this connection. The worker reads available bytes into its per-connection read buffer, appending to any partial previous read.
  3. Protocol parsing
    For ASCII protocol: scan for \r\n, split on space, extract command name and key. For binary protocol: read 24-byte header, extract opcode (0x00 = GET), key length, extras length, total body length. The binary protocol is more efficient: no text parsing, field-aligned reads.
  4. Hash computation + bucket lock
    Compute hash(key) % num_hash_buckets to find the bucket index. Acquire the per-bucket mutex (or read-lock in newer versions with reader-writer locks). This is typically the only synchronization point in the critical path.
  5. Hash table lookup
    Traverse the bucket's chain (chaining for collision resolution) comparing key strings until a match or end of chain. On match, obtain a pointer to the item struct in the slab allocator. Release the bucket lock immediately after obtaining the pointer and incrementing the reference count.
  6. TTL check
    Read the exptime field from the item header. Compare against current_time (a global variable updated by a dedicated clock thread every second to avoid expensive gettimeofday syscalls in the hot path). If expired, treat as miss, decrement reference count, return VALUE_NOT_FOUND. This is lazy expiry.
  7. LRU update
    Move the item to the head of its slab class's LRU list (or update its "last access time" for SLRU promotion logic). This requires acquiring the LRU lock for that slab class — a second, brief synchronization point.
  8. Response serialization + write
    Serialize the response: ASCII: VALUE <key> <flags> <bytes>\r\n<data block>\r\nEND\r\n. Binary: 24-byte response header + value bytes. Write to the worker's per-connection output buffer. libevent schedules the socket write when the socket is writable (edge-triggered).
  9. Client receives response
    Client deserializes the response, deserializes the value (JSON, MessagePack, protobuf, etc.), and returns it to application code. The entire round-trip: ~200–400 μs over a datacenter LAN at P50, ~800 μs at P99.

Full Request Data Flow — WRITE PATH (SET)

  1. Client sends SET command with key, flags, TTL, value
    Application serializes the value (JSON/protobuf/MessagePack), computes the destination node via consistent hashing, and sends: ASCII: set <key> <flags> <exptime> <bytes>\r\n<data>\r\n. Binary: 24-byte header + extras (flags, expiry) + key + value.
  2. Slab class selection
    Determine total item size: key length + value length + item header overhead (~57 bytes). Select the smallest slab class that fits this size. If no slab of this class exists with free chunks, either allocate a new 1MB slab page (if memory headroom exists) or trigger LRU eviction within this slab class.
  3. Memory allocation from slab
    Pop the head of the slab class's free list. Write the item struct header: flags (4B), exptime (relative seconds or absolute timestamp), key length, value length, key bytes, value bytes. This is a memcpy — extremely fast.
  4. Hash table insertion
    Compute hash(key), acquire bucket lock, check for existing entry with same key. If found, unlink it (will be freed). Link new item into bucket chain. Release lock. Update global stats (get_hits, set_cmds, bytes stored) — these are per-thread counters that are only aggregated when stats are requested, to avoid lock contention in the hot path.
  5. LRU list insertion
    Link the new item at the head of the slab class's LRU list. Acquire slab class LRU lock, do pointer surgery, release lock. Item is now the most-recently-used item in its class.
  6. Response: STORED
    Send "STORED\r\n" (ASCII) or a 24-byte success response header (binary). From client perspective, the SET is complete. There is no acknowledgment beyond this — no write-ahead log, no replication confirmation, no fsync.

CAS (Check-And-Set) — Optimistic Locking

Memcached supports CAS (Compare-and-Swap) operations, which are its only mechanism for conditional writes. Each item has a hidden 64-bit cas_token (a monotonically increasing integer, not a clock timestamp). A client does: (1) gets key — like get, but returns the current cas_token along with the value; (2) modify the value in application memory; (3) cas key <new_value> <cas_token> — Memcached atomically checks that the current item's cas_token matches the provided token, and only stores the new value if it does. If another client has modified the item since the gets (incrementing the cas_token), the CAS fails with EXISTS, and the client must retry. This is optimistic locking — it assumes conflicts are rare, which is true in most caching workloads.

Back-Pressure and Flow Control

Memcached has very limited built-in back-pressure mechanisms, which is important to understand. When a Memcached node is overwhelmed — whether by CPU saturation (worker threads maxed), connection count at the limit (-c option, default 1024), or network saturation — it will simply drop new connections or queue them in the kernel's TCP accept backlog (-b option). Clients will see connection refused errors or timeout waiting for a connection from their pool.

This means back-pressure propagates all the way back to the application server, which must handle it gracefully: either by failing open (treating the error as a cache miss and going to the database, risking a stampede) or by failing closed (returning an error to the user). The former is usually correct for availability but must be paired with rate limiting or circuit-breaking to prevent a thundering herd on the database.

Memcached 1.6.x introduced TLS support and UDP was removed (it was deprecated in 1.5.6 due to DDoS reflection amplification concerns). These are version-specific changes that interviewers appreciate candidates knowing.

Common Interview Mistake

Candidates often say "Memcached is just a hash map." The correct answer includes the slab allocator (why it exists, what fragmentation it prevents), the LRU-per-slab design (with its implications for memory efficiency), and the fact that there is NO server-side cluster coordination — Memcached nodes are completely independent; all "clustering" is in the client library. If you say "Memcached cluster" you must immediately clarify what you mean.

L7/L8 Differentiator

An L8 candidate knows that the background clock thread exists specifically to avoid gettimeofday() syscalls on the hot path (each is a kernel trap costing ~150ns), that the LRU crawler was added to address "dead but not evicted" item memory waste, and that the segmented LRU in 1.5+ was motivated by scan-resistance — preventing a viral trending item's fetch pattern from polluting the cache with one-hit-wonders. These are production operational details that signal real experience.

Key Takeaway — Pillar 2

Memcached's architecture achieves its extreme performance through three non-negotiable design choices: slab allocation for O(1) zero-fragmentation memory management, per-bucket-striped locking in the hash table for concurrent access without global lock contention, and a single-thread-per-connection model that eliminates connection-state sharing — every "clever" feature is sacrificed to protect the critical path latency.

PILLAR 03 Trade-offs & Design Decisions
The Trade-off Landscape

Every design decision in Memcached is a deliberate trade-off. Understanding these trade-offs at depth — and when to flip them — is what separates an L7 from an L6 candidate.

CAP Theorem Analysis

Memcached in its vanilla form is a CA system (Consistency + Availability, no Partition Tolerance in the traditional sense) — but this characterization needs important nuance that an L7/L8 candidate must provide.

Technically, a single Memcached node is strongly consistent — if you write a value and read it back on the same node, you get the latest value. But Memcached has no built-in replication, so there's no partition tolerance question to answer for a single node. At the cluster level (multiple nodes), Memcached does not replicate at all — each key lives on exactly one node. This means:

  • If the node holding a key partitions from the application tier, you get a cache miss — you fall through to the database. Availability of the cache data is sacrificed during the partition.
  • There's no consistency to preserve across nodes because there's no replication — so the CAP theorem's consistency dimension applies only within a single node.
  • The real framing for Memcached: it's a best-effort, high-availability, eventually-consistent cache layer operating in service of a strongly-consistent database backend. The system as a whole is AP (available + partition-tolerant) because misses fall through to the database, but the cache itself sacrifices consistency for latency.

PACELC Analysis

PACELC (Partition → Availability vs Consistency; Else → Latency vs Consistency) is more illuminating for Memcached than CAP alone.

During partition (P): Memcached chooses Availability (A) — cache misses fall through to the database rather than blocking or returning errors. Reads are available, just slower.

Without partition (E): Memcached is firmly on the Latency (L) side — it provides the lowest possible latency (sub-ms) by sacrificing any consistency guarantees. There is no linearizability, no read-your-writes guarantee across nodes, no serializability. Two clients can simultaneously SET the same key and both succeed, with the "winner" determined by which write arrived last at the node — not which was logically later.

Latency vs Throughput — The Knobs

Memcached is fundamentally optimized for low latency over high throughput, but these are not mutually exclusive. The key levers:

Connection pooling on the client: Without pooling, each request creates a new TCP connection (3-way handshake = ~100μs on LAN). With a pool of 10 persistent connections, you eliminate this overhead for 99%+ of requests. The downside of too many pooled connections: Memcached has a connection limit (-c, default 1024 per node), and each connection holds ~4KB of per-connection state in the worker. For a fleet of 200 application servers each holding 20 pooled connections = 4000 connections to one Memcached node → exceeds default limit. Scale the connection limit accordingly or use a proxy layer (mcrouter, twemproxy).

Pipeline batching: The Memcached binary protocol supports pipelining — sending multiple GET requests without waiting for each response. This dramatically increases throughput at the cost of slightly higher latency for the first request in the batch. Multi-get (get multiple keys in a single request) is even more efficient: one network round-trip for N cache lookups. This is the single most important throughput optimization for read-heavy workloads.

📐
Back-of-envelope: Throughput Ceiling

A single Memcached node on a 10Gbps NIC with 8 worker threads can handle approximately: 10Gbps / (24B header + avg 200B value) ≈ 44M GET ops/sec at network saturation. In practice, CPU saturates first: each GET takes ~5μs of CPU time → 1 core handles 200K GET/sec → 8 cores → 1.6M GET/sec. A node with 64 GB RAM storing 200-byte values holds ≈ 320M items. At 1M GET/sec with 90% hit rate → 900K hits/sec with ~200μs latency per hit → throughput is CPU-bound not network-bound for typical workloads.

Key Takeaway — Pillar 3

Memcached's trade-off philosophy is aggressive: it sacrifices durability, replication, cross-key atomicity, and consistency with zero apology, in order to deliver unbeatable single-node latency — the implicit contract is that you have a source of truth elsewhere and Memcached is disposable, warm-up cache infrastructure.

PILLAR 04 Failure Modes & Resilience
When Things Break

Memcached's simplicity means many failure modes that distributed databases handle automatically must be handled by your application and infrastructure. Understanding each failure mode in depth is a strong L7/L8 signal.

Failure 1 Single Node Crash
Detection
Client-side: TCP connection refused or timeout on the next operation to that node. The Memcached client library (e.g., libmemcached, Folly MemcacheClient) detects this as a connection failure. Most libraries implement dead-server detection with a backoff window — after N consecutive connection failures to a node, mark it as dead and stop routing to it for a configurable interval (e.g., 30 seconds) before retrying.
Auto Recovery
None within Memcached itself. Memcached has no leader election, no failover, no sentinel. The operating system (or container orchestrator) must restart the process. On restart, Memcached starts completely empty — the slab allocator is re-initialized from scratch. There is no persistence to reload from.
Data Loss / Inconsistency Risk
100% of cached data on that node is lost. All keys that hashed to this node will now be cache misses. If this node held 1/N of your total cache (for N nodes), your overall hit rate drops from H% to H - H/N %. Example: 10-node cluster, 90% hit rate → crash of 1 node → hit rate drops to 81% (10% more traffic hitting the DB for the duration).
Engineer Action
Restart the process (via systemd, Kubernetes, etc.). Ensure your deployment system restarts it with the same memory allocation and configuration. Once restarted, the cache will warm up organically over the next minutes-to-hours as traffic hits misses and repopulates keys. If warming is too slow and DB is overloaded, trigger an active cache warm-up job that pre-fetches the most-accessed keys from the database or from a warm cache snapshot if you've implemented one.
Failure 2 Simultaneous Minority Node Failures
Detection
Each client detects its own connection failures to each node independently. There is no cluster-level awareness — each Memcached node is autonomous. Multiple clients will each independently mark multiple nodes as dead. This detection is purely local to the client process.
Auto Recovery
Client libraries with consistent hashing will redistribute the keys that were owned by the failed nodes across the remaining live nodes — but this redistribution is only for new requests. The hash ring changes, so keys previously on node A may now hash to node B. But node B has no data for those keys, so they're still cache misses. No data is migrated.
Data Loss / Inconsistency Risk
Proportional to the number of nodes lost. If 3 of 10 nodes die, hit rate drops significantly. Additionally, if the client library does NOT implement proper consistent hashing (e.g., uses simple modulo hashing), the key redistribution will affect ALL keys, not just those owned by the dead nodes — causing a global cache miss storm. This is the difference between consistent hashing (Ketama) and naive modulo ring.
Engineer Action
Verify clients are using consistent hashing. Bring the dead nodes back online as quickly as possible. Monitor database QPS for sudden spikes. If database is at risk, activate DB connection pooling limits or rate limiting to prevent complete overload. Consider pre-provisioned "standby" Memcached nodes that can be added to the ring quickly.
Failure 3 Network Partition (One Side Has Majority)
Detection
Application servers on the partitioned side that cannot reach certain Memcached nodes will get TCP timeouts. With a well-configured client (short connect timeout, e.g. 50ms), these are fast failures. The unreachable Memcached nodes appear as "dead" to the partitioned app servers.
Auto Recovery
None automatic. The partitioned app servers fall through to the database for all keys owned by unreachable nodes. If the partition heals, clients retry the marked-dead nodes and reconnect. Meanwhile, the Memcached nodes themselves continue operating normally — they simply receive fewer requests. When the partition heals, app servers reconnect and the cache is as warm as it was before the partition (the data on Memcached was never lost).
Data Loss / Inconsistency Risk
During the partition, writes (SETs) from partitioned app servers go to the database but NOT to the affected Memcached nodes (they're unreachable). When the partition heals, those Memcached nodes may serve stale cache values for keys that were updated during the partition. This is the classic cache invalidation problem — the cache is now inconsistent with the DB. The fix: use short TTLs, or implement cache invalidation via delete-on-write (delete the cache key whenever you write to the DB), rather than SET-on-write (write both simultaneously).
Engineer Action
Ensure delete-on-write cache invalidation pattern is used rather than set-on-write. After partition heals, optionally flush affected keys or rely on TTL expiry to ensure staleness window is bounded.
Failure 4 Thundering Herd After Recovery
Description
When a Memcached node recovers from a crash or restart, its cache is completely empty. All keys that hash to it are now misses. If this node served 10% of your traffic with a 90% hit rate, suddenly 10% of all requests become cache misses simultaneously. These misses fall through to the database, which may have been sized for normal miss rates (10% of 10% = 1% of total traffic hitting DB for this segment). Now it's 100% of 10% = 10% of total traffic — a 10× spike for those queries. The database, already operating near capacity, can cascade into overload.
Detection
Database QPS spikes dramatically within seconds of the Memcached node restarting. Cache miss rate metrics spike. Database latency increases, causing application timeouts, which causes more retries, which causes more DB load — the classic thundering herd / feedback loop.
Mitigation
Cache stampede prevention via probabilistic early expiration: instead of all clients discovering a miss simultaneously when an item expires, add a small random jitter to TTL checks so misses are staggered. Mutex coalescing / dog-piling prevention: use a distributed lock (or Memcached's own add command as a test-and-set) so only one request triggers the DB lookup for a given key, while others wait or serve a slightly stale value. Gradual traffic ramp: when reintroducing a node to the ring, bring it in at reduced weight (10%, 25%, 50%, 100%) so it receives a fraction of traffic initially, allowing it to warm up before handling full load.
Engineer Action
Implement circuit breakers on the DB layer. Pre-warm the cache programmatically before adding the node back to the ring. Use mcrouter's pool configuration to gradually ramp weight. Monitor database latency as the leading indicator of herd impact.
Failure 5 Thundering Herd — Hot Key / Cache Stampede
Description
A specific high-traffic key (e.g., a celebrity's profile with 10M followers) expires from cache. In the microseconds between expiry and repopulation, thousands of application servers all simultaneously detect the miss and simultaneously query the database for the same row. This is the cache stampede problem — a single key expiry causes O(concurrent_requests) database queries instead of one.
Mitigation Strategies
XFetch / Probabilistic Early Expiration: As TTL approaches, probabilistically recompute early (before expiry) based on remaining TTL and computation cost. Probability of early recompute = exp(-delta/beta * log(rand)) — the closer to expiry and the more expensive the computation, the higher the recompute probability.

Memcached CAS + mutex key: When a miss occurs, immediately SET a mutex key (e.g., mutex:user:1234 with a short TTL like 5s) using add (only succeeds if key doesn't exist). Only the server that wins the add does the DB fetch and populates the cache. All others either wait briefly or serve a previously cached stale value (if you stored it with a "grace" extension).

Lease mechanism: Facebook's mcrouter uses leases — on a cache miss, the first client gets a lease token; other clients for the same key get a "wait" response and retry after a short delay. The lease-holding client fetches from DB, populates cache, and releases the lease.
Failure 6 Slow / Degraded Node (Not Dead, But Lagging)
Description
A Memcached node is responding but with high latency — perhaps it's CPU-saturated, experiencing network issues, or the OS is paging (Memcached being swapped). From the client's perspective, the node is alive but slow. Standard client timeout logic (if configured) may not trigger immediately, meaning the application is sitting and waiting for slow responses rather than falling through to the database.
Detection
Client-side P99 latency to specific Memcached nodes spikes above threshold. This requires per-node latency tracking in your metrics, not just aggregate. CPU metrics on the Memcached host spike. If the OS starts swapping (look for si/so in vmstat), the situation is critical — Memcached must NEVER be allowed to use swap.
Mitigation
Configure tight read timeouts in the client (e.g., 5ms). After N timeouts to a node, mark it as slow/dead and stop routing to it. Use Linux memlock / mlock() to pin Memcached's memory and prevent swapping. Set Memcached's memory limit (-m) conservatively below available RAM to give the OS room for page cache. Monitor with oom_kill_disable=0 so the OOM killer can kill Memcached before it starts thrashing.
Failure 7 Clock Skew Between Application Servers
Description
Memcached TTLs can be specified as either relative seconds (30 = expire in 30 seconds) or as an absolute Unix timestamp (if the value is > 30 days in seconds = 2,592,000). If your application servers have significant clock skew (e.g., NTP not configured correctly), absolute timestamp TTLs can result in items expiring immediately (if the app server's clock is ahead) or far later than intended (if behind). Relative TTLs are safe — they're interpreted by the Memcached server itself based on its own clock.
Mitigation
Always use relative TTLs (seconds from now) rather than absolute timestamps unless you have confirmed NTP is synchronized across all nodes with sub-second accuracy. Run chrony or ntpd on all Memcached hosts. Monitor for clock drift in your infrastructure monitoring.
Failure 8 Cascading Failures — DB Overload from Cache Miss Storm
Description
The most dangerous failure mode for systems using Memcached. Scenario: Multiple Memcached nodes fail simultaneously (network switch failure, AZ outage). Cache hit rate drops from 95% to 50%. Database QPS increases 10×. Database latency increases from 5ms to 50ms due to overload. Application servers wait longer for responses, holding open connections and threads. Application server thread pools exhaust. New requests get queued. Queued requests timeout. Users see errors. Error monitoring fires alerts. Engineers restart application servers. Restart causes all in-process cache to be lost. Restarted servers make even more DB requests. Database gets worse. Cascade complete — the whole system is down.
Prevention
Circuit breakers on the DB layer with fallback to degraded mode (e.g., serve cached stale data, serve simplified response, queue writes for later). DB connection pool limiting — hard cap on concurrent DB connections regardless of demand, so DB is never overwhelmed even if app tier is saturated. Rate limiting on cache miss handling — if miss rate exceeds threshold, fail fast rather than hammering DB. Replica routing — route cache miss reads to DB read replicas rather than primary, preserving primary for writes. Staged rollout of Memcached restarts — never restart the entire Memcached fleet simultaneously; do it in batches of 10–20%.
⚡
L7/L8 Differentiator — Failure Modes

An L8 candidate frames every failure mode in terms of its blast radius and the full cascade it can trigger. They also discuss dark traffic testing (sending shadow traffic through a restored node before bringing it live), synthetic cache warming (pre-populating a replacement node from a snapshot of the most-accessed keys before introducing it to the ring), and the lease mechanism from Facebook's Memcached paper (2013 NSDI) as a production-grade solution to stampedes. These are not theoretical — they're production techniques used at hyper-scale.

Key Takeaway — Pillar 4

Memcached's failure modes are almost entirely about the consequences of cache unavailability propagating to the database tier — the cache itself never loses data inconsistently, it just loses all data instantly on any node failure, and your system must be designed from day one to degrade gracefully under any miss rate up to 100%.

PILLAR 05 Scaling Strategies
Scaling to Billions

Memcached's horizontal scaling model is architecturally different from most distributed systems — because the "cluster" intelligence lives entirely in the client, scaling decisions directly shape both the client library design and the operational model.

Vertical Scaling — How Far Can You Go?

A single Memcached node can scale vertically quite far. You can give it 256–512 GB of RAM on a modern server, and the slab allocator will manage it efficiently. With 512 GB at an average item size of 500 bytes, you can cache approximately 1 billion items on a single node. CPU scales with worker threads up to about 16–24 cores before lock contention on the hash table and LRU lists becomes a limiting factor. Network is rarely the bottleneck before CPU — a single 10 GbE NIC can sustain 4–5 million GET/sec at small item sizes.

What breaks first vertically: At extremely large memory allocations (>256GB), the slab allocator's hash table grows correspondingly large — it can consume several GB of non-cached memory just for hash table overhead. Additionally, the LRU lists become very long, and LRU operations (especially the crawler's periodic walks) become more expensive. Eviction decisions become slower relative to the working set size. At the extreme, you may also see NUMA effects: a single Memcached process that doesn't pin its threads to NUMA nodes will experience cache misses across NUMA boundaries on large multi-socket servers.

Horizontal Scaling — The Ketama Consistent Hash Ring

This is where Memcached's architecture is fundamentally different from Redis Cluster or Cassandra: all cluster management lives in the client. There is no Memcached cluster manager, no gossip protocol, no leader election, no shard map stored server-side. The client library holds a consistent hash ring — a virtual ring of 2^32 positions — onto which both server nodes and keys are mapped.

The Ketama algorithm (the standard for Memcached client-side hashing) places each server at 160 virtual nodes on the ring (to ensure even distribution). To find which server owns a key: hash the key → find the next clockwise virtual node on the ring → the server that virtual node maps to is the owner. Adding a new server: insert its 160 virtual nodes. Keys previously owned by the next-clockwise server that now fall between the new server's nodes migrate to the new server — but no data is actually migrated. Those keys simply become misses on the new server, are fetched from the DB, and repopulate there. Removing a server: its virtual nodes are removed, and keys redistribute to the next clockwise nodes. Data is lost from the removed server.

📐
Back-of-envelope: Impact of Adding a Node to Ketama Ring

With N nodes and adding 1 node: approximately 1/(N+1) of all keys rehash to the new node, causing temporary misses. For N=9, adding node 10: ~10% of keys miss. If overall hit rate was 95% before, it drops to 95% × (1 - 0.10) + 0% × 0.10 = 85.5% during warm-up. DB sees a 10× traffic increase for 10% of queries until warm-up completes (typically minutes to hours depending on TTL distribution). This is a planned "mini-thundering-herd" — manage it by adding nodes off-peak, gradually weighting the new node, or pre-warming from DB before adding to the ring.

Sharding / Partitioning Strategies

Hash-based partitioning (Ketama) is the standard and should almost always be used. Its key advantage over range-based partitioning is that keys are distributed uniformly — no natural sort order means no prefix-based hotspots. A hash-based scheme ensures that even if all your users have names starting with "A", they hash uniformly across the ring.

Hotspot management: Even with good hashing, some keys will be extremely hot (viral posts, celebrity profiles, trending topics). A key receiving 1M req/sec on a node handling 2M total req/sec is monopolizing 50% of that node's capacity. Solutions:

  • Local in-process cache (L1 cache): Cache the hottest keys in each application server's memory (e.g., a Caffeine or Guava cache with a small capacity). First check local cache, then Memcached, then DB. This eliminates network round-trips for ultra-hot keys and distributes their read load across all app servers rather than concentrating it on one Memcached node. Trade-off: local cache can serve stale data for up to its TTL window; cache size adds to app server memory pressure.
  • Key replication for hot items: For identified hot keys, store them on N nodes (e.g., user:viral_post:123__replica_0, __replica_1...) and randomly pick which replica to read from. Writes must update all replicas. Detection: monitor per-key access rates (Memcached 1.5+ exposes this via the watch command or stats).
  • mcrouter hot key mirroring: mcrouter can automatically detect hot keys and mirror them across multiple backend Memcached servers, spreading the read load.

PILLAR 06 Real-World Engineering Decisions
Production Reality

Knowing when to use Memcached — and when it's the wrong tool — is as important as understanding how it works. Here are the real production patterns that inform engineering judgment.

Where Memcached Shines Above All Alternatives

Highest-throughput read caching with the lowest possible operational overhead. When you need to cache the results of expensive database queries or computation, your values are relatively small (under 1MB, ideally under 100KB), your workload is read-heavy (reads >> writes), you don't need data structures (sorted sets, lists — use Redis for that), and you have a team that doesn't want to operate a complex stateful system, Memcached is the optimal choice. It's simpler than Redis to operate (no persistence to configure, no replication to manage in the simple case), and it achieves better multi-threaded throughput than Redis for pure GET/SET workloads because Redis is single-threaded for command execution (Redis 6+ has I/O threading but command execution is still single-threaded).

Real-World Production Examples

Example 1: Facebook — The Defining Use Case (2004–Present)

Facebook deployed Memcached at what became the world's largest Memcached deployment. At peak, they ran tens of thousands of Memcached servers holding hundreds of terabytes of cached data. Their key insight from the NSDI 2013 paper: the bottleneck at Facebook scale was not cache capacity but cache invalidation and consistency across regions. Their contributions:

  • mcrouter: A Memcached protocol-compliant proxy they open-sourced, providing connection pooling (solving the C10K problem for Memcached connections), request routing, replication, and circuit breaking.
  • Lease mechanism: Solving the thundering herd problem at 1B-user scale — a client gets a lease token on miss; only the lease holder fetches from DB; others wait or serve stale.
  • Gutter pools: A set of dedicated Memcached machines that absorb traffic when a primary node fails, preventing miss storms from hitting the database — essentially a "catch all" secondary tier.
  • Why Memcached over Redis: at Facebook's scale in 2007–2010, Redis was nascent. But also, Memcached's multi-threaded architecture was critical for handling the massive parallelism of Facebook's application tier. Simple GET/SET semantics sufficed — they didn't need sorted sets or pub/sub.

Example 2: Wikipedia — MediaWiki Object Cache

Wikipedia uses Memcached as the object cache backend for MediaWiki, caching parsed wikitext, database query results, and session data. The specific fit: Wikipedia pages are read millions of times per day but edited rarely (high read-to-write ratio). The Wikimedia Foundation runs Memcached clusters in each datacenter (Amsterdam, Ashburn, San Francisco), with a custom invalidation scheme that broadcasts database changes via a message queue to all caches. They chose Memcached over alternatives specifically because:

  • The simplicity of operation — Wikipedia's infrastructure team is relatively small.
  • MediaWiki's caching needs are pure key-value — no complex data structure operations needed.
  • The ability to shard across dozens of servers without any server-side coordination is ideal for their multi-DC setup.

Example 3: Twitter (Early Architecture, 2009–2014)

Twitter used Memcached heavily for timeline caching during its high-growth years. The use case: each user's home timeline is computed by fanning in tweets from all accounts they follow. At 200M users, recomputing this in real-time for every timeline view was impossible. They cached assembled timelines in Memcached (large objects, sometimes >100KB for active users). The specific challenges this revealed:

  • Large value problem: Timeline objects exceeded the default 1MB Memcached item size limit. They had to either paginate timelines or configure larger slab sizes.
  • Write amplification: When a celebrity with 50M followers tweets, Twitter had to write the tweet into 50M cached timelines. This was the "write-fan-out" problem that eventually drove Twitter to a hybrid push/pull architecture — push for non-celebrities (cache-first), pull+compute for celebrities with massive follower counts.
  • Why not Redis: at the time (2009–2011), Redis was not yet production-hardened at this scale. Also, Memcached's multi-threaded model was critical for their write throughput needs.

When NOT to Use Memcached — Anti-Patterns

Anti-Pattern 1
Using Memcached for Session Storage

Sessions are user data that users expect to persist across server restarts. Storing sessions in Memcached means a Memcached restart logs out all users. This is catastrophic for user experience. Use a durable store (Redis with AOF persistence, DynamoDB, or a dedicated session DB) for sessions. The performance win of Memcached is not worth the user-facing impact of session loss.

Anti-Pattern 2
Storing Data That Cannot Be Regenerated

If the data in Memcached cannot be recreated from a source of truth on a cache miss, you have turned Memcached into a database without durability guarantees. This is a hidden single point of failure. Examples: processed results that are not stored anywhere else, user-generated content that is only in cache. Never store anything in Memcached that you cannot afford to lose.

Anti-Pattern 3
Using Memcached for Complex Data Structures

If your application needs sorted sets (leaderboards), lists (queues), sets (membership), or hash maps (partial updates), Memcached cannot provide these natively. You would have to get-deserialize-modify-reserialize-set for every update, creating race conditions. Redis provides all of these with atomic server-side operations. Using Memcached for these use cases leads to complex, buggy client-side logic.

Anti-Pattern 4
Using Memcached as an Application Database

Some teams under-provision their database and use Memcached as the "real" database, with the RDBMS as a write-behind. This creates an architecture where a cache restart causes complete data loss. The correct model: database is always the source of truth; Memcached is an optional acceleration layer. If Memcached goes down, the system degrades in performance, not in correctness.

Anti-Pattern 5
Mismatched Slab Configuration

Deploying Memcached with default slab class configuration for a workload with very specific value sizes (e.g., all values are exactly 700 bytes) causes massive internal fragmentation — every item goes into the ~1KB slab class, wasting 30% of memory. At 64GB per node, that's 19GB wasted. Tune the growth factor (-f) and minimum chunk size (-n) to match your actual value size distribution before putting Memcached into production.

Anti-Pattern 6
Ignoring Connection Pool Sizing

Each connection to a Memcached node consumes file descriptors and per-connection memory on the server. With 500 application servers each holding 20 pooled connections to each of 10 Memcached nodes, that's 500 × 20 = 10,000 connections per Memcached node — exceeding the default connection limit of 1,024. The result: connection refused errors, which application servers may interpret as a cache miss, causing a thundering herd. Always calculate max_connections and set -c accordingly, or route through a proxy like mcrouter.

Day-1 Monitoring Setup

On day one of a Memcached deployment, before anything else, set up these metrics and alerts:

  • Cache hit rate per node: get_hits / (get_hits + get_misses). Alert if below 80% sustained. Target: >90% for most production workloads.
  • Eviction rate: evictions counter per second. Non-zero evictions mean the cache is full and LRU is actively discarding data. Alert if eviction rate > 1% of SET rate — indicates your cache is undersized.
  • GET/SET latency P50/P99/P999: Track from the client side (not just Memcached's internal stats). P99 > 5ms is a red flag. P999 > 50ms suggests lock contention or GC on the client.
  • Connection count per node: If approaching max_connections, add more Memcached nodes or increase the limit.
  • Bytes used vs bytes limit: Track utilization. >90% means you're near eviction territory. Plan capacity expansion proactively.
  • curr_items (total items stored): Track growth rate. Sudden drops indicate eviction or restart.
  • Database QPS: The most important indirect metric. Sudden increases in DB QPS are the first indicator of cache degradation — often before the cache metrics themselves show problems, because a cascading failure can make Memcached look "healthy" (it's just not getting traffic) while the DB is dying.

Common Interview Mistake — Real-World Section

Candidates often say "Memcached vs Redis: choose Redis for persistence, Memcached for pure caching." This is superficial. The real decision factors are: (1) Do you need data structures? Use Redis. (2) Do you need multi-threaded throughput above 1M ops/sec? Use Memcached. (3) Do you need replication without a proxy? Use Redis. (4) Are you willing to add operational complexity for richer features? Redis. The honest answer for most production systems in 2024: Redis has largely superseded Memcached because its multi-threaded I/O (Redis 6+) and optional persistence make it strictly better for most teams, unless you're specifically at a scale where Memcached's simpler multi-threading model (4–16 threads vs Redis's 1 command thread) provides a meaningful throughput advantage.

Key Takeaway — Pillar 6

The most important real-world lesson from Memcached's production history is that cache invalidation — not cache storage — is where systems break: Facebook built mcrouter, the lease mechanism, and regional invalidation buses not to make Memcached faster, but to make cache invalidation correct and consistent at billion-user scale, proving that the hard problem in caching is not "how do we cache it" but "how do we know when to uncache it."

How Google Solves These Differently

Google's approach to distributed caching at hyperscale diverges significantly from the "large Memcached cluster" model. Some key insights from what's publicly known about Google's infrastructure:

Bigtable as a cache backend: For data that benefits from structured access patterns, Google uses Bigtable — which provides strong consistency, replication, and rich access patterns. Bigtable's LSM-tree (Log-Structured Merge Tree) architecture gives it better write throughput than a B-tree database, and its column family model enables partial reads of cached objects. However, it's not sub-millisecond — P50 is closer to 10ms — so it serves a different tier of the caching hierarchy than Memcached.

Edge caching with Espresso/GFE: Google's Global Front End (GFE) and CDN infrastructure cache at the edge — geographically close to users. For static and semi-static content, this eliminates the need for a Memcached cluster entirely, because the data never reaches the datacenter. Memcached handles the dynamic, user-specific data that the CDN cannot cache.

Spanner for consistent distributed data: Where Facebook uses complex invalidation schemes to approximate consistency across regions, Google's Cloud Spanner provides externally-consistent distributed transactions across regions. A system that uses Spanner as its source of truth can read a cache-busted value with a known consistency guarantee — you know exactly how stale the data can be based on Spanner's TrueTime API. This changes the caching architecture: instead of "invalidate on write", you can do "cache with a consistency timestamp" and trust that reads from Spanner with a specific TrueTime bound are correct.

Disaggregated memory pools: At Google-scale, the "RAM on the caching server" model is inefficient. Google and other hyperscalers have experimented with disaggregated memory — network-attached RAM pools (using RDMA or CXL) that any compute node can address directly. This separates the memory tier from the compute tier, allowing independent scaling. A Memcached-equivalent built on CXL memory could provide shared, persistent (across restarts) cache with microsecond access from any node in the rack. This is the frontier of 2024 distributed systems research.

Multi-Tenancy — Noisy Neighbor Isolation

When multiple teams or products share a Memcached cluster (multi-tenancy), the noisy neighbor problem is acute: team A's batch job can warm up cache with short-TTL items that evict team B's long-lived hot items. Solutions at hyperscale:

  • Namespace separation with dedicated slab classes: Assign different slab class ranges to different tenants via key prefix routing in mcrouter. Team A's keys use slab classes 0–10, Team B uses 11–20. LRU is scoped per slab class, so team A's evictions never affect team B's allocation.
  • Memory quotas enforced at proxy: mcrouter can enforce per-namespace item count or byte quotas by rejecting SETs that would cause a namespace to exceed its allocation.
  • Physical isolation for critical services: The highest-SLA services (auth, payment, core feed) run on dedicated Memcached nodes not shared with other tenants. Cost is worth the isolation guarantee.
  • Priority-weighted LRU: Emerging research (and some custom forks) implements weighted LRU where high-priority namespaces' items have higher eviction resistance. Not in vanilla Memcached — would require a custom implementation.

Data Residency and Regulatory Requirements

GDPR, CCPA, India's DPDP, and similar regulations require that personal data not leave certain jurisdictions. For Memcached, this creates specific requirements:

  • Per-region Memcached pools: EU user data is cached only in EU-region Memcached nodes. The client library (or proxy) must route EU user keys exclusively to the EU pool. Key design: include a region tag in the key (e.g., EU:user:profile:12345) or route based on key hash range in mcrouter.
  • Encryption at rest and in transit: Memcached 1.5.13+ supports TLS for in-transit encryption. For encryption at rest of cached values (GDPR requires protection of data at rest), you must encrypt the value in the application layer before storing in Memcached — Memcached itself has no at-rest encryption. This adds CPU overhead for every cache operation.
  • Right-to-erasure compliance: When a user invokes GDPR's "right to be forgotten," you must delete their data not just from the database but from all Memcached nodes. Since Memcached has no enumeration capability (you can't list all keys), you must either (a) track all keys for a user explicitly, (b) use key patterns that include the user ID and issue targeted deletes, or (c) rely on TTL expiry to eventually purge the data. Option (a) or (b) is required for strict compliance.

A "Google Spanner" Equivalent for Memcached — What Would It Look Like?

What innovations would be needed to build a globally-consistent, fault-tolerant, multi-region distributed cache that preserves Memcached's sub-millisecond access model? This is an L8-level thought experiment:

Requirement 1 — Consistent cache invalidation across regions: The fundamental problem is that writes go to the database, but cache invalidations must reach every regional cache pool. A "Google Spanner for caching" would use TrueTime-like bounded uncertainty to give every cache entry a write timestamp. Reads can assert "I need data no older than T seconds" and the cache can either return the cached value (if its timestamp is within T) or forward to the DB. This is similar to Spanner's READ AT <timestamp> semantics applied to a cache tier.

Requirement 2 — Sub-millisecond latency globally: Cross-region network latency (US-EU ≈ 80ms) fundamentally prevents sub-millisecond global consistency. A "Spanner for caching" would therefore be eventually consistent with bounded staleness at the global level, but strongly consistent within a region. This is exactly the model of Google's Cloud Memorystore (managed Redis with regional strong consistency and cross-region async replication).

Requirement 3 — CXL-enabled persistent memory: Next-generation architecture would use CXL (Compute Express Link) attached memory that survives process restarts (but not server failures), eliminating the cold-start problem. Cache warm-up after a crash becomes instantaneous because the slab allocator's memory was in persistent CXL DRAM, not regular DRAM. This is the frontier of 2024–2025 hardware research.

Cache Invalidation Strategy

I will use the delete-on-write pattern (cache-aside with invalidation) rather than set-on-write, because it avoids the race condition where:

  1. Thread A reads post from DB (old value)
  2. Thread B writes new post to DB and sets cache with new value
  3. Thread A sets cache with old value (overwrites Thread B's fresh write)

With delete-on-write: when a post is updated → DB write → DELETE the cache key. Next reader will populate from the DB with the fresh value. No stale overwrite race.

For the CDC-based invalidation path: Debezium reads the PostgreSQL WAL, publishes row-change events to Kafka, a cache invalidation consumer group reads from Kafka and issues DELETEs to mcrouter for all affected cache keys. Step 4 — Rapid-Fire Staff-Level Questions

Q01Explain Memcached's slab allocator in 60 seconds and why it exists.
The slab allocator pre-divides Memcached's total memory into fixed-size chunks organized by size class (e.g., 64B, 128B, 256B, up to 1MB). When an item is stored, it occupies the smallest slab class chunk that fits it. Allocation and deallocation are O(1) — push/pop a linked list. This exists to eliminate heap fragmentation: traditional malloc over time fragments memory so that even with enough total free bytes, you can't make large allocations. The slab allocator avoids this entirely at the cost of internal fragmentation (an item smaller than its slab class wastes the difference). The growth factor between slab classes (default 1.25×) controls the trade-off between number of classes and average wasted space per item.
Q02What is the Ketama algorithm and why does Memcached use it instead of modulo hashing?
Ketama is a consistent hashing algorithm that places servers on a virtual ring of 2^32 positions using multiple virtual nodes per server (160 by default). To find which server owns a key: hash the key, find the next clockwise position on the ring, return that server. The critical property: when a server is added or removed, only 1/N of all keys are affected (specifically, those whose clockwise neighbor changes). Modulo hashing (key_hash % N) is simpler but catastrophic for node changes: adding or removing any server remaps approximately all keys to different nodes, causing a complete cache miss storm. Ketama's O(1/N) remapping bounds the database impact of node changes.
Q03What is the "thundering herd" / "dog-piling" problem, and name three distinct solutions?
When a popular cache key expires, many concurrent requests simultaneously discover the miss and simultaneously query the database for the same data, causing a spike in DB load proportional to concurrent users. Three solutions: (1) Lease mechanism (Facebook): first client on miss gets a lease token and is the only one to query DB; others are told to wait with a retry delay. (2) Probabilistic early expiration (XFetch): before TTL expires, probabilistically recompute based on remaining TTL and computation cost, allowing one client to refresh before others see the expiry. (3) Stale-while-revalidate: serve the expired cached value to all concurrent requests while a single background request refreshes it — trading brief staleness for DB protection.
Q04How does the CAS (Check-And-Set) operation work in Memcached and when is it needed?
CAS is Memcached's optimistic locking mechanism. Each stored item has a 64-bit cas_token (monotonically incrementing counter). The flow: (1) gets key — returns value and current cas_token. (2) Client modifies value locally. (3) cas key <new_value> <cas_token> — Memcached atomically checks if the item's current cas_token matches the provided token and only stores the new value if it does. If another client modified the item in between (incrementing the token), the CAS returns EXISTS and the client must retry. CAS is needed when you need read-modify-write atomicity on a single cache key — for example, incrementing a counter that's stored as a serialized object rather than using the native incr command, or updating a list stored as a cached value where lost writes would cause data loss.
Q05You have 200 application servers and 20 Memcached nodes. What connection management problem arises, and how do you solve it?
With 200 app servers each maintaining a connection pool of 10 connections to each of 20 Memcached nodes: 200 × 10 × 20 = 40,000 total connections, but each Memcached node sees 200 × 10 = 2,000 connections — far exceeding the default limit of 1,024 (-c) and consuming significant server memory for per-connection state. Solution: introduce a Memcached proxy layer (mcrouter, twemproxy). App servers connect to local mcrouter instances (one per server), mcrouter maintains a smaller pool of connections to each Memcached node (e.g., 50 connections per node per mcrouter). Total connections per Memcached node: 200 app servers × 50/200 mcrouter connections = only 50 connections per node. This is the fan-in problem that proxies solve.
Q06What is slab_automove in Memcached 1.6.x and what problem does it solve?
slab_automove is a background mechanism introduced to address slab memory imbalance. In vanilla Memcached, memory is statically partitioned between slab classes at startup — if your workload's value size distribution shifts (e.g., you start caching larger objects than you initially configured for), the slab class for large objects fills up and evicts items while the small-item slab class has abundant free space, wasting memory. slab_automove monitors per-slab-class eviction rates and automatically migrates 1MB slab pages from underutilized classes (few evictions) to overloaded classes (high eviction rate). Enabled with slab_automove=1. The trade-off: page migration briefly takes the page's memory offline (all items in it are evicted during migration) and requires brief locking. Set slab_automove_ratio conservatively to avoid thrashing.
Q07Compare delete-on-write vs set-on-write cache invalidation and when would you choose each?
Set-on-write: On DB write, also SET the new value in cache. Problem: race condition — if two concurrent writers finish in different orders at the DB vs the cache, the cache ends up with the older value permanently until TTL expiry. Stale data is indefinitely long if the TTL is long. Delete-on-write: On DB write, DELETE the cache key. Next reader gets a miss, fetches from DB, and repopulates with fresh data. No race condition — the worst case is a cache miss, not permanently stale data. Choose set-on-write only when write amplification cost is low, the computation to regenerate the value is expensive (you want to avoid a miss), and the race condition window is negligible (very low write concurrency). For most production systems with concurrent writers, delete-on-write is safer. Set-on-write is acceptable for immutable-after-creation data (e.g., a post that can't be edited after 5 minutes).
Q08What does Facebook's mcrouter add over direct Memcached connections, and what are the trade-offs of introducing it?
mcrouter adds: (1) Connection pooling/fan-in: reduces N×M connections to N+M by acting as a proxy between app servers and Memcached nodes. (2) Replication: fan-out writes to multiple Memcached pools simultaneously for geographic redundancy. (3) Hot key detection and local shadowing: identifies keys with extreme access rates and serves them from a local cache to protect the hot Memcached node. (4) Circuit breaking and failover routing: if a Memcached node is down, route to a secondary pool instead of failing. (5) Per-namespace routing: route different key prefixes to different Memcached pools. Trade-offs: adds ~50–100μs latency to the critical path (one extra hop). mcrouter itself is a stateful process that becomes a potential SPOF and must be run with HA (multiple instances per server or per rack). Configuration management complexity increases significantly.
Q09Explain the segmented LRU (SLRU) introduced in Memcached 1.5+ and what problem it solves.
Classic LRU in Memcached had a "scan pollution" problem: if a background process fetched a large number of cold items (e.g., a reporting job scanning user records), those items would move to the head of the LRU list, pushing genuinely hot items toward the tail and causing them to be evicted. SLRU in Memcached 1.5+ divides each slab class's LRU into three sub-queues: HOT (items accessed very recently), WARM (items accessed more than once — survivors of their first LRU traversal), and COLD (candidates for eviction at the tail). New items enter HOT. Items at the tail of HOT move to COLD if not accessed again. Items in COLD that are accessed move to WARM. Evictions come only from COLD's tail. This means scan-accessed items (accessed once) drain through HOT → COLD → eviction without polluting WARM, while truly frequently-accessed items accumulate in WARM and are eviction-resistant.
Q10You're asked to implement GDPR right-to-erasure for a Memcached-backed system. What are the challenges and your solution?
Challenges: Memcached provides no key enumeration capability — you can't ask "give me all keys for user 12345." You can only do targeted get/set/delete by exact key. This means you can't just sweep the cache for a user's data. Solution: Maintain an external index (in your application DB or a KV store like Redis) that maps user_id → list of all cache keys associated with that user. Whenever you set a cache key containing that user's data, record the key in the index. On erasure request: query the index for all keys → issue targeted DELETE commands to Memcached for each key → purge the index itself. For keys you cannot enumerate (e.g., keys generated by third-party libraries): set a short global TTL on user-data-containing objects (e.g., 24 hours) so that even without targeted deletion, data expires within a legally defensible window. Additionally, encrypt values with a per-user encryption key, and destroy the key on erasure — even without deleting cache entries, undecryptable data is effectively erased.
Key Takeaway — Pillar 8

The difference between an L7 and L8 Memcached interview answer is not more details about Memcached itself — it's demonstrating that you understand the failure modes well enough to build protective infrastructure around it (lease mechanisms, gutter pools, staged scaling, circuit breakers) and that you treat cache consistency as a first-class design concern, not an afterthought to be handled by TTL alone.

  # Python example — instrumenting Memcached client
  from prometheus_client import Histogram
  import time

  memcached_latency = Histogram(
      'memcached_operation_duration_seconds',
      'Memcached operation latency',
      ['operation', 'status', 'pool'],
      buckets=[.0001, .0003, .001, .003, .01, .03, .1, .3, 1.0]
  )

  def instrumented_get(client, key, pool='default'):
      start = time.monotonic()
      try:
          value = client.get(key)
          status = 'hit' if value is not None else 'miss'
          return value
      except Exception:
          status = 'error'
          raise
      finally:
          memcached_latency.labels(
              operation='get', status=status, pool=pool
          ).observe(time.monotonic() - start)

Alert Runbook — What to Do When Alerts Fire

Alert: Hit Rate Below 80%
Immediate Check
stats on each node: Is any node down (curr_connections = 0)? Check evictions counter — high eviction rate means cache is full and LRU is aggressive. Check curr_items across all nodes — has one node lost its data?
Node Death Diagnosis
If a node is down: restart it immediately, monitor DB QPS, consider activating gutter pool if DB is at risk. Check if Ketama ring has been updated by clients to exclude the dead node.
Eviction Diagnosis
If high evictions: check memory utilization. If >95%, cache is too small for working set — either add nodes or reduce TTL on lower-value items to make room. Run stats slabs to identify which slab class is evicting — tune slab configuration or enable slab_automove.
Mass Expiry Diagnosis
If no node is down and no high eviction but hit rate dropped: check if a batch invalidation was issued. Look at delete_hits rate — a spike indicates mass invalidation. If this is expected (deployment cache clear), it will recover on its own. If unexpected, investigate what triggered the invalidations.
Alert: Eviction Rate Above 100/sec
Immediate Check
Check stats slabs to identify which slab class is evicting. Check memory utilization — if below 90%, evictions may be from a specific slab class that's misconfigured, not overall memory pressure.
Remediation
Short-term: reduce TTL on low-value items (analytics data, rendered HTML for old content). Medium-term: add more memory or more Memcached nodes. Check if slab_automove is enabled — if not, enable it to allow memory to flow from underutilized slab classes to the evicting ones.
Alert: P99 Latency Above 5ms
Immediate Check
SSH to the Memcached host. Check CPU usage (top, look for worker threads at 100%). Check if OS is swapping (vmstat 1, look for non-zero si/so columns). Check network saturation (netstat -s for retransmits). Check if hash table expansion is in progress (stats → hashtable_expansions counter incrementing).
Remediation
CPU saturation: increase -t worker threads or add more nodes to the ring to distribute load. OS swapping: CRITICAL — immediately reduce Memcached memory limit (-m) to below total physical RAM to stop swapping. Swapping Memcached is equivalent to it being down — latency will be in seconds. Network saturation: add nodes to distribute traffic or switch to 25GbE NICs.

End-to-End Simulation — Diagnosing a Production Incident

Scenario: Tuesday 2:47 PM — Database latency alert fires

Alert received: "DB P99 query latency > 500ms" and "DB connection pool at 95% saturation"

  1. Check the Cache Hit Rate Dashboard immediately
    Hit rate has dropped from 93% to 67% starting at 2:41 PM. This is the root cause: 26 percentage points less of traffic being absorbed by cache, hitting the DB directly. DB was at 70% capacity before — now receiving 140% of its normal load.
  2. Check Node Status Heatmap
    Node 7 (of 20) is red — down. Node 7 was responsible for 1/20 = 5% of keys. With 93% hit rate baseline, the cache was handling 93% of 100% of traffic. After node 7 dies: 95% of keys still have a good node, but those keys have no data yet (node 7 is empty on restart). So effective hit rate: 93% × (19/20) + 0% × (1/20) = 93% × 0.95 = 88.35%. That doesn't explain 67%. Something else is happening.
  3. Check eviction rate on remaining nodes
    stats on node 5 shows eviction rate of 800 items/sec. Node 5 is the next clockwise neighbor to the dead node 7 on the Ketama ring — it's now receiving the traffic for node 7's keys, plus its own. It's overwhelmed and evicting its own items to make room for node 7's former keys being repopulated.
  4. Immediate mitigation — restart node 7
    Restart the Memcached process on node 7's host. Check stats to confirm it's accepting connections. Clients will automatically reroute keys back to node 7 as it rejoins the Ketama ring. Node 5's eviction rate should decrease as its load reduces.
  5. Monitor DB recovery
    Over the next 10–15 minutes, as traffic organically repopulates node 7's cache, hit rate climbs back toward 90%+. DB latency decreases from 500ms toward its normal 15ms baseline. By 3:05 PM, all metrics are green.
  6. Post-mortem actions
    Implement automatic process restart (systemd Restart=always or Kubernetes liveness probe). Implement gutter pool configuration in mcrouter so that if node 7 dies again, its traffic routes to a standby pool rather than overloading node 5. Add an alert for "Memcached node down" that fires BEFORE the DB alert, so we catch node failures in seconds, not minutes.
Key Takeaway — APM & Monitoring

Monitoring Memcached correctly means treating the database's health as the ultimate health signal for your cache — because Memcached failure is always silent (it just causes misses) while DB overload is always loud (it causes user-visible failures) — and building your alerting pipeline to make the cache's hit rate visible before the DB's latency makes the problem unavoidable.
