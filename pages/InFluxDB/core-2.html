🧠 Variant 2.2 · Gorilla-Style In-Memory Blocks
Variant 2.2 · Section A

What It Is

🧠

One-Sentence Definition

The Gorilla approach stores the most recent time series data entirely in memory using a custom compression scheme (delta-of-delta for timestamps, XOR encoding for float values) that achieves ~12× compression, enabling a single server to hold ~26 hours of data for millions of series in RAM.

💡 Core Insight

For monitoring and alerting, you almost always query recent data (last 1 hour, last 24 hours). If you can keep that recent window entirely in compressed memory, you never touch disk for the hot path, and you can achieve sub-millisecond query latency.

📜 The Inventor's Problem

Facebook's existing TSDB (ODS, built on HBase) had a read latency problem: 73% of queries were for data less than 26 hours old, but every query required disk I/O. Gorilla was built as an in-memory write-through cache specifically for this recent-data read path.

Value Compression — XOR Encoding

Gorilla exploits the structure of IEEE 754 floating-point representation. When two consecutive floats are close in value, their bit representations share the same sign, exponent, and most significant mantissa bits. XOR zeros out all shared bits, leaving a tiny non-zero region.

⚠️ When XOR Fails

Gorilla compression degrades dramatically for highly volatile data (like random IDs stored as floats, or sensor data with high noise). When consecutive values differ significantly, the XOR output becomes random too—there are no shared leading bits. In this case, compression ratio falls toward ~1× or even negative (overhead from headers). Always validate compression ratio against your actual data distribution.

Blocks are typically 2 hours long. When a block's time window closes, it's marked immutable and can be flushed to disk or object storage as a backup. The in-memory store keeps approximately 13 × 2-hour blocks = 26 hours of recent data.

700M series × 10-second intervals × 26 hours = ~76 TB uncompressed → ~6.3 TB in RAM across the fleet. That's the power of 12× compression: you're hosting 76 terabytes of monitoring data in memory.

Step-by-Step Scenario: 10M series, 100K pts/sec

    t=0s: New 2-hour block starts. Each series gets a fresh stream buffer in memory.
  2. 2
    t=0.01s: First point for series X arrives: {t=1609459200, v=72.5}. Store timestamp and value as full 64-bit each (stream header).
  3. 3

  4. 4

  5. 5
    t=7200s (2h): Block closes. 720 points/series. Uncompressed: 720 × 16 bytes = 11,520 bytes. Compressed: 720 × 1.37 = ~986 bytes. ~12× compression. Background thread persists to disk.
  6. 6
    t=93,600s (26h): Oldest block evicted from memory.
Variant 2.2 · Section C

Strengths & Ideal Use Cases

✦
Blazing Read Performance

Sub-millisecond for any recent query—never touches disk. Every query for data within the 26-hour window is served entirely from compressed RAM.

✦
Incredible Compression Density

12× means you can fit 12× more data in memory, which means your "recent window" is 12× larger. The compression ratio directly determines your in-memory retention horizon.

✦
Perfect for Monitoring & Alerting

Alert rules typically evaluate over the last 5–30 minutes of data, which is always in memory. No disk I/O in the critical alert evaluation path.

✦
Simple Write Path

No WAL, no compaction, no LSM levels. Just append bits to a stream. The write path is a hash lookup + bit append — extremely low CPU overhead.

🎯 Sweet Spot

Millions of series, high write volume, queries dominated by recent time ranges (<26h), can tolerate loss of very recent data (seconds) on node crash without Paxos replication.

Real-world users: Facebook Gorilla (original paper), Prometheus 2.x local storage (adapted Gorilla compression), Uber's M3DB (Gorilla-compressed blocks + LSM for older data).

⚠️ Gotcha Most Candidates Miss

Burst of new series creation. Imagine a Kubernetes cluster where a deployment creates 50,000 new pods, each with 100 metrics. Suddenly 5 million new series appear simultaneously. Each needs a new hash map entry and stream buffer. The memory allocation spike can cause GC pauses (in Java/Go implementations) that stall writes for seconds, causing data loss from upstream buffering overflow.

Failure Scenario — Cardinality Explosion + OOM

💥 Node: 2M series, memory at 85% → new deployment triggers OOM

  • t=0sNormal. Node holds 2M series. Memory at 85% utilization.
  • t=3601sNew microservices deployment adds 500K new series. Memory allocation spikes.
  • t=3605sMemory hits 95%. GC becomes aggressive in the Go runtime.
  • t=3610sGC pause of 3 seconds. Incoming data queues in kernel buffers.
  • t=3613sGC completes. Ingestion buffer overflowed. ~150K data points dropped.
  • t=3620sMemory hits 98%. OOM killer activates. Node process dies.
  • t=3620sAll 2.5M series data lost for current block (last ~20 min of 2h window).
  • t=3625sReplica nodes (if configured) still have the data. Recovery via replicated stream.
📐 Lesson

Memory is the only critical resource in Gorilla. You MUST have hard limits on series count and pre-provision for 40–50% headroom above steady-state to absorb cardinality spikes.

Variant 2.2 · Section F

The L6/L7 Nuances

🔬 Deep Question: Why 2-Hour Blocks?

"Why did the Gorilla paper choose 2-hour blocks specifically?"

Answer: It's a trade-off between compression efficiency, memory management, and recovery granularity. Longer blocks compress better (more data points → more opportunities for delta encoding to save bits). But longer blocks mean: (1) more data lost if a block is unfinished when a node dies, (2) higher memory fragmentation as block buffers grow, and (3) longer decompression for queries needing only a few minutes. 2 hours was the empirical sweet spot at Facebook's scale where the compression plateau had been reached (diminishing returns beyond ~2h) and the loss window was acceptable with 3-way replication.

💡 IEEE 754 — Why XOR Actually Works

The XOR encoding in Gorilla isn't just "clever compression"—it's exploiting the structure of IEEE 754 floating-point representation. When two consecutive floats are close in value, their bit representations share the same sign bit, exponent bits, and most significant mantissa bits. XOR zeros out all shared bits, leaving a small non-zero region that can be encoded with just {leading zeros count, meaningful bits length, meaningful bits}.

This is why Gorilla compression degrades dramatically for highly volatile data—the XOR output becomes random too, with no shared prefix bits to exploit.

🌐 Google-Scale: Gorilla + Mixers

Google's Monarch system uses a Gorilla-like approach for its "leaves" (in-memory storage tier) but adds a distributed fan-out layer called "mixers" that can push down partial aggregation to leaves. At Google scale, you can't query all leaves—you'd have hundreds of thousands. The trick is routing queries to only the leaves that store the relevant series, which requires a global directory service that maps series to leaves. This directory service itself becomes a critical scaling challenge, introducing a new class of failure modes beyond what single-node Gorilla faces.

🏆 Staff-Level Sound Bite

Gorilla is not just a cache—it's a structural bet on workload access patterns. By accepting bounded retention and sequential-only access within a block, it eliminates the entire compaction problem that plagues LSM trees. The compression scheme works because monitoring data has a predictable regularity property: metrics arrive at known intervals and change slowly. When that regularity breaks (high cardinality, volatile values), Gorilla's advantages evaporate.

📦 Variant 2.3 · Prometheus TSDB v2
Variant 2.3 · Section A

What It Is

📦

One-Sentence Definition

A time-structured merge tree organizes time series data into immutable, time-bounded blocks on disk, each containing its own index, chunk files, and tombstone file, enabling efficient time-range queries, simple deletion (drop entire blocks), and compaction by merging adjacent time blocks rather than by LSM levels.

💡 Core Insight

Instead of organizing data by size levels (like a traditional LSM), organize by time windows. The fundamental unit of compaction is merging adjacent time blocks (e.g., three 2-hour blocks become one 6-hour block), which perfectly aligns with how time series data is queried and aged.

📜 The Inventor's Problem

Prometheus v1 used one file per time series, which created millions of tiny files and killed filesystem performance. Prometheus v2 (designed by Fabian Reinartz, inspired by Gorilla and LSM ideas) needed a storage engine that worked well on a single node, supported efficient time-range queries, allowed simple data deletion by time, and was easy to reason about operationally.

Step-by-Step Walkthrough

Tracing 50K series, 500K points/sec through the full lifecycle:

Write Path (Head Block)

  1. 1
    Point arrives: cpu_usage{host="web01"} 72.5 @1609459200
  2. 2
    WAL write: Sequential append to disk WAL file. ~20 bytes per point. Fast sequential I/O only.
  3. 3
    Head block memory: Series map lookup: hash("cpu_usage{host=web01}") → series struct. Point appended to the active Gorilla-compressed chunk for this series.
  4. 4
    Active chunk holds ~120 samples (120 × 15s = 30 min of data). When chunk fills, it's closed and a new chunk starts.

Head Block Cut (Every 2 Hours)

  1. 1
    Current head block's in-memory data serialized to disk: all closed chunks written to chunks/ files.
  2. 2
    Build inverted index — write index file with label→series mappings.
  3. 3
    Write meta.json with block metadata (time range, series count, ULID).
  4. 4
    New head block starts with fresh WAL. Old WAL segments for the flushed block are deleted.

Compaction — Merging Adjacent Blocks

COMPACTION FLOW
Compaction Plan: [Block 0h-2h] + [Block 2h-4h] → [Block 0h-4h]

For each series in either block:
  1. Read chunks from both source blocks
  2. Re-encode into new chunks
     (can improve compression: longer runs → better delta encoding)
  3. Write to new block's chunk/ files
  4. Build new inverted index for merged block
  5. Apply tombstones → physically remove deleted data

Atomic swap:
  new_block_ref = [0h-4h]
  delete old blocks: [0h-2h] + [2h-4h]

Result: 2 blocks → 1 larger block (same data, better compressed, tombstones applied)

Query Path

Query: cpu_usage{host="web01"} from t_start to t_end

  1. 1
    Block selection: Check each block's meta.json time range. Skip blocks entirely outside [t_start, t_end]. Massive I/O pruning.
  2. 2
    Index lookup in each relevant block: Look up label matcher host="web01" → posting list. Intersect with __name__="cpu_usage" → final series IDs.
  3. 3
    Chunk retrieval: For each matching series, look up chunk references that overlap [t_start, t_end]. Read and decompress those chunks only.
  4. 4
    Merge: If query spans head block + persisted blocks, merge results from memory and disk into a unified iterator.

Variant 2.3 · Section C

Strengths & Ideal Use Cases

✦
Elegant Deletion — Just rm -rf

Dropping data older than 30 days = deleting block directories. No tombstone accumulation, no compaction cycle needed. The retention policy is literally a directory deletion. Operationally, this is a superpower.

✦
Excellent Time-Range Read Performance

Block-level time pruning eliminates most I/O before even opening a file. A 30-day retention with 2-hour blocks = 360 blocks; a 1-hour query touches at most 1 block.

✦
Self-Contained, Portable Blocks

Each block is a complete, independent unit. You can back it up, move it to S3, or restore it independently. This enables easy long-term storage integration — Thanos sidecar uploads sealed blocks directly to object storage.

✦
Simple Operational Model

No complex LSM level management. Compaction is entirely predictable — merge 2-hour blocks into 4-hour, 4-hour into 8-hour, and so on. The compaction schedule is a simple exponential growth pattern.

🎯 Sweet Spot

Single-node or federated monitoring, 1M–100M series, Prometheus/Thanos/Cortex ecosystem, when retention is bounded (days to months).

Real-world users: Prometheus, Thanos, Cortex, Grafana Mimir — all use the Prometheus TSDB block format.

⚠️ Gotcha — Series Churn in Kubernetes

In Kubernetes, pods come and go. Each pod creates new series, and departed pods' series become stale. Over a day, you might see 50M unique series total but only 5M active at any moment. The head block must maintain index entries for all 50M (because it doesn't know which are "dead" until the block is cut). High series churn is the #1 operational problem in Prometheus at scale — it drives head block memory up without a corresponding increase in active data.

Variant 2.3 · Section F

The L6/L7 Nuances

🔬 Deep Question: Posting Lists vs. B-Trees

"Why does Prometheus TSDB use posting lists with bitmap intersection for its inverted index rather than a B-tree?"

Answer: Time series queries involve set operations on labels. A query like cpu{host=~"web.*", region="us-east"} requires: (1) find all series with region="us-east" → posting list A, (2) find all series matching host=~"web.*" → posting list B, (3) intersect A ∩ B. Sorted posting lists allow this intersection in O(n) with merge-join, and they compress beautifully using Roaring Bitmaps (run-length encoding for dense ranges, raw lists for sparse). A B-tree would make individual lookups fast but set intersection slow.

💡 Non-Obvious Insight That Impresses Interviewers

The Prometheus TSDB compaction strategy is deliberately vertical within a time range (merging series data across blocks of the same time window) rather than horizontal (merging series across time). This means each block's index is a self-contained snapshot of the label→series mapping for that time range.

When series churn is high, old blocks naturally contain only old series, and new blocks contain new series, without needing explicit series garbage collection. The "garbage collection" happens implicitly when old blocks are deleted by the retention policy. This is why time-structured compaction is fundamentally superior to LSM-structured compaction for high-churn workloads: churn is a feature, not a bug, in this model.

🌐 Thanos/Cortex: Distributed Prometheus TSDB

Prometheus TSDB is single-node by design, but its block format is extremely well-suited for object storage federation. Thanos uploads sealed blocks to S3/GCS as-is (no format conversion). The block's self-contained structure with its own index means Thanos store can serve queries directly from object storage by downloading only the relevant index and chunk files — never the entire block.

This is only possible because of the time-partitioned, self-contained block design. An LSM-based TSDB with cross-file index references would not have this property.

🏆 Staff-Level Sound Bite

Prometheus TSDB is the recognition that time is the natural partitioning key for monitoring data, full stop. By making time the first-class organizing dimension, you get retention-by-deletion, query-pruning, and object-storage compatibility for free. The cost is head block memory pressure at high cardinality — which is why every large-scale Prometheus deployment ultimately lands on the same solution: shard series across nodes by label hash, so each node's head block is bounded.
