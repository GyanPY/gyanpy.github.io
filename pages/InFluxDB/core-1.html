Part 1 · Section 1.1

The Elevator Pitch

For a smart non-engineer

A time series database is a system purpose-built to record and query measurements that happen over time—like a hospital's vital-signs monitor that records your heart rate every second and lets a doctor instantly see trends, anomalies, and comparisons across millions of patients simultaneously.

For a fellow engineer

A TSDB exploits the append-heavy, time-ordered, high-cardinality nature of timestamped data to achieve 10–100× better write throughput and compression than general-purpose databases by using columnar storage, aggressive time-partitioning, and specialized indexing that trades random-write flexibility for sequential-write speed.

💡 The "Aha" Moment

General-purpose databases treat every row as equally likely to be updated, queried, or deleted at any time. But time series data has a massive structural gift: new data always arrives at the "right edge" of time, old data is almost never modified, and queries almost always ask for contiguous time ranges.

Once you realize this, you understand why every single design decision in a TSDB—from the storage engine to the compaction strategy to the index structure—is an optimization exploiting these three invariants. You're not building a general database; you're building a write-optimized, range-scan-optimized, compression-optimized append log with smart indexing.

Part 1 · Section 1.2

The Perfect Analogy

🌦️ The Weather Station Logbook

Imagine you run 10,000 weather stations across the country. Each station records temperature, humidity, wind speed, and pressure every 10 seconds. You need to:

  • Write new readings as fast as they arrive (you can't tell a weather station to "slow down")
  • Query things like "show me temperature for Station #4523 over the last 24 hours"
  • Aggregate things like "what was the average humidity across all California stations last month?"
  • Eventually discard old fine-grained data (you don't need per-second resolution from 5 years ago)

A general-purpose database is like writing each reading on its own index card, filing it alphabetically by station name, and stacking them in a giant filing cabinet. Finding "Station #4523's temperature for last Tuesday" means rifling through millions of cards.

A time series database is like giving each station its own dedicated logbook, where new readings are always written on the next blank line. The pages are organized by time (one page per hour). Finding last Tuesday's temperature means pulling one logbook off the shelf, flipping to the right pages, and reading them in order. Massively faster.

Extending the Analogy to Variants

01
LSM-Tree based (InfluxDB, early Prometheus)

Each station writes onto sticky notes that pile up on a desk (memtable). Periodically, a clerk sorts the sticky notes and pastes them into the logbook (flush to SSTable). Over time, another clerk merges smaller logbooks into bigger ones (compaction).

02
Compressed columnar blocks (Gorilla / Prometheus TSDB)

Instead of writing each reading as "temp=72, humidity=55, wind=8," you have separate columns—one strip of paper just for temperatures, one for humidity. Since consecutive temperatures are close in value (72, 72, 73, 72...), you write just the differences (0, +1, -1), which compresses incredibly well.

03
Time-structured merge tree (InfluxDB TSI / TimescaleDB)

You organize the logbooks not just by station but by time blocks. All readings from "January 2024" for all stations live in one box. When January is over, you seal that box and never open it again for writes. This is time partitioning.

Failure Scenarios in the Analogy

⚡ Write Spike — The Hurricane Scenario

Suddenly all 10,000 stations report at 1-second intervals instead of 10-second. The sticky notes pile up faster than the clerk can paste them. The desk overflows—this is memtable memory pressure. You either need more desks (more RAM) or the clerk needs to flush faster (more disk I/O).

💥 Cardinality Explosion

A new requirement says to tag each reading with the individual sensor's serial number. Suddenly you don't have 10,000 logbooks—you have 10,000 × 4 sensors = 40,000. The shelf index (inverted index) blows up. This is the high cardinality problem that has killed production TSDB clusters.

🕒 Out-of-Order Data

A station lost connectivity for 2 hours and now sends backfilled data. But you already sealed that time block's box. Now you need to unseal, merge, and reseal—expensive. This is the backfill/late-arrival problem.

Part 1 · Section 1.3

The Mental Model

5 Core Concepts You MUST Internalize

1
The Write Path Is King

TSDBs are designed for massive write throughput (often 10M+ points/sec). The entire architecture—LSM trees, WAL, batching, columnar layout—serves the write path first. Reads are optimized second, but cleverly, because the time-ordered nature makes range scans inherently efficient.

2
Time Partitioning Is the Fundamental Organizing Principle

Data is partitioned into discrete, non-overlapping time blocks (e.g., 2-hour chunks). This is the single most important design decision because: writes always hit the most recent block (hot write path is tiny), deleting old data = dropping entire blocks (no expensive per-row deletion), queries span only the relevant blocks (massive pruning).

3
Series Cardinality Is the Hidden Scaling Dimension

"Series" = a unique combination of metric name + tag values (e.g., cpu.usage{host=web01, region=us-east}). The number of unique series—cardinality—determines the size of your index, not the number of data points. A system with 10M unique series is far harder to manage than one with 100K series that each have 100× more data points.

4
Compression Is Not Optional; It's Architectural

Time series data compresses 10–20× because consecutive values are strongly correlated (delta-of-delta for timestamps, XOR encoding for floats). This compression isn't just a storage saver—it means more data fits in memory, which means faster queries. Compression ratio directly determines query performance.

5
The Inverted Index Maps Tags to Series

To answer "show me CPU usage for all hosts in us-east," you need an inverted index that maps region=us-east → list of series IDs. This index is the most operationally fragile component and the one most likely to cause outages at scale.

What It GUARANTEES vs. What It Does NOT

✅ Guarantees ❌ Does NOT Guarantee
Extremely high write throughput for append-only workloads Strong consistency for concurrent reads during writes
Efficient range scans over contiguous time windows Fast random-access point lookups by arbitrary fields
Excellent compression for numerically correlated data Efficient arbitrary JOINs across series
Cheap deletion of old data via partition drops Fast updates/modifications of historical data points
Sub-second queries over recent data with proper indexing Unbounded cardinality without performance degradation

Fundamental Trade-offs

Trade-off Description
Write throughput vs. read consistency Batching and async flushing make writes fast but reads of the most recent seconds may hit the memtable path.
Compression ratio vs. random access Columnar, delta-encoded blocks compress beautifully but you can't seek to an arbitrary point without decompressing the block.
Index completeness vs. memory footprint A full inverted index enables flexible queries but consumes enormous memory at high cardinality.
Time granularity vs. storage cost Higher resolution data enables better analysis but grows storage linearly.
Query flexibility vs. write optimization Supporting SQL-like queries adds overhead to the write path; more restricted query models (like PromQL) allow tighter optimization.

The "Smell Test"

If 4 or more of these are true, you're in TSDB territory:

  • ✅Data arrives continuously with timestamps?
  • ✅Queries are overwhelmingly time-range based?
  • ✅Data is almost never updated after writing?
  • ✅You need to handle orders of magnitude more writes than reads?
  • ✅You need retention/downsampling policies?
  • ✅You need fast aggregations (avg, max, percentile) over time windows?
🔑 Interview Sound Bite

"Time series databases exploit three structural invariants—data is append-only, time-ordered, and queried in contiguous ranges—to achieve write throughput and compression ratios that are fundamentally impossible in general-purpose databases. The key architectural insight is time-partitioning: by organizing data into sealed, immutable time blocks, you turn deletes into drops, writes into sequential appends, and queries into targeted block scans."

Part 2 · Variant 2.1

LSM-Tree Based TSDB

🗂️

One-Sentence Definition

An LSM-tree based TSDB buffers incoming time series points in an in-memory sorted structure (memtable), then flushes them as immutable sorted files (SSTables) to disk, periodically merging these files to maintain query efficiency.

⚡ Core Insight

By converting random writes into sequential writes (via the memtable → flush → compact pipeline), you can achieve write throughput limited only by sequential disk bandwidth, not random I/O.

📜 The Inventor's Problem

OpenTSDB (built on HBase) needed to store billions of data points from thousands of servers, and traditional B-tree databases couldn't handle the write load because every insert required random disk seeks to update the B-tree in place.

B) How It Works — Step-by-Step

Let's say we have a TSDB receiving metrics from 5,000 hosts, each reporting 100 metrics every 10 seconds = 50,000 points/second.

  1. 1
    Write-Ahead Log (WAL): Every incoming point is first appended to a sequential WAL file on disk. This is purely for crash recovery—it's fast because it's sequential I/O.
  2. 2
    Memtable buffering: The point is simultaneously inserted into an in-memory sorted structure (typically a skip list or red-black tree), sorted by {series_id, timestamp}. At 50K points/sec with ~50 bytes/point, this accumulates ~2.5 MB/sec.
  3. 3
    Memtable flush (every few minutes or at size threshold, say 64 MB): When the memtable reaches capacity (~26 seconds of data at our rate), it's frozen, a new empty memtable takes over, and the frozen one is serialized to disk as an immutable SSTable (Sorted String Table). Each SSTable contains:
    • A data block section (sorted key-value pairs, compressed)
    • An index block (sparse index pointing into data blocks)
    • A bloom filter (for quickly checking "is series X in this file?")
  4. 4
    Compaction: Over time, you accumulate many SSTables. A read for "series X over last hour" might need to check 10+ files. Compaction merges multiple SSTables into fewer, larger ones, removing duplicates and tombstones. Two strategies:
    • Size-tiered: Merge files of similar size. Simpler, better write amplification.
    • Leveled: Organize files into levels with size limits. Better read amplification, more predictable.
  5. 5
    Read path: To query cpu.usage{host=web01} from the last hour:
    • Check the active memtable (most recent data)
    • Check each SSTable that might contain data from the last hour
    • Use bloom filters to skip SSTables that don't contain this series
    • Merge results from all sources

C) Strengths & Ideal Use Cases

  • Incredible write throughput: Sequential I/O only on the write path. 100K–1M+ points/sec per node easily achievable on SSDs.
  • Works with existing LSM infrastructure: Can leverage battle-tested engines like RocksDB, LevelDB.
  • Flexible schema: Tag-based series identification works naturally with key-value LSM layout.
🎯 Sweet Spot

10K–1M unique series, write-heavy workloads (>10:1 write:read ratio), when you need durability guarantees.

Real-world users: OpenTSDB (HBase LSM), early InfluxDB (BoltDB → custom LSM), QuestDB (partially), Apache IoTDB.

D) Limitations & Failure Modes

LimitationRoot CauseSeverity
Write amplification Each point written 10–30× across WAL → L0 → L1 → L2... Medium — burns SSD lifetime
Compaction storms Write rate exceeds compaction rate; L0 files pile up High — cascading failure
Read amplification (recent data) Query must check memtable + overlapping L0 SSTables Medium — worst at L0
Cardinality bomb Bloom filters + inverted index blow out memory at 50M+ series High — kills production clusters
Expensive deletes Tombstones not removed until compaction; space wasted during window Medium
⚠️ Gotcha Most Candidates Miss

When you drop data older than 30 days, an LSM doesn't actually delete—it writes tombstones that are only removed during compaction. During that window, the space is wasted and queries must skip tombstones. At scale, tombstones can be a significant fraction of your total data size.

Failure Scenario Walkthrough — The Spike

⚡ Monitoring system: 100K → 500K points/sec spike

  • t=0sNormal operation. 100K pts/sec. Memtable flushes every ~16s.
  • t=10sSpike hits. Memtable fills in ~3s instead of 16s.
  • t=30s6 L0 SSTables created in 20 seconds. Compaction kicks in but can only merge at ~200 MB/sec.
  • t=60s15 L0 files piled up. Read queries now must check all 15 overlapping files. P99 latency jumps from 10ms to 200ms.
  • t=120sCompaction falling behind. L0 count hits 30. TSDB applies back-pressure: slows down writes. But UDP metric emitters don't slow down. Data is now being dropped.
  • t=300sSpike ends. Compaction clears the backlog over ~10 minutes. But 40% of data from the spike window was lost.
📐 Rule of Thumb

LSM-based TSDBs need significant I/O headroom for compaction. Reserve 50% of your disk I/O budget for compaction.

E) Operational Considerations

📊 Key Metrics to Watch
  • L0 SSTable count (alert if > 20)
  • Compaction pending bytes
  • Write amplification factor
  • Memtable size + flush frequency
  • Bloom filter false positive rate
🔧 Tuning Knobs
  • Memtable size (larger = fewer flushes)
  • Compaction strategy (size-tiered vs. leveled)
  • Bloom filter bits per key
  • Compression algorithm (Snappy vs. Zstd)
  • Max L0 file count before write slowdown
🐛 Debugging Symptoms
  • "Queries are slow" → Check L0 count. Compaction behind.
  • "Disk full faster than expected" → Write amplification + tombstones.
  • "Memory climbing" → Bloom filters or memtable not flushing.
🚚 Migration Story
  • Run both in parallel (dual-write)
  • Validate query parity
  • Cut over reads, then stop writing to old DB
  • Biggest risk: relational tag schema ≠ label model

F) The L6/L7 Nuances

🔬 Deep Understanding Question

"Why does an LSM-based TSDB typically separate the inverted index (tag → series mapping) from the main LSM data store, and what are the implications?"

Answer: The inverted index needs fast point lookups and updates for tag queries, while the data store needs fast sequential writes and range scans for time-ordered values. Mixing them in the same LSM means compaction for data churn (which is massive) also churns the index (which is relatively stable). InfluxDB learned this the hard way—their TSI (Time Series Index) was eventually moved to its own separate storage to avoid coupling index I/O with data I/O.

💡 Non-Obvious Insight — Two-Level Skip

In a TSDB workload, the SSTable bloom filters are used differently than in a general KV store. In a KV store, you're checking "does key X exist in this file?" In a TSDB, you're checking "does series S have any points in this time range file?"

The time range is already pruned by the SSTable metadata, so the bloom filter is really only needed for the series dimension. This means you can use time-range metadata + series bloom filter as a two-level skip mechanism, which is much more selective than either alone.

🌐 Google-Scale vs. Startup-Scale

At startup scale, a single-node LSM-based TSDB is fine. At Google scale (billions of series), the LSM compaction becomes a distributed problem. You can't compact terabytes on a single node.

Google's Monarch uses a fundamentally different approach—it routes series to specific "leaves" (storage nodes) by consistent hashing on the series key, so each leaf's LSM only sees its shard. But cross-shard queries (like "avg CPU across all hosts") require a fan-out query layer that aggregates across all leaves, introducing the distributed aggregation problem.

🏆 Staff-Level Summary

The LSM-tree is fundamentally a write buffer abstraction: it buys you sequential write speed by deferring the cost of organization to compaction. In a TSDB context, time-partitioning already handles the biggest organizational burden (blocks are never mixed across time ranges), so the LSM only needs to organize within a time window. This is why InfluxDB's TSI and Prometheus's TSDB both evolved toward hybrid models—pure LSM carries compaction costs that time-partitioned workloads don't need to pay.
