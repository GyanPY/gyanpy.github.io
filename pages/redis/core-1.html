Part 1: Cluster Fundamentals Part 2: Data Structures & Persistence

One of the most common Redis interview mistakes: confusing hash slots with shards. They are completely different concepts that serve different roles in the cluster. This guide builds the complete mental model — from why 16,384 was chosen to live resharding mechanics, from replica placement strategy to the second-by-second failover timeline, to the decision framework for how many replicas your service actually needs.

The Core Distinction: Hash Slots ≠ Shards

Hash Slots
16,384

Logical address buckets for distributing keys. Like postal codes — a fixed address system. The number never changes regardless of how many machines you have.

Fixed forever. You never configure this.

Shards (Nodes)
2 – 1000+

Physical Redis server instances that hold the slots. Like post offices — the actual buildings that handle the mail. You decide how many you need based on your capacity requirements.

Flexible. You add more when traffic grows.

The Filing Cabinet Analogy

Imagine 16,384 folders (hash slots) distributed across 2 filing cabinets (shards). Cabinet 1 holds folders 0–8191. Cabinet 2 holds folders 8192–16383. When a key arrives, Redis computes which folder it belongs to, then routes to whichever cabinet currently holds that folder. The 16,384 folders never change. The number of cabinets is up to you.

How Keys Get Mapped to Shards

The Exact 5-Step Routing Flow
Step 1: Key arrives
  Key = "otp:01HXR3K9V7"

Step 2: Compute hash slot
  slot = CRC16("otp:01HXR3K9V7") % 16384
  slot = 11,205

Step 3: Look up which shard owns slot 11,205
  Slot mapping (configured when cluster is created):
    Shard 1: slots  0     →  8,191   (first half)
    Shard 2: slots  8,192 → 16,383   (second half)

  11,205 falls in Shard 2's range

Step 4: Client sends command DIRECTLY to Shard 2
  No proxy, no central router
  Smart client caches the slot-to-shard map locally
  Direct TCP connection to the shard that owns the slot

Step 5: Shard 2 stores/retrieves the key
  Stores in slot 11,205
  Returns OK in < 0.5ms total
L6/L7 Insight — The Client Is Smart

"Redis Cluster has no central proxy or coordinator. The client library (Jedis, ioredis, redis-py) maintains a local copy of the slot-to-shard mapping. When a key arrives, the client computes CRC16(key) % 16384 itself, looks up the slot in its local map, and connects directly to the correct shard. If the slot has moved (during resharding), the server returns a MOVED or ASK redirection and the client updates its map. This smart-client design eliminates the proxy as a bottleneck and single point of failure."

Linear Scaling — Double Shards = Double Capacity

Each Redis shard handles ~100K commands/sec. Adding shards gives you linear throughput scaling. Memory scales the same way — each new shard is a new RAM pool. There's no coordination overhead that degrades with shard count (the smart-client design ensures each request hits exactly one shard).

Why 16,384? The Engineering Reason

The Sweet Spot Between Granularity and Network Overhead
  • Too few slots (e.g. 100): With 50 shards, only 2 slots per shard → wildly uneven key distribution. Hot shards with 10× normal keys.
  • Too many slots (e.g. 1,000,000): Every node stores the slot→node mapping. Cluster heartbeat messages include a slot bitmap. At 16,384 slots, the bitmap is 2 KB. At 1M slots, it's 125 KB. Multiply by heartbeats every second × every node = enormous network overhead.
  • 16,384 is the sweet spot: 2 KB bitmap fits easily in heartbeat messages. Enough granularity for up to 1,000 nodes (16 slots each minimum). Perfectly balanced for typical 3–100 node deployments.

Redis creator Salvatore Sanfilippo specifically chose 16,384 as the balance between granularity (enough slots to distribute evenly) and cluster protocol overhead (heartbeat bitmap stays small).

Live Resharding — Adding a Shard with No Downtime

Before — 2 Shards

Shard 1: Slots 0–8,191 (8,192 slots)  |  Shard 2: Slots 8,192–16,383 (8,192 slots)  |  Shard 3: (empty, just joined)

▼
During — Slot Migration (Live)

Redis migrates slots from existing shards to the new shard — while the cluster continues serving traffic.

  • Shard 1: Moving slots 5,462–8,191 → Shard 3
  • Shard 2: Moving slots 13,654–16,383 → Shard 3
  • Keys in migrating slots move one by one
  • Requests for migrating keys get MOVED or ASK redirections automatically
  • Smart clients handle redirections transparently
▼
After — 3 Shards, Balanced

Shard 1: Slots 0–5,461 (5,462 slots)  |  Shard 2: Slots 8,192–13,653 (5,462 slots)  |  Shard 3: Slots 5,462–8,191 + 13,654–16,383 (5,460 slots)

Each shard now holds roughly 1/3 of the key space. Cluster stayed online. No downtime.

What to Watch During Resharding

The Beautiful Part — What Works Seamlessly
  • Cluster stays online throughout the migration
  • Zero downtime — no maintenance window required
  • Clients receive MOVED/ASK redirections for migrating keys and follow them automatically
  • From the application's perspective, nothing changed
The Careful Part — Watch These During Migration
  • Slightly higher latency for migrating keys: MOVED redirection adds one extra network hop
  • Avoid resharding during peak traffic: Migration is CPU and network intensive on both source and destination shards
  • Lua scripts across migrating keys: Scripts touching keys in migrating slots may receive MOVED errors — scripts must be atomic per slot
  • Migration time scales with data size: A 10 GB shard migration can take hours — plan accordingly

Replicas Per Shard — 0 / 1 / 2 / 3 Compared

0 Replicas No Redundancy — Never in Production ▾
What Happens When a Shard Dies

Slots 0–8,191 are completely unavailable. All OTPs in those slots are lost forever. Half your users can't verify OTPs. No automatic recovery possible — manual intervention required.

Total nodes: 2. Cost: 2 × $400/month = $800/month.

Acceptable for: Local development, testing environments only.

1 Replica Standard Production HA — The Right Choice for Most Services ▾
1 Replica Per Shard — Topology
Shard 1 [MASTER] ──async replication──→ Shard 1 [REPLICA]
Shard 2 [MASTER] ──async replication──→ Shard 2 [REPLICA]

Total nodes: 4 (2 masters + 2 replicas)

If Shard 1 Master dies:
  → Replica promotes to Master in 5-6 seconds ✅
  → Zero data loss if replication was caught up ✅
  → Service continues uninterrupted ✅
  → Now temporarily has NO replica (vulnerable until old master recovers)

If Shard 1 Master AND Replica die simultaneously:
  → Slots 0-8,191 unavailable 💀
  → How likely? Very unlikely — they live in different AZs
  → Requires two simultaneous failures in different data centers
Why 1 Replica Is Enough for OTP Service

OTPs are ephemeral and replaceable. TTL = 5 minutes. Even if we lost all data on a shard, users click "Resend OTP" and get a new one. The data is not precious — it's disposable by design. Worst case during a 5-second failover: a few users see a verification error, then click resend. Cost of extra replica: $800/month. Overkill for disposable data.

2 Replicas High Availability — Financial & Healthcare Systems ▾
2 Replicas Per Shard — Topology
Shard 1 [MASTER] ──→ Shard 1 [Replica A]  (AZ-1)
                 ──→ Shard 1 [Replica B]  (AZ-2)

Total nodes: 6 (2 masters + 4 replicas)

If Shard 1 Master dies:
  → Replica A promotes to Master ✅
  → Replica B still exists as backup ✅
  → You STILL have redundancy after the failure ✅

If Shard 1 Master AND Replica A both die:
  → Replica B promotes to Master ✅
  → Three simultaneous failures needed for data loss

When to use:
  → Financial systems (every transaction is irreplaceable)
  → Healthcare (patient data, lives depend on it)
  → Systems where even 1-3 seconds slot unavailability is unacceptable
3 Replicas Extreme HA + Read Scaling — Almost Never Needed ▾

Three replicas spread across three AZs — any entire AZ can go down and you still have a master or promotable replica plus two additional backups. Also used for read scaling when you want to distribute reads across replicas (Redis 7+ supports read-from-replica configuration).

Total nodes: 8 (2 masters + 6 replicas). Cost: 8 × $400 = $3,200/month. Mission-critical systems requiring 99.999% availability. Almost never needed for an OTP service or typical web application.

Placement Strategy — Where Replicas Must Live

❌ Wrong: Master and Replica on the Same Machine or Rack

If the physical machine or rack loses power: Master is dead AND Replica is also dead (same hardware). Total loss. No failover possible. This eliminates all the protection the replica was supposed to provide.

AZ Failure Analysis — Either AZ Can Go Down Safely
  • AZ-1 goes down: Shard 1 replica in AZ-2 promotes to Master ✅   Shard 2 master is already in AZ-2, unaffected ✅
  • AZ-2 goes down: Shard 2 replica in AZ-1 promotes to Master ✅   Shard 1 master is already in AZ-1, unaffected ✅
  • Notice: Masters are spread across AZs. Each shard's replica is in a different AZ from its master. Cross-shard interleaving (Shard 2's replica in AZ-1) means each AZ's failure takes down one master but always has another master present.

Replication Lag & The WAIT Command

Asynchronous Replication — The Hidden Gap
Default replication is ASYNCHRONOUS:

  T=0ms:    Client sends: SET otp:abc "data"
  T=0.1ms:  Master stores it → returns OK to client immediately
  T=0.5ms:  Master sends key to Replica (replication stream)
  T=1.0ms:  Replica receives and stores it

  The 0.5-1ms gap between "OK returned" and "replica has it" 
  is called REPLICATION LAG.

What if Master crashes in that gap?
  T=0ms:    Client writes OTP to Master
  T=0.1ms:  Master returns OK (client thinks it's saved)
  T=0.3ms:  Master CRASHES (before sending to Replica!)
  T=5s:     Replica promotes to new Master
  
  Result: That OTP is LOST.
          It existed on the dead master but never made it to the replica.
          User gets "OTP sent" but verification will fail.
  
  User impact: They click "Resend" → new OTP works fine.
  For OTPs: This is ACCEPTABLE (1 in 100,000 OTPs lost is fine).
  For banking: This is NOT ACCEPTABLE.
Redis — Synchronous Replication (When You Need It)
# For guaranteed durability (banking, critical data):
SET user_balance 5000.00
WAIT 1 0    # Wait until at least 1 replica confirms, no timeout

# Master won't return OK until the replica has the data.
# Trade-off:
#   ✅ Zero data loss on failover
#   ❌ Every write is ~1ms slower (waiting for replica network ACK)
#   ❌ If replica is temporarily down, writes BLOCK until timeout

# For OTP service: NOT worth it.
# The extra latency hurts ALL requests to protect against
# an extremely rare edge case (master crash in <1ms window).

Failover — Second by Second

Second 0

Master stops responding. Clients begin failing with connection errors.

Seconds 1–2

Heartbeats missed. Replica and other nodes notice the master hasn't sent its heartbeat. Each node independently records missed heartbeats.

Seconds 3–4

PFAIL declared. Replica has missed enough heartbeats to declare master as PFAIL (possibly failed). Other masters in the cluster are polled — do they also see the failure?

Second 4–5

Majority agrees → FAIL. If the majority of masters agree the node is unreachable, it's marked FAIL. The replica begins the FAILOVER process.

Second 5–6

Replica promotes. Replica announces to all nodes: "I am the new Master for Shard 1." All nodes update their slot maps. Clients receive MOVED redirections and reconnect to the new master.

Recovery

Old master comes back. It detects a new master exists, demotes itself to replica, and begins syncing from the new master. The shard is fully redundant again.

During the 5-6 Second Window

Requests targeting slots on the failed shard will receive connection errors. Smart clients will retry with exponential backoff. Users may see 1–2 failed requests. For OTP service: verification attempts fail → user retries → succeeds once promotion is complete.

Tuning cluster-node-timeout

The Timeout Trade-off
  • Default: 15,000ms (15 seconds) — too slow for production. Users experience 15 seconds of errors before failover completes.
  • Recommended: 5,000ms (5 seconds) — good balance. Fast enough for user experience, tolerant enough for network jitter.
  • Aggressive: 2,000ms (2 seconds) — faster failover but false positives. A 1-second network hiccup triggers unnecessary failover. Failover itself is expensive — all clients must reconnect to new master.
  • Sweet spot for OTP service: 5,000ms — failover in ~5 seconds, won't false-trigger on normal network jitter.

Interview Talking Points

🎤 "How Does Redis Cluster Work?"

"Redis Cluster divides the key space into 16,384 hash slots — this number is fixed and never changes. These slots are distributed across the actual Redis nodes, which we call shards. With 2 shards, each holds ~8,192 slots. When a key arrives, the client computes CRC16(key) % 16384 to get the slot number, then routes directly to the shard owning that slot. The client is smart — it caches the slot-to-shard mapping locally and connects directly, no central proxy needed."

🎤 "How Do You Scale Redis Cluster?"

"You add more shards and Redis migrates slots from existing shards to the new one — live, with no downtime. The cluster automatically redirects requests for migrating keys using MOVED and ASK responses. Smart clients handle these redirections transparently. It's linear scaling — double the shards, double the throughput and memory capacity. I avoid resharding during peak traffic since migration is CPU-intensive on both source and destination."

🎤 "Why Only 2 Shards for 50K RPS?"

"Each Redis shard handles about 100K commands per second. At 50K RPS, the OTP service generates ~100K Redis commands/sec. With 2 shards, each handles 50K — 50% utilization, comfortable headroom. Memory is 1.1 GB/shard on a 26 GB instance. There's no reason to over-provision. When traffic doubles, we add 2 more shards — takes minutes, zero downtime. The scaling plan is: 2 shards → 50K RPS, 4 shards → 100K RPS, 8 shards → 200K RPS. Linear and predictable."

How Many Replicas Do I Need?
Is the data replaceable? YES (OTPs, sessions, cache) → 1 replica is enough. NO (transactions, financial records) → 2+ replicas minimum.
Can users tolerate a 5-second blip? YES (most apps) → 1 replica + 5s timeout. NO (real-time trading) → 2 replicas + 2s timeout.
What's the cost of total data loss? Low (users can resend, cache can rebuild) → 1 replica. High (money lost, compliance violation) → 2 replicas + WAIT command.
For OTP service: Data is replaceable (5-min TTL) + 5-second blip acceptable + no permanent data loss risk → 1 replica per shard is the right choice.
🎯 Key Lesson — Part 1

The single most important distinction in Redis Cluster: hash slots (16,384) are a fixed distribution mechanism; shards are the physical machines you add as traffic grows. The slot count is an engineering constant — chosen to keep heartbeat bitmaps at 2 KB while providing enough granularity for 1,000-node clusters. The shard count is an operational decision — double the shards, double the capacity, linearly.

For replicas: the question is always "is the data replaceable?" Ephemeral data (OTPs, sessions, cache) tolerates 1 replica and 5-second failover. Irreplaceable data (transactions, user records) demands 2 replicas and synchronous replication via WAIT. The right answer is determined by the cost of data loss, not by a desire for maximum redundancy. Over-replicating ephemeral data is wasted money; under-replicating irreplaceable data is a production disaster waiting to happen.
