PILLAR 01

Core Concepts & Mental Models

The World Before Redis

Before Redis (2009), caching was largely done with Memcached — a flat key-value store that could only hold strings. Application data structures like leaderboards, session graphs, or rate-limit counters had to be emulated awkwardly in the application layer, requiring multiple round-trips and complex client-side logic. Relational databases handled every read, even for hot data that never changed, causing contention and latency spikes under high concurrency.

The fundamental problem: disk I/O is 1,000–10,000x slower than RAM I/O. Any database designed primarily around disk storage — even with a warm buffer pool — must pay the latency cost of disk seek + OS page fault on cache misses. For session data, counters, queues, and sorted sets, this cost is unacceptable at scale.

The Core Abstraction

Redis's central insight is to treat data structures as first-class citizens, not just opaque byte blobs. Rather than storing a counter as a JSON string that must be deserialized → incremented → re-serialized → written back (three round-trips, race-prone), Redis provides an INCR command that performs an atomic increment server-side in a single command. This server-side computation on rich types is what makes Redis more than a cache — it is a Remote Data Structure Server.

String
Binary-safe, max 512MB. Implements atomic integers/floats via INCR/INCRBYFLOAT. Foundation for all other types internally. Encoded as int, embstr, or raw depending on value.
Hash
Field-value map. Stored as listpack (ziplist) up to 128 fields/64 bytes, then converted to hashtable. Perfect for object storage with partial-field access.
List
Doubly-linked list (listpack for small sizes). O(1) push/pop from both ends. LMPOP, BLMOVE added in 7.x for atomic multi-list operations.
Set
Unordered unique collection. Intset encoding for integer-only sets (very space-efficient). SINTERCARD in 7.0 computes intersection count without materializing the set.
Sorted Set (ZSet)
Set + score, ordered by score. skiplist + hashtable dual encoding. O(log N) add/rank queries. Foundation for leaderboards, priority queues, range queries by score.
Stream 5.0+
Append-only log with consumer groups. Kafka-lite for intra-service messaging. Supports XADD, XREAD, XACK with persistent consumer group state.

Invariants That Must Always Hold

  • Single-writer invariant: Only the master/primary accepts writes in a given slot range. No two nodes can accept writes for the same key simultaneously in a correctly configured cluster (preventing split-brain requires sentinel/cluster mechanisms).
  • Key expiry monotonicity: Once a key's TTL expires, it must never return a valid value to a client. Redis enforces this lazily (on access) and actively (via a background cycle), but the invariant must be upheld even when replicas serve reads.
  • Type safety: A key has exactly one type. A LPUSH to a key of type String returns a WRONGTYPE error. Type metadata is stored per-key in the dict entry.
  • Cluster slot ownership: Each hash slot (0–16383) is owned by exactly one shard at any moment. MOVED/ASK redirections guide clients to the correct node.

🔥 Common Interview Mistake
Candidates say "Redis is single-threaded" without qualification. Since Redis 6.0, I/O threads handle network reads/writes in parallel, but command execution remains single-threaded. Since Redis 7.0, some background tasks also multi-thread. Confusing these layers loses credibility instantly.
⚡ L7/L8 Differentiator
Explain the redisObject structure: 4-bit type + 4-bit encoding + 24-bit LRU + refcount + *ptr. The encoding field is how Redis automatically switches from listpack → skiplist or intset → hashtable as data grows. This adaptive encoding is central to Redis's memory efficiency and is rarely discussed at L5/L6 level.
🔑 Key Takeaway

Redis's power comes not from being an in-memory store, but from executing rich data-structure operations server-side atomically — eliminating round-trips and race conditions that plague client-side cache patterns.

PILLAR 02

Architecture Deep Dive

Internal Components

ae Event Loop
The heart of Redis. A single-threaded event multiplexer built on epoll (Linux), kqueue (BSD/macOS), or select (fallback). Processes file events (client I/O) and time events (background tasks). All command execution happens here — no locks needed for data access.
I/O Threads (6.0+)
Configurable thread pool that handles network read/write syscalls in parallel. The main thread still dispatches commands serially. Enabled via io-threads N. Provides ~2–3× throughput improvement on multi-core machines with heavy network I/O.
Command Table
A static hash table mapping command names to handler functions, arity, flags (write/read/admin/pubsub), and first/last/step key arguments. ACL, cluster slot routing, and command statistics all rely on this metadata.
Database (redisDb)
Redis supports 16 logical databases (db 0–15) by default. Each is a struct containing: dict (main keyspace), dict (expires/TTLs), avg_ttl, and slot-to-keys mapping for cluster mode. SELECT N switches database context.
Replication Subsystem
Manages replica handshake (PSYNC2 protocol), backlog buffer (for partial resyncs), replication offset tracking, and async propagation of write commands to replicas. Backlog default: 1MB — undersized for high-write workloads.
Persistence Engine
RDB: fork() + copy-on-write snapshot. AOF: write-ahead log with configurable fsync policy (always/everysec/no). Hybrid mode: AOF file starts with RDB snapshot then appends commands. AOF rewrite compacts the log in background.
Cluster Bus
A gossip protocol over a separate port (data port + 10000) for node discovery, failure detection, and slot migration announcements. Uses binary protocol for efficiency. Every node knows the full cluster topology.
Lua / Functions Engine
Embeds Lua 5.1 (via LuaJIT in some builds) for server-side scripting. Redis 7.0 introduced Redis Functions, which are persistent, named Lua (or future engine) scripts stored in the keyspace — replacing ephemeral EVAL scripts.

On-Disk / In-Memory Data Layout

redisObject: Every value is a robj struct: 4 bits type (string/list/set/zset/hash/stream) + 4 bits encoding (int/embstr/raw/listpack/skiplist/hashtable/intset/quicklist) + 24-bit LRU clock + 10-bit refcount + pointer. Small strings ≤44 chars use embstr (single allocation for robj + SDS), avoiding fragmentation.

SDS (Simple Dynamic String): Redis's custom string type with header containing len, alloc, flags + the actual bytes. O(1) length, binary-safe, avoids buffer overflows. Multiple header sizes (sdshdr8/16/32/64) to minimize overhead for short strings.

Dict (hash table): Two hash tables (ht[0] and ht[1]) for incremental rehashing. When load factor exceeds 1.0 (or 5.0 under BGSAVE), a new ht[1] is allocated 2× size and keys are progressively migrated. During rehashing, both tables are consulted on lookup.

Skiplist (ZSet): A probabilistic data structure with O(log N) ops. Paired with a hashtable for O(1) score lookups by member. Level distribution: 25% probability per level, max 32 levels. More cache-friendly than balanced BSTs in practice due to simpler traversal.

Back-Pressure and Flow Control

Redis applies back-pressure at the client output buffer level. Each client has an output buffer with configurable hard/soft limits (client-output-buffer-limit). If a replica's output buffer exceeds the hard limit (default 256MB for replicas), the connection is dropped and a full resync is required — a major failure mode in high-write environments.

For slow clients, the client-query-buffer-limit (default 1GB) caps input. The hz config (default 10) controls how often the background cron runs key expiry and other tasks — too low means TTL-expired keys pile up.

📐 Back-of-Envelope
Redis can handle ~100K–300K ops/sec on a single core (single-threaded). With I/O threads (6.0+) and pipeline, 1M+ ops/sec is achievable on a 16-core machine. Latency: ~50–200μs for in-datacenter round-trip. Memory: ~70 bytes overhead per key + value size. A 100M key dataset with 10-byte keys and 50-byte values ≈ ~12GB RAM (with overhead).
⚡ L7/L8 Differentiator
Discuss the COW (Copy-On-Write) tax during BGSAVE/BGREWRITEAOF: When Redis forks for RDB, the child gets a read-only copy of memory pages. Every write the parent makes causes the OS to copy the modified page for the child. Under a high-write workload, this can double effective memory usage momentarily. Engineers must size Redis instances to 50–60% max memory utilization to survive fork-induced COW spikes.
🔑 Key Takeaway

Redis's single-threaded command execution is a feature, not a limitation — it eliminates locking overhead and makes all data operations naturally atomic, while I/O threads (6.0+) handle the actual network bottleneck on multi-core hardware.

PILLAR 03

Trade-offs & Design Decisions

CAP Theorem Positioning

Redis is primarily a CP system when run in Cluster mode with cluster-require-full-coverage yes (default). If a shard becomes unavailable, the cluster refuses to serve keys in those slots, prioritizing Consistency over Availability. With cluster-require-full-coverage no, it becomes more AP — serving requests even with missing shards, potentially with stale or incomplete data.

In Sentinel mode (primary-replica), Redis leans AP during partition: replicas can be promoted, but the old primary may still accept writes (split-brain window), sacrificing strict consistency for availability. The min-replicas-to-write config can move it toward CP by refusing writes if fewer than N replicas are reachable.

Consistency Models

Eventual consistency is the default in replicated Redis: replicas receive writes asynchronously and may serve stale data. This is acceptable for cache use cases where slight staleness is tolerable.

Read-your-writes can be achieved by always routing reads to the primary, or by using the WAIT numreplicas timeout command after a write to block until at least N replicas have acknowledged the write.

Strong consistency is not natively achievable across nodes — Redis does not implement Raft or Paxos for data replication (only for leader election in Sentinel). Redis Cluster uses an eventual-consistency replication model where a minority-partition primary can accept writes that will later conflict with a newly-elected primary.

Causal consistency within a single client session is achievable if that client only reads from the primary. Across clients or sessions using replicas, it is not guaranteed.

Latency vs Throughput Knobs

  • Pipelining: Batch multiple commands in a single TCP write, drastically reducing round-trips. Throughput increases from ~50K to 1M+ ops/s but individual command latency increases slightly (waiting for batch to form client-side).
  • MULTI/EXEC transactions: Queue commands and execute atomically. Blocks the event loop for the duration — long transactions hurt latency for concurrent clients. Not a distributed transaction.
  • Pub/Sub vs Streams: Pub/Sub is fire-and-forget (no persistence, no consumer groups). Streams (5.0+) add persistence, consumer groups, and acknowledgement at the cost of slightly higher complexity and memory.
  • maxmemory-policy: Controls eviction when memory is full. allkeys-lru evicts least-recently-used keys globally. volatile-lru only evicts keys with TTLs. Wrong choice = silent data loss or OOM kills.
🔥 Common Interview Mistake
Claiming MULTI/EXEC is a "transaction" in the ACID sense. Redis transactions do NOT provide isolation (other clients' reads see in-progress state in some interpretations), do NOT roll back on command errors (partial execution happens), and are NOT distributed. They only provide atomicity in the sense of not interleaving with other commands.
🔑 Key Takeaway

Redis offers a rich dial from pure performance (no persistence, async replication) to near-durability (AOF always + WAIT) — the correct setting is always dictated by your RPO/RTO requirements, and mixing durability settings across a fleet without understanding the implications is a production disaster waiting to happen.

PILLAR 04

Failure Modes & Resilience

Single Node Crash — Primary High Severity

Sentinel monitors via periodic PING (default every 1s). After down-after-milliseconds (default 30,000ms) with no response, marks as Subjectively Down (SDOWN). When a quorum of sentinels agree, it becomes ODOWN (objectively down).

Sentinel elects a new primary from replicas using a scoring algorithm: replica with highest replication offset (least lag) wins, tiebroken by run-id. Clients are redirected via Sentinel's SUBSCRIBE channel. Failover typically takes 30–60s with default config.

Any writes on the primary after the last successful replication to the elected replica are lost. With async replication, this window can be seconds. min-replicas-to-write 1 min-replicas-max-lag 10 bounds this to 10 seconds.

Investigate root cause (OOM, hardware fault, memory corruption). Repair/replace node and re-add as replica. Verify replication offset after re-join. Update monitoring.

Single Node Crash — Replica Medium Severity

Primary detects replica disconnect via TCP close event. Sentinel marks replica as DOWN but takes no failover action (only primaries trigger failover).

Replica restarts and sends PSYNC2 with its replication offset. If the offset is within the primary's repl_backlog (default 1MB), a partial resync occurs — only missing commands are sent. Otherwise, a full SYNC (RDB transfer) is required.

No data loss — replica is a consumer, not a write authority. Availability risk: if this was the only replica and the primary crashes during repair, no failover target exists.

Ensure repl_backlog is sized appropriately for expected outage duration × write rate. Consider running ≥2 replicas in production at all times.

Network Partition — One Side Has Quorum High Severity

In Redis Cluster, nodes exchange gossip. A node that can't reach peers after cluster-node-timeout (default 15s) marks them PFAIL. Once a majority confirms PFAIL → FAIL. Sentinel quorum works similarly via voting.

The majority partition elects a new primary for affected slots. The minority-side primary, now isolated, should stop accepting writes if it detects it's partitioned (via replica disconnect + min-replicas-to-write). Without that config, it continues accepting writes that will be lost.

Classic split-brain data loss window: writes to the minority-partition primary between partition start and its demotion are lost when the partition heals and the minority node resyncs to the new primary.

After partition heals, verify write counts via INFO replication. Assess extent of divergence. May require application-level reconciliation if write-loss is unacceptable.

Even Split — No Quorum Possible Catastrophic

Redis Cluster and Sentinel both require majority votes to elect a new primary. With an even split (e.g., 3 nodes on each side), neither side can reach quorum. Both partitions either: (a) keep the existing primary if it's on that side, or (b) remain leaderless for affected slots.

Slots without a reachable primary become unavailable if cluster-require-full-coverage yes. With no, partial availability. The system is frozen at split time — no new failovers.

Always deploy an odd number of nodes (3, 5, 7). Use a Sentinel setup with odd sentinel count. Consider a tie-breaker node (a small Sentinel-only instance in a third AZ).

Manual intervention: restore network connectivity, or manually promote a replica using CLUSTER FAILOVER FORCE (accepting potential data loss). Operators must decide which side's data is authoritative.

Slow / Degraded Node (Not Dead) Subtle & Dangerous

Slow nodes are difficult to detect — they respond to PING (preventing SDOWN), but commands take abnormally long. Redis 7.0+ provides LATENCY HISTORY and LATENCY LATEST commands. Client-side timeout metrics are the primary signal.

Slow key (huge list/sorted set), fork COW pressure (BGSAVE on high-write workload), swap (Redis touching swap is catastrophic — adds ms to every operation), CPU throttling in containers, or a large KEYS scan blocking the event loop.

A slow primary can cause replica lag to grow, client-side timeouts to cascade, and eventually trigger a false failover — promoting a replica that may itself be behind.

Never run KEYS in production (use SCAN). Monitor latency_percentiles_usec per command. Use slowlog (commands taking >10ms). Avoid swap via vm.swappiness=0. Ensure maxmemory is <60% physical RAM to avoid COW issues.

Thundering Herd After Recovery Architectural Risk

A Redis cluster restarts (planned maintenance or crash recovery). Every downstream service's in-memory cache is cold. All requests simultaneously miss cache and hit the database, causing a database overload that may cascade into further failures.

Cache warming: Pre-populate Redis before marking it healthy in the load balancer. Probabilistic early expiration (PER): Recompute cache slightly before actual expiry — only one worker does so. Mutex/lock on cache miss: Only one request populates the cache for a cold key; others wait.

Use SETNX (or SET NX EX) as a distributed lock on the "populate" operation. The first requester gets the lock, fetches from DB, sets the cache. Others either wait (block on BLPOP on a result list) or serve stale data if available.

At Google scale, cache warmup is a fully automated pipeline triggered by deployment systems. Traffic is gradually shifted to warmed instances. A cold Redis is never exposed to full production traffic.

Cascading Failures System-Level Risk

A Redis slowdown → client threads pile up waiting → application thread pool exhausted → upstream services time out → retries amplify load → Redis gets more requests, slows further. Prevention: circuit breakers (return stale/fallback if Redis takes >Xms), bulkheads (separate Redis connection pools per service tier), and shed load at the Redis proxy layer (Twemproxy/Envoy → return 503 rather than queuing unbounded).

⚡ L7/L8 Differentiator
Discuss clock skew implications: Redis uses gettimeofday() for TTL expiry. If system clock jumps forward (NTP correction), keys may expire prematurely. If it jumps backward, keys may live longer than expected. Redis 7.x uses CLOCK_REALTIME + monotonic clock cross-checks to mitigate. For distributed locks (Redlock), clock skew is a fundamental correctness concern — Redlock requires that clock drift across N nodes is negligible relative to the lock TTL.
🔑 Key Takeaway

Most Redis failures are not from Redis itself crashing, but from operational gaps: insufficient repl_backlog size, missing min-replicas-to-write guards, swap enabled on Redis hosts, or thundering-herd on recovery — all preventable with thoughtful configuration.

PILLAR 05

Scaling Strategies

Vertical Scaling

Redis can scale vertically to very large instances — 256GB+ RAM is common in production. The bottleneck shifts over time: at moderate sizes, the single-threaded command loop becomes the CPU bottleneck (~300K ops/s on a single core). With I/O threads (6.0+), you can push to 1M+ ops/s on large machines. Beyond ~500GB instances, the fork cost for RDB becomes dangerous: a 500GB fork can cause multi-second latency spikes due to COW pressure and TLB flush costs on large page-table structures.

What breaks first: fork latency (BGSAVE), then single-threaded CPU bottleneck, then OS memory management overhead at extreme sizes.

Redis Cluster — Horizontal Scaling

Hash Slot Key Selection — Avoiding Hotspots

Redis Cluster uses CRC16(key) % 16384 to assign keys to slots. Hash tags ({}) force related keys to the same slot, enabling multi-key operations on those keys. For example, user:{42}:profile and user:{42}:sessions both map to slot CRC16("42") % 16384. Without hash tags, multi-key commands (MGET, SUNION, EVAL) fail across slots.

Hotspot risk: If all your keys use the same hash tag (e.g., {tenant1}:*), all keys land on one slot = one shard. Redis 7.x introduced hash slot sharding hints but the fundamental solution is to use more granular hash tags or avoid them when cross-key operations aren't needed.

Resharding

Redis Cluster supports online resharding via redis-cli --cluster reshard. The process migrates slots from source to target shard by iterating keys in the source slot, sending MIGRATE commands. During migration, keys in the moving slot return ASK redirections (not MOVED) — clients must follow ASK redirections to the target while the slot is in transition. Smart clients (Jedis, StackExchange.Redis, redis-py) handle this transparently.

The challenge: large slots (millions of keys) take minutes to hours to migrate. During migration, latency on those keys increases. Use cluster-migration-barrier to control migration pace. Online resharding is safe but slow — plan for a maintenance window for large migrations.

Cross-Region Replication

Active-Passive: One region is primary; others are read-only replicas. Writes must go to the primary region — adds cross-region latency (~50–150ms). Simple, no conflict resolution needed. Failover to replica region requires promotion (sentinel/manual) and application reconfiguration.

Active-Active (Redis Enterprise): CRDTs (Conflict-free Replicated Data Types) allow writes in multiple regions simultaneously. Conflicts are resolved automatically using last-write-wins (LWW) or type-specific CRDT semantics (e.g., counters use increment merging). This is a Redis Enterprise feature — not available in open-source Redis. Replication lag between regions is expected; eventual consistency is the model.

Read Scaling vs Write Scaling

Read scaling: Add replicas — each replica handles reads, multiplying read throughput. A single primary with 5 replicas can handle 5× read throughput. Route clients to replicas using READONLY command in cluster mode or by configuring client libraries to route reads to replicas.

Write scaling: Requires sharding (Redis Cluster). You cannot scale writes on a single primary. Each shard handles a subset of the keyspace, so write throughput scales linearly with shard count. The practical limit per shard is ~300K simple writes/sec (single-threaded) or 1M+ with I/O threads and pipelining.

📐 Back-of-Envelope: Sizing a Redis Cluster
Target: 10M active sessions, avg 2KB/session, 500K reads/sec, 50K writes/sec.

Memory: 10M × 2KB = 20GB + 50% overhead = 30GB → 3 shards × 16GB or 2 shards × 32GB (with buffer).
Throughput: 50K writes/sec → 1 shard handles it comfortably (<300K/s limit). 500K reads/sec with 2 replicas per shard = 500K / (3 shards × 2 replicas) = ~83K reads/s per replica — well within range.
Recommendation: 3 shards × (1 primary + 2 replicas) = 9 nodes, 32GB RAM each, with AOF everysec + hybrid persistence.
🔑 Key Takeaway

Redis Cluster is the correct answer for write-scale, but introduces slot-affinity constraints that require careful key design from the start — retrofitting hash tags into an existing large application is costly and error-prone.

PILLAR 06

Real-World Engineering Decisions

Where Redis Shines Above All Alternatives

Rate Limiting
Sliding window rate limiting using ZADD (timestamp-scored sets) + ZREMRANGEBYSCORE. Atomic token bucket via Lua script. Redis's atomic operations prevent race conditions that plague DB-based rate limiters. Cloudflare uses Redis for its global rate limiting at billions of requests/day.
Distributed Locking
SET key value NX PX ttl as a distributed lock. Redlock algorithm for multi-node safety. Essential for leader election, job deduplication, and preventing double-booking. Must understand Martin Kleppmann's critique of Redlock for safety-critical applications.
Real-Time Leaderboards
Sorted Sets natively support ZADD (update score), ZRANK (global rank), ZRANGE BYRANK (top-N), and ZRANGEBYSCORE (score range). O(log N) per operation. Impossible to replicate this cleanly with relational DB at high write volume.
Session Store
Hashes store session fields with optional TTL. Horizontal scaling via cluster. Sub-ms reads enable transparent auth in API gateways. Used by almost every major web company as the stateless-session backbone.
Pub/Sub Messaging
Low-latency fan-out to subscribers. No persistence (use Streams for that). Ideal for real-time notifications, chat, and cache invalidation signals. Stack Overflow used Redis Pub/Sub for their entire real-time notification system.
Geospatial Indexing
GEOADD stores coordinates encoded as geohash Sorted Set scores. GEORADIUS/GEOSEARCH finds points within a radius in O(N+log M). Used by Uber, Lyft, and DoorDash for driver/restaurant proximity queries.

Production Examples

Twitter / X — Timeline Caching & Social Graph Production

Twitter uses Redis to cache the home timeline (a Sorted Set of tweet IDs scored by tweet timestamp) for each user. On tweet publish, the tweet ID is fan-out inserted into the timelines of all followers. For celebrity accounts with 50M+ followers, fanout is handled via a separate "big account" tier that computes timelines on-the-fly by merging a small cache with a bulk fetch. Redis Sorted Sets enable O(log N) range queries by time, making infinite scroll efficient.

GitHub — Distributed Job Queue & Locks Production

GitHub uses Redis as the backend for Resque (and later Sidekiq), their distributed job queue systems. Redis Lists (LPUSH/BRPOP) provide the queue primitive. Job state, retry counts, and failure queues are stored as Redis Hashes. Redis's atomic LIST operations ensure that a job is never processed by two workers simultaneously — a guarantee that's difficult to achieve with traditional message queues without transactions.

Uber — Real-Time Geofencing & Driver Location Production

Uber stores driver locations as Redis Sorted Sets (Geo commands). Every few seconds, driver apps push their GPS coordinates. Rider requests query GEOSEARCH to find drivers within N km. Redis's in-memory geospatial index handles ~1M+ location updates/min at peak. The key design challenge: driver location keys rotate (e.g., by city) to avoid hotspots, and TTLs auto-expire offline drivers' locations.

When NOT to Use Redis — Anti-Patterns

  • Primary database for critical data without application-level redundancy: If your application cannot tolerate up to 1 second of data loss (AOF everysec) or minutes of data loss (RDB), Redis should not be your primary store for writes. Use PostgreSQL + Redis as a cache layer.
  • Storing very large values (>10MB per key): Large values cause the event loop to block for milliseconds while serializing/deserializing. Use S3/GCS for large objects; store the reference URL in Redis.
  • Complex relational queries: Redis is not a query engine. Aggregations, joins, and ad-hoc filtering are not its domain. DuckDB, PostgreSQL, or BigQuery are better choices.
  • As a Kafka replacement for high-durability event streaming: Redis Streams is excellent for low-latency intra-service queuing, but lacks Kafka's multi-consumer fan-out, durable partition log, and multi-day retention guarantees at petabyte scale.
  • Unbounded key growth without eviction policy: Redis will fill RAM and either OOM-kill or silently fail without a configured maxmemory + eviction policy. This is a common production incident for teams new to Redis.

Day 1 Monitoring & Alerting

  • Memory: used_memory_rss / maxmemory >80% → alert. RSS growing faster than used_memory signals fragmentation (mem_fragmentation_ratio >1.5 → active-defrag).
  • Hit rate: keyspace_hits / (keyspace_hits + keyspace_misses) <90% → investigate eviction, TTL tuning, or cold cache.
  • Connected clients: Sudden spike → possible connection leak. Default max: 10,000.
  • Replication lag: master_repl_offset - slave_repl_offset growing → replica falling behind, replication buffer overflow risk.
  • Blocked clients: Clients blocked on BLPOP/BRPOP → queue depth growing, workers not consuming fast enough.
  • Latency: LATENCY LATEST + p99 from client libraries. Alert on p99 >5ms for production caches.
  • Evicted keys: evicted_keys counter >0 → maxmemory is tight, data is being lost silently if this is not a cache.
  • Slowlog: Commands taking >10ms indicate problematic patterns (KEYS, large LRANGE, massive HGETALL).
🔑 Key Takeaway

Redis's richness makes it the right tool for an unusually wide range of problems, but each use case demands different configuration — the Redis that works perfectly as a cache (LRU eviction, no persistence) will silently corrupt your rate limiter or message queue if those settings are inherited without review.

Multi-Tenancy and Noisy Neighbors

At Google scale, a single Redis cluster is shared by thousands of services. A noisy neighbor — a service running KEYS * or a massive SCAN, or publishing to a high-fanout Pub/Sub channel — blocks the single-threaded event loop, spiking latency for all other tenants on the same shard.

Isolation strategies: Dedicated clusters per critical service (Gmail vs YouTube vs Ads should not share Redis). Namespace-level rate limiting via a proxy (Envoy + Redis filter). Key prefix conventions enforced by ACLs (Redis 6.0+ ACL rules can restrict key patterns). Quota enforcement at the proxy layer — limit commands/sec and bytes/sec per tenant.

Google's approach: Google's internal cache systems (similar to what Memorystore is built on) use a hierarchical architecture where each service gets an isolated virtual cluster with hard resource quotas. The underlying physical infrastructure is shared but the isolation is enforced at the routing/quota layer, not at the Redis level.

Data Residency and Regulatory Compliance

GDPR, CCPA, and regional data regulations may require that EU user data never leave EU data centers. Redis Cluster, by default, is a single logical entity. For compliance:

  • Per-region clusters: Maintain separate Redis clusters per regulatory boundary. Application logic routes writes to the correct regional cluster based on user location/citizenship. No cross-region replication of regulated data.
  • Key-level encryption: Encrypt PII values client-side before storing in Redis. Redis stores ciphertext; the encryption keys are managed separately (KMS). Even a Redis breach doesn't expose plaintext PII.
  • Audit logging: Redis 6.0+ ACL logs + AOF provide command-level audit trails. Ship to centralized security logging (Chronicle/Splunk) for compliance reporting.

What a "Google Spanner for Redis" Would Look Like

Vanilla Redis lacks: globally consistent snapshots, multi-region strong consistency, and scale beyond a single cluster's management capacity. A theoretical "Google-grade Redis" would require:

  • Paxos/Raft-based replication replacing async replication, providing linearizable writes across nodes (like Spanner's TrueTime + Paxos groups).
  • Globally distributed slot map managed by a consistent metadata service (like Spanner's directory server), enabling seamless cross-region key routing.
  • MVCC (Multi-Version Concurrency Control) for snapshot isolation across data types — enabling consistent multi-key reads without locking.
  • Tiered storage: hot data in DRAM, warm data in NVMe SSD (Optane/PM), cold data in object storage. Transparent access with predictable latency SLOs per tier.
  • CRDT-native data types across all primitive types, not just counters — enabling true active-active multi-region without Enterprise licensing.

Cost Efficiency at Scale

At Google scale, the cost optimizations that matter for Redis:

  • Memory tiering (Redis 7.2+ on Redis Enterprise): Automatically spill cold keys to flash storage. Useful only when working set < total dataset. For pure caches with LRU eviction, tiering adds complexity without benefit.
  • Instance size sweet spot: Redis performance-per-dollar peaks at 32–64GB instances for typical workloads. Below 8GB, overhead is disproportionate. Above 128GB, COW fork costs require very conservative memory headroom.
  • Read replica right-sizing: Beyond 3 replicas per shard, each additional replica provides diminishing read throughput gains while adding replication bandwidth cost. Profile actual read QPS before adding replicas.
  • Compression: Redis doesn't compress values. For large string values, gzip client-side before storing can save 50–80% memory. Only cost-effective for values >1KB that are read infrequently.
⚡ L7/L8 Differentiator
Discuss Redis at the L7 proxy layer: At Google/Meta scale, Redis is fronted by an L7-aware proxy (Envoy with Redis filter, Twemproxy, or custom). The proxy handles: connection multiplexing (10K upstream connections → 10 proxy connections to Redis), slot map caching (proxy knows the cluster topology, clients don't), read-replica routing, circuit breaking, and quota enforcement. This proxy layer is more complex than Redis itself at hyperscaler deployments and is rarely discussed at L5/L6.
🔑 Key Takeaway

At Google scale, Redis stops being a database and becomes an infrastructure primitive — wrapped in layers of proxies, quota systems, and multi-tenancy controls that are more complex than Redis itself; knowing where Redis ends and the surrounding system begins is the L8 differentiator.

You are the Staff Engineer for a new mobile game launching globally. The game has real-time leaderboards at three levels: (1) global all-time top 1,000 players, (2) weekly leaderboards per geographic region (e.g., North America, Europe, Asia), and (3) a "friends leaderboard" showing a user's rank among their top-500 friends. Global Leaderboard Design

Use a single Redis Sorted Set per leaderboard type: ZADD global:alltime player_id score. ZADD with the GT flag (Redis 6.2+) atomically updates the score only if the new score is greater — perfect for monotonic scores. ZREVRANK player_id gives O(log N) rank. For top-1000, ZREVRANGE 0 999 WITHSCORES.

Scale concern: 50M players × ~100 bytes/entry in the sorted set = 5GB. Fits in a single Redis instance, but a single Sorted Set is single-shard — all writes go to one node. At 2,000 updates/sec sustained this is fine (well under 300K/s limit). At 50,000/sec during tournaments, we need to rate-limit ingest (debounce: only propagate score updates every 1 second per player) or shard the leaderboard.

Friends Leaderboard — The Hard Part

The friends leaderboard is the hardest problem. Naïvely: store each user's friend list, and on query, fetch all 500 friends' scores from the global ZSet via ZMSCORE global:alltime friend_id1 friend_id2 ... (O(N) for 500 friends), sort client-side, and return the user's rank. At 50M DAU, if 10% of users view their friends leaderboard per hour, that's 5M queries/hour = ~1,400 QPS of 500-key batch lookups.

Optimization 1: Cache the computed friends leaderboard per user with a 30s TTL: SET friends_lb:{user_id} {serialized_list} EX 30. Cache hit rate will be high for inactive users.

Optimization 2: Pre-materialize friends' leaderboards on score change using a fan-out write pattern (like Twitter's timeline fan-out). When a player scores, push score updates to a Kafka topic that a friends-leaderboard service consumes, updating cached friend leaderboards. This avoids the N+1 read pattern on query but requires write fan-out infrastructure.

Trade-off: Materialization is a write amplification of up to 500× (a player with 500 friends triggers 500 cache invalidations per score update). Rate-limit fan-out: only propagate score changes to friend caches if the delta exceeds a threshold (e.g., >100 points).

Tournament Spike Handling

At 50,000 score updates/sec during a tournament, the global leaderboard ZSet would receive too many writes if all are processed in real-time. Solution: batch-and-debounce in the Kafka consumer. Aggregate all score updates for a player within a 1-second window, taking the max. This reduces effective write rate by up to 100× (players can update 100 times/minute → 1–2 effective ZSet writes/player/minute). The consumer lag during a spike is acceptable given the 5-second staleness SLO.

Step 4: Rapid-Fire Staff-Level Q&A

Q01
Explain the difference between EXPIRE and PEXPIRE, and why TTL precision matters in distributed systems.
EXPIRE sets TTL in seconds; PEXPIRE in milliseconds. For distributed locks and rate limiters, second-level precision creates race conditions — a lock meant to hold for 500ms must use PEXPIRE to avoid premature expiry. Redis stores TTLs internally as absolute Unix timestamps in milliseconds (since Redis 2.6), so PTTL can always return sub-second remaining time. The critical edge: if a replica promotes with clock skew, TTLs may differ between old primary and new primary by up to the skew amount — short TTLs (<5s) are most vulnerable to this.
Q02
What is the Redlock algorithm and what are its safety guarantees? When is it unsafe to use?
Redlock acquires a distributed lock by attempting to SET NX PX on N/2+1 independent Redis instances within a validity window (TTL minus acquisition time). The lock is only held if a majority granted it before the TTL elapsed. It is unsafe when: (1) TTL is short and clock drift between nodes approaches TTL magnitude; (2) a GC pause or OS scheduling delay in the lock holder causes it to act after its TTL has expired while another client has already acquired the lock — Martin Kleppmann's "HLC fence token" critique. For safety-critical use cases, use fencing tokens (monotonically increasing IDs from Zookeeper/etcd) in addition to, or instead of, Redlock.
Q03
How does Redis handle the "large key" problem, and what are the performance implications?
A "large key" in Redis typically means a key whose value is very large (>1MB string) or a collection with millions of members (list/set/zset/hash with 1M+ elements). Large key operations block the event loop: a DEL on a 10M-member ZSet can take hundreds of milliseconds, halting all other command processing. Redis 4.0+ added UNLINK (async lazy deletion) and FLUSHDB ASYNC / FLUSHALL ASYNC. For large collections, use SCAN-based iteration (HSCAN, SSCAN, ZSCAN) with a COUNT hint to iterate in O(COUNT) batches rather than HGETALL/SMEMBERS. Monitor key sizes with redis-cli --hotkeys and --bigkeys in production.
Q04
Explain PSYNC2 and why it matters for production Redis deployments.
PSYNC2 (introduced in Redis 4.0) allows replicas to perform partial resynchronization after restart, even if they were previously a primary. Before PSYNC2, a restarted replica that had been a primary required a full RDB resync because its replication ID changed on restart. PSYNC2 maintains two replication IDs: the current one and the previous one (before the last promotion). This means a demoted primary can rejoin as a replica and pick up from where it left off, drastically reducing full-resync frequency in Sentinel failover scenarios. In practice, this prevents the "resync storm" that occurred in pre-4.0 clusters when multiple replicas needed full RDB transfers simultaneously after a failover.
Q05
You're seeing mem_fragmentation_ratio of 3.5 on a Redis instance. What does this mean and what do you do?
mem_fragmentation_ratio = used_memory_rss (OS-reported physical memory) / used_memory (Redis's allocator-tracked usage). A ratio of 3.5 means Redis is using 3.5× more physical memory than the logical data size — primarily due to allocator fragmentation from many small allocations and deallocations (common in workloads with frequent key deletion/creation). Mitigation: (1) Enable active defragmentation: activedefrag yes, active-defrag-ignore-bytes 100mb, active-defrag-threshold-lower 10. Redis 4.0+ uses jemalloc's background defrag. (2) If using a different allocator, switch to jemalloc. (3) For extreme cases, a full RDB save + restart resets fragmentation. Never ignore fragmentation_ratio >2.0 — it means an OOM kill can happen at 2× less memory pressure than maxmemory suggests.
Q06
How would you implement a sliding window rate limiter in Redis? What are the trade-offs vs. a fixed window approach?
Sliding window: ZADD rate:{user_id} {timestamp_ms} {request_uuid}, then ZREMRANGEBYSCORE rate:{user_id} 0 {now_ms - window_ms}, then ZCARD rate:{user_id} to get count — all in a Lua script for atomicity. TTL the key to window_size seconds for auto-cleanup. Fixed window: simpler — INCR rate:{user_id}:{window_bucket} with EXPIRE. Trade-offs: sliding window is exact but uses O(requests_per_window) memory per user and is slower (O(log N) per request). Fixed window is O(1) memory and time but has a "boundary burst" vulnerability — a user can send 2× the limit in 2× window_size seconds by timing requests around window boundaries. For most production rate limiters, a hybrid "sliding window counter" (two fixed windows with interpolation) provides 90% of sliding window accuracy at O(1) cost.
Q07
What is the OBJECT FREQ command and when would you use it? How does LFU differ from LRU for cache eviction?
OBJECT FREQ key returns the logarithmically-scaled access frequency counter for a key under LFU eviction policies. LRU (Least Recently Used) evicts the key that was accessed longest ago — it has no memory of how often a key is accessed, so a popular key accessed yesterday may be evicted over an unpopular key accessed a minute ago. LFU (Least Frequently Used, Redis 4.0+) evicts the key with the lowest access frequency, making it far more accurate for workloads with a power-law access distribution (Zipf). Redis's LFU uses a probabilistic Morris counter (8 bits per key) with configurable decay rate (lfu-decay-time). Use LFU when your access pattern has a clear "hot set" of frequently accessed keys that you want to protect from eviction — typical for content caches and session stores. LRU is simpler and works well for recency-dominant access patterns.
Q08
Explain how Redis Streams differ from Kafka. In what production scenario would you choose Streams over Kafka?
Redis Streams (5.0+) are append-only logs with consumer groups, message acknowledgement (XACK), and persistent storage. Compared to Kafka: Streams are single-node per shard (no distributed log across brokers), have no built-in retention policy beyond memory-based trimming (MAXLEN), don't support replayed consumption from time T for arbitrary consumers, and lack multi-region replication natively. Choose Streams over Kafka when: (1) latency SLO is <1ms (Kafka adds 5–50ms for replication); (2) message volume is low-to-moderate (<100K/sec per stream); (3) you want to avoid Kafka's operational complexity (ZooKeeper/KRaft, topic partition management); (4) you already have Redis in the stack and need lightweight intra-service queuing. Choose Kafka for: compliance-grade audit logs, multi-consumer replay, >1TB/day message volumes, or when consumer groups span multiple services with independent replay positions.
Q09
How does Redis Cluster handle a MOVED vs an ASK redirection, and why does this distinction matter?
MOVED indicates a permanent slot reassignment — the key definitively lives on the target node. Smart clients cache the new slot-to-node mapping and route future requests directly. ASK is a transient redirection during slot migration — the key is in the process of being migrated from source to target. The client must send an ASKING command to the target before the actual command (telling the target to serve this one request for a key it doesn't yet officially own). The client must NOT update its slot map for an ASK redirection; only MOVED updates the map. This distinction matters for correctness during resharding: if ASK were treated like MOVED, clients would permanently route to the target before migration completes, causing requests to be rejected by the source for keys not yet migrated.
Q10
What are Redis Functions (7.0+) and how do they improve on EVAL scripts?
Redis Functions (7.0+) are named, persistent server-side scripts loaded into Redis's keyspace as a library. Unlike EVAL, which requires clients to resend the script on every invocation (or cache by SHA via EVALSHA), Functions are loaded once via FUNCTION LOAD and persist across restarts (stored in RDB/AOF). They support multiple engines (Lua by default, future support for others). Key improvements: Functions are versioned and named (e.g., FCALL mylib.rate_limit 1 user:42), enabling library-style code organization. They participate in replication — FUNCTION LOAD is replicated to replicas, so the function is available cluster-wide. They support read-only functions (FCALL_RO) that can run on replicas without routing to primary. The sandboxing model is improved (restricted global variable access), reducing subtle bugs where EVAL scripts accidentally shared state between invocations.
