2.5Detailed Example Walkthrough

One Crash, Three Paradigms

🛒
Scenario: E-commerce app — "Get Order Details" fanning out across 5 services. Then, at t=10s, one service crashes.
5 services 1000 req/sec crash @ t=10s
REST gRPC GraphQL

REST Walkthrough

REST
t=0s
Normal operation
Client       → API Gateway:   GET /orders/789
Gateway      → Order Service:  GET /internal/orders/789
Order Svc    → User Service:   GET /internal/users/123
Order Svc    → Product Svc:    GET /internal/products/456

All responses JSON, ~5ms each hop.

~20ms total 4 HTTP requests
t=5s
Pattern continues

Cache hit rate: 40% (User/Product data often cached).

Effective Product Svc load: 600 req/sec
t=10s
⚠️ Product Service CRASHES
t=10.5s
Detection
  • Order Service's request to Product Service: connection refused
  • Returns 503 to Gateway
  • Gateway returns 503 to client

Impact: Order details completely unavailable. Client sees: "Error fetching order"

Complete failure
t=11s
Recovery attempt (if retry configured)
  • Retry 1: Product Service still down
  • Client timeout: Order failed
t=15s
Circuit breaker kicks in

Order Service: "Product Service is down, skip enrichment."

Returns partial response: order data without product details. Client sees: order with product ID only, no product name/image.

Graceful degradation — but requires explicit coding
t=30s
Product Service back up

Circuit breaker half-open: allow some requests.

t=35s
Product Service healthy

Circuit breaker closed: normal operation.

Recovered

gRPC Walkthrough

gRPC
t=0s
Normal operation

All services connected via persistent HTTP/2. No connection setup per request.

Client call:    orderService.GetOrderDetails(orderId: 789)
Internal calls:
  userService.GetUser(userId: 123)        — multiplexed
  productService.GetProduct(productId: 456) — multiplexed

All binary, all parallel, deadline: 100ms.

~10ms total faster serialization, no connection overhead
t=5s
Pattern continues

Single HTTP/2 connection to each service, thousands of calls multiplexed.

t=10s
⚠️ Product Service CRASHES
t=10.1s
Detection — faster than REST
  • HTTP/2 connection dies → immediate notification
  • Outstanding requests get UNAVAILABLE status
  • No timeout wait — we know it's down
~100ms detection (vs ~500ms REST)
t=10.2s
Automatic retry (gRPC retry policy)

gRPC config: retry on UNAVAILABLE, max 3 attempts.

  • Attempt 2: Still down
  • Attempt 3: Still down
  • Return UNAVAILABLE to client
t=10.5s
Client has error — but with rich context

gRPC deadline propagation ensured we didn't wait. Client knows: Product Service specifically failed.

t=11s
Load balancer removes dead instance

If multiple Product Service instances, traffic routes to healthy ones.

t=12s
If only one instance: all requests failing

Circuit breaker in service mesh (Istio / Envoy): OPEN.

t=15s
Graceful degradation

gRPC can return partial with error:

response: { order: {...}, errors: ["product unavailable"] }
t=30s
Product Service back

Health check passes → LB includes instance. gRPC connections re-established.

t=31s
Normal operation resumed
Recovered — 4s faster than REST

GraphQL Walkthrough

GraphQL
t=0s
Normal operation

Client query:

query {
  order(id: "789") {
    id
    user { name email }
    items {
      product { name price image }
      quantity
    }
  }
}

GraphQL server resolves:

  1. order → Order Service
  2. order.user → User Service (DataLoader batches)
  3. items[*].product → Product Service (DataLoader batches)

Single response: all data, exact fields requested.

~15ms total 1 round trip
t=5s
Pattern continues

DataLoader: if 10 orders share the same user, only 1 user fetch.

t=10s
⚠️ Product Service CRASHES
t=10.5s
Resolver fails — handled gracefully

Product resolver throws error. GraphQL handles it gracefully:

{
  "data": {
    "order": {
      "id": "789",
      "user": { "name": "Alice", "email": "a@b.com" },
      "items": [
        {
          "product": null,           // Failed
          "quantity": 2
        }
      ]
    }
  },
  "errors": [
    {
      "message": "Product service unavailable",
      "path": ["order", "items", 0, "product"],
      "extensions": { "code": "SERVICE_UNAVAILABLE" }
    }
  ]
}

Key: Client gets partial data with error details. This is GraphQL's superpower in failure scenarios.

t=11s
Client handles gracefully

Shows order, shows user, shows "Product temporarily unavailable".

Much better UX than complete failure
t=12s
Continued partial operation

GraphQL naturally returns what it can. No explicit circuit breaker coding needed — resolvers just fail.

t=30s
Product Service back

Resolvers start succeeding. No reconnection logic — just an HTTP request that succeeds.

t=31s
Normal operation

Full data returned.

Recovered

▣Key Differences Highlighted

Aspect REST gRPC GraphQL
Detection time ~500ms (connection timeout) ~100ms (HTTP/2 knows immediately) ~500ms (HTTP request fails)
Recovery info 503 status, no details UNAVAILABLE with metadata Partial data + error array with paths
Client UX during failure Complete failure or requires explicit partial support Complete failure or requires explicit partial Natural partial response built-in
Retry mechanism Manual (Retry-After header) Built-in (configurable) Manual at resolver level
Graceful degradation Requires explicit coding Requires explicit coding Inherent in execution model
Debug info HTTP status + body gRPC status code + details Error array with path

▣Failure Modes Comparison

Normal Operation

═══════════════════════════════════════════════════════════════
REST:          Client ──GET──→ Service A ──GET──→ Service B
                      ←─JSON──          ←─JSON──

gRPC:          Client ══RPC══> Service A ══RPC══> Service B
                      <══════          <══════
               (persistent connections, binary)

GraphQL:       Client ──query──→ GraphQL ──resolve──→ Service A
                                          ──resolve──→ Service B
                      ←─{data}────────────←─────────
               (single request, multiple resolvers)

During Failure (Service B Down)

═══════════════════════════════════════════════════════════════
REST:          Client ──GET──→ Service A ──GET──✖ Service B
                      ←─503──           (timeout)
               COMPLETE FAILURE unless explicit handling

gRPC:          Client ══RPC══> Service A ══RPC══✖ Service B
                      <═UNAVAIL═        (instant detection)
               Status code tells you EXACTLY what failed

GraphQL:       Client ──query──→ GraphQL ──resolve──→ Service A ✓
                                          ──resolve──✖ Service B
                      ←─{data: {...}, errors: [{path: "..B.."}]}──
               PARTIAL DATA RETURNED with error path
2.7Decision Framework

The Decision Tree & The Sound Bite

No paradigm is universally "better." This is the framework that turns the question "which API style?" into a small set of mechanical branches you can walk in under a minute.

▣Decision Tree for API Style Selection

═══════════════════════════════════════
Q1: Is this for external third-party developers?
├── YES → Strongly consider REST
│   └── Q2: Do they need complex nested queries?
│       ├── YES → Offer GraphQL as secondary option
│       │         (GitHub model: REST v3 + GraphQL v4)
│       └── NO → REST only (Stripe, Twilio model)
│
└── NO (internal or first-party) →
    │
    Q3: Is this service-to-service (no humans in the loop)?
    ├── YES → Strongly consider gRPC
    │   └── Q4: Do you need streaming?
    │       ├── YES → gRPC (native streaming)
    │       └── NO → gRPC still wins (performance + contracts)
    │
    └── NO (has user-facing client) →
        │
        Q5: Multiple client platforms with different needs?
        ├── YES → Strongly consider GraphQL
        │   └── Q6: Do you control all clients?
        │       ├── YES → GraphQL (full flexibility)
        │       └── NO → GraphQL + persisted queries (security)
        │
        └── NO (single client type) →
            │
            Q7: Is HTTP caching critical (CDN, edge)?
            ├── YES → REST (best HTTP caching)
            │
            └── NO → Either REST or GraphQL based on:
                    - Team familiarity
                    - Complexity of data relationships
                    - Frontend team velocity needs

▣Questions to Ask Yourself (or the Interviewer)

"Who are the clients?"
Determines accessibility requirements
  • External developers → REST
  • Internal services → gRPC
  • Multiple app platforms → GraphQL
"What's the read/write ratio?"
Determines caching strategy
  • Read-heavy → REST (HTTP caching) or gRPC (response caching)
  • Write-heavy → Matters less, all roughly equal
"How coupled can we be?"
Determines contract flexibility
  • Loosely coupled (can't coordinate deploys) → REST, GraphQL
  • Tightly coupled (shared deployment) → gRPC fine
"What's the latency budget?"
Determines protocol efficiency needs
  • <50ms strict → gRPC (binary, HTTP/2)
  • 100ms acceptable → Any
"Is the data shape fixed or variable?"
Determines query flexibility
  • Fixed → REST, gRPC (simpler)
  • Variable per client → GraphQL

The Hybrid Pattern

External: REST (public developers) + GraphQL (apps).
Internal: gRPC everywhere.

▣Migration Path Examples

REST → GraphQL

  1. Add GraphQL layer that wraps REST endpoints
  2. Migrate high-value queries first (home screen, feed)
  3. Add DataLoader to batch REST calls
  4. Gradually add direct database resolvers
  5. Deprecate redundant REST endpoints
  6. Keep REST for public API, GraphQL for apps

REST → gRPC

  1. Add gRPC endpoints alongside REST
  2. Generate REST from gRPC using transcoding
  3. Migrate internal callers to gRPC
  4. Keep REST facade for external via transcoding
  5. Eventually: gRPC is source of truth, REST is auto-generated

▣The Sound Bite

If you only memorise one thing from the entire series, memorise this. It's the answer to "compare REST, gRPC, and GraphQL" compressed into a single paragraph that signals staff-level thinking.

Sound bite · for the interview
REST is HTTP-native with great caching but chatty; gRPC is blazing fast with strong contracts but limited to internal services; GraphQL is maximally flexible but requires careful cost management. The sophisticated answer is often "all three" — gRPC for the service mesh, GraphQL for client aggregation, REST at the edge for external developers. Choose based on who's consuming and what guarantees you need.
