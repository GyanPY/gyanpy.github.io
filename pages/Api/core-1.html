Part 1: Core + REST + gRPC Part 2: GraphQL + Comparison + System Design

REST, gRPC, and GraphQL are not competing technologies — they're different answers to fundamentally different questions. This guide builds the mental model from scratch: the restaurant analogy that makes each paradigm click, the five concepts you must internalize to reason about them correctly, and complete A–F deep dives on REST and gRPC covering mechanics, failure modes, operational metrics, and the L7 nuances that separate candidates.

Elevator Pitches

REST

HTTP semantics + resource-oriented URLs + standard verbs. Optimizes for universal accessibility and cacheability. Any browser, curl, or HTTP client works natively.

gRPC

HTTP/2 + Protocol Buffers + generated stubs. Optimizes for performance and strong contracts in service-to-service communication. Binary, typed, streaming-native.

GraphQL

Query language over any transport. Optimizes for client flexibility and efficient data fetching. Clients specify exact data shapes; server returns exactly that.

The "Aha!" Moment

These aren't competing technologies — they're different answers to fundamentally different questions. REST asks: "What resources exist and what can I do to them?" gRPC asks: "What functions can I call and how fast can I call them?" GraphQL asks: "What data do I need and can I get exactly that?" Your choice depends on whether you're building for the web, for microservices, or for diverse clients with varying data needs.

The Restaurant Analogy

REST = The Classic Diner

There's a menu with numbered items (URLs) and standard actions: ORDER (POST), CHECK (GET), MODIFY (PUT), CANCEL (DELETE). Every table follows the same rules. Item #42 is always the burger. If you want the burger without pickles, you might need two trips: one to order, one to customize. Simple, everyone understands it, but sometimes inefficient.

gRPC = The High-End Chef's Kitchen

You call the chef directly on a dedicated phone line and speak in a specialized language only the kitchen understands. You can order multiple things simultaneously (streaming), the chef confirms in real-time, and communication is incredibly fast. But a random person off the street can't just walk in — they don't speak the language. Perfect for regular customers (internal services) who need speed above all.

GraphQL = The Personal Concierge

You hand the concierge a shopping list: "I want the appetizer from Table 1 but only the calories. The main course from Table 2 but only the ingredients. Reviews but only from this week." The concierge assembles exactly what you asked for — nothing more, nothing less — in one trip. Incredibly flexible, but the concierge needs sophisticated training to prevent abuse.

5 Core Concepts You MUST Internalize

Concept 1 Protocol vs Query Language vs RPC ▾

REST is an architectural style built on HTTP — not a protocol, not a specification. It's a set of constraints (stateless, cacheable, uniform interface, layered system) that, when followed, give you properties like scalability and evolvability.

gRPC is an RPC framework with its own binary protocol running over HTTP/2. It's a concrete implementation with code generation, not an architectural style you follow loosely.

GraphQL is a query language that can run over any transport (usually HTTP). It defines how clients request data — not the wire protocol for getting it there.

Concept 2 Contract Location — Where Is the Agreement? ▾

REST: Contract is implicit in URL structure and HTTP semantics. Typically documented in OpenAPI/Swagger — but documentation is optional. No compiler checks if your client matches your server.

gRPC: Contract is explicit in .proto files — code-generated stubs for both client and server from the same source. Compiler enforces compatibility at build time.

GraphQL: Contract is explicit in the schema and introspectable at runtime. Any client can query the schema itself to discover what's available. Type system enforced at query validation.

Concept 3 Coupling Spectrum ▾

REST: Loosely coupled in theory (hypermedia ideal), practically medium coupling — clients depend on URL structure.

gRPC: Tightly coupled — client and server share generated code from the same .proto file. A schema change requires both sides to update.

GraphQL: Flexibly coupled — schema provides the boundary, clients choose what they need. Clients can evolve independently as long as their fields aren't deprecated.

The interview signal: Choose coupling level based on ownership. Teams owning both client and server → tighter coupling is fine (gRPC). External developers → loose coupling required (REST or GraphQL).

Concept 4 Efficiency vs Accessibility Trade-off ▾

REST is most accessible — any browser, curl, or HTTP client works with zero setup. This universality is the reason it dominates public APIs.

gRPC is most efficient — binary encoding (Protocol Buffers) is 3–10x smaller than JSON, HTTP/2 multiplexing eliminates connection overhead, and streaming is native.

GraphQL is most flexible — clients get exactly the data they need (no over-fetching, no under-fetching), reducing both bandwidth and round trips for complex data shapes.

Concept 5 Who's In Control? ▾

REST: Server defines resources and what you can do to them. Client follows the server's prescribed URL structure and method semantics.

gRPC: Server defines functions and their exact signatures. Client calls exactly those functions — no more, no less.

GraphQL: Server defines what's available. Client decides what to fetch. Power shifts to the client — which is both the strength (flexibility) and the risk (clients can write expensive queries).

REST API — Complete Deep Dive

REST Resource-Oriented Architecture over HTTP Best for: Public APIs, CRUD, read-heavy workloads
AWhat It Is — The Mental Model ▾
One-Sentence Definition

REST is an architectural style where you model your domain as resources (nouns) and use standard HTTP methods (verbs) to perform operations on them.

The Core Insight — REST Isn't a Protocol

REST is a set of constraints (stateless, cacheable, uniform interface, layered system) that, when followed, give you scalability and evolvability. Most "REST APIs" are actually HTTP APIs that follow some REST principles. True REST includes HATEOAS — almost no one implements it.

The Inventor's Problem

Roy Fielding, while defining HTTP 1.1, asked: "What architectural style makes the web scale to billions of users without central coordination?" REST was his answer — a decentralized, stateless, cacheable system where anyone can participate without prior coordination.

BHow It Works — Mechanics ▾
Step-by-Step REST Interaction
1. CLIENT: "I need user 123's profile"
   → Constructs: GET /users/123
   → Headers: Accept: application/json, Authorization: Bearer <token>

2. NETWORK: Request travels over HTTP/1.1 or HTTP/2
   → May hit CDN cache (GET with cache headers → cache hit!)
   → Hits reverse proxy → load balancer

3. SERVER: Routes by URL path → UsersController
   → Routes by method → GET → getUser(id=123)
   → Fetches from DB/cache, serializes to JSON

4. RESPONSE:
   → Status: 200 OK
   → Headers: Content-Type: application/json, Cache-Control: max-age=300
   → Body: {"id": 123, "name": "Alice", "email": "alice@example.com"}

5. CLIENT → CDN:
┌────────┐ ──GET──→ ┌───────┐ ──GET──→ ┌──────────┐ ──GET──→ ┌────────┐
│ Client │          │  CDN  │          │ Load Bal │          │ Server │
│        │ ←─200── │(cache)│ ←─200── │          │ ←─200── │        │
└────────┘          └───────┘          └──────────┘          └────────┘
                        ↓
               Cache HIT? Return directly.
               Cache MISS? Forward to origin.
Cache Math — The Power of REST's Cacheability

E-commerce API, 5,000 req/sec: 70% GET (cacheable), 30% writes. With 80% cache hit rate on GETs: 5,000 × 0.70 × 0.80 = 2,800 req/sec never hit origin. Origin handles only 2,200 req/sec instead of 5,000. REST's cacheability is not a nice-to-have — it's a 55% reduction in origin load for free.

CStrengths & Ideal Use Cases ▾
Where REST Shines
  • Public APIs: Any developer with curl can test it immediately — zero setup
  • CRUD-heavy apps: Resources map naturally to database entities and HTTP verbs
  • Read-heavy workloads: HTTP caching at CDN/application layer dramatically reduces load
  • Existing infrastructure: Proxies, firewalls, monitoring, and load balancers all understand HTTP natively
  • Real-world examples: Stripe (developers integrate quickly), GitHub API, Twitter external API
DLimitations & Failure Modes ▾
Over-Fetching and Under-Fetching
Client needs: User name + last 3 orders

REST approach:
  GET /users/123         → Returns 50 fields, client needs 1
  GET /users/123/orders  → Returns 100 orders, client needs 3

Result: 2 round trips, 10x excess data transfer

N+1 Pattern (Client needs 10 users with departments):
  Option A (N+1):
    GET /users           → 10 users
    GET /departments/1   → for user 1
    GET /departments/2   → for user 2
    ... (11 requests total)

  Option B (Embedding):
    GET /users?include=department  → 1 request but complex API design
    Slippery slope toward rebuilding GraphQL inside REST
The Cache Consistency Failure
t=0s:  Client A GETs /products/42, price=$100, cached 5 minutes
t=1s:  Admin updates price to $150 via PUT /products/42
t=2s:  Client B GETs /products/42, cache miss → sees $150
t=3s:  Client A's app shows $100 (still cached!)
t=4s:  Client A adds to cart at $100
t=5s:  Server processes order — price mismatch!

REST's cacheability is a feature, but cache invalidation is
"one of the two hard problems in computer science."

Solution: Short TTLs, Cache-Control: no-cache for sensitive prices,
         ETags for conditional requests, purge-on-update API
The Four Gotchas
  • PUT vs PATCH confusion: PUT = full replace (send all fields), PATCH = partial update (send changed fields only)
  • POST idempotency: POST is NOT idempotent — retries can create duplicates. Use idempotency keys.
  • HATEOAS rarely implemented: True REST includes links to related actions in responses. Without it, clients are tightly coupled to URL structure.
  • Status code misuse: Returning 200 OK with an error in the body breaks all tooling that relies on HTTP semantics.

  • High latency, low errors: Slow database or missing index — check DB query logs
  • Intermittent 502s: Backend timeout or health check failing — check LB and backend logs

FL6/L7 Nuances ▾
The Deep Question

"How do you handle partial failures in a REST API when a client requests data that requires aggregating from multiple backend services?"

Strong answer: Return partial responses with a degraded indicator — include what succeeded and flag what failed. Use circuit breakers to avoid cascade failures when a dependency is down. Implement async refresh patterns for non-critical data. Distinguish user-facing consistency requirements (show something) from analytics (exact or nothing).

L6/L7 Insight — Most REST APIs Aren't Actually REST

"Most 'REST APIs' are HTTP APIs that use JSON. True REST includes HATEOAS — responses contain links to related actions, letting clients discover capabilities dynamically. Almost no one implements this, which is why REST APIs require explicit versioning and documentation. If we followed REST properly, clients would discover capabilities dynamically and breaking changes would be rarer. This is why Fielding himself has called out most APIs as not being truly RESTful."

Startup vs Google Scale

Startup: REST is fine — over-fetching isn't your problem yet. Ship fast, document well.

Google scale: REST's chattiness becomes expensive. A feed might require 50+ backend calls to assemble. Solution: BFF (Backend for Frontend) — a service that batches internal gRPC calls and exposes a coarser-grained REST (or GraphQL) API to clients. The BFF pattern is how Google/Facebook avoid building REST chattiness into their client stack.

gRPC — Complete Deep Dive

gRPC High-Performance RPC over HTTP/2 with Protocol Buffers Best for: Service-to-service, streaming, strong contracts
AWhat It Is — The Mental Model ▾
One-Sentence Definition

gRPC is a high-performance RPC framework where you define services and messages in Protocol Buffers, generate client/server code in any language, and communicate over HTTP/2 with built-in streaming and load balancing.

The Core Insight — Network as Function Call

gRPC treats the network like a function call between modules in the same program. You call userService.getUser(userId) and forget there's a network involved. The .proto file is a contract that generates matching client/server code — no manual parsing, no request building.

The Inventor's Problem

Google needed to connect millions of services in 12+ languages, handling 10^10+ RPCs per second, with microsecond-sensitive latencies. Stubby (internal predecessor) proved the model; gRPC open-sourced it in 2016.

BHow It Works — Proto → Generated Code → Transport ▾
Step-by-Step gRPC Interaction
1. DEFINE (build time) — user.proto:
   service UserService {
     rpc GetUser(UserRequest) returns (UserResponse);
   }
   message UserRequest { int64 user_id = 1; }
   message UserResponse { string name = 1; string email = 2; }

2. CODE GENERATION:
   $ protoc --go_out=. user.proto
   → Generates: UserServiceClient, UserServiceServer interfaces,
                UserRequest, UserResponse structs

3. CLIENT CALL (runtime):
   client := NewUserServiceClient(conn)
   response, err := client.GetUser(ctx, &UserRequest{UserId: 123})
   // Looks like a local function call!

   Under the hood:
   - Serialize UserRequest → binary (Protocol Buffers)
   - Send over HTTP/2 with path: /UserService/GetUser
   - Include metadata (auth, tracing headers)
   - HTTP/2 multiplexes this alongside 1000 other requests

4. SERVER HANDLER:
   func (s *server) GetUser(ctx context.Context, req *UserRequest) (*UserResponse, error) {
       user := s.db.Find(req.UserId)
       return &UserResponse{Name: user.Name, Email: user.Email}, nil
   }
   // Deserialization is automatic — generated code handles it
Performance Numbers vs REST

Service mesh with 5 services, 1,000 req/sec: REST requires new HTTP/1.1 connection per request — connection overhead 50–100ms each. gRPC maintains persistent HTTP/2 connections — 5 connections total, multiplexed. Binary serialization: 3–10x smaller payloads, 10x faster parse. Total latency: REST ~450ms vs gRPC ~150ms (3x faster) for the same multi-hop request.

The Four gRPC Streaming Patterns

1. Unary (Request/Response)

One request, one response. Equivalent to a normal HTTP request.

rpc GetUser(Request) returns (Response); Client ──Request──→ Server Client ←─Response── Server

Use: Standard lookups, mutations

2. Server Streaming

One request, stream of responses. Server sends data until done.

rpc ListUsers(Request) returns (stream Response); Client ──Request──→ Server Client ←─Response 1── Server Client ←─Response 2── Server Client ←─Response N── Server

Use: Large downloads, real-time feeds

3. Client Streaming

Stream of requests, one response. Client uploads chunks.

rpc UploadFile(stream Request) returns (Response); Client ──Chunk 1──→ Server Client ──Chunk 2──→ Server Client ──Chunk N──→ Server Client ←─Summary── Server

Use: File upload, batch writes

4. Bidirectional Streaming

Both sides stream independently and simultaneously.

rpc Chat(stream Message) returns (stream Message); Client ←──→ Server (Both directions, independently)

Use: Chat, real-time collab, gaming

CStrengths & Ideal Use Cases ▾
Where gRPC Shines
  • Polyglot service mesh: One .proto file generates compatible code in Go, Java, Python, C++ — no manual client library maintenance
  • Performance-critical paths: Protobuf parse: ~1–5μs vs JSON parse ~50–100μs. At 100K req/sec, that's 5–10 seconds of CPU saved per second.
  • Streaming use cases: Live dashboards, large file transfers with backpressure, bidirectional chat
  • Real-world users: Google Cloud APIs, etcd cluster communication, Netflix inter-service, Square backend
DLimitations & Failure Modes ▾
Deadline Propagation — The Hidden Failure
t=0ms:    Client calls ServiceA.GetData() with 1000ms deadline
t=50ms:   ServiceA calls ServiceB.Process() — inherits 950ms deadline
t=100ms:  ServiceB calls ServiceC.Compute() — inherits 900ms deadline
t=600ms:  ServiceC starts expensive computation...
t=1000ms: Client's deadline expires!

What happens:
  - Client's context cancelled → gRPC propagates CANCELLED upstream
  - ServiceA ctx.Done() triggers → cleanup, return error  ✓
  - ServiceB ctx.Done() triggers → cleanup, return error  ✓
  - ServiceC... STILL COMPUTING                           ✗

ServiceC wasted 400ms of CPU on a request nobody needed.

Solution: Check ctx.Done() or ctx.Err() periodically in long operations.
          For every gRPC handler: if ctx.Err() != nil { return nil, ctx.Err() }
Proto Schema Evolution Rules — What You Can and Cannot Do
// v1 — original schema
message User {
  int32 age = 1;
  string name = 2;
}

// v2 — WRONG: changed field type (breaks compatibility!)
message User {
  string age = 1;   ← Changed int32 → string. Old clients will misparse.
  string name = 2;
}

// v2 — RIGHT: add new field, keep old
message User {
  int32 age = 1;         ← Unchanged. Old clients still read this.
  string name = 2;       ← Unchanged.
  int32 age_years = 3;   ← New field. Old clients ignore it safely.
}

RULES you must never break:
  ✗ Never change field numbers (field 1 is always field 1)
  ✗ Never change field wire types (int32 stays int32)
  ✗ Never reuse field numbers (even for deleted fields)
  ✓ Adding new fields is always safe (old clients ignore unknown)
  ✓ Mark deprecated fields with [deprecated = true] + reserved
  ✓ Use optional fields for new additions
Browser Support & Load Balancer Gotchas
  • Browser limitation: Browsers can't access HTTP/2 binary frames directly. Need gRPC-Web + Envoy proxy to translate — adds latency and another failure point.
  • L4 LB problem: HTTP/2 multiplexes all requests on one TCP connection. L4 load balancer sees one connection and routes ALL requests to one backend. Need L7 LB (Envoy, Istio) or client-side load balancing.
  • Message size limit: Default 4MB max message size. Large payloads silently fail or cause errors. Use streaming instead.
  • Corporate proxy blocking: Some network proxies block HTTP/2 or gRPC's framing. Internal services are fine; crossing corporate networks may not be.

FL6/L7 Nuances ▾
The Deep Question

"When would you choose gRPC over REST for a service that also has browser clients?"

Strong answer: "I'd use a split architecture. gRPC for all internal service-to-service communication — this is where the performance and streaming benefits matter. For browser clients, expose a REST or GraphQL API via a BFF layer. The BFF translates browser-friendly HTTP/JSON calls into gRPC calls to internal services. This gives you gRPC's performance for 95% of traffic (internal) while maintaining REST's accessibility for browser clients."

L6/L7 Insight — gRPC Is Really About Culture

"gRPC's biggest benefit isn't performance — it's the contract culture it enforces. When every service-to-service call requires a .proto file, you can't have implicit contracts. Schema evolution rules are baked in. Code generation eliminates 'my client expects field X but your server returns field_x.' The performance is a bonus; the contract discipline is the architectural win. At Google scale, explicit contracts across thousands of services are what prevent the distributed monolith problem."

🎯 Part 1 Key Lesson

The three paradigms optimize for different things: REST for universal accessibility (any client, HTTP caching), gRPC for performance and contract discipline (binary, typed, streaming, HTTP/2), GraphQL for client flexibility (exact data shapes, single round-trip for complex queries). None is universally better — the choice is a function of who the client is, what the performance requirements are, and who owns both sides of the contract.

The L7 signal for REST: knowing that HATEOAS is the missing piece that would enable true evolvability, and that BFF is the pattern that resolves REST's chattiness at scale. The L7 signal for gRPC: knowing that deadline propagation must be explicitly checked in every handler, that L4 load balancers break HTTP/2 multiplexing, and that the real value of gRPC is the contract culture it enforces, not just the raw performance numbers.
