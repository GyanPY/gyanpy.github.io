Pillar 1
Core Concepts & Mental Models

What Problem Does ClickHouse Solve?

Before ClickHouse existed, the analytics world was divided into two very painful camps. On one side, you had OLTP databases (PostgreSQL, MySQL) — they were row-oriented, ACID-compliant, and optimized for transactional workloads. They excelled at fetching a single row by primary key but completely fell apart when you asked them to scan 10 billion rows and compute a COUNT(*) GROUP BY. On the other side, you had Hadoop/MapReduce pipelines that could handle petabyte-scale data but introduced enormous operational complexity, multi-hour query latencies, and a Java ecosystem that required a small army of engineers to maintain.

The key insight Yandex engineers had in 2008 (and open-sourced in 2016) was: analytical queries almost always touch a small number of columns but an enormous number of rows. If you rethink storage layout around columns rather than rows, and pair that with aggressive vectorized execution and compression, you can achieve query latencies measured in milliseconds over billions of rows — on commodity hardware. This is the core value proposition ClickHouse delivers.

The world before ClickHouse for real-time analytics looked like: ingest data into Hadoop → run hourly/daily batch jobs → materialize aggregated results into MySQL → serve dashboards from those pre-computed tables. This pipeline had terrible freshness (hours of lag), required teams of data engineers to manage, and any new metric required re-running the entire pipeline. ClickHouse collapsed this entire architecture into a single system with sub-second query latency on fresh data.

The Core Abstraction: Columnar MergeTree Storage

The central abstraction ClickHouse provides is a columnar, append-only, LSM-like storage engine called MergeTree. Think of it as a marriage between columnar storage (Apache Parquet's layout philosophy), an LSM-tree's write path (batching writes into immutable sorted parts), and a B-tree's read-time sparse indexing. This is a fundamentally different abstraction from a B-tree heap file (PostgreSQL), a row-oriented LSM (RocksDB), or a distributed file system (HDFS).

Why is this abstraction powerful? Because it aligns perfectly with how analytical queries behave: they filter on a few dimensions (which the sparse primary index accelerates) and aggregate over many rows in a few columns (which columnar storage and SIMD-vectorized execution makes blazing fast). The immutability of parts also means reads and writes never contend in the way they do in a page-locked B-tree, enabling very high concurrent read throughput.

Fundamental Guarantees ClickHouse Makes

  • Guarantee 1 — Ordering within partitions: Within a single data part, rows are guaranteed to be sorted by the primary key (ORDER BY expression). This guarantee is what makes the sparse primary index possible and enables merge-join like operations during background merges.
  • Guarantee 2 — Eventual deduplication (with ReplacingMergeTree): In the ReplacingMergeTree variant, rows with the same primary key will eventually be collapsed to the latest version during background merges. This is not an immediate guarantee — duplicates can persist until the next merge cycle.
  • Guarantee 3 — Monotonically increasing write visibility: Inserted data is immediately visible to subsequent queries on the same node (once the INSERT transaction completes). ClickHouse does not implement MVCC for analytical reads — it uses a simpler mechanism of snapshotting which parts exist at query start.
  • Guarantee 4 — Durable writes on ACK: When ClickHouse acknowledges an INSERT, the data has been written to the WAL (Write-Ahead Log) and the part directory on disk. The durability guarantee is local to the node unless you use Replicated tables.
  • Guarantee 5 — Horizontal partition isolation: Each partition (e.g., by month) is stored in completely separate directory trees. This means partition drops and detaches are O(1) metadata operations — no expensive tombstone cleanup or vacuuming needed.

What ClickHouse Explicitly Does NOT Guarantee

  • No multi-row ACID transactions: There is no cross-row atomicity. Two concurrent INSERTs can interleave. As of v23.x, experimental lightweight transactions (LWT) exist for single-partition operations, but these are not production-grade for complex use cases. UPDATE and DELETE are implemented as mutations that rewrite entire parts asynchronously — they are not low-latency operations.
  • No immediate deduplication: Even with ReplacingMergeTree, you can read duplicate rows between the time of insert and the next merge. You must use the FINAL modifier (which forces a merge at query time) or handle deduplication client-side.
  • No strong cross-replica read consistency: With Replicated tables, reads from different replicas can return different data if replication lag exists. There is no linearizability guarantee across replicas.
Common Interview Mistake Candidates say "ClickHouse is strongly consistent." Wrong. ClickHouse is AP in the CAP theorem — it prioritizes availability and partition tolerance. Reads from replicas can be stale. This is a core design choice, not a bug.

Invariants That Must Hold for Correctness

  • Primary key ordering within parts: If this breaks (due to a bug or forced merge manipulation), the sparse index becomes incorrect and queries silently return wrong results. ClickHouse detects this via checksums.
  • Part checksums: Every part has a checksums.txt file. If any data file's content doesn't match its checksum, the part is considered corrupt and ClickHouse refuses to use it.
  • Partition key monotonicity: Data inserted into a partition must conform to that partition's boundary. ClickHouse does not enforce this at write time for performance, but violating it (via manual part moves) corrupts query results.
  • ZooKeeper/ClickHouse Keeper quorum: In Replicated tables, the ZooKeeper (or ClickHouse Keeper) quorum must be maintained for metadata coordination. Loss of quorum makes the cluster read-only.

L7/L8 Differentiator The sparse index reads one index entry per 8192 rows (one granule). This means a table with 1 billion rows has only ~122,070 index entries — the entire index fits in memory. Compare this to a B-tree which has an index entry per row. Understanding that the index granularity is tunable (index_granularity setting) and its impact on memory vs selectivity tradeoff is what separates senior from staff candidates.
Key Takeaway: ClickHouse's power comes from the synergy of columnar storage (load only needed columns), sparse primary indexing (skip irrelevant row ranges), partition pruning (skip entire months of data), and SIMD vectorized execution (process 256 rows per CPU instruction). No single one of these is novel; combining all four in a single system is the innovation.

Write Path — Step by Step

  1. Client sends INSERT statement via HTTP or Native protocol. The payload can be in dozens of formats (CSV, JSONEachRow, Parquet, Arrow, etc.). ClickHouse performs format parsing in parallel using multiple threads if the payload is large.
  2. Block formation: Incoming rows are grouped into "blocks" — the basic unit of processing in ClickHouse. Each block is a collection of column vectors. Default block size is 65,536 rows (max_block_size). This batching is critical — ClickHouse is not designed for single-row inserts.
  3. Sorting: Rows within the block are sorted by the ORDER BY key. This is the first sort step. When multiple blocks arrive, ClickHouse writes each as a separate immutable "part" on disk.
  4. Part creation: The sorted block is serialized to disk as a "part" directory. Each column gets its own .bin file (compressed with LZ4 or ZSTD). Mark files (.mrk3) are created that store the byte offset of every 8192-row granule within the .bin file. The sparse primary index (primary.idx) stores one primary key tuple per granule. This entire write is essentially a sequential I/O pattern — extremely fast on SSDs and even spinning disks.
  5. WAL (Write-Ahead Log): As of ClickHouse 22.x+, inserts first write to an in-memory buffer backed by a WAL for crash recovery. Upon ack to the client, data is durable either in WAL or as a completed part on disk.
  6. Background Merge Thread: ClickHouse continuously runs background merge threads that select small parts and merge them into larger parts. This process re-sorts, re-compresses, and rebuilds all index structures. It's analogous to compaction in RocksDB. After a successful merge, the old small parts are deleted (after a retention period for replicas to catch up).
Back-of-Envelope: Write Amplification If you insert 1000 rows/sec in 1000 tiny INSERT statements, you get 1000 parts/sec. ClickHouse default limit is 150 parts before throttling inserts. A 1000-row part might be ~100KB compressed. Background merges read and rewrite this data ~5-7x as parts are merged from level 0 → 1 → 2... Total write amplification: 5-7x. Compare to RocksDB which is 10-30x. Batching inserts to 100K+ rows reduces this dramatically — each merged write covers more data at once.

Read Path — Step by Step

  1. SQL Parsing & Analysis: The SQL statement is parsed into an AST. The analyzer resolves table references, validates column names, and expands wildcards. In ClickHouse v22.3+, a new "Analyzer" (enabled by allow_experimental_analyzer=1, default in v23.11+) dramatically improved query planning for complex queries, CTEs, and subqueries.
  2. Query Optimization: The optimizer applies rule-based optimizations: constant folding, predicate pushdown into subqueries, filter pushdown to storage, projection pruning (only reading needed columns), and partition elimination.
  3. Partition Pruning: Using the minmax_{partition_key}.idx files (tiny files that store min/max values per part), ClickHouse eliminates entire parts whose partition range doesn't overlap with the query's WHERE clause. For a table partitioned by month, a query for "last 7 days" might skip 99% of parts instantly.
  4. Primary Index Scan: For the remaining parts, ClickHouse binary-searches the primary.idx sparse index to identify the range of granules that could contain matching rows. Since the index has only 1 entry per 8192 rows, this lookup is extremely fast and the entire index fits in the OS page cache.
  5. Skip Index Evaluation (Secondary Indices): If the table has secondary indices (BloomFilter, minmax, set, ngrambf_v1), ClickHouse evaluates them to skip additional granules. For example, a BloomFilter on a text column can skip granules that definitely don't contain a search term.
  6. Asynchronous Column Reading: Only the columns referenced in the query are read. Each column's .mrk3 mark file is used to seek directly to the relevant granule offset within the .bin file. Multiple columns are read in parallel using async I/O (io_uring on Linux since v22.x).
  7. Vectorized Execution: Decompressed column data is processed in blocks of 65,536 rows using SIMD instructions (AVX2, AVX-512). Filter conditions compile down to bit-mask operations on entire vectors. This is why ClickHouse can process hundreds of millions of rows per second per CPU core.
  8. Aggregation: Hash aggregation is used for GROUP BY. ClickHouse uses a two-stage aggregation for distributed queries: partial aggregation on each shard, then a final merge aggregation on the coordinator node.

Back-Pressure and Flow Control

ClickHouse implements several back-pressure mechanisms to prevent runaway queries from degrading the system. The parts count threshold is the most important: if the number of active parts in a single partition exceeds parts_to_delay_insert (default: 150), new inserts are slowed down with artificial delays. If it exceeds parts_to_throw_insert (default: 300), inserts are rejected with a "Too many parts" error. This is ClickHouse's primary admission control mechanism.

For queries, query complexity limits (max_rows_to_read, max_bytes_to_read, max_memory_usage) act as circuit breakers. The priority system allows marking queries with priorities (1-10) so that interactive dashboard queries can preempt long-running background analytics. The concurrent query limit (max_concurrent_queries) prevents thread exhaustion.

L7/L8 Differentiator — New in v24.x ClickHouse 24.x introduced Parallel Replicas (query-time horizontal scaling within a single table). Instead of one replica handling a query, the coordinator can split the mark range across N replicas, each reading a non-overlapping portion of granules. This is fundamentally different from sharding — same data, same replica, but query work is parallelized. This is a breakthrough for low-cardinality tables that can't be easily sharded.
Key Takeaway: ClickHouse's architecture is intentionally designed around sequential I/O, columnar access, and SIMD vectorization. Every component — from the sparse index to the async column reader to the pipeline executor — is optimized for the assumption that queries touch few columns but many rows in sequential order. Deviating from this pattern (e.g., high-cardinality point lookups) is where ClickHouse performs poorly.

Pillar 3
Trade-offs & Design Decisions

CAP Theorem Positioning

ClickHouse in its replicated configuration sits firmly in the AP (Available + Partition Tolerant) quadrant of the CAP theorem. When a network partition occurs, ClickHouse chooses to remain available for reads — potentially returning stale data from a lagging replica — rather than returning an error. Writes to the leader/quorum can still proceed (depending on configuration), while isolated replicas continue serving reads independently.

This is a deliberate design choice for the analytics use case: an analyst querying yesterday's events can tolerate reading data that's 30 seconds stale due to replication lag. They absolutely cannot tolerate their dashboard returning a 503 error. The consistency sacrifice is: different replicas can return different aggregate counts for the same time window query if replication is lagging. For financial reporting or billing, this is unacceptable. For product analytics, it's perfectly fine.

Consistency Models Offered

  • Read Your Writes (single node): Within a single ClickHouse node, after an INSERT is acknowledged, a subsequent SELECT on the same node will see the data. This holds because the part directory is created atomically before the INSERT returns.
  • Eventual Consistency (replicated): The default. Inserts to a ReplicatedMergeTree table propagate to other replicas asynchronously via the ZooKeeper/Keeper replication log. Lag is typically sub-second on healthy clusters, but can stretch to minutes under load.
  • Quorum Writes (insert_quorum=N): Forces the insert to wait until N replicas have confirmed the data. With insert_quorum=2 in a 3-replica setup, you get a write majority guarantee. Reads with select_sequential_consistency=1 will then only return data that has reached the write quorum.
  • No Serializable/Linearizable: ClickHouse does not offer serializable isolation or linearizability. Concurrent long-running aggregation queries and concurrent inserts are not serialized. A query sees a snapshot of the parts that existed at its start time — new parts inserted during the query's execution are not included.
Common Interview Mistake Candidates assume insert_quorum is equivalent to Raft consensus. It is not. insert_quorum means N replicas ACK the insert, but there's no leader election or log ordering guarantee. Two concurrent inserts can interleave across replicas. The replication order is governed by ZooKeeper's log, not by insert_quorum alone.

Durability vs Performance Trade-offs

  • fsync on insert (default: off): By default, ClickHouse calls fdatasync after writing a part but does NOT call fsync on the directory entry. This means a kernel crash could lose the last insert. Setting fsync_after_insert=1 adds safety at the cost of 2-5x higher insert latency.
  • WAL durability: The write-ahead log introduced in v22.x provides crash recovery within a node (replaying the WAL after restart recovers in-flight inserts). But the WAL itself is only fsynced based on the fsync_after_insert setting.
  • Background merge durability: Merged parts are written atomically (via rename from a temp directory). If ClickHouse crashes mid-merge, the original parts still exist and the incomplete merged part is cleaned up on restart. No data loss from interrupted merges.

Operational Complexity vs Feature Richness

ClickHouse's surface area is enormous and this is both a strength and a weakness. The MergeTree family alone has 8+ variants (MergeTree, ReplacingMergeTree, SummingMergeTree, AggregatingMergeTree, CollapsingMergeTree, VersionedCollapsingMergeTree, GraphiteMergeTree, and the Shared MergeTree introduced in ClickHouse Cloud). Each has subtly different semantics and gotchas. Choosing the wrong table engine for your use case (e.g., using ReplacingMergeTree without understanding the FINAL requirement for deduplication) leads to silently wrong query results.

Key Takeaway: ClickHouse is unabashedly an AP system that trades consistency for extreme read/write throughput. Every knob in the system (insert_quorum, select_sequential_consistency, fsync settings) is a dial on the consistency-performance spectrum. An L7/L8 engineer knows not just what these knobs do, but when to turn them and what customer SLAs justify the cost.

Failure Modes & Resilience

Failure Scenario Analysis

🔴 Single Node Crash (Leader/Follower)

(a) Detection: ZooKeeper/Keeper detects the session timeout (default: 30s). Other replicas see the ephemeral node disappear from /replicas/{name}/is_active.

(b) Auto-recovery: Followers continue serving reads. If the crashed node was the merge "leader" (who coordinates merge scheduling), another replica is elected via ZooKeeper leader election. The crashed node, on restart, replays its replication queue from the last committed log pointer in ZooKeeper, fetching any missed parts from healthy replicas.

(c) Data loss risk: Any inserts that were ACK'd to the client but not yet replicated to other nodes AND whose WAL was lost in the crash. With insert_quorum > 1, this risk is eliminated. Without it, ~30 seconds of inserts could be lost.

(d) Manual action: If the crashed node has disk corruption, run SYSTEM RESTORE REPLICA to force the replica to refetch all parts from healthy siblings. Monitor the replication queue lag with system.replication_queue.

🟠 Simultaneous Minority Node Failures

(a) Detection: Same as single node — ZooKeeper session timeouts for each node independently.

(b) Auto-recovery: The majority (2 of 3, or 3 of 5) continue operating normally. Reads and writes proceed on surviving nodes. Merge scheduling continues via leader election among survivors.

(c) Data loss risk: Low. Surviving nodes have a quorum of the replication log. Any inserts ACK'd to clients were written to the majority of nodes.

(d) Manual action: After recovering the failed nodes, they automatically catch up by fetching missed parts. Watch for "replication queue depth" — if the queue is very long, catching up can take hours and put pressure on surviving nodes' bandwidth.

🔴 Network Partition (One Side Has Quorum)

(a) Detection: Nodes on the minority side detect they've lost ZooKeeper connectivity (can't write to ZooKeeper). Nodes on the majority side continue normally.

(b) Auto-recovery: The majority side continues writes and merges. The minority side falls into read-only mode for replicated tables — inserts are rejected with "Keeper is unavailable." Reads continue serving from local data (potentially stale).

(c) Data loss risk: Zero for acknowledged writes (they were on the majority side). However, reads from the minority side return stale data — this is the AP tradeoff in action.

(d) Manual action: None typically needed. When the partition heals, minority nodes re-join, discover the replication log divergence, and catch up. Monitor system.replicas for queue_size and last_queue_update.

⛔ Network Partition (Even Split — No Quorum)

(a) Detection: Both sides detect ZooKeeper is unreachable (since ZooKeeper itself requires quorum). Both sides go read-only for writes on replicated tables.

(b) Auto-recovery: No automatic recovery possible. Both sides wait until the partition heals. ClickHouse Keeper (the built-in Raft-based replacement for ZooKeeper) handles this better — it requires N/2+1 Keeper nodes to have quorum, and Keeper itself partitions cleanly.

(c) Data loss risk: No writes are ACK'd during the partition, so no data loss. But the cluster is effectively dead for writes.

(d) Manual action: Restore network connectivity. If a Keeper node is permanently lost in a 3-node Keeper cluster, add a replacement node and let it sync from the remaining two.

🔀 Split-Brain: Two Leaders

(a) Detection: ClickHouse prevents true split-brain through ZooKeeper/Keeper's fencing tokens. All mutations to the replication log require a ZooKeeper transaction. If two nodes both think they are "leader" for merge scheduling, ZooKeeper's linearizable writes ensure only one can acquire the lock at a time.

(b) Auto-recovery: The fencing mechanism ensures split-brain doesn't cause data divergence. The node that can't acquire the ZooKeeper lock backs off. This is one reason why ClickHouse mandates ZooKeeper/Keeper for replicated tables — without it, split-brain would be a real risk.

(c) Data loss risk: Minimal due to fencing. However, if ZooKeeper itself suffers split-brain (which is extremely rare in a properly configured 3-node ZK cluster with quorum of 2), all bets are off.

(d) Manual action: If you suspect data divergence between replicas, use SELECT * FROM system.replicas to compare parts_to_check, total_replicas, and active_replicas across nodes.

🐢 Slow/Degraded Node (Not Dead, But Lagging)

(a) Detection: Not automatically detected by ZooKeeper. The degraded node keeps its ZooKeeper session alive but processes replication tasks slowly. Monitoring via system.replication_queue shows growing queue size on the slow node.

(b) Auto-recovery: ClickHouse's distributed query engine has a prefer_localhost_replica setting and a load_balancing policy. The in_order or nearest_hostname policies can route queries away from the slow node. However, this is not automatic — it requires round-robin or latency-aware routing configuration.

(c) Data loss risk: None for durability. But the slow node serves stale reads. As of v23.x, ClickHouse can detect replica staleness via max_replica_delay_for_distributed_queries — queries are not routed to replicas that are more than N seconds behind.

(d) Manual action: Identify the cause (disk I/O saturation, high CPU from a bad query, OOM-killed merge thread). Use SYSTEM STOP MERGES to reduce I/O pressure. Use KILL QUERY to terminate runaway queries.

💾 Disk Corruption or Partial Write

(a) Detection: ClickHouse checksums every part at write time (SHA256 of each .bin file stored in checksums.txt). On every read, the checksum is verified. A failed checksum immediately raises an exception. Additionally, ClickHouse runs periodic background part integrity checks (check_delay_period).

(b) Auto-recovery: In a replicated setup, when a corrupted part is detected, ClickHouse automatically adds a task to its replication queue to fetch the correct part from another replica. The corrupted part is moved to detached/ for forensic analysis.

(c) Data loss risk: Zero in a replicated setup — the data exists on other replicas. In a single-node setup, the corrupted part's data is permanently lost.

(d) Manual action: Run CHECK TABLE table_name to verify all parts. Inspect detached/ directory. In extreme corruption cases, use SYSTEM RESTORE REPLICA to force a full re-sync from a healthy replica.

⏰ Clock Skew Between Nodes

(a) Detection: ClickHouse uses ZooKeeper for all distributed coordination, and ZooKeeper itself is sensitive to clock skew (>2 seconds between ZK nodes causes ZK to fail). ClickHouse itself uses wall clock time for TTL expressions, partition function evaluation, and query timestamps.

(b) Auto-recovery: No automatic recovery from clock skew. ClickHouse does not use Hybrid Logical Clocks (HLC) like CockroachDB or TrueTime like Spanner. It relies on NTP synchronization.

(c) Data loss risk: Clock skew can cause TTL rules to delete data prematurely (if a node's clock is ahead) or retain data too long (if behind). For partitioning by date, clock skew can cause rows to land in wrong partitions. For insert_quorum with timeout, clock skew between client and server can cause spurious timeouts.

(d) Manual action: Enforce NTP synchronization across all nodes. Use Chrony with PTP for sub-millisecond accuracy. Alert on ntp_offset > 100ms. Deploy ClickHouse Keeper (which uses Raft with its own clock-independent sequence numbers) instead of external ZooKeeper for better tolerance.

⚡ Thundering Herd After Recovery

(a) Detection: When a node returns from downtime, all replicas simultaneously begin sending it missed parts. The recovering node's network interface and disk I/O become saturated. ZooKeeper shows all replicas with full replication queues pointing at the recovering node.

(b) Auto-recovery: ClickHouse has max_replicated_fetches_network_bandwidth and max_replicated_sends_network_bandwidth throttling settings. These limit the bandwidth each node uses for replication traffic, preventing recovery from overwhelming the network.

(c) Cascading risk: If the recovering node is flooded with replication traffic, its CPU and I/O are consumed, making it unable to serve production queries. This can cascade — the node appears alive to ZooKeeper but is effectively unresponsive for queries.

(d) Manual action: Set max_replicated_fetches_network_bandwidth to 50-100 MB/s on recovery. Use SYSTEM STOP FETCHES to pause recovery temporarily during business hours, then resume during off-peak. Monitor via Prometheus/Grafana — watch for disk I/O saturation on the recovering node.

🌊 Cascading Failures

(a) How they happen: A slow query consumes all thread pool slots → new queries queue up → queue fills up → new connections are rejected → load balancer marks the node unhealthy → all traffic reroutes to remaining nodes → remaining nodes become overloaded → cascade.

(b) Prevention mechanisms: (1) max_concurrent_queries hard limit per node. (2) Query complexity limits prevent single queries from consuming all memory. (3) Priority queues allow shedding low-priority queries. (4) Circuit breakers in the distributed query layer skip overloaded shards. (5) max_memory_usage_for_all_queries at the server level prevents OOM.

(c) Design patterns: Separate read and write replicas. Separate user-facing OLAP from internal batch workloads via different replica groups. Use resource groups (v23.x+) to isolate tenant workloads. Deploy a query proxy (ClickHouse Proxy, CHProxy) that enforces connection limits and query timeouts before requests reach the server.

(d) Manual action: Use KILL QUERY WHERE user='heavy_user' to evict runaway queries. Temporarily increase max_concurrent_queries if safe. Use SYSTEM RELOAD ASYNCHRONOUS METRICS to get fresh system health metrics without a restart.

Key Takeaway: ClickHouse's failure resilience is primarily built on two foundations: (1) ZooKeeper/Keeper-mediated coordination with fencing tokens that prevents split-brain, and (2) checksum-based data integrity that enables automatic corruption detection and repair from replicas. The weakest points are clock skew sensitivity, thundering herd on recovery, and the lack of automatic slow-node detection for query routing.

Scaling Strategies

Vertical Scaling — How Far Can You Go?

ClickHouse scales extraordinarily well vertically because its query executor is embarrassingly parallel — it uses all CPU cores via thread pools. A single ClickHouse node on a 96-core machine with 768 GB RAM and 8 NVMe SSDs in RAID-0 can sustain 100+ million rows per second query throughput for aggregation queries on warm cache. Yandex has reported single nodes processing 2+ PB of data.

What breaks first on a single node? In practice: RAM is the first bottleneck. ClickHouse's hash aggregation for GROUP BY with high cardinality keys can consume enormous memory. The max_memory_usage limit exists to prevent OOM, and when hit, ClickHouse spills to disk (via max_bytes_before_external_group_by). After RAM: NVMe I/O bandwidth for sequential scans. A single NVMe can sustain ~7 GB/s. 8 NVMe in RAID-0 gives ~56 GB/s — and ClickHouse can saturate this with hot queries. After I/O: network egress when the node is a distributed query coordinator receiving partial results from many shards.

Back-of-Envelope: Single Node Capacity A realistic production ClickHouse node: 32 cores, 256 GB RAM, 4x NVMe SSD (14 GB/s total I/O). Assume 10:1 compression ratio for analytics data (100 GB raw → 10 GB on disk). In a 5TB NVMe setup, that's 50 TB of raw data. At 14 GB/s I/O and 10:1 compression: reading 14 GB/s * 10 = 140 GB/s raw data throughput. A query scanning 10 TB raw data completes in ~71 seconds — with column pruning (only 5 of 50 columns), that's ~7 seconds. With partition pruning (last 7 days of a 1-year table), that's ~2 seconds. This is why ClickHouse is fast.

Horizontal Scaling — What Changes Architecturally?

ClickHouse scales horizontally through sharding and replication. The Distributed table engine is the key abstraction: it acts as a transparent proxy that routes queries and inserts to the underlying shards. The cluster topology is defined in config.xml as a static configuration of shards and replicas. This is fundamentally different from systems like Cassandra or CockroachDB that have automatic, dynamic ring-based sharding. ClickHouse's sharding is manual and static — you define the number of shards at cluster creation and resharding requires data movement.

Sharding / Partitioning Strategy

Sharding key selection is the most critical and irreversible decision in a ClickHouse deployment. The wrong sharding key leads to hot shards (one shard gets 80% of writes because all data for popular tenants maps to it) and skewed reads (cross-shard joins become expensive because related data is on different shards).

Best sharding keys:

  • intHash64(user_id) — Good for user-level analytics. Distributes data evenly. Enables shard-local joins if queries are scoped to a single user.
  • cityHash64(tenant_id) — Good for multi-tenant SaaS. Each tenant's data is co-located on one shard, enabling efficient per-tenant queries.
  • rand() — Maximum distribution uniformity. But cross-shard aggregation always needed. Good when queries always touch all shards anyway.

Anti-patterns: Using a low-cardinality column as sharding key (e.g., country with only 200 values — US gets 40% of traffic → hot shard). Using a monotonically increasing ID as sharding key without hashing (all new data goes to one shard until it overflows to the next).

Cross-Region Replication

ClickHouse's native replication is designed for low-latency intra-datacenter replication (sub-100ms network RTT). Stretching ZooKeeper replication across regions with 50-200ms RTT significantly impacts performance: every replicated insert operation requires multiple ZooKeeper round trips, and with 100ms RTT, this dominates insert latency.

The recommended pattern for cross-region ClickHouse is active-active with application-level fan-out: write to a message queue (Kafka, Kinesis), have separate ClickHouse clusters per region each consuming from their regional Kafka topic, with cross-region Kafka MirrorMaker replication. This decouples ClickHouse from inter-region latency at the cost of requiring Kafka infrastructure and accepting eventual consistency across regions.

L7/L8 Differentiator — ClickHouse Keeper vs ZooKeeper ClickHouse 22.4 introduced ClickHouse Keeper — a Raft-based coordination service embedded within ClickHouse itself, designed to replace external ZooKeeper. Key advantages: (1) No external dependency, (2) Better performance for ClickHouse-specific workloads (log replication, block number allocation), (3) Supports reconfiguration without downtime. For Google-scale deployments, ClickHouse Keeper eliminates the ZooKeeper bottleneck — a single ZooKeeper ensemble can become the coordination bottleneck when managing thousands of replicated tables across hundreds of nodes.

Read vs Write Scaling Independence

ClickHouse CAN scale reads and writes independently, but only through careful architectural design. The key pattern: write to a dedicated "write replica" (or directly to the shard's primary), and read from read-only replicas that are configured as pure read targets. This is achieved by using different replica weights in the Distributed table configuration. Read replicas can be scaled independently of write replicas — you can add 10 read replicas to a shard without adding write replicas.

Key Takeaway: ClickHouse's horizontal scaling is powerful but manual. The Distributed table engine provides transparent fan-out, but sharding key selection, resharding, and cross-region replication all require careful up-front design. The new Parallel Replicas feature in v23.3+ is a game-changer because it provides query-level horizontal scaling without the complexity of sharding — but it only works when data is replicated, not sharded.

Real-World Engineering Decisions

Where ClickHouse Shines Above All Alternatives

  • High-cardinality real-time analytics: COUNT(DISTINCT user_id) over 10 billion events in 100ms. No other open-source system matches this for fresh data.
  • Time-series aggregation with arbitrary dimensions: "Show me p99 latency by service, by region, by deployment version for the last 24 hours, with 1-minute resolution." ClickHouse handles this natively where purpose-built time-series databases (Prometheus, InfluxDB) struggle with high-cardinality dimensions.
  • Log analytics at petabyte scale: Full-text search (via ngrambf bloom filters), structured field extraction, and aggregation over billions of log lines. Better than Elasticsearch for structured analytics; worse for full-text relevance scoring.
  • Ad-tech click stream analysis: The original Yandex.Metrica use case — tracking billions of page views and user sessions with sub-second query response for real-time dashboards.

Three Real-World Production Examples

1. Cloudflare — HTTP Request Analytics at 25+ Million Requests/Second

Cloudflare processes over 25 million HTTP requests per second globally and needs to provide customers with analytics dashboards showing firewall rule effectiveness, cache hit rates, bandwidth by geography, and threat analysis — all with <1 minute data freshness and <2 second query response time for customer dashboards.

Why ClickHouse: Cloudflare evaluated Druid and BigQuery but needed a system they could deploy on-premise (customer data residency requirements), that handled high-cardinality dimensions (customer ID × country × ASN × URL path), and that supported flexible ad-hoc queries (not just pre-aggregated cubes). ClickHouse's ability to handle 50+ column schemas with arbitrary GROUP BY combinations without pre-aggregation was decisive. They run ClickHouse clusters with hundreds of nodes per region, using Kafka for ingestion buffering and the Distributed table engine for cross-shard queries.

2. Uber — Operational Metrics & Experimentation Platform

Uber's experimentation platform (for A/B testing driver/rider features) needed to compute metric deltas across treatment and control groups with statistical significance. This requires computing aggregates over hundreds of millions of trip records, grouped by experiment variant, user segment, city, and time bucket — and doing so in real-time as the experiment runs.

Why ClickHouse: The queries involve complex aggregations (quantiles, HyperLogLog distinct counts) over columnar data that changes constantly as new trips complete. Uber's alternative was a Spark batch pipeline running hourly — but this meant experiment owners couldn't detect problems (metric degradation) for an hour. ClickHouse gave them real-time metric computation with sub-minute data freshness. The key decision: using AggregatingMergeTree to pre-aggregate states (HLL, quantile sketches) at insert time, then using finalizeAggregation() at query time to compute final values. This is dramatically more efficient than re-aggregating raw rows on every query.

3. Contentsquare — UX Analytics over Trillions of Mouse Movements

Contentsquare tracks mouse movements, clicks, scroll depth, and frustration signals for enterprise website visitors. They ingest ~1.5 trillion events per day and need to compute UX heatmaps and journey analytics for their customers.

Why ClickHouse: The challenge is that each event is a tiny record (user_id, session_id, element_selector, x, y, timestamp) but there are trillions of them. Computing "what percentage of users who saw this button clicked it" requires scanning all events for a given website within a time window. ClickHouse's columnar storage means only relevant columns are read, and the sharding key on website_id means each query only hits one shard. They achieved 10x cost reduction vs their previous BigQuery setup by reducing data egress costs and compute costs simultaneously.

When NOT to Use ClickHouse

Anti-Pattern 1: OLTP Workloads (Point Lookups) If your queries look like SELECT * FROM orders WHERE order_id = 12345, use PostgreSQL. ClickHouse has no row-level caching and no efficient B-tree index for point lookups. Every "point lookup" reads an entire 8192-row granule minimum. At P99, this is 10-50ms — fine for analytics, terrible for user-facing transactional queries.
Anti-Pattern 2: Frequent Updates/Deletes If you need to update user profile data, delete GDPR-requested records frequently, or implement "last value" semantics with sub-second freshness, ClickHouse is wrong. ALTER TABLE mutations rewrite entire data parts asynchronously — a DELETE WHERE user_id = 12345 on a 1 TB table will rewrite 1 TB of data in the background over hours. Use ClickHouse's built-in TTL for bulk expiry, or design around ReplacingMergeTree for upsert patterns. For point deletes at high frequency, use PostgreSQL or Cassandra.
Anti-Pattern 3: Complex Multi-Table OLTP Joins ClickHouse's distributed JOIN performance is poor for large fact-dimension table joins where the dimension table is also large. A join between two billion-row tables is an ORDER OF MAGNITUDE slower than in a well-tuned OLAP system like DuckDB (for single-node) or BigQuery (for massive parallelism). ClickHouse is optimized for wide denormalized tables — if your schema is highly normalized (3NF with 20 tables), denormalize before loading into ClickHouse or use a different system.

Common Operational Mistakes in Production

  • Mistake 1 — Tiny INSERT statements: Sending one INSERT per event from application code. This creates thousands of tiny parts per second, exhausting the parts limit and causing "Too many parts" errors. Solution: Always buffer with Kafka, use the Buffer table engine, or batch at the application layer to >10K rows per INSERT.
  • Mistake 2 — Wrong PRIMARY KEY vs ORDER BY: In ClickHouse, you can define PRIMARY KEY (a, b) separately from ORDER BY (a, b, c). Many engineers assume these are the same. The ORDER BY determines physical sort order; PRIMARY KEY determines the sparse index. Using a high-cardinality first column in PRIMARY KEY (like UUID) makes the sparse index useless for range scans.
  • Mistake 3 — Not using Partition Pruning: Writing queries without including the partition key in the WHERE clause. If data is partitioned by month and you query without a date filter, ClickHouse scans all partitions. Always include partition key predicates.
  • Mistake 4 — Ignoring merge pressure: Not monitoring the number of parts per partition. When parts exceed parts_to_delay_insert, inserts slow down silently. The fix: increase background_pool_size to give more threads to the merge process, and ensure your data arrival rate doesn't consistently exceed merge throughput.
  • Mistake 5 — Using ReplacingMergeTree without FINAL: Reading from a ReplacingMergeTree table without the FINAL modifier and expecting deduplicated results. You will see duplicates. FINAL forces a merge at query time — expensive on large tables. Correct pattern: either use FINAL, or add a MAX(version) GROUP BY deduplication query layer, or use the optimize_on_insert setting (with caveats).

Pillar 7
Google-Scale & Beyond

What Breaks as You Scale from 1M → 100M → 1B Users

1M Users — Single Cluster Problems

At 1M active users generating events, you're likely dealing with 10-100 million events/day — roughly 1,000 events/second. A single ClickHouse node handles this trivially. The problems that emerge: ZooKeeper becomes a bottleneck when you have many small replicated tables (each table requires several ZooKeeper paths; 1000 tables × 3 replicas = 3000+ ZK watches). The solution is to move to ClickHouse Keeper (Raft-based) or carefully limit the number of replicated tables. The other early-scale problem is schema evolution — ALTER TABLE ADD COLUMN is eventually consistent in replicated setups and can take minutes to propagate.

100M Users — Multi-Cluster Problems

At 100M users generating events, you're at ~10,000-100,000 events/second. Now you need multiple shards. The challenges that emerge: Cross-shard query coordination — every SELECT on a Distributed table fans out to all shards, and if one shard is slow (stragglers), the entire query waits. The solution is implementing a "skip degraded shards" policy (skip_unavailable_shards) and timeout-based fallback. INSERT routing reliability becomes critical — if the Distributed table's async insert buffer exceeds its limit, inserts are silently dropped. This requires monitoring system.distribution_queue aggressively and implementing application-level retry.

1B Users — Hyperscaler Problems

At 1B users, you're looking at 1 million+ events/second sustained. Now you need: (1) Multi-region active-active with cross-region Kafka replication, (2) Tenant isolation to prevent one customer's massive query from impacting others, (3) Global query routing that directs queries to the region where the data lives (for data residency), and (4) Cost optimization through tiered storage (hot NVMe → warm HDD → cold S3).

How Google/Hyperscalers Solve This Differently

Google's internal analytics systems (Mesa, Dremel/BigQuery, Monarch) have several fundamental architectural differences from vanilla ClickHouse that enable hyperscale:

  • Disaggregated Storage and Compute (Google Colossus + BigQuery): BigQuery separates storage (Colossus, effectively infinite) from compute (Dremel query engines, ephemeral and elastic). This means you can scale compute up for a heavy query and back to zero. ClickHouse traditionally co-locates storage and compute — ClickHouse Cloud (ClickHouse Inc.'s managed service) introduced shared object storage (S3/GCS) as the storage layer in v23.x with "SharedMergeTree" — this is ClickHouse's first step toward disaggregation. An L8 candidate should discuss this evolution and its implications.
  • Global Query Optimization (Spanner-style): Google Spanner's query optimizer has cross-region awareness — it knows where data physically lives across global zones and optimizes data movement in the query plan. ClickHouse's distributed query layer is much simpler — it fans out to all shards and aggregates. There's no CBO (cost-based optimizer) that models cross-shard data movement cost. Adding a full CBO to ClickHouse is an unsolved engineering challenge.
  • Petabyte-Scale Compaction (Mesa architecture): Google's Mesa system (used for ads reporting) handles the same challenge as ClickHouse merges but at petabyte scale. Mesa uses a hierarchical delta compaction approach with globally coordinated compaction jobs — analogous to but far more sophisticated than ClickHouse's background merge threads. Mesa explicitly models compaction as a distributed job, not a local background task.
L7/L8 Differentiator — SharedMergeTree (ClickHouse Cloud) In 2023, ClickHouse Cloud introduced SharedMergeTree — a new table engine that stores data in shared object storage (S3/GCS/Azure Blob) instead of local disk. Parts are stored in S3 and metadata in a distributed coordination layer. This enables: (1) True compute-storage separation, (2) Zero-copy replication (replicas share the same S3 objects, no data copying needed), (3) Instant scaling of compute nodes (add a node → it immediately accesses all data in S3), (4) Dramatic cost reduction (S3 at $0.023/GB vs NVMe at $0.10/GB). The tradeoff: S3 GET latency (~5-10ms) vs NVMe local read (~0.1ms). ClickHouse mitigates this with aggressive local disk caching of hot data.

Multi-Tenancy: Noisy Neighbor Isolation

In a multi-tenant ClickHouse deployment (SaaS analytics product serving thousands of customers), the noisy neighbor problem is severe. Customer A runs a SELECT that scans 100B rows, consuming all CPU threads for 60 seconds, while Customer B's simple dashboard query gets starved. Solutions in order of complexity:

  • Level 1 — Query Complexity Limits: Per-user max_rows_to_read, max_execution_time, max_memory_usage limits in user profiles. Enforced at the session level. Simple but blunt — doesn't differentiate between a legitimately large query from a premium customer and a runaway query from a free tier user.
  • Level 2 — Priority Queues: Assign query priorities (1-10) based on customer tier. Low-priority queries yield CPU to high-priority ones. ClickHouse implements this via the priority session setting. The limitation: priorities are per-query, not per-user workload. A single high-priority query can still monopolize resources.
  • Level 3 — Resource Groups (v23.x+): ClickHouse 23.x introduced user-level and role-level resource constraints via CREATE RESOURCE CONSTRAINT. This allows capping CPU, memory, and I/O per user class. More granular than per-query limits — it limits the total resources a user class can consume concurrently.
  • Level 4 — Separate Clusters per Tier: Premium customers get dedicated ClickHouse clusters. Free tier customers share a cluster. This is operationally expensive but provides true isolation. ClickHouse Cloud's "dedicated" tier does exactly this.

Regulatory Data Residency

For GDPR (EU), CCPA (California), and sector-specific regulations (HIPAA, PCI-DSS), data residency requirements mandate that certain data never leaves a specific geographic boundary. ClickHouse's approach:

  • Physical isolation via separate clusters: EU customer data lives exclusively in Frankfurt/Amsterdam ClickHouse clusters. US customer data lives in us-east-1/us-west-2. The Distributed table's cluster configuration explicitly maps tenants to regional clusters. The routing logic is implemented at the application layer.
  • Column-level encryption: For PII fields (email, IP address), ClickHouse supports column-level encryption via the Encrypted codec. The encryption key can be per-tenant, stored in a KMS (AWS KMS, Google Cloud KMS), ensuring that even if the ClickHouse cluster is compromised, individual tenant data requires their specific encryption key to decrypt.
  • Right to be Forgotten (GDPR Article 17): This is a hard problem for ClickHouse. The recommended approach is a two-phase delete: immediately mark the user as deleted in a metadata table (Kafka event → ReplacingMergeTree), and schedule a background mutation (ALTER TABLE DELETE WHERE user_id = X) that physically removes the data within the regulatory deadline (typically 30 days). Monitor mutation progress via system.mutations.

What a "Google Spanner for ClickHouse" Would Look Like

If you were designing the hyperscale successor to ClickHouse, drawing on Google's internal systems, the key innovations needed would be:

  • TrueTime / Hybrid Logical Clocks: Replace wall-clock-based TTLs and partition functions with HLC-based monotonic timestamps. This enables correct cross-region consistent snapshots without needing to compensate for clock skew.
  • Globally Coordinated Compaction: Instead of each node running background merge independently, implement a globally-aware compaction scheduler (like Mesa's) that coordinates merges across the entire cluster to minimize total I/O amplification and ensure uniform part size distribution.
  • CBO with Data Statistics: A cost-based optimizer that collects column statistics (NDV, histogram, correlation) and uses them to make intelligent decisions about join order, aggregation strategy, and shard routing. ClickHouse's optimizer is primarily rule-based today.
  • Native ACID for Small Transactions: A lightweight transaction layer for insert+metadata updates that allows atomic "write event + update aggregate table" operations. Currently ClickHouse's lightweight transactions are experimental and limited to single-table, single-partition atomicity.
Back-of-Envelope: ClickHouse at Google Scale Google processes ~8.5 billion search queries/day. If each query generates 10 analytics events (impression, click, session, etc.) → 85B events/day → ~1M events/second. At 200 bytes/event compressed: 200 MB/s ingestion. Stored for 1 year: 200 MB/s × 86400 × 365 = ~6.3 PB/year. At 10:1 compression: ~630 TB on disk. A 50-node ClickHouse cluster with 20 TB NVMe each = 1 PB storage → 2 years of data. Query throughput: 1000 dashboard queries/second, each touching 1% of data = 6.3 TB each → at 14 GB/s per node across 50 nodes = 700 GB/s total → each query completes in ~9 seconds. With partition pruning to last 7 days: ~120 GB each → 0.17 seconds. This is the design target.
Key Takeaway: Google-scale ClickHouse requires moving beyond the traditional co-located storage+compute model toward disaggregated architecture (SharedMergeTree + S3), sophisticated multi-tenancy isolation (resource groups + dedicated clusters), and globally-coordinated metadata management. The ClickHouse Cloud product is converging on this architecture, but the open-source version still requires significant operational engineering to achieve hyperscale.

Step 4: Rapid-Fire Questions

Q1
What is the difference between PRIMARY KEY and ORDER BY in ClickHouse, and why does it matter for query performance?
In ClickHouse, ORDER BY defines the physical sort order of data within a part AND the columns indexed by the sparse primary index. PRIMARY KEY is an explicit subset of ORDER BY that tells ClickHouse which columns to include in the primary.idx file. When you set PRIMARY KEY (a, b) with ORDER BY (a, b, c), the data is physically sorted by (a, b, c) but the index only indexes (a, b). This allows the index to be smaller (omitting c) while still allowing efficient range scans using (a, b). The critical insight is that the LEFT-MOST columns of ORDER BY get the most benefit from the sparse index — a query filtering only on the 3rd ORDER BY column must scan ALL granules because the index can't narrow the range without knowing the preceding column values.
Q2
How does ReplacingMergeTree handle deduplication, and what are the correctness risks in production?
ReplacingMergeTree deduplicates rows with the same primary key (ORDER BY columns) during background merges, keeping only the row with the highest ver column value (or the last inserted row if no version column). The correctness risk: deduplication is NOT immediate — between insert time and the next merge, duplicate rows coexist in the table. A SELECT without FINAL will return duplicates. Adding FINAL forces an in-memory merge at query time, which is expensive O(total_rows) even for small result sets. The production pattern: use FINAL for correctness-critical queries, and implement application-side deduplication (GROUP BY primary key, MAX(version)) for performance-critical paths. As of v23.x, the do_not_merge_across_partitions_select_final setting can optimize FINAL for partitioned tables by performing deduplication within each partition independently in parallel.
Q3
Explain ClickHouse's "Too many parts" error and how to prevent it systematically.
Every INSERT in ClickHouse creates a new immutable "part" on disk. Background merge threads continuously merge small parts into larger ones, but if inserts arrive faster than merges can keep up, the number of parts per partition grows unboundedly. When parts exceed parts_to_delay_insert (default: 150), ClickHouse slows inserts artificially. Above parts_to_throw_insert (default: 300), all inserts are rejected. Prevention: (1) Always batch inserts to ≥10,000 rows per INSERT statement — this dramatically reduces parts/second. (2) Use Kafka + ClickHouse Kafka table engine or MaterializedPostgreSQL for batch ingestion. (3) Increase background_pool_size to allocate more CPU threads to background merges. (4) Use the Buffer table engine to absorb bursty small inserts and flush them in batches. Monitor via SELECT partition, count() FROM system.parts WHERE active GROUP BY partition.
Q4
When would you choose AggregatingMergeTree over a regular Materialized View with SummingMergeTree?
SummingMergeTree pre-aggregates rows by summing numeric columns during merges — it stores final aggregated values (e.g., total revenue = $500). AggregatingMergeTree stores partial aggregation states (e.g., a binary HyperLogLog sketch for COUNT(DISTINCT), or a quantile tdigest) that can be merged later. You choose AggregatingMergeTree when: (1) The aggregation function is not decomposable (e.g., exact median requires all data, but approximate quantiles using tdigest are decomposable — two tdigests can be merged). (2) You need COUNT(DISTINCT) — use uniqState() → stores HLL sketch → uniqMerge() at query time. (3) You need histograms, quantiles, or complex statistics that can't be expressed as simple sums. The practical rule: if your metric is SUM or COUNT, use SummingMergeTree. If it's COUNT(DISTINCT), quantile, or any statistical function, use AggregatingMergeTree with the corresponding *State/*Merge function pair.
Q5
How does ClickHouse's vectorized query execution differ from traditional row-at-a-time execution, and what hardware features does it exploit?
Traditional row-at-a-time execution (Volcano model) processes one row through the entire operator tree before fetching the next row — this is cache-unfriendly and can't exploit SIMD. ClickHouse uses vectorized execution where the query engine operates on "blocks" of 65,536 rows per column at a time. This means a filter operation loads 65,536 integers into L2/L3 cache and applies the filter using AVX2 SIMD instructions that compare 8 integers simultaneously — 8x speedup over scalar code. ClickHouse further compiles repetitive loop bodies using LLVM JIT compilation, eliminating function call overhead. Hardware features exploited: (1) AVX2/AVX-512 SIMD for parallel integer/float comparisons, (2) Cache-line prefetching for sequential column reads, (3) Branch-free SIMD operations for filter conditions, (4) POPCNT for efficient COUNT on bitmasks, (5) io_uring for async non-blocking I/O on Linux 5.1+. The result: a single CPU core can filter ~1 billion integers per second with a simple equality predicate.
Q6
How would you design a multi-tenant ClickHouse cluster serving 1000 customers with SLA isolation between customers?
A three-tier isolation architecture: (1) Logical isolation at the query layer — deploy CHProxy as a query router that authenticates customers, injects per-customer query limits (max_execution_time, max_memory_usage, max_rows_to_read) via ClickHouse user profiles, and implements per-customer query rate limiting. (2) Physical isolation via sharding key — shard by hashed customer_id so each customer's data lives on one shard. This means a customer's queries hit only one shard (no fan-out overhead), and a runaway customer query doesn't stress all shards. (3) Resource groups (v23.x+) — assign customer tiers (free/pro/enterprise) to resource constraint groups that cap total CPU shares and memory per tier group. Enterprise customers get higher CPU weights. For true noisy-neighbor isolation, deploy separate ClickHouse clusters per enterprise customer (expensive but correct). Monitor per-customer query latency via the query_log table partitioned by customer_id, and alert when any customer's P95 latency exceeds their SLA target.
Q7
What are skip indices (secondary indices) in ClickHouse, how do they work, and when are they effective?
Skip indices in ClickHouse are granule-level metadata structures that allow the query engine to skip entire granules (8192-row blocks) without reading the actual column data. Unlike B-tree secondary indices in RDBMS (which index individual rows), skip indices index per-granule aggregates. Types: (1) minmax — stores min/max value per granule; effective for range queries on numeric columns not in the primary key. (2) set(N) — stores up to N distinct values per granule; effective for low-cardinality equality filters. (3) bloom_filter(false_positive_rate) — probabilistic membership test per granule; effective for high-cardinality equality and IN list filters (e.g., search by user_id in a column not in the primary key). (4) ngrambf_v1 — n-gram bloom filter for LIKE/contains text search. Effectiveness conditions: the indexed column must have good "locality" — rows with the same value must cluster together in the granule. If values are randomly distributed, the skip index degrades (every granule passes the filter and nothing is skipped). Best paired with a correlated ORDER BY column that causes similar values to co-locate.
Q8
How does ClickHouse handle TTL (data expiration) for compliance use cases, and what are the operational risks?
ClickHouse TTL can be applied at column level (set column to DEFAULT value after expiry) or at row level (DELETE rows after expiry), and also for data tiering (MOVE TO DISK 'cold_disk' AFTER 90 DAYS). TTL evaluation happens during background merges — ClickHouse checks if any data in a part is expired and either deletes it or moves it. The operational risks: (1) TTL processing is not instantaneous — data persists until the next merge that touches the relevant part. In a low-write table, parts may not merge for months, and expired data lingers. Mitigation: OPTIMIZE TABLE ... FINAL forces a merge immediately (expensive on large tables). (2) TTL relies on system wall clock — clock skew between nodes can cause premature or delayed deletion. (3) In replicated setups, TTL merges are initiated by the merge leader; if the leader is behind, TTL-deleted data persists on other replicas longer. (4) Compliance risk: if a customer requests right-to-be-forgotten under GDPR, TTL alone is insufficient — you need a targeted mutation (ALTER TABLE DELETE WHERE user_id=X) AND a TTL as a safety net. Monitor TTL compliance via column-level checksums and compliance audit logs.
Q9
Explain ClickHouse's approach to handling JOINs in distributed queries and where it breaks down.
ClickHouse evaluates distributed JOINs using one of two strategies: (1) GLOBAL JOIN — the coordinator evaluates the right-side table (small table) in full, broadcasts the result to ALL shards, and each shard performs a local hash join. This is efficient when the right table is small (<100MB) and the left table is huge and sharded. (2) local JOIN (default) — each shard independently joins its local data with its local portion of the right table. This is ONLY correct if both tables are co-located (sharded on the same key). Using a local JOIN when tables are sharded differently produces INCORRECT results without any error — it silently misses cross-shard joins. Where it breaks down: (1) Large fact-to-fact joins (two billion-row tables) — GLOBAL JOIN broadcasts a huge dataset to all shards, overwhelming network. (2) Multi-hop JOINs (A JOIN B JOIN C) where B and C are on different shards. (3) JOINs with DISTINCT semantics across shards. The production solution: denormalize at ingestion time using Kafka Streams enrichment, eliminating the need for distributed JOINs in ClickHouse entirely. Reserve ClickHouse JOINs for dimension table lookups where the dimension fits in memory.
Q10
What is the ClickHouse Projection feature and how does it differ from a Materialized View?
A Projection is a hidden, automatically maintained secondary sort order within the same table part. When you define ALTER TABLE events ADD PROJECTION proj_by_market (SELECT market_id, event_type, count() GROUP BY market_id, event_type ORDER BY market_id), ClickHouse stores a secondary copy of the data (or aggregated data) within each part, sorted by market_id instead of the primary sort order. At query time, the optimizer automatically uses the projection if it determines the projection can answer the query more efficiently than the primary data. Key differences from Materialized Views: (1) Projections are stored INSIDE the same part as the main data — no separate table, no separate consistency management. (2) Projections are transactionally consistent with the main table (they're built together during the merge). (3) Projections can store aggregated states (GROUP BY), making them equivalent to AggregatingMergeTree MVs but without the separate table overhead. (4) Projections are limited to the same partition and part as the original data — they can't span tables. Use Projections when you have 2-3 common access patterns on the same table with different sort orders. Use MVs when you need independent lifecycle management, cross-table joins at materialization time, or separate partitioning strategy.
Key Takeaway: In a ClickHouse interview at L7/L8, the differentiator is always the combination of (1) deeply understanding WHY ClickHouse makes each design choice (columnar, sparse index, merge-tree), (2) knowing the failure modes and operational gotchas cold (too many parts, deduplication semantics, TTL timing), and (3) demonstrating that you've operated ClickHouse at scale and have war stories about what breaks. Architecture diagrams and back-of-envelope numbers are table stakes — the depth of operational knowledge is what separates L7 from L8.

Challenges and Trade-offs at L7/L8 Level

Challenge 1: Conversion Attribution Latency

Problem: Conversions happen up to 24 hours after a click. When a user clicks an ad at 9 AM and converts at 5 PM, the conversion event needs to be attributed back to the 9 AM click — and this attribution needs to update the 9 AM stats bucket in the materialized view. But ClickHouse materialized views are append-only at insert time and can't retroactively update past buckets.

Solution: Two-table pattern. Keep a "raw conversions" table alongside the materialized view. When a conversion event arrives, look up the original click in Redis (attribution cache), emit a "retroactive_credit" event with the original click's timestamp, and insert it into ClickHouse. The MV handles it as a new event in the past bucket. At query time, join the MV stats with the retroactive credits. The trade-off: query time complexity increases, and the "retroactive_credits" table can accumulate billions of rows over time. Alternative: Use CollapsingMergeTree — insert a negative-sign row to cancel the previous state, then a positive row with the new attribution.

Challenge 2: Peak Traffic Spike (Super Bowl 5x volume)

Problem: During a Super Bowl, ad event volume spikes from 5M/sec to 25M/sec for ~4 hours. The normal ingestion pipeline is designed for 5M/sec. The ClickHouse cluster can handle 5M/sec → 10 parts/sec → merges keep up. At 25M/sec → 50 parts/sec → merges fall behind → "Too many parts" error.

Solution: (1) Pre-warm: 6 hours before the event, stop all non-essential background merges (SYSTEM STOP MERGES) to free background threads, letting the merge pool accumulate capacity. (2) Kafka Lag Buffer: Kafka can absorb 4 hours of 25M/sec → 360 billion events. Kafka becomes the buffer, ClickHouse ingests at its natural rate, just with higher lag. Freshness SLA degrades from 30 seconds to 5 minutes during the spike — pre-negotiate this SLA exception with advertisers. (3) Horizontal scale pre-event: Add 2 shards 24 hours before the event, with the Distributed table config updated. New data routes to the new shards. (4) Reserve the Flink enrichment buffer: During spike, drop non-critical enrichment (geo-IP, device fingerprinting) to reduce Flink processing overhead — focus on core event ingestion at the cost of reduced dimension completeness.

Challenge 3: Advertiser Isolation for Security and Performance

Problem: A malicious or misconfigured advertiser runs a query that selects ALL events across the past year without time filter. This scans the entire table, consuming all CPU for 60 seconds and starving other advertisers' dashboard queries.

Solution: (1) Query proxy (CHProxy) with per-advertiser limits: max_execution_time=30s, max_rows_to_read=10B, max_memory_usage=10GB. (2) The advertiser's API key maps to a ClickHouse user profile with these limits. (3) Mandatory query rewriting: the API layer ALWAYS injects AND event_time >= now() - INTERVAL 90 DAY before sending to ClickHouse — no raw query passthrough. (4) Resource groups: free-tier advertisers share a resource group with 20% CPU weight; premium advertisers get 60% CPU weight; the remaining 20% is reserved for internal monitoring queries. The trade-off: the query proxy is a single point of failure (mitigated with CHProxy HA) and adds ~1ms latency (acceptable).
