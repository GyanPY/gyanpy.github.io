2.3

Variant: Slot Allocation Models

How BigQuery assigns and prices computational resources
A

On-Demand Slots

Shared pool of slots available to all BigQuery users, billed per-query based on bytes scanned.

How It Works
1
Submit queryYou send a SQL query to BigQuery
2
Slot estimationBigQuery estimates required slots based on data size and query complexity
3
Pool allocationSlots allocated from shared regional pool — your query competes with other tenants
4
Billing$5 per TB scanned — not slot-time. You pay for data read, not compute duration.
✅ WHEN IT SHINES
  • Sporadic / unpredictable query patterns
  • Small-to-medium data sizes (<10 TB queries)
  • Teams just starting with BigQuery
  • Cost-conscious with low query volume
❌ WHEN IT STRUGGLES
  • Consistent high-volume workloads (reserved is cheaper)
  • Latency-sensitive queries (may wait for slots)
  • Large concurrent workloads (slot contention)
B

Reserved Slots (Editions: Standard, Enterprise, Enterprise Plus)

Dedicated slot capacity purchased by commitment (per-second billing), guaranteeing resources for your queries.

How It Works
1
Purchase commitmentBuy a slot commitment — minimum 100 slots
2
Dedicated allocationSlots are dedicated to your organization, not shared
3
Capacity billingNo per-query billing — you pay for capacity regardless of usage
4
Reservation allocationAssign slots to specific projects or reservations for workload isolation
Slot Commitment Math — Break-Even Analysis
⚖️ Break-Even Calculation
1,000 reserved slots for 1 month
Cost: ~$20/slot/month = $20,000/month

Break-even vs on-demand:
$20,000 ÷ $5 per TB = 4,000 TB scanned

If you scan > 4,000 TB/month → Reserved is cheaper
If you scan < 4,000 TB/month → On-demand is cheaper

C

Flex Slots

Short-term slot commitments (minimum 60 seconds) for burst capacity — purchased and cancelled on-demand.

Use Case — Black Friday Example
Normal capacity: 500 reserved slots
Black Friday addition: 2,000 flex slots purchased
Duration: 7 days
Cost: 7 days × 2,000 slots × $0.04/slot-hour × 24h = $13,440
After event: Cancel — no ongoing cost

Decision Framework

⚡ Slot Model Selection Tree
Q1: Is your query volume predictable?
├── If NO  → Use On-Demand
│   └── Q2: Are you spending > $10K/month on-demand?
│       ├── If YES → Consider Reserved (calculate break-even)
│       └── If NO  → Stay on On-Demand
└── If YES → Consider Reserved
    └── Q3: Do you have burst periods?
        ├── If YES → Reserved + Flex Slots
        └── If NO  → Reserved only

Variant: Data Organization Strategies

Partitioning and clustering for dramatic query cost reduction
A

Partitioning

Physically divide a table into segments based on a column value (usually timestamp), so queries can skip entire partitions.

Partitioning Types
TypeColumn TypeGranularityMax PartitionsUse Case
Ingestion Time_PARTITIONTIMEDay/Hour/Month4,000Event data without timestamp column
Timestamp/DateDATE, TIMESTAMP, DATETIMEDay/Hour/Month/Year4,000Time-series data
Integer RangeINTEGERCustom ranges4,000ID-based partitioning
Concrete Example — Cost Impact
Table: web_events — 10 years of data, 365 TB total
❌ Without Partitioning
Query: WHERE date = '2024-01-15'
Scans: 365 TB (entire table)
Cost: $1,825
✅ With Daily Partitioning
Query: WHERE date = '2024-01-15'
Scans: 100 GB (one partition)
Cost: $0.50 · Savings: 99.97%

└────────────────────────────────────────────────────────┘

Partitioned Table (by date):
┌──────────────────────────────────────────────────────────────────────┐
│ 2024-01-01 │ 2024-01-02 │ 2024-01-03 │ ... │ 2024-12-31 │ 2025-... │

B

Clustering

Within each partition, sort and colocate data by specified columns to enable further data skipping via min/max statistics.

How Clustering Works
1
Data sorted by clustering columnsWithin each partition, rows are physically sorted by the specified columns
2
Storage blocks created~10 MB blocks contain contiguous sorted rows
3
Min/max metadata stored per blockEach block records min/max values for each clustering column
4
Query-time block pruningQuery can skip entire blocks where the filter can't match the stored min/max range
Concrete Example
EXAMPLE 
Table: web_events
- Partitioned by: date
- Clustered by: country, user_id

Partition: 2024-01-15 (100 GB, 1000 blocks)
├── Block 1:    country [AR, AU], user_id [1-10000]
├── Block 2:    country [AU, BE], user_id [10001-20000]
├── Block 3:    country [BE, BR], user_id [20001-30000]
...
└── Block 1000: country [US, ZW], user_id [990001-1000000]

Query: SELECT * FROM web_events 
       WHERE date = '2024-01-15' 
       AND country = 'US'

Without clustering: Scan all 1000 blocks (100 GB)
With clustering:    Scan only ~26 blocks where country min/max includes 'US' (2.6 GB)
Improvement: ~97% fewer bytes scanned
DIAGRAM 
Partition: 2024-01-15 (100 GB)
Without Clustering:
┌─────────────────────────────────────────────────────────────────┐
│ US │ FR │ US │ JP │ CN │ US │ BR │ US │ FR │ ... │ Randomly    │
│    │    │    │    │    │    │    │    │    │     │ distributed │
└─────────────────────────────────────────────────────────────────┘
Must scan everything to find all 'US' records

With Clustering (by country):
┌─────────────────────────────────────────────────────────────────┐
│ AR │ AU │ BE │ BR │ CN │ ... │ JP │ ... │ US │ US │ US │ ZW    │
│    │    │    │    │    │     │    │     │────────────────      │
└─────────────────────────────────────────────────────────────────┘
                                           ↑
                                    Only scan these blocks!
C

Partitioning + Clustering Decision

A combined decision framework for choosing the right data organization strategy.

⚡ Partition + Cluster Selection Tree
Q1: Do you have a timestamp or date column?
├── If YES → Partition by that column (almost always)
│   └── Q2: What other columns appear in WHERE clauses frequently?
│       ├── User_id        → Cluster by user_id
│       ├── Region/country → Cluster by region
│       └── Multiple       → Cluster by most selective first (max 4 columns)
└── If NO → Consider integer range partitioning or clustering only
    └── Q3: Do you filter by consistent columns?
        ├── If YES → Cluster by those columns
        └── If NO  → May not benefit from clustering

MechanismPhysical table divisionSorted blocks with metadata

Max columns14
Query benefitSkip entire partitionsSkip blocks within partitions
MaintenanceAutomaticAuto-reclustering
Cost impactHuge (partition pruning)Significant (block pruning)

A

Query Plan Stages

BigQuery executes queries in stages, each with distinct characteristics and cost drivers.

Stage Types
Stage TypeWhat It DoesWhen Expensive
InputRead from storage (Colossus)Large tables, no partition/cluster pruning
ComputeTransformations, filters, projectionsComplex expressions, UDFs
RepartitionRedistribute data (hash partitioning)Large intermediate results, many partitions
AggregateGROUP BY, COUNT, SUM, etc.High cardinality groups
JoinCombine tablesLarge tables, no broadcast possible
SortORDER BYLarge result sets
OutputWrite resultsLarge results, many files
How to Read a Query Execution Plan
EXPLAIN OUTPUT 
EXPLAIN SELECT country, SUM(amount) 
FROM orders 
WHERE date = '2024-01-01'
GROUP BY country;

Query Plan:
Stage 1 (Input): Read orders table
- Partition pruning: YES (date = '2024-01-01')
- Rows read: 1M (from single partition)
- Slots used: 50

Stage 2 (Aggregate): Partial GROUP BY
- Input rows: 1M
- Output rows: 200 (countries)
- Slots used: 50

Stage 3 (Aggregate): Final GROUP BY  
- Input rows: 10K (from all partial aggregates)
- Output rows: 200
- Slots used: 1
B

Join Strategies

BigQuery automatically selects between broadcast join and hash join based on table sizes — understanding this drives optimization decisions.

Broadcast Join

Mechanism

Small table (< 10 MB) is broadcast (copied) to every worker. Large table is scanned in parallel across all workers. No shuffle required.

DIAGRAM — BROADCAST JOIN 
   ┌─────────────────────────────────────────────────────────┐
   │                    Large Table (1 TB)                    │
   │ ┌────────┐ ┌────────┐ ┌────────┐       ┌────────┐      │
   │ │Worker 1│ │Worker 2│ │Worker 3│  ...  │Worker N│      │
   │ │        │ │        │ │        │       │        │      │
   │ │ Small  │ │ Small  │ │ Small  │       │ Small  │      │
   │ │ Table  │ │ Table  │ │ Table  │       │ Table  │      │
   │ │(copied)│ │(copied)│ │(copied)│       │(copied)│      │
   │ └────────┘ └────────┘ └────────┘       └────────┘      │
   └─────────────────────────────────────────────────────────┘
   
No shuffle needed! Just copy small table to each worker.

Hash Join (Shuffle Join)

Mechanism

Both tables are hashed by join key. Rows with matching keys are routed to the same shuffle bucket. Each worker then joins one bucket locally. Shuffle required — expensive but scalable.

DIAGRAM — HASH JOIN 
   Orders (1 TB)                    Users (100 GB)
   ┌──────────────┐                 ┌──────────────┐
   │ Hash by      │                 │ Hash by      │
   │ user_id      │                 │ user_id      │
   └──────┬───────┘                 └──────┬───────┘
          │                                │
          ▼                                ▼
   ┌──────────────────────────────────────────────────┐
   │              Shuffle Buckets                      │
   │ ┌──────┐ ┌──────┐ ┌──────┐       ┌──────┐        │
   │ │Bkt 0 │ │Bkt 1 │ │Bkt 2 │  ...  │Bkt N │        │
   │ │O + U │ │O + U │ │O + U │       │O + U │        │
   │ └──────┘ └──────┘ └──────┘       └──────┘        │
   └──────────────────────────────────────────────────┘
          │
          ▼
   Each bucket joined independently by one worker

C

Materialized Views

Pre-computed query results that BigQuery automatically maintains and transparently uses to answer matching queries.

How It Works
SQL — MATERIALIZED VIEW 
-- Create materialized view
CREATE MATERIALIZED VIEW sales_by_region AS
SELECT region, DATE(order_date) as date, SUM(amount) as total
FROM orders
GROUP BY region, DATE(order_date);

-- This query automatically uses the MV:
SELECT region, SUM(amount) 
FROM orders 
WHERE order_date >= '2024-01-01'
GROUP BY region;
-- BigQuery rewrites to read from sales_by_region
Maintenance Details
01
Auto-Maintained

BigQuery maintains materialized views automatically with incremental refresh — no manual intervention needed.

02
Refresh Delay

Near real-time for streaming inserts; immediate for batch DML. Freshness depends on source write pattern.

03
Storage Cost

Charged like regular tables — you pay for the pre-aggregated data stored in Colossus.

04
Query Cost

Significantly reduced by reading pre-aggregated data instead of scanning the full source table.
