Part 1

Core Understanding

Make It Click

1.1 The Elevator Pitch

For a smart non-engineer

BigQuery is like having a massive team of workers who can instantly split up any question you ask, each tackle a tiny piece simultaneously, and combine their answers in seconds—no matter how much data you're asking about.

For a fellow engineer

BigQuery is a serverless, petabyte-scale analytics warehouse built on Dremel (columnar execution engine) and Colossus (distributed file system), using a separation of storage and compute architecture with aggressive column pruning and tree-structured aggregation to achieve sub-minute queries over multi-petabyte datasets.

💡 The "Aha Moment"

The magic isn't just parallelism—it's the combination of columnar storage (only read what you need), disaggregated architecture (storage and compute scale independently), and tree-shaped query execution (thousands of workers aggregate upward like a tournament bracket).


Traditional databases are like one librarian reading entire books to answer questions; BigQuery is like having 10,000 librarians who each memorize one column of data and can instantly shout up their partial answer through a hierarchy of summarizers.

1.2 The Perfect Analogy

The Concert Census Analogy

Imagine you need to count how many people wearing blue shirts attended a 100,000-person concert, organized by section.

Traditional Database (Row-Store)

You have ONE census taker who must walk to every single person, look at their entire outfit (shirt, pants, shoes, hat, accessories), write it all down, then check if the shirt was blue. They visit 100,000 people, writing 10 fields each = 1,000,000 data points examined.

BigQuery (Columnar)

You have a TEAM organized like this:

  • Storage Layer (Colossus): 100 photographers have each taken pictures of ONLY shirts—nothing else. Each photo is labeled with the section.
  • Compute Layer (Dremel Workers): 100 shirt-counters, each assigned 1,000 photos. They count blues in parallel.
  • Mixer Tree: Results flow up: 100 workers → 10 regional managers (aggregate) → 1 final manager (combines all).

Total data examined: 100,000 shirt photos × 1 field = 100,000 data points. Time: Seconds (parallel) vs hours (sequential).

Extending to Main Variants

ConceptConcert Analogy
Column PruningPhotographers only sent shirt photos, not full outfits
Partition PruningIf you only want Friday concert data, ignore Saturday photo albums entirely
ClusteringWithin each section, photos are pre-sorted by shirt color—blue shirts are already grouped together
SlotsEach counter is a "slot"—Google dynamically assigns counters based on workload
ShuffleWhen counting "blues per section," counters must pass their section data to the right regional manager—like physical mail sorting

Failure Scenarios Through the Analogy

⚡ Worker (Slot) Failure AUTO-RETRY

One counter collapses. The manager notices missing reports, reassigns that batch of 1,000 photos to another counter. Other counters unaffected.

🗄️ Storage (Colossus) Failure REPLICATED

Photos are stored in triplicate across three warehouses. If one warehouse burns down, photos retrieved from backup warehouse—query continues.

🔀 Shuffle Overload PARALLELISM

Too many counters trying to pass papers to one regional manager creates a bottleneck. Solution: add more regional managers (increase shuffle parallelism) or reduce what's being passed (push aggregation earlier).

🔥 Hot Partition PRE-SPLIT

Everyone wore blue on Friday. The Friday regional manager drowns in reports while others are idle. Solution: pre-split the Friday section.

1.3 The Mental Model

The 5 Core Concepts You MUST Internalize

01
Separation of Storage and Compute

Data lives in Colossus, completely separate from query execution. Run 1,000 queries simultaneously on the same data—no locks, no contention. Pay and scale independently.

02
Columnar Storage (Capacitor Format)

Data stored column-by-column. SELECT name FROM users reads only the name column. Columns compress better and enable vectorized processing.

03
Tree-Structured Query Execution (Dremel)

Root (coordinator) → Mixers (intermediate aggregators) → Leaf workers (data scanners). Thousands of leaves scan in parallel, aggregate upward. Tree depth: typically 3–4 levels.

04
Slots = Units of Computational Resource

A slot is roughly: 1 vCPU + some RAM + network capacity. BigQuery dynamically allocates slots from shared pool (on-demand) or dedicated pool (reserved). Query parallelism is slot-limited.

05
Shuffle = Data Redistribution Between Stages

GROUP BY, JOIN require data to shuffle between workers. Expensive—writes to disk, network transfer. BigQuery uses Colossus for shuffle storage: durable, not ephemeral like Spark shuffle.

What BigQuery Guarantees vs. Does Not Guarantee

✅ GUARANTEES
  • Strong read-after-write consistency
  • ACID transactions within DML statements
  • Exactly-once query execution
  • Automatic fault tolerance (worker failures transparent)
  • Petabyte-scale queries without cluster management
❌ DOES NOT GUARANTEE
  • Low latency for small queries (min ~200ms overhead)
  • Point-lookup efficiency (analytics, not OLTP)
  • Infinite concurrent slots (contention under heavy load)
  • Predictable query performance (data distribution matters)
  • Real-time freshness (streaming insert ~few seconds delay)

Fundamental Trade-offs Baked In

Trade-offBigQuery ChoosesSacrificesWhy
Throughput vs LatencyHigh throughputSub-100ms latencyAnalytics workloads favor throughput
Flexibility vs EfficiencySchema-on-read flexibilitySome scan efficiencyAnalysts need ad-hoc queries

  • Queries scan large portions of tables (not point lookups)
  • Query patterns are unpredictable (ad-hoc analytics)
  • Read:write ratio is >100:1
  • You care about total throughput, not p99 latency
  • Data is append-mostly (not frequent updates)

2.1 Variant: Storage Architecture

A

Capacitor (Storage Format) vs Other Columnar Formats

Capacitor is Google's proprietary columnar storage format that stores each column separately with heavy compression and embedded statistics for query optimization.

The Core Insight

Unlike Parquet which stores row groups (~128MB chunks of multiple columns), Capacitor is more aggressively columnar with finer-grained statistics and is optimized for Dremel's query patterns.

The Inventor's Problem

Google needed a format that could enable predicate pushdown at extremely fine granularity, support nested/repeated fields (Protocol Buffers style), and compress to minimize Colossus storage costs.

B

Colossus vs Other Distributed File Systems

Colossus is Google's next-generation distributed file system (successor to GFS) that provides durable, high-throughput storage with location-aware data placement.

How It Works — Step by Step
Write Path
1
Data Splitting Data split into chunks (~64MB)
2
Replication / Erasure Coding Each chunk replicated to 3+ storage nodes (Reed-Solomon erasure coding for large files)
3
Metadata Storage Metadata stored separately in Bigtable-like metadata service
4
Acknowledgement Write acknowledged when replication is complete
Read Path
1
Chunk Location Lookup Query worker asks metadata service for chunk locations
2
Nearest Replica Read Worker reads from nearest replica (datacenter-aware)
3
Lock-Free Parallel Access Multiple workers can read same chunk simultaneously (no locks)
Fault Tolerance
!
Failure Detection & Re-replication Chunk server fails → read from replica. Background scan detects under-replicated chunks. Re-replicate to maintain redundancy.
Concrete Example
EXAMPLE 
Table: user_events (100 TB)
├── event_id column:   10 TB → stored as 156,250 chunks (64 MB each)
├── timestamp column:   5 TB → 78,125 chunks
├── user_id column:     8 TB → 125,000 chunks
└── payload column:    77 TB → 1,203,125 chunks

Each chunk replicated 3x → actual storage: 300 TB

Query: SELECT user_id, timestamp FROM user_events WHERE timestamp > '2024-01-01'
- Only reads: user_id + timestamp columns = 13 TB (not 100 TB!)
- Colossus returns chunks from nearest replicas
- 10,000 workers each read ~1.3 GB
Colossus vs HDFS
AspectColossusHDFS

ReplicationErasure coding for large filesAlways 3x replication
Compute localityLess important (network fast)Critical (co-locate compute)
ScalabilityExabytesPetabytes (NameNode limit)

2.2 Variant: Query Execution Models

A

Dremel Tree Execution

Dremel executes queries through a multi-level tree where leaf nodes scan data in parallel and intermediate "mixer" nodes aggregate results upward.

The Core Insight

Instead of shuffle-heavy MapReduce stages, Dremel uses a persistent tree topology where aggregation happens incrementally as results flow upward—reducing shuffle data dramatically.

Step-by-Step Walkthrough

Query: SELECT country, COUNT(*) FROM web_logs GROUP BY country

TREE STRUCTURE (3 levels) 
Level 0: Root Server (1)
         └── Final aggregation: combines results from Level 1

Level 1: Mixer Servers (50)
         └── Partial aggregation: combine results from ~40 leaves each

Level 2: Leaf Servers (2,000)
         └── Scan data, local pre-aggregation
Concrete Example with Numbers
EXECUTION TRACE 
Table: web_logs
- Size: 10 TB
- Rows: 100 billion
- Countries: 200 unique values

Execution:
1. Root creates execution plan, distributes to tree
2. 2,000 leaf servers each scan 5 GB (10 TB / 2,000)
3. Each leaf pre-aggregates: 50M rows → 200 country counts
4. Each leaf sends 200 counts (~3 KB) to its mixer
5. Each mixer aggregates 40 leaves: 40 × 200 = 8,000 counts → 200 counts
6. Root aggregates 50 mixers: 50 × 200 = 10,000 counts → 200 final counts

Shuffle data: Only 2,000 × 3KB + 50 × 3KB = ~6 MB
(Compare to shuffling 100B rows through traditional approach!)

Where It Shines vs. Struggles
✅ WHERE IT SHINES
  • Aggregation queries (GROUP BY reduces data)
  • Queries with high selectivity filters (WHERE clause)
  • Single large table scans
❌ WHERE IT STRUGGLES
  • Large JOINs (requires shuffle to different tree)
  • Results with high cardinality (millions of groups)
  • Complex multi-stage transformations
B

Shuffle-Based Execution (Dremel + Shuffle)

For operations requiring data redistribution (JOINs, high-cardinality GROUP BY), BigQuery uses Colossus-backed shuffle that writes intermediate results to disk.

The Core Insight

Unlike Spark's ephemeral shuffle (lost on failure), BigQuery writes shuffle data to Colossus, making it durable and enabling automatic retry without recomputation.

Step-by-Step Walkthrough for JOIN
JOIN EXECUTION STAGES 
Query: SELECT o.*, u.name 
       FROM orders o 
       JOIN users u ON o.user_id = u.user_id

Stage 1: Scan + Hash Partition
├── Workers read orders table, hash(user_id) → write to shuffle buckets
└── Workers read users table, hash(user_id) → write to shuffle buckets

Stage 2: Join Execution
├── Each join worker reads ONE bucket from orders + ONE bucket from users
├── Same hash = same bucket = can join locally
└── Output written to result set

Shuffle Data Written to Colossus:
- Orders: 1 TB → 1 TB shuffle files (partitioned by hash)
- Users: 100 GB → 100 GB shuffle files (partitioned by hash)
- Total shuffle: 1.1 TB written to persistent storage
