Industry Deep-Dive

Google BigQuery — Dremel + Colossus + Jupiter ▾
Architecture: Four Layers of Google Infrastructure
  • Dremel: Distributed SQL execution engine — the query optimizer, slot scheduler, and execution graph coordinator
  • Colossus: Google's distributed file system (successor to GFS) — stores Capacitor columnar files, handles shuffle intermediates
  • Borg: Google's cluster management system (predecessor to Kubernetes) — allocates physical machines to Dremel workers
  • Jupiter: Google's datacenter network fabric — provides the high-bandwidth, low-latency shuffle backbone that makes distributed joins practical
Key Design Decisions That Shaped BigQuery
  • Separation of concerns: Storage team (Colossus) vs Compute team (Dremel) vs Query team (optimizer) — allows each to evolve independently
  • Multi-tenancy: Shared infrastructure with isolation via slots — same physical hardware serves millions of queries from different customers
  • Serverless model: Users manage SQL queries, not clusters. Google manages everything else.
  • SQL-first: Leverages 40+ years of SQL optimization research — cost-based optimizer, join reordering, predicate pushdown
Public Scale Numbers
Scans:    Petabytes per second across the Dremel fleet
Queries:  Millions per day across all customers
Latency:  Sub-second queries on terabyte datasets (common case)
Slots:    Hundreds of thousands of slots across Google's fleet
Storage:  Exabytes of customer data on Colossus
Amazon Redshift + Athena — Two Approaches ▾
Redshift — Provisioned MPP Cluster
  • Originally provisioned clusters — you pick node type (ra3, dc2) and count upfront
  • MPP (Massively Parallel Processing) architecture with columnar storage
  • Requires capacity planning: over-provision = waste money, under-provision = slow queries
  • Now has Serverless option (added 2022) but it's an add-on, not the native model
Athena — Serverless Presto Over S3
  • Serverless Presto engine that queries data directly from S3
  • Pay per query (like BigQuery on-demand) — no cluster to manage
  • Open formats: Parquet, ORC, Avro — no proprietary lock-in
  • Less integrated optimization than BigQuery — relies on data format and partitioning
  • No streaming inserts natively — needs Kinesis/Firehose pipeline

Meta / Facebook Presto → Trino — Open Source MPP ▾
Architecture: Open Source Data Lake SQL
  • Presto (now Trino): Interactive SQL engine — open source, no vendor lock-in
  • Hive: Batch processing — MapReduce-based, handles heavy ETL
  • S3-compatible storage: ORC and Parquet files — fully open formats
  • Hive Metastore: Separate metadata layer — stores schema, partition info
Key Differences from BigQuery — Where Trino Falls Short
  • Memory-based execution: Faster for data that fits in cluster memory, but fails when memory runs out (no Colossus shuffle persistence)
  • No built-in shuffle persistence: A failed worker during a large shuffle = recompute from scratch. BigQuery writes shuffle to Colossus = transparent recovery.
  • Requires separate orchestration: Airflow or similar for pipeline scheduling. BigQuery has Scheduled Queries and integrated workflows.
  • Portable: Run on your own AWS/GCP/Azure infrastructure. BigQuery is Google-only.

L6/L7 Insight — When to Recommend OSS Over BigQuery

"OSS solutions win when: data sovereignty or vendor lock-in is a hard constraint (regulated industries), open format portability is required (multi-cloud data mesh), or the query pattern matches a specialized engine (Druid for sub-second OLAP, ClickHouse for high-insert dashboards). BigQuery wins when: operational simplicity matters, Google's integrated ecosystem (Dataflow, Vertex AI, Looker) is valuable, or scale exceeds what you can manage operationally."

L6/L7 Insight — The $50K vs $2M Architecture

"The architecture that works at 1 TB will bankrupt you at 1 PB. At startup scale, BigQuery is just a SQL database that never runs out of disk. At Google scale, it's a carefully orchestrated system where every layer — partitioning, clustering, materialized views, slot allocation — must be tuned together. The difference between $50K/month and $2M/month in BigQuery spend is usually just schema design and query patterns."

Red Flags to Avoid

❌ "Just use BigQuery" without understanding the workload
Shows: Lack of critical thinking — defaulting to a tool without requirements analysis
Better: "BigQuery is right if we have ad-hoc analytics with >1s latency tolerance. If we need sub-100ms, we'd pair it with Bigtable or BI Engine for the serving layer."
❌ "We'll just add more slots"
Shows: Doesn't understand the cost model — slots affect parallelism, not scan volume
Better: "More slots reduce query time but don't reduce cost — bytes scanned determines cost. The optimization is at query/schema level, not slot count."
❌ "Partition by everything"
Shows: Doesn't understand the 4,000 partition limit or cardinality issues for clustering
Better: "Partition on the one dimension most queries filter — usually date. Cluster for secondary dimensions. Clustering supports up to 4 columns; pick the highest-frequency filter columns."
❌ "Real-time means streaming inserts"
Shows: Doesn't understand BigQuery's streaming buffer latency (~10 minutes for MVs)
Better: "Streaming inserts have ~10s table read latency. Materialized views see streaming data only after batch commit (~10 min). For true real-time serving, Bigtable or Redis; BigQuery for analytics."
❌ "BigQuery is strongly consistent so we're fine globally"
Shows: Doesn't understand multi-region replication lag
Better: "Strong consistency is within a region. Cross-region replication is async with observable lag. For consistency-critical workloads, single-region with failover — not multi-region active-active."
❌ Not asking about query patterns before schema design
Shows: Schema-first thinking instead of workload-first — the schema encodes query assumptions
Better: "Before I design the schema, let me understand the top 5 queries by frequency. The partition key, clustering columns, and materialized views should all be derived from those patterns."

7 Impressive Nuances to Demonstrate

Nuance 1Slot Pool Segmentation

Segment reservations by workload type — dedicated slots for SLA-bound dashboards, background slots for ETL with idle slot sharing. A large ETL consuming all slots while dashboards queue is an organizational problem solved by infrastructure isolation, not by adding more slots.

Nuance 2HyperLogLog for Cardinality

For COUNT(DISTINCT user_id) on billions of rows, exact computation requires massive memory and shuffle. APPROX_COUNT_DISTINCT uses HyperLogLog++ with ~1% error at 10x performance. For 99% of analytics use cases, the error is invisible to business decision-making.

Nuance 3Streaming Buffer Visibility

Data in the streaming buffer isn't immediately visible to materialized views or table snapshots. There's a ~10-minute window where streaming data is in a different consistency domain. Affects dashboards claiming "real-time" on MV-powered queries.

Nuance 4Partition Filter Enforcement

Enable require_partition_filter on all large tables to prevent accidental full scans. One bad query without a partition filter on a petabyte table costs $5,000. Governance isn't optional at scale — it's enforced at the schema level.

Nuance 5Clustering Column Order

Clustering uses leftmost columns first — only a prefix of clustering columns enables block pruning. Clustering by (country, user_id) provides zero benefit for queries filtering on user_id alone. Order must match query patterns, not just list the important columns.

Nuance 6Shuffle Persistence to Colossus

Unlike Spark where shuffle is ephemeral (failed executor = recompute from source), BigQuery shuffle writes to Colossus. This makes large joins more resilient to worker failures, but means shuffle writes have higher latency than in-memory shuffle alternatives.

Nuance 7BI Engine for Sub-Second Dashboards

For dashboard workloads needing sub-second latency, BI Engine caches data in memory and changes BigQuery from 1–10s queries to ~100ms queries for cached data. It's not a replacement for query optimization — it's an in-memory acceleration layer layered on top of properly partitioned, clustered tables. Best deployed for known high-frequency dashboard queries.

Clarifying Questions to Ask

"What are the primary query patterns and top queries by frequency?"
Shows: Workload-driven schema design thinking — schema is derived, not guessed
Determines: Partition column, clustering order, which materialized views to build, where to add indexes
"What's the acceptable query latency — is sub-second required?"
Shows: Understanding that BigQuery has a latency floor (~1–5s typical)
Determines: Whether BigQuery is the right tool, or needs BI Engine, or needs a different serving layer entirely
"What's the read-to-write ratio and write frequency?"
Shows: Understanding BigQuery's analytics optimization (read-heavy) vs write challenges
Determines: Whether to use batch loading or streaming inserts, and which streaming architecture (if any)
"How predictable is the query load — bursty or steady?"
Shows: Thinking about slot allocation strategy and cost optimization
Determines: On-demand (unpredictable/bursty) vs Reserved (steady, high-volume) vs Flex slots (burst on top of reserved)
"What are the data retention and compliance requirements?"
Shows: Thinking about storage cost, compliance conflicts, and time travel configuration
Determines: Partition expiration settings, time travel window, GDPR deletion vs audit retention conflict design
"Who are the users and what's their SQL proficiency?"
Shows: Thinking about query efficiency, governance, and cost accountability
Determines: Whether to enable require_partition_filter, whether to use authorized views, whether to build abstractions over raw tables

Sound Bite Collection

On the Core Concept

"BigQuery is three innovations in one: columnar storage that only reads what you query, separated storage and compute that allows thousands of parallel workers, and tree-structured execution that aggregates results efficiently without sending all data to a single node."

On Partition vs Clustering

"Partitioning is your coarse-grained knife — it eliminates entire date ranges before a single byte is scanned. Clustering is your scalpel — it finds the exact blocks within a partition. Use both, in that order. Never cluster without partitioning."

On Trade-offs

"BigQuery trades latency for throughput and simplicity for control. If you need sub-100ms, you're using the wrong tool. But if you need to query a petabyte in under a minute without managing infrastructure, nothing else comes close."

On Scale and Cost

"The architecture that works at 1 TB will bankrupt you at 1 PB. Schema design isn't just about correctness — it's about economics. A well-designed table can be 100x cheaper to query. I've seen companies spend 10x their necessary BigQuery budget because they didn't enforce partition filters."

On Failures

"BigQuery handles node failures transparently. What it doesn't handle is bad queries. A single full-table-scan on a petabyte table costs $5,000 and takes an hour. The failure mode you design for is the human writing the query — not the hardware."

On Distributed Complexity

"The shuffle is where distributed systems get expensive. Every GROUP BY and JOIN potentially shuffles data across the network. The art is pushing computation as close to the data as possible — filter early, aggregate early, join late."

On Operational Maturity

"Monitoring BigQuery isn't about uptime — it manages itself. It's about cost efficiency. Track bytes scanned per dollar of business value. If a dashboard costs $1,000/day to refresh but influences $100 in decisions, something's structurally wrong."

On Common Mistakes

"The most common BigQuery mistake is schema design that ignores query patterns. Second is not using approximate functions for high-cardinality aggregations. Third is streaming inserts when batch loading would suffice — streaming costs 10x more per row."

On the Key Insight

"BigQuery isn't a database you query — it's a database you design queries for. The schema IS the performance tuning. Get it right and you pay 4 cents for a query. Get it wrong and you pay $25 for the same answer."
