Sample Interview Problem

Design Problem
Analytics Platform for a Global E-Commerce Company
Functional Requirements
  • Product managers query any historical data with SQL
  • Dashboards refresh every 5 minutes with key metrics
  • Data scientists run ad-hoc queries for experiments
  • Finance team needs daily revenue reports
  • Real-time fraud detection on transactions
Non-Functional Requirements
  • Ad-hoc queries complete in <60s P95
  • Dashboard queries <5s P95
  • Data queryable within 15 minutes of generation
  • 3-year data retention
  • Cost-efficient at scale
Scale Numbers
  • 500M products
  • 100M daily active users
  • 2B transactions/day × 2KB = 4 TB/day
  • 50B page view events/day × 500B = 25 TB/day
  • Total: ~30 TB/day ingestion
  • 3-year retention: ~33 PB
6 Regions, Multiple Teams
  • Teams across 6 geographic regions
  • All wanting to analyze the same data
  • Different query patterns per team
  • Governance and cost attribution needed

Ambiguities to Clarify First

The strong hire answer starts with questions, not schema. These four ambiguities materially change the architecture.

"What's the acceptable latency for fraud detection?"
If <100ms → Real-time fraud needs a separate system (Bigtable + ML serving). BigQuery is not the tool.
If minutes → Can use BigQuery streaming + BigQuery ML. Single platform.
"Do all 6 regions need all data, or only their local data?"
All data → Multi-region BigQuery dataset with cross-region replication. Higher cost, ~2x storage.
Local data → Regional BigQuery instances with data federation for cross-region analysis. Lower cost, adds query complexity.
"Are the dashboards pre-defined or fully ad-hoc?"
Pre-defined → Materialized views + BI Engine. Dashboard queries become sub-second.
Ad-hoc → More flexible optimization needed. Clustering strategy must cover wider query patterns.
"How much historical data is queried frequently?"
Recent 30 days → Hot partitions (recent SSD-backed) + aggressive clustering. Cold data can be cheaply archived.
Full 3 years equally → Different optimization — tiered storage less helpful, need broader clustering strategy.

Step 2 BigQuery Table Design — Schema is the Optimization ▾
BigQuery SQL — events table definition
CREATE TABLE events (
  event_id          STRING      NOT NULL,
  event_timestamp   TIMESTAMP   NOT NULL,
  user_id           STRING,
  event_type        STRING,        -- 'page_view' | 'transaction' | 'search' | 'add_to_cart'
  product_id        STRING,
  session_id        STRING,
  region            STRING,        -- 'us-east' | 'eu-west' | 'ap-south' etc.
  properties        JSON,          -- Flexible schema for arbitrary event properties
  -- Nested structs for type-specific data (avoid separate tables)
  page_view         STRUCT,
  transaction       STRUCT
)
PARTITION BY DATE(event_timestamp)
CLUSTER BY event_type, user_id, product_id
OPTIONS(
  partition_expiration_days = 1095,  -- 3 years exactly
  require_partition_filter = TRUE     -- CRITICAL: prevent $5K accidental scans
);

-- Why this clustering order?
-- (event_type, user_id, product_id)
-- 90% of queries filter on event_type first
-- User-specific analysis filters event_type + user_id
-- Product analysis filters event_type + product_id
-- Leftmost columns must match most common filter pattern
Why STRUCT Over Separate Tables

Joining event table with a transactions table requires a JOIN on 50B rows — expensive shuffle. Nested STRUCTs keep all event data in one row, enabling columnar pruning within the struct without a cross-table join. BigQuery reads only the struct columns you query.

Step 3 Materialized Views — Dashboard <5s SLA ▾
BigQuery SQL — Materialized View for Dashboard
-- Pre-aggregate for dashboard performance
-- This MV is incrementally refreshed every 5 minutes
CREATE MATERIALIZED VIEW daily_revenue_by_region
OPTIONS(
  enable_refresh = TRUE,
  refresh_interval_minutes = 5
) AS
SELECT
  DATE(event_timestamp)          AS date,
  region,
  SUM(transaction.amount)        AS total_revenue,
  COUNT(DISTINCT user_id)        AS unique_customers,
  COUNT(*)                       AS transaction_count,
  APPROX_COUNT_DISTINCT(user_id) AS approx_unique_customers  -- 10x faster cardinality
FROM events
WHERE event_type = 'transaction'
GROUP BY 1, 2;

-- Additional MVs for other dashboard panels:
-- daily_top_products_by_region (APPROX_TOP_COUNT)
-- hourly_fraud_rate_by_payment_method
-- weekly_cohort_retention (pre-computed rolling window)

-- Cost impact:
-- Without MV: 50B rows × daily dashboard refresh = expensive per-query cost
-- With MV: Pre-aggregated to region × date grain = tiny scan, <1s response
L6/L7 Insight — MV Refresh vs Streaming Buffer Lag

"Materialized views only see committed data — not the streaming buffer. With a 5-minute refresh interval and ~10-minute streaming buffer commit time, dashboards may show data up to 15 minutes old. This satisfies the '15 minutes to queryable' NFR exactly. If the requirement were stricter, we'd query the base table directly for the most recent window and merge with MV for historical."

Step 4 Slot Strategy — Reserved + Flex ▾
Reserved Slots (Enterprise Edition)
├── Dashboard Reservation: 500 slots — HIGH priority
Serves: 5-min dashboard refreshes via MVs
SLA: <5s P95 | Never preempted by ETL | Dedicated
│
├── Analyst Reservation: 1,000 slots — MEDIUM priority
Serves: Data scientist ad-hoc queries, PM analysis
SLA: <60s P95 | Can borrow idle Dashboard slots
│
├── ETL Reservation: 2,000 slots — LOW priority, idle sharing
Serves: Daily aggregation pipelines, ML training jobs
SLA: Best effort | Borrows from Analyst when idle
│
└── Flex Slots: On-demand for burst periods
Serves: End-of-quarter reporting, annual analysis, Black Friday peaks
Billed per second | Auto-provision | Decommission after burst
Why Segment Reservations?

Without segmentation, a single large ETL pipeline consuming all 3,500 slots will block dashboard refreshes, causing SLA violations. Reservations provide hard isolation: the Dashboard pool guarantees 500 slots are always available for 5-second refreshes, regardless of what ETL is doing.

The Business Case: $1.4M/month Savings

Reserved slots at $415K/month vs on-demand at $1.83M/month = $1.4M/month savings ($16.8M/year). Break-even calculation: 3,500 slots × $20 = $70K/month reserved. On-demand equivalent: $1.5M/month compute. Reserved makes sense at any scan volume above ~14 TB/month for this slot count.

Step 6 Monitoring & Governance ▾
Five Governance Layers
  • Query cost attribution by team: BigQuery labels on every query. Dashboard: cost per team per day. Creates accountability without permission changes.
  • Partition filter enforcement: require_partition_filter = TRUE on all tables >1 TB. Prevents $5K accidental full scans. Enforced at schema level, not policy level.
  • Slow query alerts (>60s): INFORMATION_SCHEMA.JOBS monitoring. Alert fires → auto-label query → notify team. Identifies optimization candidates.
  • Cost anomaly detection: Daily spend per team vs 7-day rolling average. 3σ deviation triggers PagerDuty to data engineering on-call.
  • Slot utilization dashboards: Per-reservation slot utilization P50/P95. If Dashboard reservation >80% sustained → add slots before SLA degrades.
BigQuery SQL — Cost Attribution Query
-- Run daily: cost per team, identify top spending queries
SELECT
  labels['team']                           AS team,
  DATE(creation_time)                      AS query_date,
  COUNT(*)                                 AS query_count,
  SUM(total_bytes_processed) / POW(10,12)  AS tb_scanned,
  SUM(total_bytes_processed) / POW(10,12) * 5.0 AS estimated_cost_usd,
  MAX(total_slot_ms / 1000 / 60)           AS max_slot_minutes
FROM
  `region-us`.INFORMATION_SCHEMA.JOBS
WHERE
  creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 1 DAY)
  AND job_type = 'QUERY'
  AND state = 'DONE'
GROUP BY 1, 2
ORDER BY estimated_cost_usd DESC;

The 5-Minute Refresher

Architecture Capacitor (columnar storage on Colossus) + Dremel (tree-structured distributed execution) + Slots (dynamically allocated compute units). Separated storage and compute — they scale independently.
Why It's Fast Only reads columns you query (projection pruning) + thousands of parallel workers + tree aggregation reduces shuffle volume by aggregating at each level before forwarding to root mixer.
Key Optimizations Partition (eliminate entire date ranges before any scan) → Cluster (skip blocks within partitions using min/max metadata) → Materialized Views (pre-compute aggregations, incremental refresh). Apply in that order.
Cost Model Pay per TB scanned (on-demand: $5/TB) or per slot commitment (reserved: $20/slot/month). Schema design determines scan volume. Effective cost = $5/TB × (1 - pruning efficiency). With 95% pruning, effective cost is $0.25/TB.
Failure Handling Worker failures: automatic (shuffle persisted to Colossus, work reassigned). Slot exhaustion: queries queue and degrade. The failure mode you design for is the human writing a full table scan on a petabyte — governance, not infrastructure.

Interview Cheat Sheet

If They AskAnswer With
"How does BigQuery achieve scale?" Columnar storage reads only queried columns + separated storage/compute allows thousands of parallel workers + tree-structured Dremel execution aggregates without full shuffle to root
"How would you optimize this query?" Partition pruning (date filter) → Cluster pruning (secondary filter, leftmost columns) → Projection pruning (SELECT only needed columns) → Approximate functions (APPROX_COUNT_DISTINCT) → Materialized views (pre-compute if repeating)
"How do you handle cost at scale?" Schema design (partition/cluster) to maximize pruning efficiency → Materialized views to pre-compute common aggregations → Reserved slots (break-even ~24 TB/month for 500 slots) → require_partition_filter to prevent accidental scans → Cost attribution labels by team
"What about real-time data?" Streaming inserts: ~10s latency for table reads, ~10min before visible to MVs. True real-time serving → Bigtable. BigQuery for analytics on near-real-time data. Streaming costs 10x more per row than batch loading.
"How do you monitor BigQuery?" Slot utilization per reservation (alert >80% sustained), bytes scanned efficiency (TB scanned / rows returned), P50/P99 query duration by team, cost attribution via labels, cost anomaly detection vs rolling baseline
"BigQuery vs [Redshift/Athena/Spark]?" BigQuery wins: serverless from day 1, integrated ML, native streaming, deep Google ecosystem (Dataflow/Looker/Vertex). Redshift wins: co-location benefits, AWS-native. Athena wins: open formats, no lock-in. Spark wins: general-purpose ML+ETL+SQL, portable.
🎯 The Final Lesson — Apply the Framework, Don't Memorize the Answers

Every element of the practice problem answer above was derived from the same framework: requirements → query patterns → schema → optimization → cost → monitoring. The specific SQL, the slot numbers, the cost estimate — these change for every system. The framework doesn't.

When an interviewer gives you a novel system you haven't seen before, start with: What are the query patterns? What's the latency tolerance? What does the scan volume look like? What are the retention requirements? The answers to those questions determine the partition key, the clustering columns, whether MVs are needed, and whether reserved slots are worth it. The schema and the slot strategy are derived conclusions, not starting points.

That's the difference the rubric measures: average candidates start with the schema; strong hires start with the questions that make the schema obvious.
