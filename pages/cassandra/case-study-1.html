Real-World System 1 · Discord Messages

The Problem Discord Faced

Users send messages in channels (1:1 or groups with 10 to 500K+ members). Messages appear instantly (<100ms). Users scroll back through history (range queries). Messages persist for years. Some channels have millions of messages (gaming communities).

Option EvaluatedWhy Discord Rejected It
PostgreSQLSingle-master bottleneck, sharding complexity — 100M messages/day would kill it
MongoDBBetter, but master-slave replication + write scaling issues
DynamoDBCost prohibitive at their scale, less operational control
Cassandra✅ Linear write scaling, no master, perfect for append-heavy messaging
L6/L7 Insight — Why Cassandra Was the Right Call

"The choice of Cassandra wasn't just about scale — it was about the access pattern. Messages are append-only (writes), read in time order (clustering column), and accessed by channel (partition key). This maps perfectly to Cassandra's storage model. A system with lots of updates or cross-channel queries would need a different choice."

Data Model Deep-Dive

Attempt 1 Naive Model — channel_id as Partition Key
CQL · First Attempt
Partition Key:  channel_id
Clustering Key: message_id (snowflake, time-ordered)

channel_id | message_id | author_id | content | timestamp
-----------+------------+-----------+---------+-----------
12345      | 1001       | user_a    | "Hello" | 2024-01-01 10:00:00
12345      | 1002       | user_b    | "Hi!"   | 2024-01-01 10:00:01
12345      | 1003       | user_a    | "Sup?"  | 2024-01-01 10:00:02
The Problem: 10GB+ Partitions on Large Servers

Large Discord servers (100K+ members) generate millions of messages per channel. One partition could grow to 10GB+. Here's what happens when you read the last 50 messages from a 10GB partition:

Read Trace · 10GB Partition
Read "last 50 messages" from channel 12345:

1. Coordinator hashes "12345" → token 847293
2. Finds replicas: Node A, Node B, Node C
3. Sends read to Node A
4. Node A must:
   - Check memtable (fast, in RAM)
   - Check Bloom filters for ALL SSTables (fast)
   - For "maybe" SSTables, load index (might not fit in RAM for 10GB partition!)
   - Seek to end of partition across multiple SSTables
   - Merge results

Problem: Index for 10GB partition = ~100MB
         Doesn't fit in memory → disk I/O for index
         Multiple SSTables → multiple seeks
         Latency: 500ms+ instead of 5ms ❌
Attempt 2 Bucketed Model — (channel_id, bucket) as Partition Key
CQL · Bucketed Model
Partition Key:  (channel_id, bucket)
Clustering Key: message_id

Where: bucket = message_id >> 22  (roughly 10-day buckets based on Snowflake ID)

channel_id | bucket  | message_id | content
-----------+---------+------------+---------
12345      | 100     | 1001       | "Hello"
12345      | 100     | 1002       | "Hi!"
12345      | 101     | 5001       | "Later message" ← newer bucket

Why this works:
  Partition size: ~100MB instead of 10GB (bounded by time window)
  Index size: ~1MB (fits in memory — no disk I/O for index)
  Read latency: <10ms consistently ✓

Query "last 50 messages":
  1. Start with most recent bucket (known from current time)
  2. Read from bucket 101 → get 30 messages
  3. Read from bucket 100 → get remaining 20 messages
  4. Return combined 50 messages

Trade-off: 2 partition reads instead of 1, but each is fast
L6/L7 Insight — Bucket Sizing Is Critical

"Bucket sizing requires access-pattern-driven analysis. Too small (1 hour) = many partitions to scan for history. Too large (1 year) = back to giant partitions. Discord chose ~10 days because most queries hit recent messages, and scrolling back rarely goes beyond a few buckets. This is access-pattern-driven design — not a default setting."

Write Path: Sending a Message

Tracing exactly what happens when User A sends "Hello" in channel 12345, with per-step latency.

Steps 1–2 · Client → Coordinator → Replica Identification
Step 1: Client → Coordinator (any Cassandra node)                   t=0ms
════════════════════════════════════════════════

  Client (Discord server) sends:
  INSERT INTO messages (channel_id, bucket, message_id, author, content)
  VALUES (12345, 101, 6001, 'user_a', 'Hello')
  WITH CL=LOCAL_QUORUM

  Coordinator: Node X (in us-east datacenter)

Step 2: Coordinator identifies replicas                              t=0.1ms
═══════════════════════════════════════════════

  murmur3_hash(12345, 101) = token 7482910384

  NetworkTopologyStrategy lookup:
  - us-east (RF=3): Node A (rack1), Node B (rack2), Node C (rack3)
  - us-west (RF=3): Node D (rack1), Node E (rack2), Node F (rack3)
  - eu-west (RF=3): Node G (rack1), Node H (rack2), Node I (rack3)
Steps 3–5 · Parallel Writes → Quorum Achieved
Step 3: Coordinator sends to local DC replicas (parallel)           t=0.5ms
════════════════════════════════════════════════════════

  Coordinator X → Node A: "Write this row"
  Coordinator X → Node B: "Write this row"
  Coordinator X → Node C: "Write this row"
  (Non-blocking sends — all 3 in parallel)

Step 4: Each replica processes the write                            t=1–15ms
═══════════════════════════════════════════

  On Node A:
    4a. Append to Commit Log (sequential write to disk)
        → Durability guarantee: survives node crash
        → ~0.1ms on SSD

    4b. Write to Memtable (in-memory sorted structure)
        → Red-Black tree keyed by (partition_key, clustering_key)
        → ~0.01ms

    4c. Send SUCCESS to Coordinator
        → Does NOT wait for SSTable flush
        → Data is durable (commit log) but not yet in SSTable

  Node A: responds SUCCESS at t=2ms
  Node B: responds SUCCESS at t=2.5ms
  Node C: responds SLOWLY (disk I/O contention) at t=15ms

Step 5: Coordinator evaluates LOCAL_QUORUM                          t=2.5ms
══════════════════════════════════════════════

  LOCAL_QUORUM with RF=3: need (3/2)+1 = 2 responses

  Got 2 responses (Node A + Node B) at t=2.5ms
  → Return SUCCESS to client ✓
  (Node C's response arrives later but client already has success)

  Total write latency: ~3ms
Step 6 · Async Cross-DC Replication
Step 6: Async cross-DC replication (non-blocking)

  Coordinator X sends to us-west and eu-west asynchronously:

  → Node D (us-west):  receives at t=50ms   (cross-country latency)
  → Node G (eu-west):  receives at t=120ms  (transatlantic latency)

  Each remote DC replica: same commit log + memtable process

  By t=150ms: All 9 replicas globally have the message ✓

Why LOCAL_QUORUM? The Three Options Compared

CL = ONE
Latency~2ms ✓
DurabilityOnly 1 replica
RiskNode crash = msg lost
Rejected
CL = QUORUM (global)
Latency~150ms ✗
Durability5 of 9 globally
RiskTransatlantic wait
Rejected
CL = LOCAL_QUORUM ✓
Latency~3ms ✓
Durability2 of 3 local
RiskDC failure (rare)
Discord's Choice
L6/L7 Insight — LOCAL_QUORUM Is the Sweet Spot

"LOCAL_QUORUM is the sweet spot for most multi-region systems. You're betting that a single node failure is 1000× more likely than a full DC failure, so you optimize for the common case while async replication handles DC-level durability. If you needed CL=QUORUM globally, the 150ms transatlantic wait would make real-time chat unusable."

Read Path: Loading a Channel

User opens channel 12345 and wants to see the last 50 messages. Notice how the query uses CL=LOCAL_ONE — reads can tolerate more staleness than writes.

Full Read Trace · Steps 1–4 with Per-Step Timing
Step 1: Client → Coordinator                                        t=0ms
════════════════════════════════

  SELECT * FROM messages
  WHERE channel_id = 12345 AND bucket = 101
  ORDER BY message_id DESC
  LIMIT 50
  WITH CL=LOCAL_ONE

  Why LOCAL_ONE?
  Stale reads (seeing 49 vs 50 messages) are acceptable.
  Read latency matters more than perfect consistency for chat history.

Step 2: Coordinator routes to fastest replica                       t=0.1ms
════════════════════════════════════════════

  Dynamic snitch scores each replica by recent latency:
  - Node A: avg 2ms, healthy        ← picks this
  - Node B: avg 3ms, healthy
  - Node C: avg 50ms, under compaction

Step 3: Node A processes the read                                   t=0.1–3ms
═════════════════════════════════════

  3a. Check Memtable                                                0.01ms
      Found: 3 recent messages (not yet flushed to SSTable)

  3b. Check Bloom Filters for each SSTable

      SSTable-1 (1GB, from yesterday):
        "Is partition (12345, 101) possibly here?" → YES (might be there)
        → 0.001ms

      SSTable-2 (500MB, from 3 days ago, different bucket range):
        "Is partition (12345, 101) possibly here?" → NO (definitely not)
        → SKIPPED ENTIRELY ✓  0.001ms saved

      SSTable-3 (2GB, from last week):
        Bloom filter → YES

  3c. For "YES" SSTables, check partition index

      SSTable-1:
        Summary (in RAM): partition (12345, bucket 100-110) → index page 47
        Index lookup → data offset 847293
        → 0.1ms (all in RAM)

  3d. Read data from SSTables (disk I/O)

      SSTable-1: Seek to offset 847293 → 40 messages for (12345, 101)
        → 1ms (SSD random read)

      SSTable-3: Seek to offset 129483 → 25 messages for (12345, 101)
        → 1ms

  3e. Merge results
      Memtable:  3 messages  (timestamps 1001, 1002, 1003)
      SSTable-1: 40 messages (timestamps 500–990)
      SSTable-3: 25 messages (timestamps 100–400)

      Merge by clustering key (message_id DESC):
      Return top 50: timestamps 1003, 1002, 1001, 990, 989...

      → 0.1ms

  Total Node A processing: ~3ms

Step 4: Return to client                                            t=4ms
════════════════════════════════

  Coordinator receives 50 messages from Node A
  Returns to Discord server
  Total end-to-end latency: ~4ms ✓

Read Path Optimization: Speculative Retry

Speculative Retry · Handling Slow Nodes
Scenario: Node A is under disk compaction (usually responds in 3ms, now slow)

t=0ms:   Send read to Node A
t=10ms:  No response yet (Node A usually responds in 3ms → timeout exceeded)
         → Speculatively send same read to Node B
t=12ms:  Node B responds (fast)
         → Return Node B's result to client immediately
t=50ms:  Node A finally responds (ignored—already returned)

Result:  Client sees 12ms latency instead of 50ms  ✓
Trade-off: Extra network + disk I/O for the speculative read
L6/L7 Insight — Speculative Retry Tuning

"Speculative retry is Cassandra's answer to tail latency. P99 drops dramatically because slow nodes are routed around. The tuning question is the speculation threshold — too aggressive wastes resources and creates I/O storms; too conservative doesn't help. The sweet spot: speculate after 2-3× the p50 latency for that table."

Compaction: Keeping Reads Fast

TWCS for Discord Messages — Why It's Perfect
Time Window: 1 day

Day 1 accumulation:
  Hour 0:   [SSTable-1a] 50MB
  Hour 6:   [SSTable-1a][SSTable-1b] 50MB each
  Hour 12:  [SSTable-1a][SSTable-1b][SSTable-1c]
  Hour 24:  [SSTable-1a][SSTable-1b][SSTable-1c][SSTable-1d]
              ↓ COMPACT within Day 1 window ↓
            [SSTable-Day1] 200MB ← clean, time-bounded

Day 2: Same pattern → [SSTable-Day2] 200MB
Day 3: Same pattern → [SSTable-Day3] 200MB
...
After 30 days: [Day1][Day2][Day3]...[Day30]
               30 SSTables total (manageable!)

WHY TWCS IS PERFECT FOR MESSAGES:

  1. Messages are time-ordered (Snowflake IDs)
     → Each SSTable contains a clean time range
     → Queries for "recent messages" only hit recent SSTables

  2. Messages are rarely updated
     → No cross-SSTable version merging needed
     → Each message exists in exactly one SSTable

  3. Future TTL support is free
     → If Discord adds 1-year retention, Day1 SSTable drops entirely
     → Zero tombstone processing — just delete the file!

  4. Predictable compaction pattern
     → Compaction happens once per day per node
     → No surprise compaction storms

What Would Happen with STCS?

STCS (what Discord tried first)

How it grows:

[50MB]×4 → compact → [200MB]
More + [200MB] → compact → [800MB]×4 → BIG COMPACT → [3.2GB]

Problem 1: Old and new messages mixed in same SSTables → can't drop old data without reading entire 3.2GB SSTable

Problem 2: Huge compaction = read 3.2GB + write 3.2GB simultaneously → disk I/O saturated for minutes → read latency spikes

TWCS (Discord's final choice)

How it grows:

Day 1 messages → [Day1 SSTable 200MB]
Day 2 messages → [Day2 SSTable 200MB]
Each day: one clean 200MB file

Benefit 1: Temporal locality — recent messages stay in small, recent SSTables that stay in OS page cache

Benefit 2: Compaction is daily, predictable, 200MB — never triggers multi-GB I/O storms

L6/L7 Insight — TWCS Is About More Than TTL

"TWCS isn't just about TTL optimization — it's about temporal locality. When your access pattern is 'recent data is hot, old data is cold,' TWCS keeps hot data in small, recent SSTables that stay in page cache. STCS mixes hot and cold data, evicting hot messages from cache to make room for cold ones. Discord tried STCS first, hit these problems, and switched to TWCS."

Failure Scenario: Node Crash During Write

Node B Crashes Mid-Write — Full Timeline
Initial State:
  Replicas for channel 12345: Node A (rack1), Node B (rack2), Node C (rack3)
  All healthy, gossip shows all UP

t=0ms:   User sends "Hello"
         Coordinator X sends write to A, B, C

t=1ms:   Node B crashes (power failure / kernel panic)
         - Commit log: partial write (or nothing — depends on timing)
         - Memtable: LOST (RAM cleared)

t=2ms:   Node A responds SUCCESS
t=2.5ms: Node C responds SUCCESS
         Coordinator has LOCAL_QUORUM (2 of 3) ✓
         → Returns SUCCESS to client
         Message IS durable (on A and C)

t=5s:    Gossip detects Node B missing heartbeats
         φ (phi) accrual score: 3... 5... 7...

t=30s:   φ > 8, Node B marked DOWN in gossip
         All nodes stop routing to B
Hinted Handoff — Writes While B Is Down
New message arrives while B is DOWN:
  1. Coordinator finds replicas: A, B (DOWN), C
  2. Sends to A and C only
  3. LOCAL_QUORUM: needs 2, gets 2 → SUCCESS

But B is missing this write!
→ HINTED HANDOFF activates:

  Coordinator stores a "hint" locally:
  {
    target_node: "Node B",
    mutation: {channel_id: 12345, bucket: 101, message: "...", ...},
    timestamp: now(),
    expires: now() + 3_hours  // max_hint_window_in_ms (default)
  }
  
  Stored in: system.hints table (local to coordinator)

NODE B RECOVERS at t=10min:

  1. B reads its commit log → replays any complete entries
  2. B starts gossiping → new generation number
  3. Coordinator sees B is back → replays stored hints to B
  4. Hints deleted after successful replay

t=15min: B is caught up ✓

What if B was down > 3 hours?
  - Hints expired and dropped
  - B is missing some writes
  - Must run: nodetool repair  → syncs B with A and C via Merkle trees

Repair Deep-Dive: How Merkle Trees Work

Merkle Tree Repair — Finding What's Missing
Step 1: Build Merkle Tree for each replica (for a partition range)

Node A's Merkle Tree for range [0-1000]:

                  [Root Hash: abc123]
                 /                    \
       [Hash: def]                    [Hash: ghi]
       /         \                    /         \
 [Range 0-250] [Range 251-500] [Range 501-750] [Range 751-1000]
    Hash: x       Hash: y         Hash: z         Hash: w

Node B's Merkle Tree (same structure):

                  [Root Hash: abc999]  ← DIFFERENT!
                 /                    \
       [Hash: def]                    [Hash: ghi]
       /         \                    /         \
 [Range 0-250] [Range 251-500] [Range 501-750] [Range 751-1000]
    Hash: x       Hash: y         Hash: zzz        Hash: w
                                       ↑
                                   DIFFERENT!

Step 2: Compare trees top-down (O(log N) comparisons)

  Root hashes differ → descend
  Left subtrees match → SKIP (no differences in ranges 0-500)
  Right subtrees differ → descend further
  Range 501-750 hashes differ → stream only those rows

Step 3: Stream differences

  Node A sends its rows for range 501-750 to Node B
  Node B writes them (or keeps newer version if B has data too)

Result:
  Full table sync with minimal data transfer
  For a 100GB table: might only transfer 1GB of differences
  O(log N) comparison instead of O(N) full table scan
L6/L7 Insight — Merkle Tree Repair Trade-offs

"Merkle tree repair is Cassandra's eventual consistency mechanism. The O(log N) comparison is brilliant — for a 100GB table you compare trees in milliseconds, not stream the whole table. The trade-off: building the Merkle tree itself is expensive — it scans all SSTables to compute hashes. That's why full repairs are scheduled during low-traffic windows, and incremental repair maintains trees per-SSTable to only repair deltas."

The Discord Operational Result
  • Write latency: ~3ms (LOCAL_QUORUM)
  • Read latency: ~4ms (LOCAL_ONE, dynamic snitch)
  • Node failure: transparent to users (CL handles it)
  • Compaction: predictable daily windows, no latency spikes (TWCS)
  • Recovery: hinted handoff for <3h outages, repair for longer
🔑 The Discord Design Principles

"Discord's Cassandra design is a masterclass in access-pattern-first thinking: the partition key (channel + bucket) was chosen to bound partition size for the specific query pattern (recent 50 messages). The consistency level (LOCAL_QUORUM for writes, LOCAL_ONE for reads) was calibrated to the durability-latency budget. The compaction strategy (TWCS) aligned with temporal access patterns. Each decision flowed from the next, not from defaults."

🎯 End-to-End Lesson — What Makes a Principal Engineer Answer

An L5 engineer would say "use Cassandra for Discord messages." An L6 engineer would add the data model. An L7 engineer traces every millisecond through the system: how the partition key determines which nodes get the write, why 2 of 3 ACKs at t=2.5ms is sufficient, why Node C's slow response at t=15ms doesn't block the user, why TWCS keeps the most recent SSTable in page cache while old days stay cold on disk, and why Merkle trees make repair O(log N) instead of O(N).

That depth of causal reasoning across the full stack — not just knowing the components exist — is the L7 standard.
