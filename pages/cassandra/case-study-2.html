Real-World System 2 · Netflix Viewing History

The Problem Netflix Faced

Every Netflix member's viewing history is tracked: every play, pause, stop, and rewind. This data powers "Continue Watching," "Because you watched X" recommendations, and ML training pipelines. Retention is forever — the ML team needs years of viewing history.

Scale Numbers
Members200M+ worldwide
Events per day24 billion
Writes/sec (sustained)280K writes/sec
Writes/sec (peak)1.4M writes/sec (prime time)
Storage per day4.8TB (24B × 200 bytes)
Storage per year1.75PB
How We Derive 24B Events/Day

200M members × 2 hours/day average × 1 event/minute = 24B events/day = 280K writes/sec sustained. Peak is 5× at prime time (Friday 8-10 PM) → 1.4M writes/sec. This is why CL=ONE is the only viable consistency level for write path — any higher CL would require 5× more infrastructure just for peak.

Data Model

CQL · viewing_history Table Design
Table: viewing_history

Partition Key:   member_id
Clustering Key:  (view_date DESC, timestamp DESC)

member_id | view_date  | timestamp           | title_id | event_type | position
----------+------------+---------------------+----------+------------+----------
12345     | 2024-01-15 | 2024-01-15 20:30:15 | tt001    | PLAY       | 0
12345     | 2024-01-15 | 2024-01-15 20:45:00 | tt001    | PAUSE      | 900
12345     | 2024-01-15 | 2024-01-15 21:30:00 | tt001    | STOP       | 3600
12345     | 2024-01-14 | 2024-01-14 19:00:00 | tt002    | PLAY       | 0

Key design decision: member_id as partition key
→ All of one member's history is co-located on the same 3 nodes
→ All member queries are single-partition lookups (no scatter-gather)
→ Even with 10 years of history, one member's partition stays bounded

Three Query Patterns — All Served by One Table

Query 1
"Continue Watching"
CQL
SELECT * FROM viewing_history
WHERE member_id = 12345
LIMIT 10

Clustering by (view_date DESC, timestamp DESC) returns most recent first. Single partition, ultra-fast.

Query 2
"What Did User Watch on Date X?"
CQL
SELECT * FROM viewing_history
WHERE member_id = 12345
AND view_date = '2024-01-15'

Clustering key prefix match — scans only that date slice within the partition. No SSTable cross-partition seeks.

Query 3
ML Training Batch Job
CQL
SELECT * FROM viewing_history
WHERE member_id = 12345
AND view_date >= '2023-01-01'

Range scan on clustering key — sequential read within partition. Sequential I/O on SSD = very fast for even years of history.

L6/L7 Insight — Three Queries, One Table

"All three query patterns are served by a single table because the partition key (member_id) and clustering key (view_date DESC, timestamp DESC) are chosen to align with the access pattern. No secondary indexes, no materialized views, no JOINs needed. This is Cassandra data modeling done right: identify the queries first, then design the table structure to serve them with a single partition scan."

Partition Sizing — Does a Power User Break This?

Worst-Case Power User Calculation
10 years × 365 days × 5 titles/day × 3 events/title = 54,750 rows
Row size: ~200 bytes (member_id + dates + title_id + event_type + position)
Partition size: 54,750 × 200 bytes = ~11MB ✓
Well under the 100MB guideline — no bucketing needed unlike Discord!
Why No Bucketing Needed (Unlike Discord)

Discord's hot channels generate millions of messages and needed time-bucket partitioning. Netflix's member-centric partition stays bounded naturally — even the heaviest Netflix user generates only ~11MB of history over 10 years. The access pattern (one member at a time) and write volume (5 events/day) make member_id a safe partition key. Always calculate before adding complexity.

This is a single Cassandra cluster for one use case. Netflix runs many clusters — recommendations, billing, user profiles each have their own clusters. This is the "multiple smaller clusters" pattern (Apple uses it too) rather than one monolithic cluster, which provides better failure isolation and simpler per-cluster operations.

Consistency Strategy — Three Tiers for Different Operations

CL = LOCAL_ONE
Writes (viewing events)
Losing one PAUSE event is acceptable. 280K writes/sec requires minimal latency. Async replication to other DCs is fine. Events have timestamps — duplicates and out-of-order events are resolvable.
CL = LOCAL_ONE
Reads ("Continue Watching")
Must be fast (<50ms). Showing slightly stale data is fine — user just watched this content, expecting it to appear. LOCAL_QUORUM would add 3-5ms unnecessary latency for no user-perceptible benefit.
CL = LOCAL_QUORUM
Billing-related reads
Billing decisions need consistency — you can't charge a user twice or miss a cancellation. Worth the latency trade-off. Only a small fraction of total reads hit this tier.
L6/L7 Insight — CL Tiering Is the Right Pattern

"Netflix doesn't use a single consistency level for everything. They tier by business impact: viewing events (eventually consistent is fine) use CL=LOCAL_ONE for maximum throughput. User-facing reads (slightly stale is OK) also use LOCAL_ONE. Billing (must be correct) uses LOCAL_QUORUM. This tiered approach is what L7 thinking looks like — not 'use QUORUM for everything' (too slow) or 'use ONE for everything' (too risky for billing)."

Throughput split:
  EVCache: 95% × total_reads = very high QPS absorbed by Memcached
  Cassandra: 5% × total_reads = manageable (Cassandra not the bottleneck)

Write Path — Invalidate, Don't Write-Through

Write Strategy Breakdown
Event Write
Viewing event → Cassandra (CL=LOCAL_ONE) → async invalidate EVCache key for that member
Cache Miss
Next read for that member → EVCache miss → fetch from Cassandra → populate cache with 30s TTL
Cache Hit
Subsequent reads within 30s → served from EVCache → <5ms, Cassandra not touched
Why Not Write-Through Caching?
  • Volume: Viewing events are 280K writes/sec — writing to both Cassandra AND EVCache doubles write load
  • Cache unit mismatch: Cache stores the entire member's profile (aggregated), not individual events — can't meaningfully cache a single PAUSE event
  • Invalidation is simpler: Just delete the cache key — next read rebuilds the aggregated view from Cassandra
  • Slight staleness acceptable: 30-second TTL means worst case, user sees history from 30 seconds ago. Acceptable for viewing history.
Complete Read Path with Timing
Cache HIT path (95% of requests):
  t=0ms:   Client requests "Continue Watching"
  t=0.5ms: EVCache lookup: key "cw:member:12345" → HIT
  t=1ms:   Return cached viewing history
  Total: <5ms

Cache MISS path (5% of requests):
  t=0ms:   Client requests "Continue Watching"
  t=0.5ms: EVCache lookup: key "cw:member:12345" → MISS
  t=1ms:   Query Cassandra with CL=LOCAL_ONE
  t=2ms:   Dynamic snitch picks fastest replica
  t=3ms:   Replica checks memtable (no recent history)
  t=5ms:   Bloom filter checks identify 2 candidate SSTables
  t=8ms:   Read from 1 SSTable (member's history, sequential read)
  t=10ms:  Merge results, return to coordinator
  t=15ms:  Coordinator receives 10 most recent events
  t=20ms:  Populate EVCache (async, non-blocking)
  t=22ms:  Return to client
  Total: ~25ms (well under 50ms target)

L6/L7 Insight — Same Database, Different Designs

"Discord and Netflix both use Cassandra for high-write event data, but make completely different data modeling choices. Discord needs bucketing because channels generate unbounded data; Netflix doesn't because member usage is naturally bounded. Discord uses LOCAL_QUORUM for writes because losing a message is unacceptable; Netflix uses LOCAL_ONE because a lost PAUSE event is fine. Same tool, different trade-offs, both optimal for their specific access patterns."

🔑 The Netflix Design Principle

"Netflix doesn't query Cassandra directly for most reads — EVCache absorbs 95% of traffic. Cassandra is the durable source of truth, but you design the system assuming cache hits. This is the right architectural pattern: Cassandra for durability and scale-out writes, a fast cache for read latency. If you're designing a system and Cassandra seems too slow for your read SLAs, the answer is usually 'add a cache layer,' not 'replace Cassandra.'"

🎯 Netflix Lesson — Cassandra as a Foundation, Not a Monolith

Netflix's viewing history system illustrates the mature Cassandra pattern: Cassandra handles durable writes at massive scale (280K/sec), while EVCache handles read latency (<5ms for 95% of reads). The data model is member-centric for query alignment. Consistency is tiered by business impact. The cluster is partitioned into separate, purpose-built clusters (not one giant shared cluster).

The L7 insight: no single component does everything. Cassandra does writes and durability. EVCache does read throughput and latency. The art is understanding which component owns which responsibility and designing the data flow between them.
