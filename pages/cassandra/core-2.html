§2.5 Compaction Strategies

One-Sentence Definitions

Compaction strategies determine when and how Cassandra merges SSTables to reclaim space, remove obsolete data, and keep read performance manageable. The wrong choice is the #1 source of mysterious Cassandra performance problems.

StrategyAnalogyBest For
STCSSnowball — small SSTables merge into bigger onesWrite-heavy, space-tolerant workloads
LCSFiling cabinet — strict organization, fixed-size drawersRead-heavy, predictable latency
TWCSDaily newspaper archive — group by time, never merge across daysTime-series with TTL
Strategy 1 · Write-Heavy
SizeTieredCompactionStrategy (STCS)
Snowball Merge
▾
Philosophy

Wait until you have ~4 SSTables of similar size, then merge them. Minimizes write amplification at the cost of space and read performance.

Diagram · STCS Snowball Pattern
Time 0:  ○ ○ ○ ○        (4 small SSTables, ~50MB each)
              │
              ▼ compact!
Time 1:  ●               (1 larger SSTable, ~200MB)

Time 2:  ● ○ ○ ○ ○       (1 big + 4 new small)
              │
              ▼ compact smalls
Time 3:  ● ●             (2 medium SSTables)

Time 4:  ● ● ○ ○ ○ ○     (2 medium + 4 new small)
              │
              ▼
...eventually...

Time N:  ◉ ◉ ◉ ◉         (4 huge SSTables accumulate)
              │
              ▼ BIG compaction!
Time N+1: ◯              (1 massive SSTable — hours of I/O!)
Strengths
  • Minimizes write amplification — data rewritten rarely
  • Good for heavy write loads — compaction doesn't compete with writes much
  • Default strategy — works reasonably well for mixed workloads
Weaknesses
  • Space amplification: Need 50-100% free disk space for worst-case compaction (when the 4 huge SSTables merge)
  • Unpredictable: Small compactions are fast (milliseconds), big ones can take hours — impossible to predict latency
  • Read amplification: Reads must check many SSTables between compaction cycles — sawtooth latency pattern
Strategy 2 · Read-Heavy
LeveledCompactionStrategy (LCS)
Filing Cabinet
▾
Philosophy

Organize SSTables into "levels" with strict size limits and non-overlapping key ranges within each level. Maximum read performance at the cost of write amplification.

Diagram · LCS Level Layout
Level 0:  [A-Z] [A-Z] [A-Z] [A-Z]     ← Recent flushes, may overlap, max 4
              │
              ▼ compact into L1
Level 1:  [A-D] [E-H] [I-L] [M-P] [Q-T] [U-Z]     ← 10 × 160MB = 1.6GB total
              │                                        Non-overlapping ranges!
              ▼ compact into L2
Level 2:  [A-B][C-D][E-F]...............[Y-Z]        ← 10 × L1 = 16GB total
              │
              ▼
Level 3:  ....                                        ← 10 × L2 = 160GB total

Rules:
• SSTables WITHIN a level have non-overlapping key ranges
• Each level is 10× the size of the previous
• Compaction picks an L(n) SSTable, merges with overlapping L(n+1) SSTables
Strengths
  • Read amplification: max ~10 SSTables per read (one per level) — consistent, predictable
  • Space efficient: ~10% overhead vs STCS's 50-100%
  • Predictable performance — no sawtooth latency pattern
Weaknesses
  • Write amplification: ~10× — every row is rewritten once per level traversal
  • Heavy on I/O: constant small compactions running in background
  • Not good for write-heavy workloads — compaction competes with writes for disk bandwidth
Strategy 3 · Time-Series + TTL
TimeWindowCompactionStrategy (TWCS)
Daily Archive
▾
Philosophy

Group data by time window, compact only within each window, and drop entire files when TTL expires — no tombstone processing needed.

Diagram · TWCS Time-Window Grouping
Day 1 window:  [Morning] [Afternoon] [Evening] → compact → [Day1 ████]
Day 2 window:  [Morning] [Afternoon] [Evening] → compact → [Day2 ████]
Day 3 window:  [Morning] [Afternoon] [Evening] → compact → [Day3 ████]
Day 4 window:  [Morning █] [Afternoon █] [Evening...]  ← in progress

When TTL expires on Day 1:
  DROP THE ENTIRE FILE — no compaction needed!
  Zero tombstone overhead. No read penalty.

Queries for "last 24 hours":
  Only Day4 window SSTables in page cache (hot)
  Day3 and earlier: cold storage (never read)
Strengths
  • Perfect for TTL-based data: Drop whole files, zero tombstone processing
  • Predictable, time-based compaction — no surprise big compactions
  • Efficient for time-series queries (range scans within a window hit one file)
  • Hot/cold data naturally separated: Only recent windows in page cache
Weaknesses
  • Updates/deletes across windows create tombstones (the very thing TWCS avoids)
  • Window size must be set correctly: too small = many files, too large = slow deletes
  • Only works well for time-ordered, append-mostly data

§2.5D Walkthrough Scenarios

Scenario 1: IoT Time-Series, 10K writes/sec, 7-day TTL

STCS vs TWCS · Time-Series
STCS approach:
  Hour 0:   10 small SSTables (36M rows each)
  Hour 1:   Compaction merges → 3 medium SSTables
  Hour 6:   Some huge compactions, 4 large SSTables
  Day 7:    TTL expires → tombstones everywhere!
            Must compact to remove tombstones
            Meanwhile, reads hitting tombstones = SLOW

TWCS approach:
  Hour 0:   Window-0 has 10 small SSTables
  Hour 1:   Compaction merges Window-0 → 1 SSTable
  Hour 24:  Window-0 through Window-23 each have 1 SSTable
  Day 7:    TTL on Window-0 expires
            DROP THE ENTIRE FILE — zero compaction needed!
            Zero tombstone overhead

Winner: TWCS by a mile for time-series + TTL

Scenario 2: User Profiles, 1K writes/sec, Frequent Random Reads

STCS vs LCS · Random Reads
STCS approach:
  Week 1:  Many SSTables accumulate
           Read for user_123: Check 20 SSTables
           Latency: 15-30ms (inconsistent, sawtooth)
  Week 2:  Big compaction happens
           Read latency drops to 5ms, then climbs again
  Pattern: Sawtooth latency — unacceptable for SLOs

LCS approach:
  Week 1:  Continuous background compaction
           Max 10 SSTables per read (one per level)
           Read latency: 3-5ms (consistent!)
  Week 2:  Same pattern — totally predictable
  Cost:    Each row written ~10× (write amplification)
           But read SLO is met reliably

Winner: LCS for predictable read latency
L6/L7 Nuances — Compaction

Question that shows deep understanding: "When would you use STCS over LCS even for a read-heavy workload?"

Answer: "When data has high temporal locality — like a cache where recent data is read much more frequently than old data. STCS keeps recent data in small, hot SSTables. LCS would constantly rewrite these hot rows as it levels them up, wasting I/O. The trick is recognizing that 'read-heavy' doesn't mean 'uniform reads.'"

Non-obvious TWCS insight: "TWCS isn't just about TTL efficiency — it's about query pattern alignment. If your time-series queries always hit the last 24 hours, only the most recent window's SSTables stay in page cache. Older windows are cold storage. STCS would mix hot and cold data in the same SSTables, defeating your cache."

§2.6 Gossip Protocol & Failure Detection

Mechanism 5
Gossip Protocol
Epidemic Propagation
▾
A) What It Is

Gossip is Cassandra's decentralized mechanism for nodes to share cluster state — who's alive, who's dead, who's overloaded — by periodically exchanging information with random peers.

Core insight: Instead of a central registry (single point of failure), every node is a "rumor mill." Information spreads exponentially: tell 2 friends, they tell 2 friends. In a 100-node cluster, a piece of information reaches everyone in ~7 rounds.
B) How Gossip Rounds Work
Gossip · Every Second Per Node
Every second, each node:
  1. Pick a random LIVE node
     → Exchange state: "Here's everything I know about every node"
     → Merge: adopt any info with a higher version number

  2. Maybe also: Pick a random SEED node
     → Ensures isolated clusters can reconnect

  3. Maybe also: Pick a random DEAD node
     → Check if it's come back to life
State Exchanged Per Node
JSON · Gossip State Structure
{
  "node": "10.0.0.5:9042",
  "generation": 1623456789,    // When node last started (wall clock)
  "heartbeat": 5423,           // Increments every second
  "applicationState": {
    "STATUS":  "NORMAL",       // NORMAL | LEAVING | MOVING | BOOTSTRAPPING
    "LOAD":    "152.3GB",      // How much data this node stores
    "SCHEMA":  "abc123",       // Schema version hash (detects mismatches!)
    "DC":      "us-east",      // Data center name
    "RACK":    "rack1",        // Rack name
    "TOKENS":  "[...]"         // Token ranges this node owns
  }
}
Phi Accrual Failure Detection
Why Phi Accrual Is Brilliant

Traditional fixed timeout: "No heartbeat for 10 seconds = dead." But Node A (fast network) could detect failure in 2s, and Node B (slow network, 2s jitter) triggers false positives at 12s.

Phi Accrual adapts per node: "Based on historical heartbeat arrival times for THIS specific node, how likely is it dead right now?" Each node gets a personalized threshold based on observed network behavior.

Phi Accrual Math
φ = -log10(probability node is alive)

If heartbeats usually arrive every 1s ± 0.1s,
and it's been 3s since the last heartbeat:

φ = 5  →  99.999%     likely dead
φ = 8  →  99.999999%  likely dead

Cassandra default: phi_convict_threshold = 8

Node A (fast network, low jitter):  adapted timeout ~2s
Node B (slow network, high jitter): adapted timeout ~15s
→ No false positives, but still fast detection when real failure occurs
9-Second Failure Detection Timeline
Timeline · Node D Crashes in 5-Node Cluster
t=0s:  Node D crashes (power failure)

t=2s:  C gossips to A: "D's heartbeat hasn't increased"
       A notes: "D's last heartbeat was 2s ago, φ=3 (probably fine, wait)"

t=5s:  Multiple nodes report D's heartbeat unchanged
       A calculates: φ=8 for D → marks D as DOWN
       A gossips to B: "D is DOWN"

t=7s:  All nodes agree: D is DOWN
       Coordinator stops routing to D
       Hinted handoff ACTIVATES for D's token ranges

t=100s: Node D reboots, starts gossiping with generation=NEW_TIMESTAMP
        Nodes see D is back with new generation
        D enters BOOTSTRAPPING state
        D streams data from hints and replicas
        Eventually D is NORMAL again
D) Failure Modes
Network Partition Scenario
Split-Brain Timeline
Initial: A ←→ B ←→ C ←→ D ←→ E  (all healthy)

Partition: Cable cut between C and D
           [A, B, C] |  [D, E]

What happens:
  [A,B,C]: Gossip works normally within partition
           Eventually marks D,E as DOWN
  [D,E]:   Gossip works normally within partition
           Eventually marks A,B,C as DOWN

Both sides continue serving clients!
→ With RF=3 and data split across partition: some writes may fail (quorum)
→ With CL=ONE: both sides may accept conflicting writes

Partition heals:
  A gossips to D → exchange state → detect conflict
  Both sides wrote to same key during partition!
  Resolution: Read repair + Last Write Wins (LWW)
  Winner: whichever side had the later timestamp
Gotcha 1: Seed Nodes Are Critical

Gossip uses "seed" nodes to bootstrap. If all seeds are unreachable, new nodes can't join the cluster. But seeds aren't operationally special — any healthy node can serve as seed. Best practice: 3 seeds per DC in different racks, and seeds should never all be in the same failure domain.

Gotcha 2: Schema Disagreements Need Manual Fix

Cassandra detects schema mismatches via the SCHEMA version in gossip state. But it won't auto-resolve — you'll see warnings in logs and may see query failures. Requires manual nodetool describecluster + schema alignment.

Gotcha 3: Clock Skew Corrupts Gossip

Generation numbers use wall-clock timestamps. If a node's clock is significantly off, its gossip may be ignored entirely (looks like old data) or may override valid state from other nodes. Monitor NTP across all Cassandra nodes.

E) Operational Considerations
Key Commands & Metrics
  • nodetool gossipinfo — raw gossip state for every node
  • nodetool status — derived cluster health (U/D per node)
  • nodetool describecluster — schema version agreement across nodes
  • Alert on: phi_convict_threshold breaches, schema version mismatches, generation changes (node restarts)
  • Ports required: 7000 (internal gossip), 7001 (TLS gossip) — firewall issues = nodes can't see each other
F) The L6/L7 Nuances
L6/L7 Nuances — Gossip

Convergence time is O(log N): In a 1000-node cluster, full propagation of a state change takes ~10-15 seconds (log₂(1000) ≈ 10 rounds). During this window, different coordinators may have inconsistent views of cluster membership — this is expected and fine for most operations.

Gossip storm on mass restart: If many nodes restart simultaneously (deployment, region recovery), gossip traffic spikes as everyone shares full state with random peers. With 1000 nodes each gossiping state for 1000 nodes, you get ~1M gossip messages per second. Use ring_delay_ms to stagger startup.

Generation numbers as a debugging tool: When a node restarts, its generation increments. You can detect unexpected restarts by monitoring generation changes in nodetool gossipinfo. A node that's restarting repeatedly (crash loop) will show rapidly incrementing generation numbers.

Questions to Ask (or Your Interviewer)

  • "Write-to-read ratio?"
    Determines STCS vs LCS. >10:1 → STCS. <10:1 → LCS. The most important compaction question.
  • "TTL on the data?"
    Any time-series data with natural expiration → TWCS eliminates tombstone overhead entirely.
  • "Strong consistency or eventual?"
    Determines consistency level strategy. If strong: CL(write) + CL(read) > RF. If eventual: ONE or QUORUM per path.
  • "How many DCs, latency requirements?"
    Determines LOCAL_QUORUM vs EACH_QUORUM. Cross-DC adds 80-150ms per operation — avoid paying this tax for every query.
  • "Acceptable data loss window (RPO)?"
    Determines RF and CL combination. RPO=0 → synchronous replication, QUORUM or higher.
  • "Hot partition risk in the data model?"
    Consistent hashing doesn't save you from bad partition key choices. High-cardinality partition keys are the #1 Cassandra performance problem.
🔑 Sound Bite — Comparing All Variants

"I pick compaction strategy based on the write-to-read ratio: STCS for write-heavy, LCS for read-heavy, TWCS for time-series with TTL. For multi-region, I always use NetworkTopologyStrategy with LOCAL_QUORUM to avoid cross-DC latency tax on every operation — only escalating to EACH_QUORUM for operations that truly need global consistency. And I choose partition keys with high cardinality upfront, because consistent hashing can't save you from a hot partition once you're in production."

🎯 Complete Cassandra Internals — Final Lesson

Cassandra's architecture is a set of deliberate, composable trade-offs: Consistent Hashing (where data lives) → NetworkTopologyStrategy (how it survives failures) → Consistency Level (how many copies confirm before success) → LSM-Tree (how writes hit disk) → Compaction Strategy (how SSTables are organized over time) → Gossip (how nodes know about each other). Each layer adds one capability while accepting one limitation. The L6/L7 insight: you don't configure Cassandra, you compose its primitives for your access patterns.

The final soundbite: "Cassandra is the right tool when you need write scalability and multi-region availability, and you're willing to pay with schema rigidity and eventual consistency. If you need ACID transactions or ad-hoc queries, reach for Postgres or Spanner instead."
