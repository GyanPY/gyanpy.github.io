§3.2 The Distributed Systems Twist

Single-Node Intuitions That Break in Distributed Cassandra
  • New failure mode: Partial writes. Write succeeds on 2 of 3 replicas; third replica is stale until repair runs.
  • New failure mode: Replica divergence. Network partition allows conflicting writes on both sides.
  • New failure mode: Coordinator bottleneck. One coordinator handles all requests for a partition — single coordinator can be the bottleneck even when replicas have capacity.
The Concrete Example — Why CL=QUORUM Isn't Just About Durability

"On a single node, an LSM-tree guarantees once I read a value, I'll never see an older version. But across 3 replicas with CL=ONE, I might read from replica A (version 1), then read from replica B (version 0). My second read returns older data than my first. CL=QUORUM gives me not just durability — it gives me monotonic reads."

Fail-Open vs. Fail-Closed Analysis

Fail-Open (CL=ONE, RF=3)

If 2 replicas are down, still serve reads from the 1 remaining. Potentially stale data, but system stays available. Right for logging, analytics, non-critical reads.

Fail-Closed (CL=QUORUM)

If 2 replicas are down, reject the request. No stale data risk, but system becomes unavailable. Right for writes to critical data to prevent split-brain.

The L7 Answer

"For Cassandra, fail-open (CL=ONE or LOCAL_ONE) is usually the right choice for reads because availability matters more than perfect consistency. For writes to critical data, I'd fail-closed (QUORUM) to prevent split-brain. The exception: logging and metrics can fail-open on writes too (CL=ANY), because a lost log entry is acceptable."

§3.4 Edge Cases That Impress Interviewers

Why These Matter

These reveal whether you've operated Cassandra under production pressure vs just designed with it. Mention one proactively and you signal L7 experience.

Edge Case 1: Read-Your-Writes Failure Consistency Signal
▾

Scenario: Client writes to coordinator A (replicates to B and C). Next read goes to coordinator D, which queries replica C — which hadn't received the write yet. Client sees their own write missing.

Sophisticated answer:

  • CL=LOCAL_QUORUM for both reads and writes — guarantees at least one replica overlaps (preferred)
  • Sticky sessions — route a user's requests to the same coordinator (fragile)
  • Client-side caching — cache the just-written value with a short TTL
The Formula to Quote

Read-your-writes requires: Write CL + Read CL > RF. With RF=3: QUORUM(2) + QUORUM(2) = 4 > 3 ✓. With ONE+ONE: 2 < 3 ✗.

Edge Case 2: Lightweight Transactions (LWT) Performance Cliff Performance Signal
▾

Scenario: Someone uses INSERT ... IF NOT EXISTS for uniqueness guarantees on every write. System designed for 100K writes/sec.

What happens: LWT uses Paxos — 4 network round-trips instead of 1. Throughput drops 10-20×. System does 5K writes/sec instead of 100K. No errors, just silent degradation.

Sophisticated answer:

  • LWTs are for rare operations (account creation), not every write
  • For uniqueness at scale: use the partition key itself (same key = conflict via QUORUM)
  • Or: accept eventual consistency with client-side conflict resolution
The Hidden Cost

Paxos: PREPARE → PROMISE → PROPOSE → ACCEPT → COMMIT. Each step = network round-trip. In multi-DC: 200-500ms per LWT. This is not a bug — it's the cost of distributed consensus.

Edge Case 3: Tombstone Avalanche Operations Signal
▾

Scenario: Application deletes millions of rows. System degrades silently over weeks. Reads eventually timeout. Cassandra logs show: "Reading X live rows and Y tombstone cells."

What happens: Each delete creates a tombstone marker. Reads scan 2M tombstones before finding live data → timeouts.

Sophisticated answer:

  • Long-term fix: Use TTL for automatic expiration with TWCS — files drop at the filesystem level, zero tombstone processing
  • Alternative: Add a deleted column and filter at application level instead of row deletion
  • Immediate relief: nodetool compact on the affected table — treats symptom, not cause
Why TWCS Is the Real Solution

With TWCS, TTL-expired data is deleted by dropping entire SSTable files — zero tombstone processing, zero compaction overhead. The delete happens at filesystem level, not row level.

Edge Case 4: Clock Skew + Last Write Wins Corruption Data Integrity Signal
▾

Scenario: Server A's clock is 1 hour ahead. A writes "X=1" at real t=100 (its clock says t=3700). Server B later writes "X=2" at real t=200 (clock says t=200). LWW keeps "X=1" because 3700 > 200.

What happens: Correct, newer write (X=2) is silently overwritten by stale value. No errors, no warnings — data is just wrong.

Sophisticated answer:

  • Require NTP on all nodes with monitoring for clock drift >1 second
  • At Google scale: TrueTime provides bounded clock uncertainty (±7ms). Cassandra doesn't have this natively — known limitation of LWW conflict resolution
  • For truly critical data: use a CP database (Spanner, CockroachDB) where consensus timestamps solve this problem correctly

🔑 Sound Bite — Key Trade-off

"The fundamental trade-off in Cassandra is W + R > RF for consistency vs latency. If I need strong consistency, I pay the quorum tax on every operation. If I need speed, I accept eventual consistency and design for conflict resolution. There's no free lunch — I just get to choose which lunch I'm paying for."

Back-of-Envelope Reasoning

Two scenarios with full math. Critical question answered: does this workload even need Cassandra?

Scenario A
1M Users — Does This Need Cassandra?
▾

100K × 50 reads/day  = 5M reads/day  = 58 reads/sec avg
100K × 5 writes/day  = 500K writes/day = 6 writes/sec avg

Peak (10× average): 580 reads/sec, 60 writes/sec

Storage per user: 10KB profile + 1MB content = ~1MB
Total:   1M × 1MB = 1TB raw
With RF=3: 3TB
With 50% compaction overhead: 4.5TB

Nodes needed:
  Single node: 5-10K reads/sec, 10-20K writes/sec capacity
  3 nodes minimum for RF=3 (2TB SSD each = 6TB capacity)
  
Traffic headroom: ~100× before stress
The Honest Conclusion

This scale doesn't need Cassandra. PostgreSQL or MySQL would handle 1M users trivially with far more flexible querying. Reach for Cassandra when single-machine databases become the bottleneck — usually at >50K writes/sec sustained or >10TB data.

§4.3 Industry Deep-Dive

🎬 Netflix — 1000+ Node Clusters, Petabytes

Scale: 1000+ node clusters, petabytes of viewing history, multi-region (US-East, US-West, EU)

Key decisions:

  • LOCAL_QUORUM for most operations (avoids cross-DC latency on every read)
  • TWCS for viewing history — old data drops cleanly, zero tombstone overhead
  • Custom client driver with automatic region failover
  • EVCache (Memcached) in front for the hottest reads
💡 Lesson: Cassandra is the source of truth, but you cache heavily in front of it. At Netflix scale, even Cassandra needs a caching layer for hottest reads.
🍎 Apple — 75,000+ Nodes (World's Largest Cassandra Deployment)

Scale: 75,000+ nodes for iCloud, iTunes, Siri. Multiple separate clusters.

Key decisions:

  • Multiple smaller clusters vs one giant cluster — better isolation, simpler operations per cluster
  • Heavy investment in automation — node replacement is fully automated
  • Custom extensions for specific workload patterns
💡 Lesson: At extreme scale, cluster management is the bottleneck, not Cassandra itself. Apple's engineering investment is mostly in automation and tooling, not Cassandra internals.
💬 Discord — Messages, Hot Partition Problem Solved

Problem: Large channels (millions of messages) = hot partitions with simple channel_id partition key. Three nodes hammered for popular channels.

The solution:

  • Bucket messages by time window within each channel
  • Partition key = channel_id + bucket_id (bucket = time-based, e.g., day)
  • Recent messages efficient; old buckets become cold storage
💡 Lesson: Partition key design is everything. Wrong partition key = system doesn't scale regardless of cluster size. Discord's fix was adding one dimension to the partition key.
📸 Instagram — Direct Messages, Hybrid Storage Pattern

Architecture: Multi-DC for latency (user's messages stored in nearest DC). Cassandra + Redis hybrid.

Key decisions:

  • LOCAL_QUORUM for low-latency reads (no cross-DC round trips for normal operations)
  • Aggressive TTL on read receipts — let them expire, don't store forever
  • Cassandra for durable storage, Redis for caching hot conversations
💡 Lesson: Hybrid storage (Redis + Cassandra) is the practical production pattern. No one uses Cassandra alone for high-read workloads at scale.

🔑 Sound Bite — Scaling

"Cassandra's superpower is that doubling nodes roughly doubles write capacity — true linear horizontal scaling. But non-linear effects bite you at scale: gossip overhead, repair time, and managing thousands of nodes. At Netflix/Apple scale, the bottleneck shifts from Cassandra internals to cluster management automation."

🎯 Complete Cassandra Internals Deep Dive — Master Lesson

Cassandra is the right answer for a narrow but important class of problems: write-heavy, multi-region, known-query-pattern workloads where availability matters more than strong consistency. Within that class, the design stack is: choose your partition key (determines linear scaling vs hot partitions), choose replication strategy (always NetworkTopologyStrategy), choose consistency level (usually LOCAL_QUORUM), choose compaction strategy (STCS/LCS/TWCS based on read-write ratio), monitor the five warning signs.

The L7 insight: 80% of Cassandra performance is data modeling, 20% is configuration. Discord's hot partition problem wasn't solved by adding nodes — it was solved by adding a time bucket to the partition key. Get the data model right first; tune configuration second.
