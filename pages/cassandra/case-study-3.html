Real-World System 3 · Uber Driver Locations

The Problem Uber Faced

Real-time location tracking: 5M+ drivers worldwide, millions online simultaneously. Location updates every 4 seconds. Riders need to see nearby drivers instantly. Must work across network partitions (driver in a tunnel).

Scale Numbers
Online drivers (peak)2M+ simultaneously
Update frequencyEvery 4 seconds per driver
Write rate2M ÷ 4s = 500K writes/sec
Event size~100 bytes (lat, lng, status, timestamp)
Daily storage (raw)500K × 100B × 86,400s = 4.3TB/day

Data Model: Geohash Partitioning

CQL · driver_locations Table
Partition Key:   (city_id, geohash_prefix)
Clustering Key:  driver_id

city_id | geohash_prefix | driver_id | lat       | lng        | status    | updated_at
--------+----------------+-----------+-----------+------------+-----------+---------------------
1       | 9q8y           | driver_a  | 37.7749   | -122.4194  | AVAILABLE | 2024-01-15 10:00:00
1       | 9q8y           | driver_b  | 37.7751   | -122.4190  | ON_TRIP   | 2024-01-15 10:00:01
1       | 9q8z           | driver_c  | 37.7800   | -122.4100  | AVAILABLE | 2024-01-15 10:00:02

Query: "Find available drivers near rider at (37.775, -122.419)"
1. Compute geohash of rider location: "9q8y"
2. SELECT * FROM driver_locations
   WHERE city_id = 1 AND geohash_prefix = '9q8y'
   AND status = 'AVAILABLE'
   → Single partition read → application filters by exact distance
Three Design Decisions — Why (city_id, geohash_prefix)?
  • Why city_id? Bounds partition size (SF drivers vs NYC drivers stay in separate partitions). Enables city-level operations. Keeps data local — SF Cassandra cluster handles SF traffic.
  • Why 4-char geohash prefix? Full geohash (8 chars) is too specific — single building. 4-char prefix = ~1km × 1km cell. Busy urban areas: 100–500 drivers per cell. Correct balance between partition density and query recall.
  • Why not lat/lng range query? Cassandra can't efficiently filter on two columns simultaneously. Geohash converts a 2D spatial query to a 1D partition key lookup — that's the entire trick.

Handling Driver Movement: TTL vs Tombstones

The Tombstone Trap — Wrong vs Right Approach
WRONG APPROACH — Explicit delete when driver changes geohash cell:

Driver moves from geohash "9q8y" to "9q8z":
  Step 1: DELETE FROM driver_locations
          WHERE city_id=1 AND geohash_prefix='9q8y' AND driver_id='driver_a'
  Step 2: INSERT INTO driver_locations
          VALUES (1, '9q8z', 'driver_a', ...)

PROBLEM: 500K updates/sec = 500K deletes/sec = 500K tombstones/sec
         Tombstones accumulate in SSTables
         Every query for '9q8y' must scan tombstones
         Within 24 hours: billions of tombstones → cluster becomes unreadable

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CORRECT APPROACH — TTL-based expiry:

Every location update: INSERT with TTL=30s (no explicit delete ever)

INSERT INTO driver_locations
  (city_id, geohash_prefix, driver_id, lat, lng, status, updated_at)
VALUES (1, '9q8y', 'driver_a', 37.7749, -122.4194, 'AVAILABLE', now())
USING TTL 30;

Why 30 seconds?
  Update frequency: every 4 seconds
  → Location refreshed 7× before TTL expires
  → If driver loses connectivity: they disappear from search ✓  (correct!)
  → If driver restores: next INSERT brings them back ✓
  → Driver crosses into new cell: old cell row expires in ≤30s ✓
  → Zero tombstones. Zero. Forever.
L6/L7 Insight — TTL Is the Right Delete for Time-Bound Data

"Cassandra tombstones are not free. At 500K writes/sec with explicit deletes, you create 500K tombstones/sec. They accumulate in SSTables and every read must scan them to establish what's been deleted. TTL-based expiry is the correct pattern for ephemeral data — the storage engine handles expiry during compaction, dropping whole SSTables with no tombstone overhead. The 30s TTL also gives you correct behavior on driver disconnect for free."

Compaction Strategy: TWCS for Location Data

TWCS — Perfect Fit for 30s TTL Location Data
Configuration:
  TWCS time_window = 1 minute (2× the TTL, to capture a full expiry cycle)

What happens:
  t=0:   New SSTable created — 30M rows, TTL=30s each
  t=1m:  Another SSTable created — 30M rows
  t=2m:  TWCS compacts minute-0 + minute-1 → single SSTable
         TTL has now expired on minute-0's data
         Compaction drops expired rows — entire minute-0 data gone
  t=3m:  Minute-2 SSTable compacted similarly

Steady state:
  Always ≤ 2 SSTables active (1-2 minutes of data)
  Old data removed automatically by compaction
  Zero tombstones — TTL expiry is not a tombstone
  Read path: 2 SSTables to check (not 50+)

Why TWCS and not STCS here?
  STCS: Mixes all time windows → driver from 10 minutes ago might
        be in same SSTable as current drivers → reads must filter expired rows
  TWCS: Time-aligned SSTables → entire SSTable of expired data is
        dropped as a file → no per-row scanning needed

Cross-DC Design: Why Uber Uses Single-DC for Locations

Multi-DC Challenge — Why Cross-DC Replication Doesn't Help Here
Scenario: Driver and rider both in San Francisco (US-WEST)

Normal operation:
  Driver writes → US-WEST Cassandra
  Rider reads  → US-WEST Cassandra  ✓ (same DC, consistent)

During US-WEST maintenance (traffic routed to US-EAST):
  Driver writes → US-EAST (eventually)
  Rider reads  → US-EAST
  Cross-DC replication lag: ~100ms
  Data TTL: 30 seconds

  Problem: Cross-DC lag = 100ms, TTL = 30,000ms
           Replication completes with 99.7% of TTL remaining → OK?

  But: Peak lag during failures can exceed 5-10 seconds
       During that window, rider sees stale driver positions
       For ride-hailing, stale positions = wrong driver dispatching

Uber's solution:
  driver_locations → SINGLE DC, no cross-DC replication
  RF = 3 (within the regional cluster)
  If SF cluster is down: SF matching is degraded (not wrong)

Why this is the right trade-off:
  Ephemeral data (30s TTL) loses its value during cross-DC lag
  Regional outage → degrade gracefully, not serve stale locations
  Netflix viewing history (durable, hours old = still useful) ≠ driver positions
🔑 Uber Location Sound Bite

"Not all Cassandra tables should be multi-DC. For ephemeral real-time data like driver locations, cross-DC replication adds latency without benefit — by the time data arrives in another DC, it may already be stale. The correct design degrades gracefully on regional failure rather than serving incorrect data. This is a nuanced understanding of when global consistency doesn't matter."

End-to-End Flow: Instagram DM — Alice → Bob "Hey!"

Real-World System 4 · Instagram Direct Messages

Putting It All Together

A complete request lifecycle across a messaging system like Instagram DMs. Five distinct phases from Alice's tap to Bob seeing the message — each with exact latency accounting and the hidden consistency challenge in Phase 5.

Phase 1Client → API Servert=0 → t=20ms

Alice's phone (NYC) sends: POST /api/messages with {conversation_id: "conv_alice_bob", sender: "alice", content: "Hey!", client_timestamp: "2024-01-15T10:00:00.123Z"}. Routed to API server in US-EAST (geographically closest). Network latency: 20ms.

Phase 2API Server Processingt=20ms → t=22ms

API server: (1) validates auth + rate limit, (2) generates server-side message_id (Snowflake ID: 1234567890), (3) sets server_timestamp as the authoritative timestamp — client timestamp is untrusted, (4) computes bucket: message_id >> 22 = bucket_42. Latency: 2ms.

Phase 3 · Write to Cassandra — Step by Step
Table: messages
Partition Key:   (conversation_id, bucket)
Clustering Key:  message_id DESC

INSERT INTO messages (conversation_id, bucket, message_id, ...)
VALUES ('conv_alice_bob', 42, 1234567890, ...)
WITH CL=LOCAL_QUORUM

Coordinator (Node X, US-EAST):
  1. Hash('conv_alice_bob', 42) → token 847293
  2. Replicas: A(rack1), B(rack2), C(rack3) in US-EAST
  3. Parallel sends to A, B, C

  Node A: Commit log append (0.5ms) + Memtable write (0.01ms) → ACK at t=23ms
  Node B: Commit log append (0.6ms) + Memtable write (0.01ms) → ACK at t=23.5ms
  Node C: (disk I/O contention) ... still writing ...

  Coordinator has 2 ACKs = LOCAL_QUORUM → SUCCESS returned to client ✓

  t=25ms: SUCCESS sent to Alice's app (total: 25ms round-trip)

  Async (non-blocking): Coordinator sends to EU-WEST, AP-SOUTH replicas
  EU-WEST receives at t=125ms (transatlantic)
  AP-SOUTH receives at t=175ms (trans-pacific)
Phase 4Push Notification (parallel to Cassandra write)t=22ms → t=30ms (push sent)

While Cassandra write executes: (1) publish "new_message" event to Kafka, (2) push service consumes event, (3) look up Bob's registered device tokens, (4) send push notification to Bob's phone via APNs/FCM. Bob in Tokyo (AP-SOUTH) receives push at approximately t=30ms: "Alice: Hey!".

Phase 5 · Bob Opens App — The Consistency Gap
Bob's phone (Tokyo) receives push at t=30ms and opens app immediately.

Bob's query:
  GET /api/conversations/conv_alice_bob/messages?limit=50
  Routed to AP-SOUTH (closest API server)

  SELECT * FROM messages
  WHERE conversation_id = 'conv_alice_bob' AND bucket = 42
  ORDER BY message_id DESC LIMIT 50
  WITH CL=LOCAL_ONE

AP-SOUTH Cassandra status at t=100ms (Bob opens app):
  Async replication from US-EAST → AP-SOUTH was sent at t=25ms
  Cross-DC propagation time: ~150ms
  AP-SOUTH receives the message at t=175ms

  Bob opens at t=100ms → AP-SOUTH does NOT yet have the message!
  Bob's query returns 0 results for the new message ← PROBLEM
Edge Case: Three Solutions to the Cross-DC Consistency Gap

Timeline of the problem: t=0ms Alice sends → t=5ms push sent → t=30ms Bob receives push → t=100ms Bob opens app → t=120ms Bob's query reaches AP-SOUTH Cassandra → t=175ms async replication completes. Gap: 55ms where Bob sees nothing.

  • Option A — Include message in push payload (Instagram's approach ✓): Push contains the full message content. App renders push content immediately. Background sync with Cassandra catches up. No perceived inconsistency — the push IS the delivery mechanism.
  • Option B — Route Bob's query to US-EAST: Detect that notification source was US-EAST, route message fetch there. Consistent, but adds ~200ms extra latency for Bob's message load. Trade-off: complexity for consistency.
  • Option C — Delay push until all DCs replicated: Don't send push until AP-SOUTH confirms receipt. Simple, but adds 150ms to push latency. Unacceptable for real-time messaging UX.
L6/L7 Insight — The Cross-DC Gap Is Where Most Engineers Miss the Complexity

"You can't just 'use Cassandra' and call it done. You need to design around eventual consistency at the application layer. Instagram's solution is elegant: treat the push notification as the primary delivery mechanism, with Cassandra as the durable backup. The app shows push content immediately and syncs with Cassandra in the background. The user never perceives a gap — but only if the push payload carries the message, not just a notification."

Anti-Patterns: What Goes Wrong in Production

Anti-Pattern 1 Using Cassandra as a Queue ▾
BAD Pattern
Table: job_queue
  job_id | status | created_at | payload

Workers poll:
  SELECT * FROM job_queue WHERE status = 'PENDING' LIMIT 10
  → FULL TABLE SCAN — no partition key filter!

Workers update:
  UPDATE job_queue SET status = 'PROCESSING' WHERE job_id = X
  → Not in-place modification — creates a new version in memtable
Four Failure Modes
  • Full table scan on every poll: WHERE status = 'PENDING' has no partition key — Cassandra fans out to all nodes, each does a local scan. This is a full cluster scan on every worker heartbeat.
  • Cassandra can't "find rows matching condition": It's a distributed key-value store, not a query engine. Filtering without a partition key is always O(N) across the cluster.
  • UPDATE is not in-place: Cassandra appends a new version — old version persists until compaction. No true update semantics.
  • No distributed locking: Two workers can grab the same job simultaneously. No transactions means no SELECT FOR UPDATE equivalent.

Fix: Use Kafka, SQS, or Redis for queues. Cassandra is for key-value access with known partition keys — not for finding rows by arbitrary conditions.

Anti-Pattern 2 Secondary Index Abuse ▾
BAD Pattern
Table: users (Partition Key: user_id)
CREATE INDEX ON users(city);

Query:
  SELECT * FROM users WHERE city = 'San Francisco';

What actually happens:
  1. Query sent to coordinator
  2. Coordinator fans out to ALL nodes
  3. Each node scans its LOCAL secondary index for 'San Francisco'
  4. Results merged back at coordinator
  → Full cluster scan disguised as an index lookup
Why Cassandra Secondary Indexes Are Not Like Relational Indexes
  • Local, not global: The index exists separately on each node — it only indexes data that node holds. There is no global B-tree spanning the cluster.
  • High cardinality = poor selectivity: If users are spread across 1000 cities, each node's local index for 'San Francisco' contains only a small fraction of SF users. The merge at coordinator is expensive.
  • Every node involved: Unlike a partition key query that routes to 1-3 nodes, a secondary index query always touches ALL nodes.

When secondary index IS acceptable: Low cardinality (status = 'ACTIVE' — only 3 values), combined with partition key (WHERE user_id = X AND status = 'ACTIVE'), or small tables (<1M rows).

Fix: Denormalize. Create users_by_city with city as partition key, user_id as clustering key. Two writes per user, fast single-partition reads per city.

Anti-Pattern 3 Unbounded Partition Growth ▾
BAD Pattern
Table: events
  Partition Key: customer_id
  Clustering:    event_timestamp

SELECT * FROM events WHERE customer_id = 'big_enterprise';

What happens for an enterprise customer with 100M events:
  Partition size: 100M × 500 bytes = 50GB
  Single partition read: coordinator must merge data across
    all SSTables that ever held this partition
  Compaction of 50GB partition: read 50GB, write 50GB simultaneously
    → disk I/O saturated for minutes
    → read latency for ALL reads on that node spikes
    → Cassandra emergency memory pressure
The Cascading Failure
  • The node hosting the 50GB partition runs out of memory during compaction
  • GC pauses lengthen → other queries on that node time out
  • Coordinator marks node as slow → routes traffic to replicas
  • Replicas also have the 50GB partition → same problem there
  • Entire cluster latency degrades for all customers, not just the enterprise one

Fix — Add time bucketing to partition key:

GOOD Pattern
Partition Key: (customer_id, year_month)

Now each partition: ≤ 1M events per month (bounded)
Partition size: 1M × 500B = 500MB → still high, consider year_week

Recent query:
  WHERE customer_id = 'X' AND year_month = '2024-01'  ← single partition

Historical query:
  Run parallel queries for each month → coordinator merges results
  Each partition is small and fast → total latency acceptable

Incident Response: Compaction Storm

Runbook · P99 Read Latency Spikes 10ms → 500ms
SCENARIO: P99 read latency spikes from 10ms to 500ms in 20 minutes

STEP 1 — DIAGNOSE: Check pending compactions
  $ nodetool compactionstats
  pending tasks: 347  ← WAY too high, normal is <10

STEP 2 — CONFIRM: Check SSTable count
  $ nodetool cfstats messages | grep "SSTable count"
  SSTable count: 89  ← Should be <20 for healthy cluster

STEP 3 — FIND ROOT CAUSE
  Check deployment log: New feature shipped 30 min ago
  Increased write rate 10×: 50K/sec → 500K/sec
  Compaction throughput (default 64MB/s) can't keep up

IMMEDIATE MITIGATION (within 5 minutes):
  $ nodetool setcompactionthroughput 256   # Raise from 64 to 256MB/s
  $ nodetool setconcurrentcompactors 4     # Raise from 2 to 4 threads

  If disk I/O is saturated and reads need priority:
  $ nodetool stop COMPACTION               # On the hottest table only
  (This pauses compaction — SSTable count grows but reads improve)

  Watch: pending compactions should start dropping within 5-10 min

LONG-TERM FIX (within 24-48 hours):
  1. Scale cluster — add nodes to spread write load per node
  2. Review compaction strategy: STCS → LCS for read-heavy tables
  3. Add alerting: pending_compactions > 20 → PagerDuty
  4. Capacity plan new features BEFORE shipping: estimate write rate delta

Incident Response: Hot Partition

Runbook · 3 Nodes at 90% CPU, Rest at 20%
SCENARIO: 3 specific nodes running at 90% CPU, rest of cluster at 20%
          P99 reads rising only for specific conversations

STEP 1 — IDENTIFY the hot partition
  $ nodetool toppartitions messages 1000
  
  Top partitions by read count (last 1000ms sample):
  (conversation_id='celeb_123', bucket=42): 50,000 reads/sec  ← HOT
  (conversation_id='conv_alice_bob', bucket=42): 3 reads/sec
  
  celeb_123 is a celebrity with 10M followers — viral moment triggered

ROOT CAUSE: All followers refreshing the celebrity's DM simultaneously

IMMEDIATE MITIGATION:

Option A — Cache the hot partition (if reads are cacheable)
  1. Add Redis cache: key = conversation:celeb_123:bucket_42
  2. TTL = 30 seconds (acceptable stale window for a read-heavy feed)
  3. Application reads from Redis first, falls back to Cassandra on miss
  4. Impact: 50K/sec → ~500/sec hitting Cassandra (99% cache hit)

Option B — Rate limit at application layer (if not cacheable)
  1. Limit refresh rate for this conversation to 1 req/5sec per user
  2. Serve stale content with "updated X seconds ago" indicator
  3. Background polling with exponential backoff

LONG-TERM FIX:

1. Partition splitting for celebrity accounts:
   Multiple buckets per time window: bucket = (time_window, random(0,9))
   Application writes to random bucket → reads from all 10 buckets
   Scatters the hot partition across 10× more nodes

2. Dedicated VIP infrastructure:
   Identify celebrity/high-traffic accounts programmatically
   Route to dedicated cache layer or dedicated Cassandra cluster
   Pre-warm cache on viral content detection

The L6/L7 Cassandra Design Checklist

When designing with Cassandra, a Principal Engineer considers all of these. Click any checkbox to track your review progress.

📐 Data Modeling ▾
Partition key has high cardinality (millions of unique values — no single key receives disproportionate traffic)
Partition size bounded (<100MB, <100K rows — calculated for worst-case user/entity, not average)
Clustering key matches query ORDER BY — physical sort order matches access pattern (DESC for recent-first)
Denormalized for each query pattern — one table per query shape, not one table for everything
No cross-partition queries in hot path — every hot-path query specifies the full partition key
TTL used instead of DELETE for ephemeral data — tombstone accumulation calculated and acceptable
Time bucketing applied where partition growth is unbounded by nature (events, logs, metrics)
⚖️ Consistency & Replication ▾
RF ≥ 3 in each DC for production — RF=1 or RF=2 are not safe for production workloads
LOCAL_QUORUM for critical writes — default safe choice unless write rate makes it prohibitive
LOCAL_ONE only for non-critical reads (logs, metrics, ephemeral data) — documented decision, not default
EACH_QUORUM only when global consistency required — rare, adds cross-DC latency, explicitly justified
Cross-DC replication lag considered — application handles eventual consistency window (push payloads, retry logic)
Single-DC evaluated for ephemeral data — real-time data may not benefit from cross-DC replication
💾 Storage & Compaction ▾
Compaction strategy matches workload: TWCS for time-series/TTL, LCS for read-heavy, STCS for write-heavy append-only
TTL window matches TWCS window — TWCS time_window should be 2× the TTL to capture a full expiry cycle
Tombstone accumulation monitored — tombstones/read alert set, data model reviewed if >1K tombstones/read P99
Compaction throughput sized for write rate — default 64MB/s insufficient for high-volume tables
Disk headroom for compaction — STCS requires 2× data space temporarily; disk utilization target is ≤70%
🔧 Operational Excellence ▾
Monitoring for pending compactions, SSTable count, P99 latency — all three correlated in dashboards
Repair scheduled — incremental repair preferred at scale, full repair on new nodes, scheduled during low-traffic
Backup strategy — snapshots (nodetool snapshot) + streaming backups; recovery procedure tested quarterly
Capacity planning accounts for compaction overhead — new features: write rate estimated before deploy
Runbooks exist for: compaction storm, hot partition, node down >3h (hint expiry), repair failure
🏗️ Architecture Patterns ▾
Cache layer for read-heavy access — EVCache/Redis in front of Cassandra; cache-miss rate monitored
Not used as a queue — no polling patterns with status filters; Kafka/SQS for async job processing
Secondary indexes used sparingly — only for low-cardinality + partition-key-constrained queries; denormalized otherwise
Application handles eventual consistency — retries with idempotency keys; push payloads carry data; stale reads tolerated or bounded
Hot partition mitigation plan exists — bucket splitting for celebrities/VIPs, cache layer for viral content
🎯 Series Complete — The Principal Engineer Mindset

Seven parts. Discord messages, Netflix history, Uber locations, Instagram DMs. Four real production systems, each teaching a different Cassandra lesson: partition key design, clustering key as physical layout, TTL over tombstones, cache as the read layer, cross-DC consistency gaps, TWCS temporal locality, single-DC for ephemeral data.

The through-line: there is no universal "correct" Cassandra configuration. Every choice — partition key, consistency level, compaction strategy, replication topology — is derived from the access pattern, durability requirement, write volume, and acceptable staleness window of the specific system. Cassandra gives you the most expressive set of levers in distributed databases. The L7 signal is knowing which ones to pull, in which direction, and being able to defend the trade-off you're making when you pull them.

The anti-patterns exist not because engineers are careless, but because Cassandra looks like a database and people use it like a database. It is not a database. It is a distributed, append-only, eventually-consistent key-value store with tunable consistency and pluggable compaction. Design to its strengths and it will handle 1.4M writes/sec without breaking a sweat. Fight against its model and a 50GB partition will bring down your whole cluster at 3am.
