Spatio-Temporal Indexing

What It Is — Time as a Dimension

One-sentence definition: Extending spatial indexes to include time, enabling queries like "Where was vehicle X at 3 PM?" or "What vehicles were within 1km of this point between 2–4 PM?"

The core insight: Time is just another dimension. A moving object traces a path in 3D (x, y, t) space. But time has special properties: it only moves forward, and it's the query dimension most frequently used for filtering — which enables aggressive time-based pruning that purely spatial indexes can't exploit.

Four Approaches to Spatio-Temporal Indexing

Approach 1 Time-Partitioned Spatial Indexes ▾
Time-Bucketed Spatial Index Structure
Index structure:
├── 2024-01-15-08:00 → GeoHash/S2 spatial index for this hour
├── 2024-01-15-09:00 → GeoHash/S2 spatial index for this hour
├── 2024-01-15-10:00 → GeoHash/S2 spatial index for this hour
└── ...

Query "drivers near P between 14:00–16:00":
  1. Retrieve index for 14:00, 15:00 buckets
  2. For each: query spatial index for nearby GeoHash cells
  3. Union results, deduplicate by driver_id
  4. Calculate exact distances, filter to radius
Pros and Cons
  • Pro: Simple — reuses existing spatial indexes. Each bucket is just a regular GeoHash/S2 lookup.
  • Pro: Old data easily archived or deleted — just drop old time buckets.
  • Pro: Bucket size is tunable — 1 minute for high-res, 1 hour for historical.
  • Con: Cross-boundary queries (e.g., 13:45–14:15) need two index lookups and a union.
  • Con: Storage grows linearly — every time bucket stores every object's location.
Approach 2 Space-Partitioned Temporal Indexes ▾
Per-Cell Time-Series
For each spatial cell, maintain a time-series:
├── Cell u09tun → [{t:08:00, drivers:[A,B,C]}, {t:08:01, drivers:[A,D]}, ...]
├── Cell u09tup → [{t:08:00, drivers:[E,F]}, {t:08:01, drivers:[E,G,F]}, ...]
└── ...

Query "history of cell u09tun from 14:00–16:00":
  1. Direct cell lookup — O(1)
  2. Scan time-series for the cell within range
  3. Very fast — all data is co-located by cell
Pros and Cons
  • Pro: "What happened in this location over time?" queries are extremely fast.
  • Pro: Natural fit for geofencing — "when was anyone in region R?"
  • Con: Moving objects cross many cells. A driver covering 10 cells in 1 hour writes to 10 different time-series — expensive storage and fan-out writes.
  • Con: "Where was driver X at time T?" requires finding which cell X was in — can't index by driver directly.
Approach 3 Combined Space-Time Indexing (3D R-Tree / Octree) ▾

Treats (x, y, t) as three dimensions and applies a 3D R-Tree or Octree directly. True spatio-temporal queries in one index pass — no partition lookups or unions needed.

The Fundamental Problem: Time Doesn't Behave Like Space
  • Space is symmetric — you can query north just as easily as south. Time is asymmetric — you only ever query the past.
  • Space is bounded — Earth has finite coordinates. Time is unbounded — it keeps growing.
  • 3D R-Trees degrade as the time dimension grows without bound — MBRs in the time direction become huge.
  • Result: Theoretically elegant, practically limited. Used in academic research more than production systems.
Approach 4 Trajectory Indexing — Index Paths, Not Points ▾

Instead of indexing point locations at each timestamp, index the trajectory — the entire path an object took as a curve in (x, y, t) space. Specialized structures include TB-Tree, SETI, STRIVE.

When Trajectory Indexing Makes Sense
  • Best for: "What path did vehicle X take between 2 and 4 PM?" (route reconstruction)
  • Best for: "Did any vehicle ever enter region R?" (trajectory-polygon intersection)
  • Not for: Real-time tracking — trajectory indexes are built over completed paths, not live positions
  • Used in: Fleet management analytics, sports analytics (ball tracking), animal movement ecology

Four Canonical Query Patterns

Pattern 1 — "Where was X at time T?"

Index by (entity_id, time). Retrieve location directly. No spatial index needed — it's a pure entity-time lookup. Very cheap.

Pattern 2 — "What was near point P at time T?"

Use a time-partitioned spatial index. Look up the appropriate time bucket. Query spatial index for that bucket. Two-step lookup.

Pattern 3 — "What was near P between T1–T2?"

Query all time buckets in range. Union the results. Deduplicate. Cost grows linearly with the time window — budget accordingly.

Pattern 4 — "Did X ever enter region R?"

Either retrieve all of X's positions and check against R (expensive), or use a trajectory index if this query is frequent. Trajectory intersection is the right abstraction.

L6 vs L7 Answer — Delivery Driver Complaint System

Interview question: "Design a system to track 10M delivery drivers and answer 'what drivers were within 1km of this complaint location in the past 2 hours?'"

L6 Answer — Correct but Surface Level

"I'd partition by time — hourly buckets. Each bucket has a GeoHash index. The query scans 2 hourly indexes, filters by GeoHash prefix and 8 neighbors, then calculates actual distances to keep only those within 1km."

L7 Answer — Access Pattern Reasoning + Quantified Expectations

"Time partitioning is right, but let me reason about access patterns first. Complaints almost always arrive within hours of the incident — hot data is recent. That shapes the architecture:

Real-time tier (last 24 hours): All driver positions in memory, GeoHash-indexed per 5-minute bucket. Query hits 2–3 buckets, GeoHash prefix at precision 5 (~5km) narrows to ~0.001% of drivers, exact distance filter finalizes. Expected latency: <20ms p99.

Archive tier (older data): GeoHash-partitioned Parquet files in columnar storage (BigQuery). Queried rarely — the 1% of complaints that are delayed. Higher latency acceptable.

Quantified: 99% of complaint queries hit only the real-time tier. 2-hour window = 24 five-minute buckets = ~0.0001% of the daily driver-position dataset. The GeoHash precision-5 filter reduces to ~0.001% of that. We're looking at perhaps 500 candidate drivers from 10M, then exact-distance filtering to ~30. Total data touched: ~50KB. This is why <20ms is achievable."

Complete Decision Framework with Reasoning

Decision Tree — Every Question With Its Reasoning
Q1: Are objects frequently moving? (>10K updates/sec)
│
├── YES → Avoid ALL tree structures (QuadTree, R-Tree, KD-Tree)
│         Why: Tree updates cascade. R-Tree: 7-step cascade per update.
│         QuadTree: can cause rebalancing. KD-Tree: no rebalance = degrades.
│   │
│   └── Q2: Do you need complex spatial queries (polygon, nearest-neighbor)?
│       ├── YES → Use S2 with cell covering
│       │         Why: S2 handles moving objects via cell ID update (O(1))
│       │         and supports sophisticated range queries with cell hierarchies.
│       │         Accept implementation complexity.
│       │
│       └── NO  → Use GeoHash or H3 grid — simple cell membership
│                 GeoHash: simpler, works with any KV store (Redis, Dynamo)
│                 H3: uniform cell size → better for aggregation and pricing
│
└── NO → Tree structures are viable
    │
    └── Q3: Are objects complex shapes (polygons, lines, not just points)?
        ├── YES → Use R-Tree / PostGIS
        │         Why: R-Tree is designed for shapes. MBRs bound complex geometry.
        │         Spatial joins (intersects, contains, within) are native.
        │         PostGIS GiST index is production-hardened R-Tree.
        │
        └── NO  → Points only
            │
            └── Q4: What is your primary query type?
                ├── Nearest-neighbor / k-NN → KD-Tree
                │   Why: Alternating splits create tighter NN pruning regions.
                │   Backtrack pruning eliminates entire branches for NN queries.
                │
                ├── Range / region queries → QuadTree
                │   Why: Uniform quartering maps naturally to rectangular regions.
                │   Easy to determine which quadrants overlap a search rectangle.
                │
                └── Both NN and range, or uncertain → S2
                    Why: Most versatile. S2 cell hierarchy supports both.
                    Accepts complexity cost for flexibility.

Q5: Do you need spherical Earth accuracy?
├── YES → Use S2 or H3 (NOT GeoHash)
│         GeoHash distorts near poles. Cells are not equal-area.
│         S2/H3 are designed for the sphere — cells are approximately equal-area.
│
└── NO  → GeoHash is fine for regional (non-polar, non-antimeridian) services

Q6: Is uniform spatial aggregation important?
├── YES → Use H3 (hexagonal uniformity)
│         Every H3 hexagon has 6 equidistant neighbors.
│         Pricing zones, surge areas, density maps work better with H3.
│
└── NO  → S2 or GeoHash based on other factors

Q7: Do you need tight SQL/database integration?
├── YES → GeoHash (works with any KV or string-indexed DB)
│         or PostGIS R-Tree (if using PostgreSQL)
│
└── NO  → Any option based on above factors
🔑 Sound Bite — Comparing All Variants

"The choice between spatial indexes comes down to three questions: Are objects moving frequently? Do I need complex shape queries? Does spherical accuracy matter? For static data with complex shapes, R-Tree wins. For moving objects with simple proximity queries, GeoHash or H3 grids win. For anything at Google scale with accuracy requirements, S2 is the robust choice. Trees give you query flexibility; grids give you update performance. You rarely get both."

L6/L7 Insight — The Coordination Cost Is the Hidden Cost

"On a single node, an R-Tree insert might trigger a split — this takes microseconds. In a distributed system, that same split requires: (1) Lock the leaf node, (2) Lock the parent, (3) Decide split algorithm, (4) Update routing table, (5) Replicate changes to followers. The microsecond operation becomes 50ms minimum, and you've created a hot-path bottleneck on every write. This is why GeoHash and S2 dominate in distributed systems — their O(1) cell ID computation has no coordination requirement."

The Replication Lag Consistency Challenge

Single-node: "Is driver A in cell X?" has one source of truth. Distributed with replication: Driver A's location is on shard 1. A rider querying shard 2 may see stale data — driver A moved 30 seconds ago but still appears at old location. For Uber, 30-second staleness is acceptable (the driver is on the same street). For a nuclear plant perimeter alert, it is not. Always ask the interviewer: what's the staleness tolerance?

Fail-Open vs Fail-Closed

Fail-Open — Return Results Even If Degraded

If spatial index is unavailable, scan all data.

Pro: Results are still returned — user experience not completely broken.

Con: 1,000x latency spike. Scanning all data instead of 0.1% of data. Downstream systems overloaded.

When correct: Safety-critical queries — "find nearest hospital" should fail-open, even if slow. A wrong answer (no hospitals) is worse than a slow answer.

Fail-Closed — Return Error if Index Unavailable

If spatial index is unavailable, return an error immediately.

Pro: Predictable failure. No cascading load on downstream systems.

Con: Feature completely unavailable during index outage.

When correct: Most location-based services — "find nearby restaurants" should fail-closed. Returning 10 million restaurants is worse than returning an error with a retry message.

Edge Cases Most Candidates Miss

Edge Case 1 The Antimeridian Problem — The 180° Boundary ▾
The problem The antimeridian is the 180°E / -180°W longitude line, running through the Pacific Ocean. A query centered near Fiji spans roughly +170° to -170° longitude — but naively, a bounding box from -170° to +170° includes almost the entire Earth. GeoHash failure GeoHash prefixes "u" and "b" represent locations on opposite sides of the antimeridian — they are geographically adjacent (across the Pacific) but share no common prefix. A prefix-based search near the antimeridian returns nothing from the other side. Bounding box failure min_lng = -170, max_lng = +170 misses the antimeridian crossing entirely, scanning almost the entire planet instead of a local area. Solution Detect crossing: If max_lng < min_lng after a standard query expansion, the query crosses the antimeridian. Split into two separate queries: one for [lng_min, 180°] and one for [-180°, lng_max]. Union results. S2 and H3 handle this correctly internally.
Edge Case 2 The Pole Singularity — Search Radius Near lat=90° ▾
The problem At the North Pole (lat=90°), all longitude lines converge at a single point. A "1km radius" search becomes a spherical cap — not a circle in lat/lng space. GeoHash failure GeoHash cells become infinitely thin wedges near the poles. A query for nearby objects may require checking hundreds of cells, each covering a tiny sliver of longitude. Cell sizes are wildly non-uniform. Index failure Any flat-Earth index (GeoHash, naive bounding box) returns wildly incorrect results or misses nearby objects entirely near the poles. Solution Detect polar region (|lat| > 85°) and switch to spherical math explicitly. S2 and H3 are designed for the sphere — they handle poles correctly without special cases. For GeoHash-based systems: add a warning that the service doesn't support polar regions, which is acceptable for most commercial applications.
Edge Case 3 The Duplicates Problem — 10,000 Drivers in One Airport Lot ▾
The problem 10,000 Uber drivers wait in the same airport holding lot. They all report the exact same (or nearly identical) GPS coordinate — within centimeters of each other. QuadTree failure QuadTree keeps splitting trying to separate the points — eventually reaching maximum depth with many nodes each holding a single point. For truly identical coordinates, the split is infinite — you hit the minimum cell size limit. R-Tree failure All 10,000 objects end up in one massive MBR — a single point. This MBR matches every query near the airport, causing massive false positive fan-out. Solution Bucket multiple objects at the same location — store a list, not individual entries, per cell below a minimum precision. Or add random jitter (±0.0001°) to avoid exact collisions. Or use a separate data structure for the "waiting pool" pattern — a simple queue is better than a spatial index for objects that aren't spatially distinguishable.
Edge Case 4 The Dense City Problem — Manhattan vs Montana ▾
The problem Manhattan has ~1M restaurants in 20 km². Montana has ~1,000 restaurants in 300,000 km². A fixed GeoHash precision returns 10,000 results in NYC for a "nearby" query, and 0 results in rural Montana. Fixed precision failure GeoHash precision 6 (city block) works perfectly in Manhattan but fails in Montana — the search cell has no results. Precision 3 (province) works in Montana but returns the entire borough in NYC. User experience failure NYC user sees 10,000 results with no way to filter. Montana user sees 0 results with no way to expand. Both are bad UX driven by the same fixed-precision choice. Solution Adaptive precision: Start at high precision, expand outward until you have N results (e.g., N=20). Cap expansion at max radius. Alternatively, use a two-pass approach: coarse pass to find density, fine pass for results. Or use pagination with a radius cursor — return the first 20 results and the radius used, letting the client request more.
Edge Case 5 The International Date Line — Time Partitions at 180° ▾
The problem A ship in the Pacific crosses the International Date Line at 23:30. West of the line: it's January 16th. East of the line: it's January 15th. The ship's events span two calendar dates even though only minutes elapsed. Time-partition failure A time-partitioned spatial index has the ship's events in two different date buckets. A query for "all events in the past 2 hours" may miss events in the adjacent date partition. Why it matters Shipping, aviation, maritime tracking — any system where objects cross the date line. Also applies to Daylight Saving Time transitions (clocks jump forward/back, creating duplicate or missing hours). Solution Always use UTC internally — never local time for time partitions. UTC has no date line or DST ambiguities. For queries near partition boundaries: query both adjacent partitions and union results. Add a buffer (e.g., extend both ends of the time range by one partition width) when forming partition keys from user-provided local times.
🔑 Sound Bite — Trade-offs

"The fundamental tension in spatial indexing is between query flexibility and update performance. Tree structures like R-Trees adapt beautifully to data distribution and support sophisticated queries, but every insert potentially cascades through the tree. Grid systems like GeoHash give O(1) updates — just change the cell ID — but range queries become 'query all relevant cells.' At Uber scale with millions of moving drivers, update performance dominates. For Google Maps indexing static roads, R-Trees make sense because you bulk-load once and query forever."

The L7 pattern across all three parts: every spatial index decision is a workload decision, not a data structure preference.
