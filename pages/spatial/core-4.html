2.6

H3 — Uber's Hexagonal System

Hierarchical hexagons with exactly 6 equidistant neighbors for uniform spatial analysis
A

What It Is — The Mental Model

H3 divides Earth into hierarchical hexagonal cells, where each hexagon has exactly 6 equidistant neighbors, making it ideal for uniform proximity analysis and spatial aggregation.

The Core Insight

Hexagons are the only regular polygon that tiles a plane where every neighbor is equidistant. Squares have corner neighbors that are √2 farther than edge neighbors. Hexagons don't have this problem.

The Inventor's Problem

"We're Uber. We need to aggregate supply/demand by region, compute surge pricing by area, and match riders to drivers. Square grids have annoying corner effects. We need uniform spatial regions."

B

Icosahedron projection → hexagonal tiling → hierarchical refinement (aperture 7) → 64-bit cell ID.

Why Hexagons? The Geometry Argument
❌ Square Grid Problem

Edge neighbors: distance = 1

Corner neighbors: distance = √2 ≈ 1.41

Corner neighbors are 41% farther — "nearby" isn't uniform. Algorithms that treat all 8 neighbors equally introduce systematic bias.

✅ Hexagonal Grid Solution

All 6 neighbors: distance = 1 (exactly equal)

Every neighbor is equidistant. Proximity analysis, averaging, and gradient computations are all geometrically unbiased.

K-ring queries (all cells within k steps) produce truly uniform coverage.

The Pentagon Problem
Why Pentagons Exist

You can't tile a sphere purely with hexagons. H3 has exactly 12 pentagons at each resolution (at the vertices of the underlying icosahedron). These pentagons have 5 neighbors instead of 6 and are deliberately placed in oceans to minimize their impact on real-world queries.

H3 Core Operations
▸
Point to Cell

(lat, lng, resolution) → H3 index. Deterministic. Different resolutions return different cells for the same point.

▸
K-ring

Get all cells within k steps (neighbors of neighbors). K=1 → 7 cells. K=2 → 19 cells. K=n → 3n²+3n+1 cells.

▸
Polyfill

Given a polygon boundary, return all H3 cells that intersect it. Essential for "find all drivers in this city zone" queries.

▸
Parent / Children

Navigate hierarchy — each cell has one parent at coarser resolution and ~7 children at finer resolution (aperture 7).

D

Limitations & Failure Modes

No natural total ordering means range queries don't work — you must enumerate cells manually.

01
Not Great for Range Queries

H3 indices aren't ordered like S2/GeoHash. "Find all cells in range X–Y" doesn't work naturally. Must enumerate k-rings or use polyfill.

02
Pentagon Edge Cases

12 pentagons per resolution with only 5 neighbors. Placed in oceans, but all algorithms must handle the pentagon case explicitly.

03
Unclean Hierarchy

Aperture 7 means 1 hex → ~7 children (not exact). Some children hexes stick out of parent boundary. Parent-child containment isn't perfect.

04
Less Ecosystem Support

Fewer databases natively support H3 vs GeoHash. More client-side computation required. Fewer native index optimizations.

⚠️ Gotcha Scenario — Sharding by H3

"You shard your database by H3 cell at resolution 4. You have 122 shards (122 level-4 cells cover your region). But H3 cells don't have a total ordering — you can't do range queries across shards easily. When you need 'all drivers in San Francisco,' you must identify all relevant cells and query each shard individually."

E

Operational Considerations

Cell population monitoring, k-ring tuning, and integration with analytics stacks.

Monitoring
📊
Cell Population Distribution

Detect hot cells — high-density urban cores can dwarf surrounding cells at the same resolution.

🔢
K-ring Query Sizes

K=1 → 7 cells, K=2 → 19, K=3 → 37. Monitor fan-out to prevent runaway query costs at high k values.

⬠
Pentagon Proximity

Rare, but track when queries touch pentagon cells — neighbor counts differ (5 vs 6) and can cause off-by-one bugs.

Integration Patterns
PostgreSQL / PostGIS

H3 extension available. Store cell IDs as bigint, use standard B-tree index for point lookups. Range queries require explicit cell enumeration via h3_k_ring().

Spark / Flink

H3 UDFs for streaming pipelines. Ideal for real-time aggregation: tag each event with H3 cell, then GROUP BY h3_cell for demand heatmaps.

Redis

Store as int64, but no native range advantage unlike S2. Use sorted sets keyed by resolution for hierarchical lookups.

F

The L6/L7 Nuances

The H3 vs S2 decision and the non-obvious aperture-7 hierarchy trap.

L6/L7 "When would you choose H3 over S2?"

"H3 when my primary operations are aggregation and neighborhood analysis — average surge in this cell, drivers within 2 hexes. The uniform 6-neighbor property simplifies algorithms considerably."

"S2 when I need efficient range queries and database integration — S2's Hilbert ordering means range scans work naturally on B-tree indexes. S2 also has a cleaner hierarchy (parent always contains children)."

"In practice, Uber uses H3 for analytics and visualization and likely S2-like approaches for real-time driver matching — they use different tools for different problems."

Non-Obvious Insight — Aperture-7 Containment Trap

"H3's aperture 7 creates a peculiar property: a child cell might be closer to a different parent's center than its actual parent. This matters for algorithms that assume hierarchical containment. If you're doing 'find the level-4 cell containing this point and count level-8 children,' the count isn't exactly 7⁴ — the boundary children may belong to adjacent level-4 cells."

2.7

Z-order Curve vs Hilbert Curve

Space-filling curves deep dive — the mathematical engine behind GeoHash and S2
A

What They Are

Space-filling curves that map 2D spatial coordinates to 1D values — the key choice that separates GeoHash (Z-order) from S2 (Hilbert).

Z-ORDER (MORTON CODE) — used by GeoHash
  • Interleave bits of coordinates
  • Simple to compute — pure bit operations
  • Has "jumps" where sequential values aren't spatial neighbors
  • 2D locality → 1D has discontinuities
HILBERT CURVE — used by S2
  • Continuous fractal curve visiting every cell exactly once
  • More complex to compute (lookup table or recursive)
  • Preserves locality better than Z-order
  • Consecutive values are always spatial neighbors
The Core Trade-off

Z-order: Fast to compute, simple bit operations, but 2D locality → 1D has discontinuities.    Hilbert: Better locality preservation, but more complex computation.

B

How They Work

Concrete encoding walkthrough and cell numbering — the diagrams that make the locality difference visual.

Z-order (Morton) Encoding — Step by Step
Z-ORDER ENCODING 
Point (x=5, y=3):
x = 5 = 101 in binary
y = 3 = 011 in binary

Interleave (y₂x₂y₁x₁y₀x₀):
y₂=0, x₂=1, y₁=1, x₁=0, y₀=1, x₀=1
= 010111 = 23 in decimal

So (5,3) → Morton code 23

The Key Difference

In Z-order, cells 3 and 4 are adjacent in 1D but live in completely different spatial quadrants. In Hilbert, every step from N to N+1 is a move to a spatially adjacent cell — guaranteed, with no exceptions.

C

Why Hilbert Wins for Databases

Range queries on Hilbert IDs require fewer ranges — translating directly to fewer disk seeks, fewer B-tree lookups, and better cache utilization.

The Range Query Problem — Concrete Examples
Query: "Find all points in the upper-left 2×2 quadrant"

Z-order: Cells 0,1,2,3 → range query "0 to 3" ✓ works   |   Hilbert: Cells 0,1,2,3 → range query "0 to 3" ✓ works   |   Tie for this shape.

Query: "Find all points in the left 2×4 column"

Z-order: Cells 0,1,2,3,8,9,10,11 → needs TWO ranges: "0–3" AND "8–11"   |   Hilbert: Path is more spatially coherent → fewer ranges needed on average

The Key Metric

Average number of contiguous ranges needed for arbitrary rectangular queries. Hilbert curve requires fewer ranges on average because its path minimizes jumps. This means fewer disk seeks, fewer B-tree lookups, and better cache utilization.

30–50% Fewer ranges needed (Hilbert vs Z-order)
~40% Faster wall-clock (5 ranges vs 8 ranges = 40% speedup)
Empirical For typical rectangular spatial queries
Why This Isn't Just Big-O Academic

"The difference between Z-order and Hilbert seems academic until you're doing range scans on Bigtable or Spanner where each range requires a separate tablet lookup. A query that needs 5 ranges instead of 8 ranges is 40% faster in wall-clock time, not just big-O equivalent."
