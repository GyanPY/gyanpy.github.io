2.4

GeoHash

Encoding lat/lng as strings where shared prefixes mean proximity
A

What It Is — The Mental Model

GeoHash encodes latitude/longitude into a single alphanumeric string where shared prefixes mean proximity — "dr5ru" and "dr5rv" are neighbors because they share "dr5r".

The Core Insight

By interleaving bits of latitude and longitude and encoding as base32, we convert 2D coordinates to 1D strings where lexicographic proximity approximates spatial proximity. This means standard database indexes (B-trees on strings) become spatial indexes.

The Inventor's Problem

"I want to use my existing database infrastructure — standard indexes, range queries, caching — for spatial queries. I don't want to implement a custom spatial index."

B

How It Works — Mechanics Deep-Dive

Interleaved binary encoding of latitude and longitude, base32-encoded into an alphanumeric string.

Step-by-Step Encoding

Encoding the Eiffel Tower: lat=48.8584, lng=2.2945

1
Binary latitude — bisection Range [-90, 90], midpoint 0. 48.8584 > 0 → bit=1, new range [0,90]. Range [0,90], midpoint 45. 48.8584 > 45 → bit=1, range [45,90]. Continue… bits: 1 1 0 0 0 1…
2
Binary longitude — same process Range [-180, 180], midpoint 0. 2.2945 > 0 → bit=1, range [0,180]. Range [0,180], midpoint 90. 2.2945 < 90 → bit=0, range [0,90]. Continue… bits: 1 0 0 0 0 0…
3
Interleave bits (lng-lat alternating) lng[0], lat[0], lng[1], lat[1]… → 1, 1, 0, 1, 0, 0, 0, 0, 0, 1…
4
Base32 encode — group 5 bits → character 11010 = 26 = 'u' · 00000 = 0 = '0' · Result: "u09t…"
Result

Eiffel Tower GeoHash (precision 8): u09tunqu

C

Strengths & Ideal Use Cases

Spatial queries on standard infrastructure — no custom indexes required.

✅ WHERE IT SHINES
  • Existing relational databases (just add a string column)
  • Sharding (prefix-based partitioning is natural)
  • Caching (GeoHash is perfect cache key)
  • Simple proximity queries at fixed precision
  • High write throughput (update = just changing a string)
✅ THE SWEET SPOT
  • Redis / Elasticsearch / DynamoDB (no custom indexes)
  • Need to shard by geography
  • Queries are "find in general area" not exact distances
  • High update volume (moving objects)
Real-World Users
MongoDB (uses GeoHash internally) Redis GEO commands (GEOADD, GEORADIUS) Elasticsearch geo_point Couchbase spatial views
D

Limitations & Failure Modes

The boundary problem is fundamental and always requires neighbor queries — understanding this is the L6/L7 differentiator.

The Critical Boundary Problem
DIAGRAM — BOUNDARY PROBLEM 
┌────────┬────────┐
│ u09tun │ u09tup │
│        │   A    │  ← A and B are 50 meters apart
├────────┼────────┤     but in different cells
│ u09tuk │ u09tum │
│   B    │        │
└────────┴────────┘

Prefix of A: u09tup
Prefix of B: u09tuk
Shared prefix: u09tu (5 chars)

But points C and D might share "u09tun" 
and be 1km apart (diagonal corners of cell)
Key Insight

Shared prefix means possible proximity, not certain proximity. And a different prefix means NOTHING — neighbors can have wildly different prefixes. You must always query neighboring cells.

Where It Breaks Down
01
Boundary Problem (Critical)

Two points 10 meters apart across a cell boundary have completely different GeoHashes at that precision level. Must always query neighboring cells.

02
Z-order Discontinuities

GeoHash uses Z-order curve (Morton code). Z-order has "jumps" where sequential cells aren't spatial neighbors. Must always check all 8 neighbors.

03
Precision Trade-off

Too low → cells are huge, too many results. Too high → many cells to query for radius search. No "just right" for variable-radius queries.

04
Earth Edge Cases

Antimeridian (180°/−180°): Neighbors have completely different prefixes. Poles: Cell shapes become wedges, behavior is unpredictable.

⚠️ Gotcha Scenario

"Your 'find drivers within 2km' query uses GeoHash precision 6 (1.2km cells). A user is at the corner of a cell. You query 9 cells (center + 8 neighbors). But 2km radius can extend into cells beyond immediate neighbors! Your query misses valid drivers."

Common Mistake

Assuming GeoHash prefix equality means proximity. "u09tun" and "u09tup" might be neighbors. "u09tun" and "gbpbp" might ALSO be neighbors (across prime meridian edge case).

E

Operational Considerations

Tuning precision, monitoring post-filter ratios, and database indexing strategies.

Monitoring
📊
Cell Population Histogram

Detect hotspots — cells with disproportionate entity counts cause unbalanced shard load

🔢
Neighbor Query Overhead

How many cells are queried per request? Spikes indicate precision mismatch with query radius

🎯
Post-Filter Ratio

Results fetched vs. results returned. A high ratio (e.g. 100 fetched, 5 returned) means wrong precision level

Tuning
Precision Level

Balance cell count vs. result set size. For a 1km radius query, precision 6 (1.2km cells) is typically the sweet spot — 9 cells cover the area without massive over-fetch.

Neighbor Depth

Query 8 neighbors? 24? Depends on radius vs. cell size. If your search radius exceeds cell diagonal, extend to 24 (second ring of neighbors).

Database Optimization
▸
B-tree on GeoHash string

Prefix range queries (LIKE 'u09t%') use the B-tree efficiently. This is the core value proposition of GeoHash.

▸
Composite Index

(GeoHash, entity_type) for filtered queries — e.g. "find only restaurants near me, not all entities."

▸
Multiple Precisions

Store GeoHash at multiple precisions (e.g. 4, 6, 8) for different query types. Trade storage for query flexibility.

F

The L6/L7 Nuances

Deep questions and non-obvious insights that separate strong candidates from exceptional ones.

L6/L7 "If GeoHash has the boundary problem, why is it still so popular?"

"Because GeoHash's boundary problem is predictable and fixable — always query neighbors. The benefit is you get spatial indexing for FREE in any database that supports string indexes. The alternative is implementing a custom spatial index, which is complex, or using a specialized spatial database, which adds operational burden."

"For most applications, querying 9 cells instead of 1 is a tiny price for using standard infrastructure. The boundary problem becomes critical only when you need exact-distance queries or variable-radius searches — then GeoHash's limitations matter more."

Non-Obvious Insight — The Real Power

"GeoHash's real power is as a sharding key. 'All data with GeoHash prefix u09' goes to shard 1. Geographic locality means queries typically hit one shard. This is why Redis, DynamoDB, and similar systems love GeoHash — it solves the distributed systems problem of 'how do I partition spatial data' elegantly."

2.5

S2 Geometry — Google's Approach

Cube projection + Hilbert curves for mathematically rigorous spherical indexing
A

What It Is — The Mental Model

S2 projects Earth onto a cube, then unfolds the cube into a Hilbert curve, creating a hierarchical cell system where every point on Earth has a 64-bit ID and nearby points have similar IDs.

The Core Insight

The Earth is a sphere, not a flat plane. S2 handles this properly. By projecting onto a cube and using Hilbert curves, S2 maintains consistent cell sizes across the entire planet and preserves locality better than Z-order (GeoHash).

The Inventor's Problem

"GeoHash distorts near poles, has discontinuities, and cell sizes vary wildly by latitude. We need a system that's mathematically rigorous for a spherical Earth at Google scale."

B

How It Works — Mechanics Deep-Dive

Cube projection → face assignment → recursive subdivision → Hilbert ordering → 64-bit integer cell ID.

Step-by-Step
1
Cube projectionProject Earth's sphere onto an enclosing cube (6 faces)
2
Face assignmentEach point on Earth maps to one of the 6 cube faces (0–5)
3
Recursive subdivisionEach face subdivides into 4 quadrants, recursively, up to level 30
4
Hilbert orderingCells numbered using a Hilbert space-filling curve, not Z-order — preserving locality
5
Cell ID assigned64-bit integer: 3 bits face + up to 60 bits position along Hilbert curve
S2 CELL ID STRUCTURE 
S2 Cell ID structure (64 bits):
┌──────────┬──────────────────────────────────────────────────────┐
│ Face (3) │ Position along Hilbert curve (up to 60 bits)        │
└──────────┴──────────────────────────────────────────────────────┘

Level determines precision:
- Level 0:  ~7,840 km² (1 face, huge)
- Level 10: ~22 km²
- Level 15: ~0.06 km² (few blocks)
- Level 20: ~6 m² (room)
- Level 30: ~1 cm² (max precision)
Why Hilbert Curve Instead of Z-order (GeoHash)?
DIAGRAM — HILBERT vs Z-ORDER 
Z-order (GeoHash uses this):        Hilbert curve (S2 uses this):
┌───┬───┬───┬───┐                   ┌───┬───┬───┬───┐
│ 0 │ 1 │ 4 │ 5 │                   │ 0 │ 1 │14 │15 │
├───┼───┼───┼───┤                   ├───┼───┼───┼───┤
│ 2 │ 3 │ 6 │ 7 │  Jump!            │ 3 │ 2 │13 │12 │  No jumps!
├───┼───┼───┼───┤  3→4 are          ├───┼───┼───┼───┤  Every step
│ 8 │ 9 │12 │13 │  neighbors in     │ 4 │ 7 │ 8 │11 │  is a neighbor
├───┼───┼───┼───┤  1D but not 2D    ├───┼───┼───┼───┤
│10 │11 │14 │15 │                   │ 5 │ 6 │ 9 │10 │
└───┴───┴───┴───┘                   └───┴───┴───┴───┘

In Z-order, querying "3 to 7" includes 4,5,6 which are far away!
In Hilbert, querying "3 to 7" = {3,4,5,6,7} all spatially connected!
Z-ORDER (GeoHash)
  • Has spatial "jumps" — 3→4 are 1D neighbors but spatially distant
  • Range queries include non-nearby cells
  • Requires checking all 8 neighbors always
  • 2D proximity ≈ 1D proximity (approximate, with gaps)
HILBERT CURVE (S2)
  • No spatial jumps — every sequential step is a spatial neighbor
  • Range queries return spatially coherent results
  • Better locality preservation for range scans
  • 2D proximity ≈ 1D proximity (tighter mapping)
S2 Core Operations
▸
Point to Cell

(lat, lng) → S2CellId at any desired level (0–30). Deterministic, reversible.

▸
Covering

For any shape (polygon, radius circle), compute minimal set of S2 cells that cover it. Sophisticated algorithm with configurable min/max level.

▸
Range Query

Cell IDs are integers → efficient range queries. WHERE cell_id BETWEEN a AND b on B-tree index.

▸
Neighbors

Find adjacent cells — guaranteed to be spatially adjacent due to Hilbert ordering. No "fake" neighbors from curve jumps.

C

Strengths & Ideal Use Cases

Global applications, arbitrary shape covering, and systems already integrated with Google's stack.

✅ WHERE IT SHINES
  • Spherical Earth accuracy (poles, antimeridian, int'l date line)
  • Better locality preservation (Hilbert > Z-order)
  • Variable precision — choose any level 0–30
  • Region covering (cells covering arbitrary shapes)
  • Google-scale systems (built by Google for Google)
✅ THE SWEET SPOT
  • Global apps where consistency across latitudes matters
  • Need to cover complex regions (polygons, circles)
  • Range queries on cell IDs (Bigtable, Spanner)
  • Systems already using the Google stack
Real-World Users
Google Maps Google Earth Uber (partially) Tinder Foursquare / Swarm Google internal systems
D

Limitations & Failure Modes

Complexity, square cells, and choosing the right covering level are the operational challenges.

01
Complexity

More complex than GeoHash to understand and implement. Library required. Debugging 64-bit integer cell IDs is harder than reading alphanumeric GeoHashes.

02
Square Cells

S2 cells are distorted squares. Circles require many cells to cover. Hexagonal systems (H3) can be a better fit for some use cases.

03
Cell Boundary Still Exists

Hilbert reduces but doesn't eliminate boundary issues. Still need to handle neighbors for exact proximity queries.

⚠️ Gotcha Scenario — Covering Level Selection

"You compute an S2 covering for a 1km radius circle. At level 15, you get 5–12 cells depending on position. Your database query becomes WHERE cell_id IN (c1, c2, …, c12). If you use too fine a level, you get 50+ cells. If too coarse, cells include too much area outside your circle. Choosing the right level requires understanding your query patterns."

E

Operational Considerations

Covering size monitoring, level tuning, and integration patterns with Google infrastructure.

Monitoring
📐
Covering Size Distribution

Track cells per query. Outliers indicate complex shapes or suboptimal level choices

🔬
Level Distribution

Are you using consistent levels? Mixed levels complicate index efficiency and query planning

⚡
Range Query Efficiency

Sequential cell ID reads = fast. Scattered cell IDs = random I/O. Monitor index access patterns.

Tuning
Min/Max Covering Levels

Set min_level and max_level on the covering algorithm to balance precision vs. cell count. For 1km radius queries, level 13–15 is typical.

GetCovering vs GetInteriorCovering

GetCovering: All cells that intersect the shape (over-estimates slightly). GetInteriorCovering: Only cells fully inside the shape (under-estimates). Choose based on whether false positives or false negatives are more costly.

Integration Patterns
▸
Bigtable / Spanner

S2 cell IDs as row keys. Range scans on integers are highly efficient. Geographic locality means physically co-located data.

▸
PostgreSQL

Store as int64, use standard B-tree index. Range queries work natively without PostGIS extension.

▸
Redis

Sorted set with cell ID as score. ZRANGEBYSCORE becomes a spatial range query. Elegant and fast.

F

The L6/L7 Nuances

The deep questions that test whether you truly understand S2 vs. GeoHash — and Google's engineering decisions.

L6/L7 "Why did Google create S2 when GeoHash already existed?"

Three reasons:

(1) Spherical accuracy — GeoHash treats lat/lng as a flat grid, which severely distorts at high latitudes and has edge case nightmares at poles and the antimeridian. S2's cube projection handles the sphere properly.

(2) Hilbert curve — S2 uses Hilbert instead of Z-order, giving better locality preservation. Fewer "jumps" means range queries on cell IDs return more spatially coherent results.

(3) Covering — S2 has sophisticated algorithms to compute minimal cell sets covering any shape. GeoHash requires manual neighbor computation. For a system like Google Maps indexing every road segment on Earth, these differences compound massively.

Non-Obvious Insight — Hierarchical Bit Operations

"S2's cell ID is structured so that parent-child relationships are simple bit operations. Cell ID 'X' at level N has children 'X×4', 'X×4+1', 'X×4+2', 'X×4+3' at level N+1. This makes hierarchical queries trivial — 'find all POIs in California' can be expressed as range queries on cell ID ranges, with no custom spatial logic needed."

"The GeoHash vs. S2 choice comes down to operational simplicity vs. mathematical correctness. GeoHash gives you spatial indexing for free in any database that supports strings — the boundary problem is a small, predictable tax. S2 is the right choice when you need global accuracy, arbitrary shape covering, or are already on Google infrastructure — its Hilbert curve and cube projection eliminate entire classes of edge cases that bite you at scale."
