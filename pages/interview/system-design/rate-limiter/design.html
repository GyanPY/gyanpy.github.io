01

Problem Clarification

Phase 1 · 0–5 minutes

The interviewer leans back and says: "Design a rate limiter." That's it. That's all you're given. The clock is ticking. What do you do first?

❌ No Hire
L4 / Weak L5 response
NO HIRE ▼

What they say: "Okay, so a rate limiter limits the rate of requests. We'll need to track how many requests each user makes. I'm thinking we can use Redis to store counters — it's fast and supports atomic increments. We'll have an API gateway that checks Redis before forwarding requests. If the count exceeds the limit, we return a 429. For the algorithm, I'll use a sliding window because it's more accurate than fixed windows. Let me draw the architecture..."

Why this fails
  • Jumps straight to solution without understanding the problem
  • Assumes "user" is the rate limit key without asking
  • Assumes single algorithm fits all cases
  • No clarification on where this runs (API gateway? Service level? Client?)
  • Picked Redis and sliding window without knowing requirements
Interviewer's notes

"Jumped to implementation. No requirements gathering. Concerning — likely builds features without understanding problem space."

Fatal Mistakes

  • Zero clarifying questions asked
  • Technology choices made with no context
  • Assumed scope (user-based rate limiting only)
⚠️ Lean Hire
Meets L5, not L6
LEAN HIRE ▼

What they say: "Before I dive in, I'd like to ask some clarifying questions. First, what's the scale we're designing for — how many requests per second? Second, is this for a specific service or a general-purpose rate limiter? Third, what should happen when a user is rate limited — hard reject or graceful degradation? And what are we rate limiting by — user ID, IP address, API key?"

What's good
  • Pauses to ask questions (good instinct)
  • Thinks about scale
  • Considers the rate limit key dimension
What's missing for L6
  • Questions are a checklist, not structured thinking
  • Doesn't categorize questions (functional vs. non-functional)
  • Doesn't ask about existing infrastructure or constraints
  • Doesn't ask WHY we need a rate limiter (threat model, cost control, fairness?)
  • Misses distributed systems complexity (is this single-node or global?)
Interviewer's notes

"Asked questions, which is good. But felt like a memorized checklist. Didn't demonstrate deep understanding of problem space."

Upgrade needed

To reach L6: structured approach to clarification, understanding of business context (WHY), and awareness of where the hard problems lie in distributed rate limiting.

✅ Hire
Meets L6 bar
HIRE · L6 ▼

What they say: "Thanks for the problem. Rate limiting is deceptively complex because the hard parts depend heavily on context. Before I architect anything, I want to make sure I understand the problem space.

Let me structure my questions into three areas:

First, the business context: What's the primary goal here? Are we protecting against abuse and DDoS? Enforcing API quotas for a paid tier system? Ensuring fairness across tenants? Or protecting downstream services from overload? The goal shapes everything.

Second, the scope: Is this a centralized rate limiting service that multiple teams will use, or embedded rate limiting for a specific API? Will this run at the edge, at the API gateway level, or within individual services? And what's our deployment model — single region or globally distributed?

Third, constraints: Do we have existing infrastructure I should leverage — an API gateway like Kong, a service mesh like Istio? What's our tolerance for false positives versus false negatives? In other words, is it worse to accidentally rate limit a legitimate user, or to let some abuse through?

The reason I ask these upfront is that a rate limiter for a single-region API with 10K QPS looks completely different from a globally distributed system handling millions of QPS with strict consistency requirements."

L6 signals demonstrated
  • Structured thinking (three clear categories)
  • Business context awareness (asks WHY, not just WHAT)
  • Understands where complexity hides (distributed, consistency, edge vs. origin)
  • Shows experience by noting the problem is "deceptively complex"
Interviewer's notes

"Strong start. Structured approach to ambiguity. Understands that requirements drive architecture. Has clearly thought about rate limiting before."

Key phrases that signal L6

"The goal shapes everything" "Let me structure my questions into..." "A rate limiter for X looks completely different from Y" "What's our tolerance for false positives vs. false negatives?"
What separates this from L5

"L5 asks WHAT questions (scale, format, features). L6 asks WHY questions (business goal, tradeoff preferences) and shows awareness of WHERE the hard problems are."

🏆 Strong Hire
L7 signal — Exceptional
STRONG HIRE · L7 ▼

What they say: "Great problem — rate limiting sits at an interesting intersection of distributed systems, product policy, and operational reality. Before I go anywhere, I want to understand what success looks like.

On the problem itself: Rate limiters serve different masters. If we're protecting against DDoS, we care about L3/L4 rate limiting at the edge — probably not what you want me to design. If we're enforcing API quotas for a SaaS product, we care about accuracy, fairness, and customer experience. If we're protecting a fragile downstream service, we care about load shedding and backpressure. Each of these has a different architecture. Which flavor are we discussing?

On the distributed systems angle: The fundamental tension in distributed rate limiting is that truly accurate global rate limiting requires coordination, but coordination kills performance and availability. Most real systems accept some inaccuracy — the question is how much. Should I assume we're okay with a system that might let through 105 requests when the limit is 100, in exchange for being highly available and low-latency?

On the deployment context: Where does this run in the request path? If we're building into an existing API gateway, I'll focus on the algorithm and storage layer. If we're building a standalone rate limiting service that other services call, I need to think about the additional network hop, failure modes, and whether this becomes a SPOF.

On operational maturity: Rate limiters are one of those things that seem simple but cause severe incidents when they misbehave. I've seen rate limiters that failed closed and caused total outages, and ones that failed open and let DDoS traffic through. What's the blast radius if this component fails? That shapes how much redundancy and graceful degradation I build in."

What makes this exceptional
  • Opens with insight about why rate limiting is interesting (shows depth)
  • Explicitly names the core distributed systems tradeoff (accuracy vs. availability)
  • Brings in operational experience (failure modes, blast radius)
  • Offers a framework, then asks for guidance (collaborative, not performative)
  • Distinguishes between L3/L4 vs. L7 rate limiting (technical precision)
Interviewer's notes

"Exceptional problem decomposition. Has clearly built and operated rate limiters. Understands failure modes. Want this person on my team."

The "wow" moments

"Rate limiters serve different masters" "Truly accurate global rate limiting requires coordination, but coordination kills performance" "I've seen rate limiters that failed closed and caused total outages"
What separates L6 from L7

"L6 asks good questions and shows structure. L7 offers insight — they name the core tensions, share relevant experience, and help the interviewer understand that they've internalized the problem space at a deep level. They're teaching, not just asking."

The non-obvious insight

"Most candidates ask 'how many requests per second?' L7 candidates ask 'what's our tolerance for inaccuracy?' because they know that's the question that actually determines the architecture."

Functional Requirements

P0 — Must have for launch:

  • Enforce rate limits by API key — this is the core function
  • Support configurable limits per customer tier (free: 100 req/min, pro: 1000 req/min, enterprise: custom)
  • Return HTTP 429 with Retry-After header when limit exceeded
  • Support at least two time windows: per-second (burst protection) and per-day (quota enforcement)

P1 — Important but can fast-follow:

  • Rate limit response headers (X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset)
  • Configuration API to update limits without deployment
  • Per-endpoint rate limiting

P2 — Nice to have:

  • Real-time analytics dashboard for rate limit events
  • Alerting when customers approach limits
  • Quota pooling across API keys for enterprise (org-level limits)

The standout insight — graceful degradation as a spectrum

Most candidates define accuracy as a fixed number. Strong hires define it as a sliding scale that degrades gracefully:

  • Normal operation: ±5% accuracy
  • One region down: ±10% accuracy (that region's data is stale)
  • Cross-region sync broken: ±20% accuracy (regions enforce independently)
  • Rate limiter down: 0% accuracy, 100% availability (fail open)

This graceful degradation is more valuable than trying to maintain perfect accuracy during failures.

Multi-Region Strategy

Three options considered:

  1. Fully independent — each region tracks its own limits. Customer gets 5x their limit globally. Simple but inaccurate.
  2. Async sync — regions share counts periodically. Accurate over time, but can overshoot during sync gaps.
  3. Global coordination — single source of truth. Accurate but adds cross-region latency.

Recommendation: Async sync with local enforcement. Each region enforces locally. Every 1-2 seconds, regions publish deltas to a global aggregator. If a region gets 30% of traffic, it gets ~30% of the quota budget.

Rate Limit Client (in Gateway)

python
class RateLimitClient:
    def __init__(self, redis_cluster, config_cache):
        self.redis = redis_cluster
        self.config = config_cache  # TTL 30s, background refresh
        
    def check(self, api_key, resource, tokens):
        # Step 1: Get config (local cache, <1ms)
        config = self.config.get(api_key)
        if not config:
            config = self.config.get_default_tier()
            
        # Step 2: Execute rate limit check (Redis, ~2ms)
        try:
            result = self.redis.execute_lua(
                RATE_LIMIT_SCRIPT,
                keys=[f"rl:{api_key}:burst", f"rl:{api_key}:daily"],
                args=[config.burst_limit, config.daily_limit, tokens, now()]
            )
        except RedisError:
            # Fail open — log and allow
            metrics.incr("ratelimit.failopen")
            return RateLimitResult(allowed=True, degraded=True)
            
        # Step 3: Emit async event for aggregation (non-blocking)
        self.emit_usage_event(api_key, tokens, result)
        
        return result

Decision Deep Dive
Rate Limiting Algorithms
Fixed Window Counter

Divide time into fixed windows. Maintain a counter per window.

✓ O(1) memory, extremely simple

✗ Boundary problem: 2x throughput at window edges

Best for: daily quotas, coarse-grained limits

Sliding Window Log

Store timestamp of every request in a sorted set. Count entries in last N seconds.

✓ Perfect accuracy, no boundary problem

✗ O(n) memory — at 500K RPS, 30M entries/key possible

Best for: low-volume, compliance/audit

Sliding Window Counter

Combine fixed window counters with weighted average: prev × overlap% + curr

✓ O(1) memory, ~95-99% accuracy, smooths boundary

✗ Slightly approximate (assumes even distribution)

Best for: most production rate limiters

Token Bucket

Bucket fills at refill_rate tokens/sec. Each request consumes tokens. Reject if empty.

✓ Natural burst handling, intuitive for customers

✗ Two parameters to tune (bucket_size + refill_rate)

Best for: APIs with burst tolerance

Leaky Bucket

Requests queue and are processed at fixed rate. If queue full, reject.

✓ Smoothest output rate, protects fragile services

✗ Introduces latency — we're a gate, not a queue

Best for: traffic shaping, not rate limiting

Mitigation: Local Aggregation with Bounded Sync

python
class HotKeyHandler:
    def __init__(self, sync_interval_ms=50, sync_batch_size=50):
        self.local_counts = {}
        self.sync_interval = sync_interval_ms
        self.sync_batch = sync_batch_size
        
    def check_and_increment(self, api_key, limit) -> bool:
        if api_key not in self.local_counts:
            global_count = self.redis.get(f"rl:{api_key}") or 0
            self.local_counts[api_key] = {
                "global_count": global_count,
                "pending": 0,
                "last_sync": now()
            }
        
        state = self.local_counts[api_key]
        estimated_total = state["global_count"] + state["pending"]
        
        if estimated_total >= limit:
            return False  # Reject
            
        state["pending"] += 1
        
        if (state["pending"] >= self.sync_batch or 
            now() - state["last_sync"] > self.sync_interval):
            self._sync_to_redis(api_key)
            
        return True
        
    def _sync_to_redis(self, api_key):
        state = self.local_counts[api_key]
        pending = state["pending"]
        new_global = self.redis.incrby(f"rl:{api_key}", pending)
        state["global_count"] = new_global
        state["pending"] = 0
        state["last_sync"] = now()

Accuracy Math

20 gateways, sync every 50 requests, limit 10,000/min:

  • Worst case: 20 × 49 = 980 overshoot = ~10% of limit
  • To hit 5% target: reduce sync_batch to 25 → 20 × 24 = 480 = ~5%

Read Path Problem

Each gateway knows its own pending count plus a potentially stale global count. Accept bounded staleness. On sync, we get the true global count. Between syncs, we're optimistic. Staleness is bounded to sync_batch × num_gateways. Our ±5% target allows this.

Edge Cases

  • Gateway restart: Pending count is lost. Worst case: lose sync_batch from count. Acceptable given accuracy tolerance.
  • Hot key stops being hot: LRU cache naturally evicts keys, falls back to direct Redis.
  • Clock skew: Sync interval is relative, not absolute. Clock skew doesn't affect correctness.

Hierarchical Counting with Probabilistic Admission

Three layers: L1 (probabilistic admission filter, microsecond), L2 (local counters with adaptive sync), L3 (Redis ground truth, periodic sync only).

L1: Probabilistic Admission

python
class ProbabilisticAdmission:
    def should_admit(self, api_key, local_state) -> bool:
        headroom_pct = 1 - (local_state.estimated_count / local_state.limit)
        
        if headroom_pct > 0.2:
            return True  # Plenty of headroom — always admit
        elif headroom_pct > 0:
            # Approaching limit — probabilistic admission
            admit_prob = headroom_pct / 0.2  # Linear decay
            return random.random() < admit_prob
        else:
            return False  # At or over limit

Why probabilistic? With 20 gateways trying to use the last 5% of quota, exact counting creates coordination overhead. Probabilistic admission means each gateway naturally claims its fair share without coordination.

class AdaptiveRateLimiter:
    def check(self, api_key, limit) -> Decision:
        if not self._under_limit(api_key, limit):
            return DENY
            
        health = self.backend_health.current_health()
        
        if health < 0.5:  # Backend struggling
            effective_limit = limit * health
            if not self._under_limit(api_key, effective_limit):
                return DENY_WITH_BACKPRESSURE
                
        return ALLOW

Probabilistic Early Expiration ("Probabilistic Immortality")

python
def should_refresh(cache_entry):
    age = now() - cache_entry.created_at
    ttl = cache_entry.ttl
    
    # Probability of refresh increases as we approach expiry
    # At 50% TTL: 0% chance. At 95% TTL: 80% chance.
    if age < ttl * 0.5:
        return False
    
    remaining_pct = 1 - (age / ttl)
    refresh_prob = 1 - (remaining_pct / 0.5) ** 2
    
    return random.random() < refresh_prob
    # Entries almost never actually expire — they get refreshed.
