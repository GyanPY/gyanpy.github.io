Variant 4: Hash-Based (MD5/SHA256 of URL)

Variant 4
Hash-Based Generation
Idempotent · Birthday Paradox Risk
▾
A) What It Is

Hash the original URL and take the first N characters of the hash as the shortcode, with collision detection/resolution.

Core insight: Cryptographic hashes already produce uniformly distributed, deterministic outputs. We're borrowing their entropy for uniqueness — but truncation destroys the collision-resistance properties.

The inventor's problem: "I want idempotency — the same URL should always get the same shortcode. Also, I don't want any coordination or counters."

B) How It Works
Step-by-Step · Hash + Collision Resolution
1. User submits: https://example.com/my-long-path
2. Compute: MD5("https://example.com/my-long-path") = "a1b2c3d4e5f6..."
3. Take first 7 characters: "a1b2c3d"
4. Check DB: "Is a1b2c3d taken?"
   Case A: NO → Insert and return "a1b2c3d"  ✓
   Case B: YES, same URL → Return existing (idempotent!) ✓
   Case C: YES, different URL (COLLISION!):
     Option 1: Slide window: take chars 1-8 → "1b2c3d4"
     Option 2: hash(hash + nonce): hash("a1b2c3d4..." + "1")
     Repeat until unique

Concrete collision scenario:
  URL1 = "https://example.com/page-a"
    MD5 = "a1b2c3d4e5f6..."
    shortcode "a1b2c3d" → NOT IN DB → SUCCESS

  URL2 = "https://example.com/page-b"
    MD5 = "a1b2c3d999..."   ← same first 7 chars!
    shortcode "a1b2c3d" → IN DB, different URL → COLLISION
    Try:  "1b2c3d9" → NOT IN DB → SUCCESS
Birthday Paradox Collision Probability

With 7 hex characters: 16⁷ = ~268 million possibilities. Collision probability after N URLs:

URL CountCollision ProbabilityStatus
10,000~0.02%Safe ✓
100,000~2%Watch ⚠
1,000,000~86%Broken ✗
10,000,000>99.99%Unusable ✗

Formula: P(collision) ≈ 1 − e^(−n² / 2 × 268M)

C) Strengths & Ideal Use Cases
The Sweet Spot
  • Idempotency is a hard requirement
  • Moderate scale (<10M unique URLs)
  • Read-heavy workload (same URL → same code, fast)
  • Natural deduplication of identical URLs
Where It Shines
  • Idempotency: Same URL always gets same shortcode
  • Truly stateless computation: Before the DB check, no coordination needed
  • Content-based addressing: Similar to git (SHA-1 of content)
  • Deduplication: Identical URLs naturally collapse to one code
D) Limitations & Failure Modes
Where It Breaks Down
  • Collision rate scales with N²: Birthday paradox makes this untenable at scale
  • Collision resolution is expensive: Each collision = DB check + retry
  • Not time-sortable: Hash order is meaningless
  • Short codes = more collisions: The length vs. collision trade-off is brutal
Failure Scenario: Collision Storm at Scale
Full Parking Lot Problem
State: 50M URLs in system, using 7-char shortcodes

New URL submitted:
  Attempt 1: Hash prefix = "a1b2c3d" → COLLISION
  Attempt 2: Shifted prefix = "x7y8z9a" → COLLISION
  ...
  Attempt 47: Finally find unique code

When 90% of codes are taken, finding an empty spot
takes 10x longer on average.

System slows dramatically as namespace fills.
The Gotcha Candidates Miss

MD5/SHA256 are cryptographic hashes designed to resist collisions on the full hash. When you truncate to 7 characters, you lose all those security properties. The 7-character prefix collides at birthday paradox rates, not cryptographic rates. Using "SHA256" doesn't make 7 characters any safer than random characters.

E) Operational Considerations
Key Metrics to Monitor
  • Collision rate — % of generations requiring retry (alert at >1%)
  • Average retries per success — should stay <1.1
  • Namespace utilization — % of possible shortcodes used
  • p99 latency — tail latency grows with collision rate

Uneven shortcode distributionPoor hash functionAnalyze character distribution

Shortcode Length vs. Namespace
Each Character Adds 62x Namespace (base62)
  • 6 chars: 56 billion possible

F) The L6/L7 Nuances
L6/L7 Nuances

Question that shows deep understanding: "At what namespace utilization percentage would you switch from hash-based to a different strategy, and how would you calculate that threshold?"

Non-obvious insight: Hash-based generation seems "coordination-free" but actually requires coordination at the collision-check step. The DB check IS coordination — you've just deferred it. As collision rate rises, you do MORE coordination, not less. The "coordination-free" claim is misleading.

Google-scale vs. startup-scale: At Google scale, hash-based is generally avoided for primary key generation — collision rates at billions of URLs are unacceptable. At startup scale, it's fine and has the nice property of idempotency. The key: migrate around 0.1% collision rate, not when things are broken.

Variant 5: Pre-Generated Pool (Key Service)

Variant 5
Pre-Generated Pool
Zero Collision · Consistent Latency
▾
A) What It Is

Generate millions of random shortcodes in advance, store them in a pool, and hand them out on demand.

Core insight: Separate generation from allocation. Generate codes when you have spare capacity; allocate them when users need them. This moves the expensive collision check to the cold path.

Real examples: Gift card numbers (pre-generated, handed out at purchase), lottery tickets, beta invite codes.

B) How It Works
Step-by-Step · Pool Architecture
Pool State at t=0:
  code    | status
  a1B2c3D | available
  x7Y8z9A | available
  m4N5o6P | available
  ... (10 million more)

Request arrives at t=1s:
  BEGIN TRANSACTION
    SELECT code FROM pool
    WHERE status='available'
    LIMIT 1
    FOR UPDATE SKIP LOCKED  ← critical: avoid hot row problem
  → Returns "a1B2c3D"
    UPDATE pool SET status='used' WHERE code='a1B2c3D'
  COMMIT

Pool State at t=1s:
  a1B2c3D | used      ← Now taken
  x7Y8z9A | available
  m4N5o6P | available

Background Generator (runs when pool < 1M available):
  1. Generate 1M random 7-character strings
  2. Check against used codes (one-time check)
  3. INSERT valid codes with status='available'
  4. Continue until pool refilled
Hot Row Problem — Critical Implementation Detail

Naive implementation: SELECT ... WHERE status='available' LIMIT 1 FOR UPDATE creates a hot row problem. Every server wants the same "first" available row → lock contention → latency spikes.

Solution: Use SKIP LOCKED (Postgres) — servers skip rows locked by other transactions and grab the next available one. Or partition the pool across servers.

C) Strengths & Ideal Use Cases
The Sweet Spot
  • Security-sensitive applications (codes can't be enumerated)
  • Moderate scale (10K–1M allocations/day)
  • Have capacity for background processing
  • Need consistent, predictable allocation latency
Where It Shines
  • Unpredictable codes: No sequence, no pattern, can't enumerate
  • Guaranteed uniqueness: Collision check happens at generation (cold path), not allocation (hot path)
  • Consistent latency: Allocation is O(1), no retry loops
  • Graceful degradation: Pool depletes gradually, not suddenly
D) Limitations & Failure Modes
Where It Breaks Down
  • Pool exhaustion: If allocation rate exceeds generation rate, pool empties
  • Storage overhead: Must store millions of pre-generated codes
  • Hot key problem: Pool table becomes contention point without SKIP LOCKED
  • Wastage: Codes generated but never used
Failure Scenario: Traffic Spike Exhausts Pool
Pool Exhaustion Timeline
t=0:     Pool: 10M available codes
         Normal traffic: 100 allocations/sec
         Generation rate: 1,000 codes/sec
         Steady state: pool grows ✓

t=60m:   Viral event! Traffic → 50,000 allocations/sec
         Pool draining at 49,000 codes/sec

t=63m:   Pool EXHAUSTED (0 available)
         New requests fail or block

t=63m+:  Generation still only 1,000/sec
         Recovery time: 10M / 1,000 = ~166 minutes!

Impact: ~3 hours of degraded service after exhaustion

Fix: Pre-warm pool to 100M+ codes, scale generator
     during high traffic, alert at 20% depletion
The Gotcha Candidates Miss

Naive SELECT ... LIMIT 1 FOR UPDATE creates a hot row problem. At 10,000 req/sec, all servers compete for the same "first" row → lock contention → latency spikes. Fix: use SKIP LOCKED in PostgreSQL, which lets servers skip locked rows and grab the next available one atomically.

E) Operational Considerations
Key Metrics to Monitor
  • Pool depth — available codes remaining (alert at 20%)
  • Allocation rate — codes used per second
  • Generation rate — codes added per second
  • Pool depletion ETA — at current rate, when does pool hit zero?
SymptomLikely CauseInvestigation
Pool depleting fastTraffic spike or generation failureCheck both rates
High allocation latencyHot key contentionCheck lock wait times
Duplicate codesConcurrent allocation bugCheck transaction isolation
Generation failingCollisions with existing codesCheck namespace utilization
Tuning Guide
Sizing Rules of Thumb
  • Pool size: 10× expected daily allocation rate
  • Refill threshold: Start generating at 80% depletion
  • Partition count: 1 partition per 1,000 req/sec capacity
  • Code length: Start with 7 chars; increase if utilization >1%
F) The L6/L7 Nuances
L6/L7 Nuances

Question that shows deep understanding: "How would you design the pool allocation to avoid hot spots while maintaining transactional guarantees across multiple datacenter regions?"

Non-obvious insight: Pool approach moves collision resolution from hot path (user request) to cold path (background generation). This is a classic latency optimization pattern: do expensive work when you have time, cache the results, serve them quickly. Most candidates don't name this pattern explicitly — doing so signals L7 thinking.

Google-scale vs. startup-scale: At Google scale, you'd need distributed pools per region/datacenter with careful capacity planning — the pool becomes its own distributed system. At startup scale, a single partitioned database table with SKIP LOCKED works fine.

Variant 6: Counter + Random Suffix

Variant 6
Counter + Random Suffix
Best of Both Worlds
▾
A) What It Is

Combine a sequential counter for uniqueness with random characters for unpredictability.

Core insight: Sequential counters are great for uniqueness; random bits are great for security. Combine both to get both properties without sacrificing either.
B) How It Works
Strategy 1: Concatenate Counter + Random
Strategy: 4-char counter + 3-char random

Request 1:
  Counter: 1 → base62: "0001"
  Random 3 chars: "xYz"
  Shortcode: "0001xYz"

Request 2:
  Counter: 2 → base62: "0002"
  Random 3 chars: "aBc"
  Shortcode: "0002aBc"

Result: Unique (counter) but hard to predict (random)
Attacker must guess "0001???" — 62³ = 238,328 possibilities per code
Strategy 2: XOR Obfuscation
Counter: 12345
Secret key: 98765 (fixed, stored securely)
XOR:    12345 XOR 98765 = 110876
base62(110876) = "Rxq"

Counter: 12346
XOR:    12346 XOR 98765 = 110879
base62(110879) = "Rxt"

Output looks random, but deterministically REVERSIBLE with key.
Counter is recoverable: base62_decode("Rxq") XOR 98765 = 12345
Where It Shines
  • Best of both worlds: Unique (counter) AND unpredictable (random/XOR)
  • Simple to implement: Small tweak to counter approach
  • Reversible (if needed): With XOR, can recover counter from code for debugging
  • Real examples: YouTube video IDs, analytics tracking codes
Where It Breaks Down
  • Longer codes: Need space for both counter and random bits
  • More complex: More logic than pure counter, more failure modes
  • XOR is NOT cryptographic: Attackers can collect codes and potentially recover the key
The Gotcha Candidates Miss

Simple XOR with a fixed key is NOT cryptographically secure. Attackers can collect multiple shortcodes and potentially recover the key through statistical analysis. For real security, use proper Format-Preserving Encryption (FPE) like FF1 or FF3 — this gives you XOR-like simplicity with cryptographic strength.

F) The L6/L7 Nuances
L6/L7 Nuances

Question that shows deep understanding: "How would you design the obfuscation to prevent statistical attacks while maintaining the ability to debug production issues?"

Non-obvious insight: Simple XOR is not cryptographically secure. For real security, use Format-Preserving Encryption (FPE) like FF1/FF3. FPE takes a number, encrypts it, and outputs a number of the same length — so base62(7) in gives base62(7) out, but with true cryptographic guarantees. This is what URL shorteners that claim "unpredictable codes" should be using.

Decision Framework — How to Choose in an Interview

The 5-Question Decision Process

Walk through these questions in order. The first question that has a definitive answer determines your strategy. Don't skip ahead.

1
Do you need idempotency? (Same URL = same code?)

If YES → Hash-Based (V4). Accept the collision trade-off; migrate away when utilization hits 0.1%.

If NO → continue to step 2.

2
What's your write throughput requirement?

If <10K writes/sec → Auto-Increment (V1). Simplest, most debuggable, proven.

If 10K–100K writes/sec → Pre-Allocated Ranges (V2). Amortize coordination cost.

If >100K writes/sec → continue to step 3.

3
Can you tolerate clock synchronization requirements?

If YES (have NTP/GPS infrastructure) → Snowflake (V3). Zero coordination, time-sortable, scales to millions/sec.

If NO (clock reliability uncertain) → Pre-Generated Pool (V5). Consistent latency, no clock dependency.

4
Do you need codes to be unpredictable but sequential-ish?

If YES → Counter + Random (V6) or Counter + FPE (V3 variant). Best of both worlds.

5
Is this for a URL shortener specifically at L6/L7 interview scale?

At 500M URLs/month: Pre-Allocated Ranges (V2) + FPE obfuscation.

This gives zero collisions (counter-based), minimal coordination (range allocation), and unpredictable codes (FPE encryption).

The Complete Interview Script — URL Shortener at 500M URLs/Month

Use This Verbatim (Adapted for Your System)

This is the complete L7-signal script for choosing a shortcode generation strategy in a URL shortener interview at the given scale.

Complete Interview Script
"For shortcode generation at 500M URLs/month, let me walk through my requirements
and eliminate options:

I need: zero collisions (non-negotiable), <10ms claim latency on the hot path,
unpredictable codes (security — can't enumerate URLs), and multi-region support.

I can rule out Auto-Increment (V1): predictable codes are a security risk for a
public service — competitors can enumerate all URLs.

I can rule out Hash-Based (V4): at 500M URLs, collision probability approaches
~1%. The birthday paradox makes this untenable at our scale. Also, I don't
actually need idempotency — I'm fine with the same URL getting different codes.

I can rule out Snowflake (V3) as my primary shortcode mechanism: while it's great
for internal IDs, the full 64-bit ID encodes creation time (privacy leak) and
converts to 11+ base62 characters (too long for a URL shortener).

Between Pre-Allocated Ranges (V2), Pre-Generated Pool (V5), and Counter+FPE:

Pre-Generated Pool gives me unpredictability built-in, but pool exhaustion
during traffic spikes is a reliability risk I'd rather avoid.

My choice: Pre-Allocated Ranges + Format-Preserving Encryption (FPE)

Architecture:
  - Range Allocator: Backed by a Postgres sequence or etcd
  - API servers request ranges of 10,000 IDs at startup
  - Pre-fetch next range when 80% exhausted (zero blocking)
  - Apply FPE (FF1 or FF3) to convert sequential counter to a
    7-character base62 code that looks random but is reversible

Why FPE over XOR? FPE is cryptographically secure — you can't recover the
counter without the secret key, even with millions of samples.

Operational properties:
  - Zero collisions: mathematically guaranteed (counter-based)
  - Minimal coordination: range fetch every 10K URLs, ~0.02% of requests
  - Unpredictable: FPE output is indistinguishable from random
  - Multi-region: each region gets non-overlapping counter ranges
  - Fallback: if range allocator unavailable, serve from pre-fetched range

Monitoring:
  - Alert when range pool <20% capacity
  - Track range fetch latency (should be <50ms, happens rarely)
  - No collision monitoring needed (counter = impossible to collide)

The tradeoff I'm accepting: more operational complexity than auto-increment,
but the properties are exactly right for a public URL shortener at this scale."
🔑 The Complete Final Soundbite

"Every shortcode strategy is making one bet: either I trust coordination to guarantee uniqueness, or I trust math. At URL shortener scale, I trust math — counter-based generation with FPE obfuscation gives me mathematical uniqueness guarantees without the coordination cost. The key insight is that 'coordination-free' doesn't mean 'magic' — you're trading synchronization overhead for the careful design of a system where uniqueness is baked into the structure itself."

🎯 Complete Deep Dive Summary

The 6 variants in order of sophistication: Auto-Increment (DB as coordinator) → Pre-Allocated Ranges (amortize coordination) → Snowflake (eliminate coordination with physics) → Hash-Based (trade coordination for idempotency) → Pre-Generated Pool (defer coordination to cold path) → Counter + FPE (uniqueness + security without trade-off). For a public URL shortener at L6/L7 interview scale: Pre-Allocated Ranges + FPE is the answer, and you should be able to explain why every other option was eliminated.
