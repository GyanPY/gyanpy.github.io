3. Incident Response: The 3 AM Playbook

Scenario: P99 Latency Spiked from 50ms to 2 Seconds

Walk through minute by minute. The goal is mitigation first, root cause second.

Minute 0-2Assess Scope (Immediate Parallel Checks)
▾
  • Is it us or external? Check CDN status page + AWS status page. If external: update status page, wait.
  • Which endpoint? redirect_p99 vs create_p99. Narrows to read or write path.
  • Which region? Single region = regional issue. All regions = shared dependency.
  • When did it start? Correlate with deployments, traffic spikes, cron jobs.
Minute 2-5Identify Bottleneck via Traces
▾

Query slow traces: trace.duration > 2000ms AND service = "redirect-service"

  • If Redis span slow (>500ms instead of <1ms): Check redis_connections, redis_memory, network latency to Redis
  • If Postgres span slow (>1500ms instead of <10ms): Check active_connections, slow_query_log, lock_waits, replication lag
  • If no single span slow: Check GC pauses, CPU throttling, NIC saturation
Minute 5-10Mitigate (Restore Service First)
▾
  • If recent deployment suspect: kubectl rollout undo deployment/redirect-service
  • If database slow: Kill long-running queries with pg_terminate_backend, failover to replica, enable circuit breaker
  • If cache unavailable: Restart Redis nodes (rolling), enable fallback to DB directly
  • If DDoS/traffic spike: Enable CDN rate limiting, block IPs at edge, scale API servers
SQL · Kill Long Queries
SELECT pg_terminate_backend(pid) FROM pg_stat_activity
WHERE state = 'active'
AND query_start < now() - interval '30 seconds';
Minute 10-15Verify Resolution
▾
  • P99 returning to normal? (<50ms)
  • Error rate declining?
  • Synthetic monitors succeeding?
  • Social media / support complaints stopping?
  • If NOT resolved: escalate, page additional engineers, update status page

4. Deployment Strategy: Canary

Diagram · Canary Deployment Pipeline
Stage 1: Integration Tests (CI)
  → Unit tests, integration tests, contract tests (API schema)

Stage 2: Staging Deployment
  → Full E2E test suite, load test, security scan (OWASP)

Stage 3: Canary (1% traffic, 10 minutes)
  → Auto-rollback triggers:
    - Error rate > 2x baseline
    - P99 latency > 1.5x baseline
    - Memory growth > 20%

Stage 4: Progressive Rollout
  → 1% → 10% (10 min) → 50% (10 min) → 100%
  → Same auto-rollback triggers at each stage
  → Manual approval gate before 50%

Stage 5: Bake (24 hours)
  → Full production traffic
  → Keep old version ready for quick rollback

Database Migrations (Zero Downtime)

Rule: Never Make Breaking Schema Changes in One Deployment

Example: Renaming column "url" to "long_url"

SQL · Zero-Downtime Migration — 5 Steps
-- Step 1: Add new column (safe, old code still works)
ALTER TABLE urls ADD COLUMN long_url VARCHAR(2048);

-- Step 2: Backfill + dual-write
UPDATE urls SET long_url = url WHERE long_url IS NULL;
-- Deploy code that WRITES to both columns, READS from "url"

-- Step 3: Switch reads
-- Deploy code that READS from "long_url", still writes to both

-- Step 4: Stop writing to old column
-- Deploy code that only uses "long_url"

-- Step 5: Drop old column (weeks later, when safe)
ALTER TABLE urls DROP COLUMN url;

1. SLO Framework with Burn Rate Alerting

SLOTargetWindowError Budget
Redirect Success Rate99.99%30 days4.32 min outage OR 0.01% errors
Redirect Latency p99<50ms30 days0.01% requests can exceed
Create Success Rate99.9%30 days43.2 min outage
Data Durability99.9999%InfiniteZero tolerance
Burn Rate Alerting — Better Than Threshold Alerts

Traditional alert: "P99 > 50ms" → fires on any spike, too noisy

Burn rate alert: "At the current error rate, we'll exhaust our monthly budget in X hours"

Burn Rate = (current error rate) / (allowed error rate)

If SLO = 99.99%, allowed error rate = 0.01%. If current error rate = 0.05%, burn rate = 5x. At 5x, we exhaust monthly budget in 6 days instead of 30.

Burn RateTime to Budget ExhaustionAlert LevelAction
14.4x2 hoursP1Page immediately
6x5 hoursP2Page during work hours
3x10 hoursP3Slack notification
1x30 daysNormalNo alert
Why Burn Rate Is Better
  • A 1-minute spike at 10x error rate is ignorable (burns 10 min of budget)
  • A sustained 2x error rate for 2 hours is serious (burns 4 hours of budget)
  • Traditional alerting can't distinguish these — burn rate can

Continuous Profiling — The Secret Weapon

Most candidates don't mention this. Profiling captures CPU flame graphs, memory allocation, goroutine counts, and lock contention in production continuously.

Use case for latency spike: Traditional debugging → look at traces, see everything is slow. With profiling → see that 80% of CPU is in JSON marshaling → found the bug: accidentally serializing a 10MB response object.

3. Incident Response — Minute-by-Minute (L7 Detail)

SeverityDefinitionExampleResponse Time
SEV1Complete outageAll redirects failingImmediate
SEV2Partial / degradationP99 >500ms, single region down15 min
SEV3Elevated errorsError budget burning 3x1 hour
SEV4AnomalyCache hit rate dropped 5%Next biz day
Minute 0Acknowledge & Assess (60 seconds)
▾
  • Acknowledge page (stops escalation timer)
  • Check status pages: AWS, Cloudflare, PagerDuty (rule out providers)
  • Open primary dashboard
  • Answer: Is it affecting users? Is it still ongoing? Is it getting worse?
Minute 1-3Initial Triage — Open 4 Browser Tabs Simultaneously
▾
  • Tab 1 — Metrics Dashboard: Which endpoint? Which region? Started when?
  • Tab 2 — Traces (filtered: latency >2s, last 5 min): Sample 5 slow traces. Which span is slow? Is there a pattern (same shard? same pod)?
  • Tab 3 — Recent Changes: Any deploys in last 2 hours? Config changes? Infrastructure changes (cert rotations, scaling events)?
  • Tab 4 — External Factors: Traffic spike (DDoS or viral)? Time-based pattern (cron job)? Upstream dependency?
Minute 3-5Root Cause Hypothesis (Decision Tree)
▾
  • IF Redis span slow (>100ms): Check redis_cpu, redis_memory, connections, hot key detection
  • IF Postgres span slow (>500ms): Check active connections, slow query log, lock waits, replication lag, disk I/O throttling
  • IF App slow (no external span slow): Check CPU (GC?), memory (swapping?), continuous profiler flame graph, goroutine count (deadlock? leak?)
  • IF CDN slow: Check CDN status page, origin health from CDN perspective, certificate issues
Minute 5-10Mitigation — Restore Service First, Investigate Later
▾
bash · Common Mitigation Commands
# Rollback deployment (if suspect)
kubectl rollout undo deployment/redirect-service

# Kill long-running DB queries
SELECT pg_terminate_backend(pid) FROM pg_stat_activity
WHERE state = 'active' AND query_start < now() - interval '30 sec';

# Enable CDN rate limiting for DDoS
cf-cli rate-limit --zone=short.url --rate=1000/min --action=block

If nothing works and impact is severe: escalate (page additional engineers), update status page, consider "degraded mode" (disable non-critical features like detailed analytics).

**Severity:** SEV2     |  **Error budget consumed:** 12% of monthly budget

## Summary
Redirect p99 latency increased from 50ms to 2.1 seconds,

4. Game Days — Proactive Failure Testing

Philosophy

"Don't wait for 3 AM incidents. Practice them in controlled conditions during business hours when you have a full team."

Scenario 1: Database Primary Failure
  • Terminate primary database instance
  • Verify automatic failover to replica
  • Measure: time to detect, time to recover, requests affected
  • Target: <30 seconds failover
Scenario 2: Redis Cluster Failure
  • Kill 2 of 3 Redis primary nodes
  • Verify system degrades gracefully (DB fallback)
  • Verify no data loss, no user-facing errors
Scenario 3: Runaway Traffic (Hot URL)
  • Simulate 10x traffic spike to a single URL
  • Verify rate limiting engages for other users
  • Verify other URLs unaffected
Scenario 4: Bad Deploy
  • Deploy code with intentional memory leak
  • Verify canary catches it before full rollout
  • Verify auto-rollback triggers correctly
Scenario 5: Dependency Failure
  • Block network to external safe browsing API
  • Verify URL creation continues (timeout, fallback)
  • Verify no cascading failures downstream

Expand-Contract Migration Pattern

Diagram · Expand-Contract: Zero-Downtime Column Rename
Goal: Rename column "url" to "long_url" with zero downtime

PHASE 1: EXPAND
─────────────────
• ALTER TABLE urls ADD COLUMN long_url VARCHAR(2048)
• Deploy: writes to BOTH columns
• Backfill in batches of 10K (avoid locking):
  UPDATE urls SET long_url = url WHERE long_url IS NULL LIMIT 10000;

PHASE 2: MIGRATE
─────────────────
• Deploy: READS from new column, writes to both
• Verify all reads successful

PHASE 3: CONTRACT
─────────────────
• Deploy: only uses new column
• ALTER TABLE urls DROP COLUMN url
  (wait 1-2 weeks to ensure no rollback needed)

Total time: 2-4 weeks  |  Downtime: Zero  |  Risk: Low (reversible)

On-Call Best Practices & Alert Hygiene

On-Call Load Targets
  • <2 pages per week per person
  • <1 page requiring action per week
  • If exceeded → reliability sprint
  • Monthly alert review: count pages per type, check if actionable, automate where possible
Anti-Patterns to Eliminate
  • "This always fires on Tuesday at 3 AM, we ignore it" — tune or delete it
  • "Page for X, but we never do anything about it" — add runbook or delete
  • "50 alerts fired but they were all the same issue" — aggregate alerts
