Short Code Generation
Algorithms

An exhaustive analysis of every approach to generating unique, compact, unpredictable identifiers across distributed servers — with full math, trade-offs, and interview signals.

🔬 Deep Dive 6 Approaches Production Scale
Overview · 01

The Core Problem Statement

This is one of the most critical design decisions in a URL shortener. Generate billions of unique, compact identifiers across distributed servers with:

  • Zero collisions — a collision means two URLs map to the same code → data loss
  • Low latency — <10ms on the hot path
  • No single point of failure
  • Unpredictable codes — security: can't enumerate URLs
  • Compact representation — 7 characters ideal
Overview · 02

The Math That Constrains Us

Base62 alphabet: a-z, A-Z, 0-9 = 62 characters Keyspace by length: 6 chars: 62^6 = 56.8 billion 7 chars: 62^7 = 3.52 trillion 8 chars: 62^8 = 218 trillion Our scale: 500M URLs/year × 10 years = 5 billion URLs With 7 chars (3.52T keyspace): 0.14% utilization With 6 chars (56.8B keyspace): 8.8% utilization → collision risk grows Decision: 7 characters gives us ~700x headroom. Safe choice.
Approach · 01

Pure Random Generation

How It Works
Python 
def generate_short_code():
    alphabet = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789"
    return ''.join(random.choice(alphabet) for _ in range(7))
The Collision Mathematics (Birthday Problem)
Probability of at least one collision after n codes: P(collision) ≈ 1 - e^(-n² / 2k) Where: n = number of codes generated k = keyspace size (62^7 = 3.52 trillion) At various scales: n = 1 million: P ≈ 0.00000014 (negligible) n = 100 million: P ≈ 0.0014 (1 in 714) n = 1 billion: P ≈ 0.13 (13% - concerning) n = 10 billion: P ≈ 1.0 (virtually certain collision)
Random + Collision Check
Python 
def generate_with_retry(db, max_retries=5):
    for attempt in range(max_retries):
        code = generate_random_code()
        if not db.exists(code):
            if db.insert_if_not_exists(code, url):  # Atomic!
                return code
    raise ExhaustedRetriesError()
Expected retries at various scales: With n existing codes in keyspace k: Collision probability per attempt = n / k Expected attempts = 1 / (1 - n/k) At 500M URLs: 500M / 3.52T = 0.014% collision rate Expected attempts = 1.00014 (essentially 1) At 10B URLs: 10B / 3.52T = 0.28% collision rate Expected attempts = 1.0028 (still fine) At 100B URLs: 100B / 3.52T = 2.8% collision rate Expected attempts = 1.029 (3% overhead)

The REAL Problems with Random + Check

Problem 1: Race Condition

Race Condition Timeline 
Time    Server A                    Server B
────────────────────────────────────────────────
T1      Generate code "Xk9Lm2n"     Generate code "Xk9Lm2n" (same!)
T2      Check exists? → NO          Check exists? → NO
T3      Insert → SUCCESS            Insert → FAILS (duplicate key)
T4      Return to user              Retry...

Solution: Use atomic INSERT ... ON CONFLICT or distributed lock. But this adds latency.

Problem 2: Database Round-Trip on Every Write

Even with 0.014% collision rate, you always need to check. That's a DB query on the hot path.

Latency breakdown: Generate random: 0.01ms DB existence check: 5-20ms (network + query) DB insert: 5-20ms Total: 10-40ms just for collision handling

Problem 3: Thundering Herd on Popular Codes

If multiple servers generate the same code simultaneously (rare but possible), you get multiple retries, increased DB load, and latency spikes.

Problem 4: Cryptographic Randomness Cost

Python 
# WRONG - predictable if seed is known
import random
random.choice(alphabet)

# RIGHT - cryptographically secure
import secrets
secrets.choice(alphabet)

Crypto-random is ~10x slower than pseudo-random. At 2K QPS, this matters.

Verdict on Pure Random

AspectRatingNotes
Simplicity⭐⭐⭐⭐⭐Dead simple
Collision Safety⭐⭐⭐Requires retry logic
Latency⭐⭐DB check on every write
Scalability⭐⭐⭐Fine to 10B, degrades after
Unpredictability⭐⭐⭐⭐⭐Cryptographically random
When to Use
Prototypes, low-scale systems (<100M URLs), when simplicity > performance.
❌Weak: "I'll generate random codes"

✅Strong: "Random works at our scale but requires DB round-trip on every write. I'd use it for a prototype, then migrate to pre-generated pool for production."
Approach · 02

Sequential Counter (Auto-Increment)

How It Works
Python 
def generate_short_code(db):
    next_id = db.get_next_sequence_value()  # Atomic increment
    return base62_encode(next_id)

def base62_encode(num):
    alphabet = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"
    if num == 0:
        return alphabet[0]
    result = []
    while num:
        result.append(alphabet[num % 62])
        num //= 62
    return ''.join(reversed(result)).zfill(7)
ID 1 → "0000001" ID 62 → "0000010" ID 1000000 → "00004c92" ID 3521614606208 → "ZZZZZZZ" (max 7-char base62)

The Problems

Problem 1: Predictability (CRITICAL SECURITY ISSUE)

Attack Scenario 
Attacker observes:
  Created URL: short.url/0004c92
  That's ID 1,000,000

Attacker can now:
  - Enumerate ALL existing URLs: /0000001, /0000002, ...
  - Estimate your total URL count (business intelligence leak)
  - Predict future URLs before they're created
  - Scrape your entire database systematically
Real-world impact: In 2018, a URL shortener was scraped completely because of sequential IDs. Millions of private documents exposed.

Problem 2: Single Point of Coordination

DIAGRAM
Server A ──┐
Server B ──┼──→ [Postgres Sequence] ──→ Next ID
Server C ──┘

At 2000 QPS writes:
  - Single sequence becomes bottleneck
  - Cross-region writes need round-trip to sequence owner
  - Sequence owner failure = write outage

Problem 3: ID Leakage Reveals Scale

Business Intelligence Leak 
Competitor creates URL on Monday: short.url/0005000
Competitor creates URL on Friday: short.url/0008500

"They created 3,500 URLs this week"
"At that rate, ~700/day, ~250K/year"
"Small player, not a threat"

Mitigations (That Don't Fully Work)

Mitigation 1: Random Offset

Python
def generate():
    id = get_next_sequence() + RANDOM_OFFSET
    return base62_encode(id)

Problem: Still sequential, just shifted. Enumeration still works.

Mitigation 2: Multiply by Prime

Python
def generate():
    id = get_next_sequence()
    scrambled = (id * LARGE_PRIME) % MAX_ID
    return base62_encode(scrambled)

Better, but reversible. Attacker can figure out the prime.

Mitigation 3: Format-Preserving Encryption (FPE)

Python
from ff3 import FF3Cipher

cipher = FF3Cipher(secret_key, tweak, radix=62)

def generate():
    id = get_next_sequence()
    plaintext = base62_encode(id).zfill(7)
    encrypted = cipher.encrypt(plaintext)  # Still 7 chars, looks random
    return encrypted
This WORKS! But now we've added complexity. At that point, why not use a better approach?

Verdict on Sequential Counter

AspectRatingNotes
Simplicity⭐⭐⭐⭐⭐Trivial to implement
Collision Safety⭐⭐⭐⭐⭐Mathematically impossible
Latency⭐⭐⭐Sequence fetch is fast
Scalability⭐⭐Single coordination point
Unpredictability⭐Completely predictable
When to Use
Internal tools, admin systems, anywhere enumeration is acceptable.
NEVER Use For
Public-facing URL shorteners, anything with private/sensitive URLs.
❌Weak: "Auto-increment is simple and guaranteed unique"
✅Strong: "Auto-increment has zero collisions but exposes us to enumeration attacks and leaks business metrics. I'd only use it for internal tools. For public-facing, I need unpredictability."
Approach · 03

Hash-Based (MD5/SHA256 + Truncate)

How It Works
Python
import hashlib

def generate_short_code(long_url):
    hash_bytes = hashlib.md5(long_url.encode()).digest()
    # Take first 42 bits (enough for 7 base62 chars)
    num = int.from_bytes(hash_bytes[:6], 'big') % (62 ** 7)
    return base62_encode(num)

The Appeal

  • Deterministic: Same URL → same short code (deduplication!)
  • No coordination: Any server can compute independently
  • No storage for generation: Hash is computed, not looked up

The Problems

Problem 1: Collisions Are Inevitable (Pigeonhole Principle)

MD5 output: 128 bits = 3.4 × 10^38 possibilities We truncate to: 42 bits = 4.4 × 10^12 possibilities (62^7) We're mapping 3.4 × 10^38 → 4.4 × 10^12 Every short code maps to ~10^26 possible long URLs Birthday problem strikes again: At 1 billion URLs: ~11% chance of collision At 2 billion URLs: ~37% chance of collision This is WORSE than random! Because you can't retry — the hash is deterministic.

Problem 2: No Collision Resolution

Collision Example
url1 = "https://example.com/page-that-hashes-to-abc1234"
url2 = "https://evil.com/carefully-crafted-to-collide"

hash(url1)[:7] == hash(url2)[:7]  # Both produce "abc1234"

# Now what? You CANNOT store both with the same short code.

Options: reject the second URL (bad UX), append a disambiguator (codes aren't just hashes), use longer codes (defeats the purpose).

Problem 3: MD5 is Cryptographically Broken

Attack Scenario
Attacker creates legitimate URL first
Later, replaces with phishing URL that has same hash
Or finds existing legitimate short URL and creates colliding phishing URL
Note: Even SHA256 truncated has this problem — truncation reintroduces collision risk.

Problem 4: Same URL = Same Code (Sometimes Unwanted)

Privacy Issue
User A shortens: example.com/secret-doc
  Gets: short.url/Xk9Lm2n

User B shortens: example.com/secret-doc  
  Gets: short.url/Xk9Lm2n (SAME!)

Now User B knows User A's short code.
Click analytics are merged.
No user isolation.

Verdict on Hash-Based

AspectRatingNotes
Simplicity⭐⭐⭐⭐Easy to implement
Collision Safety⭐Collisions inevitable
Latency⭐⭐⭐⭐⭐No external calls
Scalability⭐⭐⭐⭐⭐No coordination
Unpredictability⭐⭐⭐Looks random, but deterministic
When to Use
Content-addressable storage (like Git), deduplication systems where collision = same content anyway.
NEVER Use For
URL shorteners where different users can submit same URL, or where collisions cause data loss.
❌Weak: "I'll hash the URL and take the first 7 characters"
✅Strong: "Hashing seems appealing for deduplication, but truncating to 7 chars reintroduces collision risk — worse than random because we can't retry. Also, MD5 is cryptographically broken. I'd use a different approach for the short code and hash only for dedup checks."
Approach · 04

Snowflake-Style Distributed IDs

How It Works (Twitter's Snowflake)
DIAGRAM
64-bit ID structure:
┌─────────────────┬───────────┬──────────────┐
│ Timestamp (41b) │ Machine(10b)│ Sequence(12b)│
└─────────────────┴───────────┴──────────────┘

- Timestamp: Milliseconds since custom epoch (41 bits = 69 years)
- Machine ID: Unique per server (10 bits = 1024 machines)
- Sequence: Counter within millisecond (12 bits = 4096/ms)
Python
class SnowflakeGenerator:
    def __init__(self, machine_id):
        self.machine_id = machine_id
        self.sequence = 0
        self.last_timestamp = 0
        self.epoch = 1609459200000  # 2021-01-01
    
    def generate(self):
        timestamp = current_time_ms() - self.epoch
        
        if timestamp == self.last_timestamp:
            self.sequence = (self.sequence + 1) & 0xFFF  # 12 bits
            if self.sequence == 0:
                # Wait for next millisecond
                timestamp = wait_next_millis(self.last_timestamp)
        else:
            self.sequence = 0
        
        self.last_timestamp = timestamp
        
        id = (timestamp << 22) | (self.machine_id << 12) | self.sequence
        return id

Strengths

  • Globally Unique Without Coordination — each machine generates IDs independently
  • Time-Sortable — great for DB indexing, debugging, range queries
  • Extremely High Throughput — 4096 IDs/ms per machine = 4M IDs/second; 1024 machines = 4 billion/second

The Problems for URL Shortening

Problem 1: 64 Bits → 7 Characters

Snowflake ID: 64 bits 7 char base62: log2(62^7) = 41.7 bits We need to truncate or transform 64 bits → 42 bits This reintroduces collision risk! Option A: Take lower 42 bits short_code = base62_encode(snowflake_id & ((1 << 42) - 1)) Problem: Different snowflake IDs can have same lower 42 bits. Option B: Hash the snowflake ID short_code = base62_encode(hash(snowflake_id) % (62**7)) Problem: We're back to hash truncation collisions. Option C: Use full ID, accept longer codes short_code = base62_encode(snowflake_id) # 11 characters 11 characters is too long for a "short" URL.

Problem 2: Machine ID Coordination

In Kubernetes with auto-scaling pods, assigning stable machine IDs is genuinely hard. Options include manual configuration (error-prone), ZooKeeper/etcd (new dependency), DB assignment (overhead), or MAC address hash (collision risk in cloud).

Problem 3: Clock Skew

Clock Skew Issue
Server A's clock: 10:00:00.000
Server B's clock: 10:00:00.050 (50ms ahead)

Server B generates IDs that look "from the future"
If clocks adjust backward, IDs can collide

Snowflake handles this by refusing to generate IDs if clock moves backward — but that means outage during NTP corrections.

Verdict on Snowflake

AspectRatingNotes
Simplicity⭐⭐⭐Needs machine ID coordination
Collision Safety⭐⭐⭐⭐⭐Perfect for 64-bit IDs
Latency⭐⭐⭐⭐⭐Pure computation
Scalability⭐⭐⭐⭐⭐Billions per second
Unpredictability⭐⭐Time component is predictable
When to Use
Internal primary keys, distributed databases, anywhere you need 64-bit unique IDs.
Not Ideal For
7-character short codes (truncation reintroduces collisions).
❌Weak: "Use Snowflake IDs and convert to base62"
✅Strong: "Snowflake is perfect for our internal id column — 64-bit, time-sortable, globally unique. But for the 7-character short_code, truncating 64 bits to 42 bits reintroduces collisions. I'd use Snowflake for id and a separate mechanism (pre-generated pool) for short_code."

Detailed Architecture

Component 1: Generator Service

Python
class CodeGenerator:
    def __init__(self, redis, counter_db):
        self.redis = redis
        self.counter = counter_db  # Postgres sequence
        self.cipher = FF3Cipher(SECRET_KEY, TWEAK, radix=62)
    
    def generate_batch(self, batch_size=10000):
        codes = []
        for _ in range(batch_size):
            counter_value = self.counter.next()
            plain_code = base62_encode(counter_value).zfill(7)
            encrypted_code = self.cipher.encrypt(plain_code)
            codes.append(encrypted_code)
        
        self.redis.sadd("code_pool:available", *codes)
        return len(codes)
    
    def run_forever(self):
        while True:
            pool_size = self.redis.scard("code_pool:available")
            if pool_size < MIN_POOL_SIZE:  # e.g., 5 million
                self.generate_batch(BATCH_SIZE)  # e.g., 100,000
            sleep(POLL_INTERVAL)  # e.g., 1 second

Component 2: Claimer (API Hot Path)

Python
def create_short_url(long_url):
    # Claim a code - O(1) operation
    code = redis.spop("code_pool:available")
    
    if code is None:
        # Pool exhausted - fallback to synchronous generation
        code = generate_code_synchronously()
        metrics.increment("code_pool.fallback")
    
    # Mark as claimed (for reclaim if we crash before DB write)
    redis.setex(f"code_pool:claimed:{code}", 300, "pending")  # 5 min TTL
    
    try:
        db.insert_url(code, long_url)
        redis.delete(f"code_pool:claimed:{code}")
        return code
    except Exception:
        # Failed to persist - code will be reclaimed after TTL
        raise

Component 3: Reclaimer (Background)

Python
def reclaim_abandoned_codes():
    """
    Find codes that were claimed but never persisted.
    This handles worker crashes between claim and DB write.
    """
    for key in redis.scan_iter("code_pool:claimed:*"):
        code = key.split(":")[-1]
        
        if not db.code_exists(code):
            # Code was claimed but never persisted
            redis.sadd("code_pool:available", code)
            redis.delete(key)
            metrics.increment("code_pool.reclaimed")

Why Format-Preserving Encryption (FPE)?

The counter gives us uniqueness. FPE gives us unpredictability.

FPE Demo
# Without FPE:
counter 1       → "0000001"
counter 2       → "0000002"
counter 1000000 → "004c92"  # Sequential, predictable

# With FPE:
counter 1       → "Kx9mL2p"
counter 2       → "Qw3nR7s"  # Looks completely random
counter 1000000 → "Ht8vB4n"

# But it's reversible with the key!
decrypt("Kx9mL2p") → "0000001" → counter 1
FF3-1 is the NIST-approved FPE algorithm. It preserves format (7-char base62 in → 7-char base62 out), is cryptographically secure, and is reversible (useful for debugging).

Implementation

Central Allocator

Python
class RangeAllocator:
    def __init__(self, db, range_size=10000):
        self.db = db
        self.range_size = range_size
    
    def allocate_range(self, worker_id):
        # Atomic increment and return
        # Uses Postgres: UPDATE counters SET value = value + 10000 RETURNING value
        new_end = self.db.atomic_increment("code_counter", self.range_size)
        new_start = new_end - self.range_size
        
        self.db.log_allocation(worker_id, new_start, new_end)
        
        return (new_start, new_end)

Worker (API Server)

Python
class WorkerCodeGenerator:
    def __init__(self, allocator, worker_id):
        self.allocator = allocator
        self.worker_id = worker_id
        self.current_range = None
        self.current_position = None
        self.cipher = FF3Cipher(SECRET_KEY, TWEAK, radix=62)
    
    def get_next_code(self):
        if self.current_range is None or self.current_position >= self.current_range[1]:
            self._allocate_new_range()
        
        counter = self.current_position
        self.current_position += 1
        
        plain_code = base62_encode(counter).zfill(7)
        return self.cipher.encrypt(plain_code)
    
    def _allocate_new_range(self):
        self.current_range = self.allocator.allocate_range(self.worker_id)
        self.current_position = self.current_range[0]
        # Proactive: request next range when 80% through current

Complete Strong Hire Answer

🏆 Strong Hire Response

"For short code generation, I'll use range allocation with format-preserving encryption."

Why not simpler approaches?

  • Random + collision check: Requires DB round-trip on every write
  • Auto-increment: Predictable codes enable enumeration attacks
  • Hash truncation: Collisions are mathematically inevitable

The architecture:

A central allocator service (backed by Postgres sequence) assigns counter ranges to workers. Each API server gets a range like 0–10,000, uses it locally (no coordination), then requests a new range when 80% exhausted.

For unpredictability, I apply FF3-1 format-preserving encryption before returning the code. Counter 1 becomes "Kx9mL2p", counter 2 becomes "Qw3nR7s" — looks random, but is deterministically derived from the counter with our secret key.

Failure modes:

  • Allocator down: Workers continue with current range, alert if <20% remaining
  • Worker crash: Lose unused portion of range (<10K codes, negligible)
  • Clock issues: None — we don't depend on time
  • Multi-region: Each region's allocator manages a separate counter range (US: 0–1B, EU: 1B–2B, APAC: 2B–3B). No cross-region coordination.

Monitoring:

  • Range allocation latency (should be <50ms)
  • Range exhaustion events (should be smooth, not bursty)
  • Total codes allocated per region (capacity planning)

Trade-off acknowledged: Slightly more complex than random-with-check. But at our scale, the latency consistency (no collision retries) and operational simplicity (no pool to monitor) are worth it.

Alternative if we valued simplicity over performance: Start with random + collision check + bloom filter for fast negative lookups. Migrate to range allocation when we hit latency issues.
