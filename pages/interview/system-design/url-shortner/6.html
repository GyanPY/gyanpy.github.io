🔴
First to Break: PostgreSQL Writes (breaks around 10-20x)
Single Postgres instance maxes out at 30-50K write QPS — we hit the wall at 10x

Every URL creation MUST write to Postgres synchronously (RPO=0 requirement). No caching helps the write path. At 20K write QPS (10x), we're already stressed. At 200K (100x), it's impossible.

Solution: Horizontal sharding with short_code as shard key.

Why short_code as the Shard Key?
  • Every redirect query has short_code — no scatter-gather needed
  • Uniformly distributed (thanks to our FPE encoding)
  • No hot partitions — unlike user_id which would hot-partition on viral users

Start with 16 shards: 200K QPS ÷ 16 = 12.5K QPS per shard — well within Postgres limits, room to grow to 64 without re-architecture.

Custom alias challenge: Aliases need global uniqueness check. Solutions: (1) Route to dedicated "alias coordinator" shard, (2) Reserve a prefix range for aliases, (3) Use a distributed lock in Redis for alias claims.

🟡
Second to Break: Cache Size (breaks around 20-50x)
15GB working set → 1.5TB at 100x. Cache hit rate drop cascades to DB.

Solution: Redis Cluster + Tiered caching

Tiered Cache Architecture
Tier 1: Local in-process cache (LRU, 1GB per server)
  - Catches extreme hot URLs
  - < 0.1ms latency

Tier 2: Redis Cluster (distributed)
  - 32 nodes × 64GB = 2TB capacity
  - Handles 90% of requests
  - < 1ms latency

Tier 3: Database (source of truth)
  - Only for cache misses
  - < 10ms latency
🟢
Third to Break: Origin Traffic (breaks around 30-50x)
1M QPS to origin at 100x — this is the biggest lever to optimize

Solution: Drive CDN hit rate from 90% → 99%

The Key Math

Current: 10M QPS × 10% miss = 1M QPS to origin (too high)

Target: 10M QPS × 1% miss = 100K QPS to origin (same as current total!)

Going from 90% → 99% CDN hit rate brings origin traffic back to today's level. This is the biggest single lever.

How to improve CDN hit rate: Increase TTL (24h → 7 days), stale-while-revalidate, edge validation at CDN (reject invalid formats), prefetch on URL creation, negative caching for 404s.

🔵
Fourth to Break: Kafka Throughput (breaks around 50x)
10M events/sec beyond single Kafka cluster practical limits

Solutions: Regional Kafka clusters (each region has its own, merge for analytics), sampling (at extreme scale, sample 10% of events), client-side batching (100 events per message).

What Doesn't Break (Surprisingly)

These Scale Naturally
  • CDN — CloudFront/Fastly handle billions of requests. Cost scales, but it works.
  • Range Allocator — Already distributed. Each region has its own. 200K QPS = 20K per region = fine.
  • API Servers — Stateless, horizontal. Add more servers. Auto-scaling handles it.

📊
Pre-Analysis: Power Law Distribution
Traffic is NOT uniformly distributed — this changes everything
The Crucial Insight L5/L6 Candidates Miss

"100x scale" doesn't mean all URLs get 100x traffic. URL access follows a power law:

  • Top 1% of URLs → 50% of traffic
  • Top 10% of URLs → 90% of traffic
  • Remaining 90% of URLs → 10% of traffic

This is GOOD NEWS — caching is incredibly effective. But it creates HOT PARTITION problems — a single viral URL could receive 10M QPS alone.

Key insight from power law for cache sizing: We don't need to cache ALL URLs, just the hot ones. At 500M total URLs, the top 10% (50M) × 300 bytes = 15GB. Same as today! With smart eviction, the cache doesn't need to grow linearly with URL count.

Custom Alias Challenge — The Hard Part

Option C: Redis-Based Alias Registry (Best Solution)
Python · Custom Alias Claim with Redis + DB
def claim_custom_alias(alias, short_code, long_url):
    # Try to claim in Redis (distributed lock - atomic NX)
    result = redis.set(f"alias:{alias}", short_code, nx=True, ex=86400)
    if not result:
        raise AliasAlreadyTaken()

    # Redis claim successful, now write to database
    # If DB write fails, release the Redis key
    try:
        shard = get_shard(short_code)
        shard.insert(short_code, long_url, alias=alias)
        # Make Redis claim permanent (remove TTL)
        redis.persist(f"alias:{alias}")
    except Exception:
        redis.delete(f"alias:{alias}")
        raise

Zero-Downtime Migration Path

Migration Steps · Zero Downtime
Step 1: Add shard routing logic → route EVERYTHING to "shard 0" (current DB)
Step 2: Set up new shard primaries (shards 1-15) with replication
Step 3: Double-write — new URLs go to correct shard + shard 0
Step 4: Backfill — migrate existing URLs from shard 0 to correct shards
Step 5: Verify — read from both, compare results
Step 6: Cutover — read from correct shard, stop writing to shard 0
Step 7: Cleanup — remove duplicates from shard 0

Timeline: ~2-4 weeks with proper testing
Rollback: At EACH step, we can revert to reading from shard 0

Hot Key Detection + Replication

Python · Hot Key Detector
class HotKeyDetector:
    def __init__(self):
        self.hot_keys = set()
        self.counts = defaultdict(int)

    async def track(self, key):
        self.counts[key] += 1
        if self.counts[key] > HOT_THRESHOLD:  # e.g., 10K hits/minute
            if key not in self.hot_keys:
                self.hot_keys.add(key)
                await self.broadcast_hot_key(key)

    async def broadcast_hot_key(self, key):
        # Tell all servers to cache this locally
        # Also replicate to multiple Redis slots
        value = await redis.get(key)
        for i in range(NUM_REPLICAS):
            await redis.set(f"{key}:replica:{i}", value)

The Thundering Herd Scenario (Unprompted — L7 Signal)
What happens when the primary DB recovers after a 10-minute outage?
Why Most Candidates Miss This

The outage was bad. But the recovery can be worse. When DB comes back, all buffered writes + all expired cache entries hit it simultaneously. This can knock the DB down again — a cascading failure.

Python · Circuit Breaker Implementation
class CircuitBreaker:
    def __init__(self):
        self.state = "CLOSED"
        self.failure_count = 0
        self.last_failure = None

    async def call(self, func):
        if self.state == "OPEN":
            if time.now() - self.last_failure > BACKOFF:
                self.state = "HALF_OPEN"
            else:
                raise CircuitOpen()
        try:
            result = await func()
            self.reset()
            return result
        except Exception:
            self.record_failure()
            raise

    def record_failure(self):
        self.failure_count += 1
        self.last_failure = time.now()
        if self.failure_count > THRESHOLD:
            self.state = "OPEN"

# Usage: wrap all DB calls
result = await circuit_breaker.call(lambda: db.query(sql))
Post-Outage Recovery Playbook
  • Gradual traffic admission: Don't accept 100% traffic immediately. Start at 10%, ramp up over 5 minutes
  • Request coalescing: Multiple requests for same short_code during recovery get collapsed into one DB query
  • Cache warming: Proactively warm cache from DB backup or read replica before promoting primary

The One Thing to Practice

Before answering "what breaks at scale," ask yourself: "Is traffic uniformly distributed?" For most systems, the answer is NO (power law). This single insight means caching is incredibly effective, hot partitions are the real problem, and your cache doesn't need to grow linearly with your URL count.

🎯 Key Lesson — Phase 5

L5 engineers scale by multiplying resources. L6 engineers identify bottlenecks and solve them efficiently. L7 engineers recognize that traffic follows power laws, design adaptive systems that self-tune, plan zero-downtime migrations, and achieve sub-linear cost scaling. 100x traffic should never cost 100x.
