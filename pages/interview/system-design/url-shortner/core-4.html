Part 4 · Scale & Real-World Application

Back-of-Envelope Reasoning

Two scenarios — 1M users and 1B users — with full math for traffic, storage, and infrastructure. This is the math you do before choosing a strategy.

Scenario A
1M Users — Toy Scale
▾

Traffic Estimate

Math
1M users × 10% DAU = 100K DAU
100K DAU × 1 URL/day = 100K writes/day
Peak = 10× average: 100K / 86,400 × 10 ≈ 12 writes/sec peak
Reads (redirect): 10× writes = 120 reads/sec peak

Storage Estimate

Math
Per record: shortcode (8B) + URL (avg 200B) + metadata (50B) = ~260 bytes
After 1 year: 100K/day × 365 = 36.5M records
Storage: 36.5M × 260 bytes ≈ 10 GB
Required Infrastructure
  • Single database instance — handles this trivially
  • One application server (two for HA)
  • No caching needed (but Redis helps)
  • Which variant: DB Auto-Increment — simplest, 12 writes/sec is nothing
  • What's comfortable: Everything. This is toy scale. Don't over-engineer.
Scenario B
1B Users — 1000× Scale
▾

Traffic Estimate

Math
1B users × 10% DAU = 100M DAU
100M writes/day = 1,200 writes/sec average, 12,000 writes/sec peak
Reads: 10× = 120,000 reads/sec peak
What Breaks
  • Auto-increment on single DB: Bottleneck — can't handle 12K writes/sec
  • Range-based with single range server: Still a bottleneck at the range service
  • Hash-based: Collision rate becoming noticeable

Evolution Required

Infrastructure Evolution
1M scale                    1B scale
─────────────────           ─────────────────────────────
Single DB                   Sharded DB cluster (10+ shards)
Auto-increment              Snowflake IDs
No cache                    Multi-tier cache (local + Redis)
One region                  Multi-region with geo-routing
Non-Linear Effects — The Tricky Part
  • Cache invalidation complexity: O(n) → O(n²) as you add regions
  • Hot key management becomes critical (viral URLs)
  • Database sharding strategy becomes crucial (by shortcode prefix? by time?)

§4.3 Industry Deep-Dive

🔵 How Google Does It (Inferred)

ID generation: Likely Snowflake-variant backed by TrueTime (GPS + atomic clocks for bounded clock uncertainty)

Key design decisions:

  • Globally unique IDs without coordination
  • Use of Spanner for globally consistent storage where needed
  • Heavy caching at edge (Google Global Cache)
💡 What we can learn: Clock synchronization at scale requires hardware investment (GPS receivers). Software alone isn't enough.
🐦 How Twitter Does It

Architecture: Snowflake — they invented it

Key design decisions:

  • Machine IDs assigned via Zookeeper (ephemeral nodes)
  • Datacenter ID embedded in the ID structure
  • Custom epoch = Twitter's launch date
  • Scale: Billions of tweet IDs generated
💡 What we can learn: 41 bits of timestamp gives ~69 years. Plan your epoch carefully — document it so your successor in 2089 isn't surprised.
🔗 How bit.ly Does It (from public info)

Architecture: Initially MySQL auto-increment, evolved to distributed

Key design decisions:

  • Heavy caching (most clicks are reads, not writes)
  • Separate write path (shortening) from read path (redirect)
💡 What we can learn: Start simple (auto-increment), scale when needed. bit.ly was huge before it needed distributed ID generation.

🔑 Sound Bite About Scale

"At 1M users, use auto-increment and don't overthink it. At 100M users, you need Snowflake-style distributed generation. At 1B users, your bottleneck isn't ID generation — it's everything else: caching, sharding, global consistency. If you're still thinking about shortcode generation at 1B users, you've under-invested in the rest of your architecture."

§5.3 Impressive Nuances to Demonstrate

💡 Drop One of These to Signal Deep Expertise

Each of these is a signal that you've thought about this problem beyond the interview prep surface. Pick 1–2 that are relevant to your conversation and drop them naturally.

Epoch planning: "Our Snowflake epoch of Jan 2020 gives us until ~2089. That's probably fine, but should be documented so the next engineer isn't surprised."

Birthday paradox awareness: "With 7-char base62 and 10M URLs, we're at ~0.01% collision probability. Still acceptable, but worth a quarterly namespace utilization check."

Clock uncertainty bounds: "Google's TrueTime provides BOUNDED clock uncertainty — they KNOW their time is accurate ±7ms. That's what enables global consistency without coordination."

Sharding implications: "If we shard by shortcode prefix, Snowflake IDs have time-clustering — traffic spikes could create hot shards. May need range-based sharding instead."

Namespace economics: "We're 'spending' namespace. At 100K codes/day, 56B namespace lasts 1500 years. But waste 50M/day to server churn → that's 3 years of runway."

Information leakage: "Sequential codes leak our traffic volume. Hash codes leak nothing. Snowflake leaks creation time. Each has privacy implications."

Tail latency awareness: "Range allocation causes p99 latency spikes at exhaustion. Pre-fetch at 80% usage to smooth this tail — never let the allocation happen on a user request."

Structural vs. statistical uniqueness: "The aha moment: uniqueness can be structural (Snowflake — impossible to collide by construction) or statistical (hash — very unlikely). Both acceptable; know which you're choosing."

§5.4 Clarifying Questions to Ask

Ask these before committing to a strategy. Each question changes the answer.

  • "Idempotency required?"
    Same URL submitted twice: same shortcode or different? If yes → hash-based or dedup layer. If no → simpler. This is the single most impactful clarification.
  • "Predictable codes OK?"
    Can attackers enumerate URLs? If no → need random/hash/FPE. Shows security awareness.
  • "Max code length?"
    6 chars vs 8 chars changes the namespace by 62²= ~3844×. Shows awareness that namespace affects strategy.
  • "Time-ordering needed?"
    Do we query by creation time? Snowflake gives this for free. Shows B-tree index awareness.
  • "Availability during region failure?"
    99.9% or 99.99%? Determines coordination tolerance. Shows you understand the CAP trade-off.
  • "5-year growth rate?"
    500M URLs/month growing 2× annually changes namespace planning dramatically. Shows you're designing for the future, not just today.

Part 6 · Practice & Application

Sample Interview Problem

A complete mock problem with requirements, ambiguities to clarify, and a strong hire rubric. Use this to self-evaluate before your interview.

Problem Statement: Design bit.ly

Functional requirements: Given a long URL, generate a short URL. Given a short URL, redirect to the original. Optional: custom short URLs, URL expiration, analytics (click counts).

Non-functional requirements: Highly available (99.9% uptime). Low latency redirects (<100ms p99). Short URLs as short as possible. URLs globally unique.

Scale: 500M new URLs/month · 50B redirects/month · 5-year data retention

Ambiguities to Clarify First

"Is idempotency required? Same URL submitted twice: same shortcode or different?"

Changes if yes: Need hash-based generation or a separate dedup layer. Changes if no: Simpler generation (counter-based or Snowflake).

"What's the minimum acceptable code length? 6 chars? 7? 8?"

Why it matters: 6 chars = 56B max. 500M/month × 60 months = 30B. 6 chars works but tight — a viral event could exhaust it faster than projected.

"Do codes need to be unpredictable? Can attackers enumerate URLs?"

Changes if yes: Need random/hash approach or FPE obfuscation. Sequential codes leak business metrics.

"Is time-ordering valuable for queries?"

Changes if yes: Snowflake gives time-ordering for free and improves B-tree index locality. If no, any approach works.

📋 The One-Page Cheat Sheet
Default Choice
Auto-increment + base62 — simplest, works to 10K writes/sec. Don't over-engineer early.
When to Upgrade
Hitting write throughput limits, or need global distribution across regions.
Upgrade Path
Range-based (simpler, some wastage) or Snowflake (elegant, needs clock sync)
Special Cases
Need idempotency? → Hash-based · Need unpredictability? → Pool or Counter+FPE · Need time-sorting? → Snowflake
Key Metrics
Generation latency · Collision rate · Namespace utilization · Clock drift (Snowflake only)
Key Insight
Uniqueness can be structural (Snowflake) or statistical (hash). Both work — know which you're choosing and why.

🎯 Complete Deep Dive Series — Master Lesson

The entire shortcode generation problem reduces to one question: where do you sit on the coordination spectrum? Auto-increment trusts a single coordinator. Snowflake trusts physics. Hash trusts probability. Pool trusts pre-computation. Every design choice flows from that position. Knowing not just where each approach sits, but why — with numbers, failure modes, and migration paths — is what separates L6 from L7.

The final soundbite: "Single-node uniqueness is trivial. Distributed uniqueness is a consensus problem. Snowflake elegantly sidesteps consensus by encoding enough structure that collisions are physically impossible. That insight — that structure can replace coordination — is the core of modern distributed systems design."
