Phase 4 · 15–30 min

High-Level Architecture

15 – 30 minutes
Interviewer Prompt
"Good. Now walk me through your high-level architecture. I want to see the end-to-end flow for both creating a URL and redirecting. Show me every component, explain why it's there, and tell me where your single points of failure are."
❌ No Hire L4 / Weak L5
▾

"Okay, so here's my architecture. We'll have a load balancer in front that distributes traffic. Behind that, we have our API servers that handle the requests. The API servers talk to a MySQL database to store the URL mappings.

For caching, I'll use Redis to cache the frequently accessed URLs so we don't hit the database every time.

For the create flow: User sends request → Load balancer routes to API server → API server generates short code → Stores in MySQL → Returns short URL.

For the redirect flow: User clicks short URL → Load balancer → API server checks Redis cache → If not in cache, query MySQL → Return redirect.

I'll use Kafka for the analytics to track clicks. Should I go deeper on any part?"

Why This Fails
  • No justification for any component — "I'll use Redis" but why Redis vs Memcached?
  • No scale numbers — How many API servers? What's the cache hit rate?
  • MySQL without reasoning — Why not Postgres? Why not NoSQL?
  • "Kafka for analytics" — How? What's the data flow? Pure buzzword dropping
  • No failure analysis — What happens when MySQL dies?
  • No multi-region consideration — Users are global, architecture is single-region
  • No CDN — Redirect path doesn't leverage edge caching — the biggest miss
  • No rate limiting — How do we prevent abuse?
Interviewer's Notes "Architecture is a checklist of components without reasoning. 'I'll use Kafka' without explaining how. No failure modes discussed. No numbers. This is tutorial-level thinking, not production design."
Fatal Mistakes
  • No "why" for any technology choice
  • No scale/capacity reasoning
  • No failure mode analysis
  • Single-region design for global users
  • Kafka mentioned but never integrated into data flow
⚠️ Lean Hire Meets L5, Not L6
▾

"Let me draw out the architecture for both read and write paths.

Components: Load Balancer (AWS ALB), API Servers (stateless Node.js/Go, auto-scaled), Redis Cache, PostgreSQL (primary data store), Kafka (click events), Analytics Service.

Write Path: Client → Load Balancer → API Server → Generate Code → Write to PostgreSQL → Return Short URL

Read Path: Client → Load Balancer → API Server → Check Redis → (Cache miss?) → Query PostgreSQL → Return 301 Redirect

Analytics Path: On redirect → API Server → Kafka → Analytics Service → Aggregates clicks

Why these choices: PostgreSQL for ACID, Redis for sub-millisecond cache, Kafka for async durable event streaming.

Scaling: API servers 10-20 instances. Redis single instance with replica for HA. PostgreSQL primary with read replicas.

For single points of failure, the PostgreSQL primary is one — we'd need automatic failover."

What's Good
  • Separated read/write paths clearly
  • Included analytics as async path
  • Some reasoning for technology choices
  • Mentioned read replicas and HA
  • Identified PostgreSQL primary as SPOF
What's Missing for L6
  • No CDN for redirects — The single biggest gap for a read-heavy system
  • No estimation tie-in — "10-20 instances" but why? Based on what QPS?
  • No cache sizing or hit rate — How big is Redis? What hit rate do we expect?
  • No multi-region — Global users but single-region architecture
  • Rate limiting not shown — Where and how?
  • No code generation component — Where does the range allocator live?
  • Consistency model not explicit — What happens between write and cache populate?
  • Shallow failure analysis — Only mentioned one SPOF
Interviewer's Notes "Reasonable architecture with read/write separation. Missing CDN is a significant gap for a read-heavy service. No estimation backing the component counts. Failure analysis is superficial. Meets L5 bar but needs more depth for L6."
Upgrade Needed to Reach L6

Add: CDN for redirect caching, estimation-backed component sizing, multi-region strategy, explicit cache population strategy (write-through vs write-behind), and comprehensive failure mode analysis for each component.

✅ Hire Meets L6 Bar
▾
L6 Opening — Requirements Drive Architecture

"Let me walk through the architecture with estimations, then analyze failure modes. The 50:1 read:write ratio tells me this is heavily read-optimized. My architecture will prioritize the redirect path."

  • Write QPS: 2K peak (URL creation)
  • Read QPS: 100K peak (redirects)
  • Latency: p99 < 50ms for redirects
  • Availability: 99.99% for redirects
  • Global users (US, Europe, Asia)

Write Path (Create URL) — Step by Step

Flow · Create URL
1. Client POST /api/v1/urls with Idempotency-Key header
   │
   ▼
2. Rate Limiter checks token bucket in Redis
   - Key: user_id or IP
   - Limit: 100/hour anonymous, 1000/hour authenticated
   - If exceeded: return 429 immediately
   │
   ▼
3. API Server receives request
   ├─▶ 3a. Idempotency check: Redis GET idempotency:{key}
   │       If exists: return cached response (200, not 201)
   ├─▶ 3b. Validate long_url format and length
   └─▶ 3c. Duplicate check: Query by url_hash if allow_duplicate=false
   │
   ▼
4. Generate short code
   - API server has pre-allocated range from Range Allocator
   - Increment local counter, apply FPE
   - If range exhausted: request new range (async pre-fetch at 80%)
   │
   ▼
5. Write to PostgreSQL (primary)
   - INSERT INTO urls (id, short_code, long_url, ...)
   - Synchronous replication to at least 1 replica (durability)
   │
   ▼
6. Write-through to Redis cache
   - SET url:{short_code} {long_url} EX 86400
   - Ensures immediate read-your-writes consistency
   │
   ▼
7. Store idempotency record
   - SET idempotency:{key} {response} EX 86400
   │
   ▼
8. Return 201 Created with short URL

Latency breakdown:
  Rate limit check:   1ms
  Idempotency check:  1ms
  Code generation:    0.1ms
  DB write:           10-20ms
  Cache write:        1ms
  ─────────────────────────────
  Total:              ~15-25ms p50, <200ms p99

Read Path (Redirect) — Step by Step

Flow · Redirect
1. Client GET /{shortCode}
   │
   ▼
2. CDN Edge (CloudFront)
   - Check cache for this short code
   - Cache key: /{shortCode}
   - If HIT (90% of requests): return 301 immediately (5-20ms)
   - If MISS (10%): forward to origin
   │
   ▼ (cache miss: 10% of requests)

3. Origin: API Server
   ├─▶ 3a. Redis cache: GET url:{short_code}
   │       If HIT (99% of origin requests): proceed to redirect
   ├─▶ 3b. Cache MISS: Query PostgreSQL
   │       SELECT long_url, redirect_type, expires_at
   │       FROM urls WHERE short_code = ?
   │       — Uses covering index (no heap access) — Latency: 5-10ms
   └─▶ 3c. Populate Redis: SET url:{short_code} {long_url} EX 86400
   │
   ▼
4. Check expiration
   - If expires_at < now(): return 410 Gone
   │
   ▼
5. Return 301 Redirect
   - Location: {long_url}
   - Cache-Control: public, max-age=86400
   - CDN caches this for future requests
   │
   ▼
6. Async: Emit click event to Kafka (fire-and-forget)
   - Event: {short_code, timestamp, ip, user_agent, referrer}
   - If Kafka unavailable: drop event (analytics can be lossy)

Latency breakdown (CDN miss):
  Redis lookup:           1ms
  DB lookup (cache miss): 10ms
  Return redirect:        1ms
  Total:                  ~12ms p50 (cache hit), ~25ms p50 (cache miss)

With 90% CDN hit rate:
  Effective p50: 0.9 × 10ms (CDN) + 0.1 × 12ms (origin) ≈ 10ms ✓

Component Sizing (Estimation-Backed)

ComponentSizingReasoning
CDNCloudFront, 200+ PoPsGlobal users, edge caching target
API Servers20 instances100K QPS ÷ 5K per instance
Redis32GB, 3-node cluster15GB working set + buffer, HA
PostgreSQLPrimary + 2 replicas2K write QPS, 5K read QPS overflow
Kafka3 brokers, 12 partitions100K events/sec, 7-day retention
Why Estimation-Backed Sizing Signals L6

L5 says "10-20 instances." L6 says "20 instances because 100K QPS ÷ 5K per instance." The number comes from math, not intuition. Every component has a reason behind its specification.

Failure Mode Analysis

ComponentFailureImpactMitigation
CDNPoP failureTraffic routes to next PoPBuilt-in CDN redundancy
CDNTotal outageAll traffic to originOrigin overprovisioned at 100K QPS
API ServerInstance crashLoad balancer routes aroundHealth checks, 20 instances

RedisTotal outageAll requests hit DB (degraded)DB handles it, alert triggers

PostgreSQL PrimaryFailureReads continue from replicasRead path unaffected

Range AllocatorFailureCode gen stops when buffer exhaustedWorkers have 10K buffer, alert at 80%
Single Points of Failure (Explicitly Named)
  • PostgreSQL Primary — Mitigated by automated failover, but writes pause ~30 sec during promotion
  • Range Allocator DB — If Postgres sequence unavailable, new ranges can't be allocated. Mitigation: workers have pre-fetched ranges; monitoring alerts well before exhaustion
Consistency Model (Write-Through)

After a successful create, the URL is immediately resolvable. This is guaranteed by write-through cache on create — Redis is populated synchronously before returning 201. CDN will miss on first access (new URL not yet at edge), but origin Redis will serve it.

AOMB Completeness Checklist — Verified
  • Structured opening tying back to requirements (50:1 ratio)
  • CDN layer as first component (biggest win for read-heavy)
  • Separate read and write paths with detailed numbered steps
  • Component sizing backed by estimations from Phase 2
  • Rate limiting shown with mechanism (token bucket, Redis)
  • Idempotency flow shown in write path
  • Cache strategy explicitly stated (write-through, not write-behind)
  • Async analytics path (Kafka, fire-and-forget)
  • Latency breakdown for each path with CDN hit rate math
  • Failure mode table with mitigations for every component
  • SPOFs identified by name with specific mitigations
  • Consistency model discussed (read-your-writes guarantee)
  • Clear transition offering to go deeper
Interviewer's Notes "Strong architecture with estimation backing. CDN placement shows understanding of read-heavy optimization. Failure analysis is systematic. Write-through cache shows consistency thinking. Would like to probe multi-region and deep dive on redirect hot path."

What Separates This From L5

L5 draws boxes and arrows. L6 draws boxes and arrows with numbers — latency per step, sizing per component, and failure probabilities. L6 also makes explicit trade-offs: analytics is lossy, and that's intentionally acceptable. The write-through vs write-behind decision shows they understand consistency models, not just caching patterns.

The One Thing to Practice

After drawing your architecture, walk through ONE request end-to-end with latency at each step: "Request arrives at CDN, 5ms. Cache miss, forwarded to origin. Redis lookup, 1ms. Cache hit, return 301, 1ms. Effective p50: 10ms." This demonstrates you understand where time goes.

🎯 Key Lesson — Phase 4

L5 engineers draw architectures that work. L6 engineers draw architectures that scale and handle failures. L7 engineers draw architectures that are operable — they include monitoring, alerting, cost analysis, and evolution paths. They think about the full lifecycle, not just the happy path.
