💥 Failure Mode Analysis — Three Tiers

L7 Signal: Three-Tier Classification

Most candidates lump all failures together. L7 separates availability (service down) from correctness (wrong data returned) from operational (degrading slowly). Each tier has different detection strategies, response urgency, and customer impact.

Why Three Tiers Matter

Tier 1 failures (availability) trigger immediate P1 pages. Tier 2 failures (correctness) may go undetected without explicit monitoring — they're the scariest. Tier 3 failures (operational) accumulate silently until they cause Tier 1 events. Naming this taxonomy in an interview signals you've operated systems under real production pressure.

Why Alert Thresholds Are Set Below SLO

Notice: the redirect latency alert fires at p99 > 40ms, but our SLO is 50ms. This gives us a 10ms warning window to investigate before we breach the SLO and burn error budget. Setting alerts at the SLO itself is too late — you want to catch the trend, not react to the violation.

🌍 Future Evolution: Multi-Region

When We Need <20ms Redirects Globally

Current: Single region (US-East). This works because the CDN handles 90% of traffic at edge. When we need sub-20ms globally even for cache misses, we evolve to Active-Passive Multi-Region.

• Writes always go to US-East (single writer)
• Replication lag: 100-200ms (acceptable for our use case)
• User creates URL in EU → request routed to US → 100ms added latency
• URL readable in EU within 200ms (async replication)

What Separates This From L6

L6 designs a working system with proper failure handling. L7 designs a system that is operable — complete with dashboards, alert thresholds, runbooks, cost analysis, and evolution roadmap. They think not just about Day 1 (launch) but Day 365 (operate) and Day 730 (evolve).

L5 engineers draw architectures that work. L6 engineers draw architectures that scale and handle failures. Decision Deep Dive

🔬 Database Selection

The Problem in Plain English

Store billions of URL mappings durably. Query by short code in <10ms. Handle 2K writes/sec and 10K reads/sec (after cache miss). Never lose data. Five approaches — only one is right for this system at this scale.

Approach 1
PostgreSQL
▾

Open-source relational database with ACID transactions, strong consistency, and excellent indexing.

Benefits
  • ACID transactions for strong consistency
  • Mature, battle-tested (Instagram runs on it)
  • Excellent B-tree indexes, partial indexes, covering indexes
  • Synchronous replication for durability
  • Rich ecosystem: PgBouncer, pg_auto_failover, JSONB
Limitations
  • Vertical scaling has limits (~50K QPS on single instance)
  • Horizontal write scaling requires sharding (complex)
  • Replication adds latency (sync) or risks data loss (async)
When to Use / When It Breaks

Use: Strong consistency required, data fits in single cluster (~10TB), query patterns known. Breaks: 100K+ write QPS, data exceeds 10-20TB, need automatic sharding.

Operational Cost

Medium. Need to manage connections (PgBouncer), vacuuming, replication lag, failover.

Interview Signal
I'll use Postgres because I know it
Postgres for ACID compliance. At 2K writes/sec with covering indexes, a single primary handles our load. Synchronous replication to one replica for durability (RPO=0). If we outgrow it, migration path to CockroachDB for horizontal scaling.
Approach 2
MySQL
▾

Mature relational database, particularly strong in read-heavy workloads.

Benefits
  • Mature, proven at scale (Facebook, Twitter historically)
  • InnoDB is solid for most workloads
  • Good replication options; wide hosting availability
Limitations
  • Less feature-rich: no partial indexes, weaker JSON support vs Postgres
  • Replication setup more complex than Postgres
  • No native UPSERT until recently
When to Use / When It Breaks

Use: Read-heavy workloads, team has MySQL expertise. Breaks: Same ceiling as Postgres — horizontal write scaling.

Interview Signal
MySQL or Postgres, they're basically the same
MySQL works, but I prefer Postgres for partial indexes (WHERE deleted_at IS NULL) and covering indexes with INCLUDE clause. Both handle our scale — the difference is in the operational niceties.
Approach 3
DynamoDB
▾

AWS managed NoSQL key-value store with automatic scaling to millions of QPS.

Benefits
  • Fully managed — zero database operations
  • Automatic scaling to millions of QPS
  • Single-digit millisecond latency; built-in replication
Limitations
  • No transactions across items (limitations with transactions API)
  • No secondary indexes without data duplication
  • Query patterns must be designed upfront; vendor lock-in
  • Cost can explode at very high scale
When to Use / When It Breaks

Use: Simple key-value lookups at massive scale, zero ops desired. Breaks: Complex queries, strong consistency needs, cost-sensitive at very high scale.

Interview Signal
DynamoDB for NoSQL scalability
DynamoDB works for the redirect lookup (pure key-value). But custom alias uniqueness needs strong consistency — DynamoDB's conditional writes work but are more complex than a Postgres unique constraint. For our scale, Postgres is simpler. If we needed 100K+ write QPS, I'd reconsider.
Approach 4
Cassandra
▾

Distributed NoSQL database designed for high write throughput and multi-datacenter availability.

Benefits
  • Linear horizontal scaling; no single point of failure
  • High write throughput (100K+ QPS per node)
  • Multi-datacenter replication built-in
Limitations
  • Eventual consistency by default — tunable but complex
  • No transactions; secondary indexes are problematic
  • High operational complexity: JVM tuning, compaction, repair jobs
When to Use / When It Breaks

Use: Write-heavy workloads at massive scale, multi-datacenter deployments. Breaks: Strong consistency needed, complex queries, transactions.

Operational Cost

High. Requires significant Cassandra-specific expertise.

Interview Signal
Cassandra for distributed data
Cassandra's eventual consistency complicates custom alias uniqueness. We could use Lightweight Transactions (LWT), but they're significantly slower. For URL shortener scale, Postgres with read replicas is simpler and sufficient. Cassandra makes sense if we 10x write volume.
Approach 5
CockroachDB / Spanner
▾

Distributed SQL databases offering horizontal scaling with strong consistency — the best of both worlds.

Benefits
  • Horizontal scaling with automatic sharding
  • Strong consistency (serializable by default)
  • SQL interface — familiar and powerful
  • Multi-region with automatic data placement
Limitations
  • Higher write latency than single-node Postgres (~10-20ms)
  • Complexity: distributed transactions, Raft consensus overhead
  • Cost: Spanner is expensive; CockroachDB requires more nodes
  • Overkill for small-to-medium scale
When to Use / When It Breaks

Use: SQL + horizontal scaling + strong consistency all required simultaneously. Doesn't break — but expensive for workloads that don't need it.

Operational Cost

Medium (CockroachDB self-hosted) to Low (Spanner fully managed).

Interview Signal
CockroachDB for scale
At our current scale, Postgres handles it with room to spare. If we 10x and need horizontal write scaling, CockroachDB gives us a migration path without abandoning SQL or strong consistency. I'd abstract the data layer behind a repository interface now to make that migration cheaper later.

Decision Framework Script

⚙️ The Complete Decision Script

"For our database, I need: strong consistency for uniqueness guarantees, sub-10ms reads after cache miss, 2K writes/sec, 10K reads/sec, and durability (RPO=0).

I can rule out Cassandra immediately — eventual consistency complicates custom alias uniqueness. We could use Lightweight Transactions, but they're slow and add complexity without benefit at our scale.

I can rule out DynamoDB — our team prefers SQL semantics, and the conditional write logic for alias uniqueness is more complex than a Postgres unique constraint. Also, DynamoDB cost can surprise you at scale.

Between PostgreSQL and CockroachDB: Postgres is simpler and sufficient at our current scale. We're at 2K writes/sec, 15TB over 5 years — well within Postgres limits. CockroachDB adds distributed transaction overhead that would increase write latency 5-10ms for no benefit yet.

For THIS system: PostgreSQL with synchronous replication to one replica for durability (RPO=0), plus two async replicas for read scaling.

The trade-off is horizontal write scaling — if we 10x to 20K writes/sec, we'd need to migrate to CockroachDB. I'd mitigate migration pain now by: (1) using standard SQL with no Postgres-specific features in application code, (2) abstracting data access behind a repository interface so we can swap the implementation layer.

That's a good problem to have."
