Phase 1

Problem Clarification

0 – 5 minutes
The Problem — Interviewer Prompt
"Design a URL shortener." — That's it. That's all you get.

In Study/AOMB mode, the breakdown below shows exactly how candidates at each level approach the initial clarification phase — what questions they ask, how they structure their thinking, and what separates No Hire from Strong Hire.

❌ No Hire L4 / Weak L5
▾

"Okay, a URL shortener. So we need to take long URLs and make them short. I'm thinking we can use a hash function to generate the short codes. We'll need a database to store the mappings. I'll use MySQL for that since it's reliable.

For the architecture, we'll have a web server that receives requests, generates the short URL, stores it in the database, and returns it to the user. For redirection, we just look up the short code in the database and redirect.

We should probably use Redis for caching since URL lookups need to be fast. And we'll put a load balancer in front of the web servers for scalability.

Should I start drawing the architecture?"

Why This Fails
  • Zero clarifying questions asked — jumped straight to solution
  • Assumed scope without validation — didn't ask about custom aliases, analytics, expiration
  • No understanding of scale — didn't ask about QPS, storage requirements, geographic distribution
  • Technology choices before requirements — mentioned MySQL, Redis without understanding the actual needs
  • No discussion of success criteria — what does "working" mean for this system?
Interviewer's Notes "Candidate immediately jumped to implementation. Did not demonstrate the ability to scope a problem or identify ambiguity. Concerning pattern — will they build the wrong thing in a real project?"
Fatal Mistakes
  • Started designing before understanding the problem
  • Never asked a single clarifying question
  • Mentioned specific technologies (MySQL, Redis) with no justification
⚠️ Lean Hire Meets L5, Not L6
▾

"Before I dive in, I'd like to ask a few clarifying questions.

First, what's the expected scale? How many URLs do we expect to shorten per day, and how many redirects? This will help me understand the read/write ratio.

Second, how long should the shortened URLs be valid? Do they expire, or are they permanent?

Third, do users need to be authenticated to create short URLs, or is this a public service?

Fourth, do we need analytics — like tracking how many times a short URL was clicked?

And finally, are there any latency requirements I should know about?"

What's Good
  • Asked clarifying questions before designing
  • Covered scale, expiration, authentication, and features
  • Showed awareness that requirements matter
What's Missing for L6
  • Questions are generic/checklist-style — not demonstrating deep thinking about this problem
  • Didn't ask about success criteria or what problem we're actually solving
  • No questions about existing systems or constraints — is this greenfield or integration?
  • Didn't ask about geographic distribution of users
  • No prioritization thinking — which questions are MOST important?
  • Didn't ask about custom short codes or collision handling
  • No business context questions — internal tool vs public service vs enterprise product?
Interviewer's Notes "Candidate asked reasonable questions but they felt rehearsed. Didn't demonstrate ownership-level thinking about what could go wrong or what makes this problem uniquely challenging."
Upgrade Needed to Reach L6

Questions that show you understand the HARD parts of this problem — ID generation collisions, consistency requirements for short codes, abuse prevention — and questions that reveal business context rather than just technical parameters.

✅ Hire Meets L6 Bar
▾

"Before I start designing, I want to make sure I understand the problem space and scope appropriately. Let me ask some clarifying questions, starting with the most important ones.

First, what problem are we actually solving, and for whom? Is this an internal tool for a company to share links, a public consumer service like bit.ly, or an enterprise product? This fundamentally changes our security model, abuse concerns, and scale characteristics.

Second, what does success look like? Are we optimizing for redirect latency, URL creation throughput, analytics richness, or cost efficiency? I want to understand the primary metric we're designing for.

Third, what's the consistency requirement for short code uniqueness? If two users try to create short URLs at the same time, we need to guarantee no collisions. Is strong consistency required, or can we tolerate eventual consistency with conflict resolution?

Fourth, are we supporting custom aliases? For example, can a user request 'mysite.co/promo2024' specifically? This significantly complicates the ID generation strategy.

Fifth, what's the expected lifespan of URLs? Permanent storage has very different cost and design implications than 30-day expiration with cleanup.

Sixth, where are our users? Single region, multi-region, or global? This affects our consistency model and latency targets.

Seventh, is there an existing system I'm replacing or extending? Are there data migration concerns or API compatibility requirements?

Let me pause there — these answers will significantly shape my design. Which of these are most relevant to scope for today's discussion?"

AOMB Completeness Checklist — Verified
  • Structured opening that frames the approach
  • Questions prioritized by impact on design
  • Business context questions (not just technical)
  • Consistency requirements asked explicitly
  • Geographic distribution considered
  • Existing systems/constraints explored
  • Clear handoff asking interviewer to help scope

L6 Signals Demonstrated

  • Prioritization — asked about success criteria BEFORE scale numbers
  • System-level thinking — immediately identified collision/uniqueness as a hard problem
  • Business awareness — distinguished internal vs public vs enterprise use cases
  • Ownership mentality — asked about existing systems, not assuming greenfield
  • Scoping instinct — explicitly asked interviewer to help narrow focus
Interviewer's Notes "Strong clarification approach. Candidate identified the genuinely hard parts of this problem (consistency for uniqueness, custom aliases). Questions were prioritized and showed real experience with distributed systems. Good signal."
Key Phrases That Signal L6 — Memorize These
  • "What problem are we actually solving, and for whom?"
  • "What does success look like? What's the primary metric?"
  • "What's the consistency requirement for [X]?"
  • "This fundamentally changes our [design aspect]..."
  • "Let me pause — which of these are most relevant for today?"
Communication Technique Demonstrated

Candidate signposted ("starting with the most important"), numbered their questions for clarity, and explicitly invited the interviewer to help scope — showing they can drive a conversation while remaining collaborative.

What Separates This From L5

L5 asks "What's the scale?" L6 asks "What does success look like?" BEFORE asking about scale. L6 recognizes that understanding the why shapes the how — a 99.9% availability internal tool looks completely different from a 99.99% public-facing service.

🏆 Strong Hire L7 Signal · Exceptional
▾

"Thank you. Before I design anything, I need to understand the problem space deeply. I'll organize my questions into three categories: business context, technical constraints, and scope boundaries.

Business Context — what and why:

What's the primary use case we're solving for? Internal link sharing, marketing campaign tracking, or general-purpose public shortening? Each has radically different threat models — a public service needs abuse prevention and rate limiting that an internal tool doesn't.

Who's the customer? If it's B2B enterprise, we might need tenant isolation, custom domains, and audit logging. If it's consumer, we're optimizing for simplicity and speed.

What's the business model? Free with ads means we need high-volume, low-cost infrastructure. Paid tiers mean we need metering and entitlements.

Technical Constraints — the hard parts:

What's the consistency requirement for short code uniqueness? I'm specifically thinking about the case where two users or two datacenters try to claim the same short code simultaneously. Do we need CP or can we tolerate AP with conflict resolution?

Are we supporting custom vanity URLs? This is often a premium feature, but it introduces a much harder coordination problem than random ID generation.

What's the URL validation requirement? Do we just check URL format, or do we need to verify the destination is live, safe (not phishing/malware), and appropriate?

What's the data retention policy? GDPR right-to-deletion for European users? Can we actually delete URLs or do we need audit trails?

Scope Boundaries — what we're NOT building:

Are analytics in scope? Click tracking, geographic breakdown, referrer analysis, device breakdown? This could be 50% of the system complexity.

Are we building the link preview/unfurling service, or just redirect?

Are we handling abuse at this layer — detecting spam campaigns, blocking malicious URLs — or is that a separate service?

My mental model going in: The core URL shortening is straightforward — it's a key-value lookup. The interesting design challenges are: (1) globally unique ID generation at scale, (2) low-latency redirects with high availability, (3) abuse prevention without adding latency, and (4) analytics that don't slow down the redirect path.

Based on your answers, I'll timebox our discussion to the areas where I can demonstrate the most depth. Which aspects would you like me to prioritize?"

AOMB Completeness Checklist — Verified (L6 requirements plus)
  • Non-obvious concerns: GDPR deletion, URL safety verification, tenant isolation
  • Business model implications on architecture
  • Explicit threat model thinking (abuse, phishing)
  • Proactively stated what's NOT in scope as questions
  • Articulated their mental model of where complexity lies
  • Drove the conversation by offering to prioritize

What Makes This Exceptional

  • Categorized questions into business/technical/scope — showing structured thinking
  • Identified the REAL hard problems before being told — ID generation, abuse, analytics isolation
  • Mentioned compliance (GDPR) proactively — most candidates never think of this
  • Threat model thinking — distinguished internal vs public threat surfaces
  • Business model to architecture connection — "free with ads means high-volume, low-cost"
  • Drove the interview — offered to prioritize and timebox
Interviewer's Notes "Exceptional problem decomposition. Candidate clearly has experience building and operating systems at scale. Identified GDPR, abuse prevention, and analytics isolation without prompting. This is someone who's been burned by missing requirements before."

The "Wow" Moments

"Each has radically different threat models" — shows security-first thinking
"The core URL shortening is straightforward — it's a key-value lookup" — immediately identified that the OBVIOUS part isn't the HARD part
"Analytics that don't slow down the redirect path" — understood the performance isolation requirement
What Separates This From L6

L6 asks good clarifying questions. L7 articulates their mental model of where the complexity lies BEFORE starting the design. They say "here's what I already know is hard, and here's what I need you to tell me." This shows genuine expertise, not just a checklist.

The Non-Obvious Insight

The candidate recognized that analytics isolation is a key architectural challenge. Most candidates think of analytics as "just log the click." But at scale, synchronous analytics kills redirect latency. The Strong Hire candidate already knows this will require async processing, sampling, or a separate write path — before even starting the design.

🎯 Key Lesson — Phase 1

L5 engineers ask "what should I build?" L6 engineers ask "what does success look like and what's actually hard about this?" L7 engineers articulate their mental model upfront, showing they already understand the complexity landscape before being told.

Phase 2

Requirements Gathering

5 – 10 minutes
Interviewer Prompt
"Alright, you understand the problem. Give me your requirements — functional and non-functional. I want your top 3 non-functional requirements with specific numbers, justified. And tell me explicitly what you're NOT building."
❌ No Hire L4 / Weak L5
▾

"Okay, so the functional requirements are: Users can create short URLs from long URLs. Users can click short URLs and get redirected. Users can see analytics for their URLs. Users can create custom short URLs. Users can set expiration dates. Users can delete their URLs. Users need to sign up and log in. Users can see a dashboard of all their URLs. The system should track click counts. We need an API for programmatic access.

For non-functional requirements: The system should be fast. The system should be scalable. The system should be reliable. The system should be secure.

I think that covers everything. Should I start on the design now?"

Why This Fails
  • Feature soup — listed 10 functional requirements with no prioritization
  • Vague NFRs — "fast," "scalable," "reliable" are meaningless without numbers
  • No SLAs — no latency targets, availability percentages, or throughput numbers
  • No scope control — didn't say what's out of scope
  • No justification — why these requirements? What drives them?
  • Time mismanagement — would take months to build everything listed
Interviewer's Notes "Candidate provided a feature wishlist, not prioritized requirements. No specific SLAs. 'Should be fast' is not an engineering requirement. Serious concern about ability to scope work."
Fatal Mistakes
  • Listed 10+ features without prioritization (P0/P1/P2)
  • Non-functional requirements had zero numbers
  • Never mentioned what's explicitly OUT of scope
  • No connection between requirements and business goals
⚠️ Lean Hire Meets L5, Not L6
▾

"Let me separate functional and non-functional requirements.

Functional Requirements: Create a short URL from a long URL — this is core functionality. Redirect from short URL to original URL — also core. Support custom aliases for premium users. Track click counts for each URL. Optional URL expiration.

Non-Functional Requirements: Latency: Redirects should be fast — under 100ms. Availability: System should have high uptime — 99.9%. Scalability: Should handle millions of requests.

Out of scope: Detailed analytics (geographic, device breakdown). Link preview/unfurling. User accounts for basic usage.

Does this look right?"

What's Good
  • Separated FR and NFR clearly
  • Provided some numbers for NFRs
  • Listed out of scope items
  • Kept functional requirements reasonable
What's Missing for L6
  • SLAs not specific enough — "under 100ms" but p50? p99? p999?
  • No read/write breakdown — what's the ratio? What's the QPS?
  • No storage requirements — how much data, how long retained?
  • No justification — WHY 99.9%? What's the business impact of 99.99%?
  • No consistency model — what happens on failure? Strong vs eventual?
  • Availability math not done — 99.9% = 8.7 hours/year downtime, is that acceptable?
  • Missing durability requirement — URLs should never disappear
Interviewer's Notes "Candidate structured the requirements but lacked precision. '99.9% availability' — have they calculated what that means in downtime? No reasoning for why these specific numbers."
Upgrade Needed to Reach L6

Specific percentile latencies (p50/p99), calculated QPS with read/write ratio, justified availability target with business reasoning, explicit consistency model, and data durability requirements.

✅ Hire Meets L6 Bar
▾

"Let me define prioritized requirements with specific, justified targets.

Functional Requirements — prioritized:

P0 — Must have for launch: URL shortening: Accept a long URL, return a short URL with a unique 7-character code. URL redirection: Given a short URL, redirect to the original URL with HTTP 301/302. Uniqueness guarantee: No two URLs ever map to the same short code.

P1 — Important, can fast-follow: Custom aliases. Basic analytics: Track click count per URL. URL expiration: Optional TTL on URLs.

P2 — Nice to have: Detailed analytics. User accounts and dashboard. API rate limiting tiers.

Non-Functional Requirements — my top 3 with justification:

1. Availability: 99.99% (four nines) — Why 99.99% and not 99.9%? 99.9% = 8.7 hours downtime/year. For a public URL shortener, that's unacceptable — a marketing campaign could fail. 99.99% = 52 minutes downtime/year. 99.999% would require active-active multi-region with complex consistency — overkill for this use case.

2. Latency: p50 < 10ms, p99 < 50ms for redirects — Redirects are in the critical path of user experience. For URL creation: p99 < 200ms is acceptable — it's not user-facing latency-sensitive.

3. Durability: Once created, a URL mapping must NEVER be lost — If a short URL stops working, it breaks links across the internet — in emails, documents, ads. This is different from availability. This means replicated storage with at least 3 copies across failure domains.

Derived Capacity Requirements: Write QPS: 500M / (30 × 86,400) ≈ 200 writes/sec average, ~1,000/sec peak. Read QPS: 10B / (30 × 86,400) ≈ 4,000 reads/sec average, ~20,000/sec peak. Read:Write ratio: ~20:1 — this is a read-heavy system. Storage: 500M × 12 × 5 years × ~500 bytes ≈ 15 TB over 5 years.

Strong consistency for writes. A newly created short URL must be immediately resolvable. Eventual consistency acceptable for analytics.

Explicitly NOT building: Link safety scanning, link preview/unfurling, detailed analytics beyond click counts, multi-tenancy/white-labeling, user authentication for URL creation.

These requirements shape my architecture. Should I proceed to API design?"

AOMB Completeness Checklist — Verified
  • P0/P1/P2 prioritization
  • Specific numbers for SLAs with percentiles
  • Business justification for availability target
  • Read/write QPS calculated with peak factors
  • Storage estimation done
  • Explicit consistency model stated
  • Clear list of what's NOT being built
  • Transition to next phase
Interviewer's Notes "Candidate demonstrated clear requirements thinking. Particularly impressed by the durability vs availability distinction and the business justification for 99.99%. Good estimation approach. Ready to design."
Key Phrases That Signal L6 — Memorize These
  • "Let me prioritize as P0, P1, P2..."
  • "Why 99.99% and not 99.9%? Because..."
  • "p50 < X, p99 < Y for [operation]"
  • "This means [specific architectural implication]"
  • "Strong consistency for [X], eventual consistency acceptable for [Y]"
  • "Explicitly NOT building: [list]"
What Separates This From L5

L5 says "high availability." L6 says "99.99%, which means 52 minutes/year downtime, and here's why that's the right target for THIS system." The difference is justification and precision — L6 engineers can defend their numbers.

🏆 Strong Hire L7 Signal · Exceptional
▾

"Let me structure this as prioritized requirements with success metrics, justified targets, and explicit trade-offs.

Functional Requirements — with success criteria (P0 Launch Blockers):

RequirementSuccess CriteriaWhy P0
URL shorteningGiven valid URL, return unique short code in <200ms p99Core value prop
URL redirectionGiven valid short code, redirect in <50ms p99User-facing critical path
Uniqueness guaranteeZero collisions, ever, globallyOne collision destroys trust
IdempotencySame long URL + user = same short URL (optional)Prevents link duplication

P1 — Fast-follow (week 2–4): Custom aliases with conflict detection. Click counting (async, eventually consistent). URL expiration with automatic cleanup.

P2 — Future roadmap: Rich analytics, user accounts, API tiers, abuse detection.

Non-Functional Requirements — the three that drive architecture:

1. Availability: 99.99% — but let me break this down. I'm actually setting different SLOs for different operations:

OperationAvailability SLOJustification
Redirects99.99%Revenue-critical, brand-critical
URL Creation99.9%Can tolerate brief outages
Analytics Dashboard99.5%Non-critical, can degrade

Why this tiering? Redirects are in the critical path of potentially millions of marketing campaigns, embedded links, QR codes. URL creation is only visible to our direct users. Analytics is internal tooling. This tiered approach lets me optimize architecture — redirects can be served from a heavily cached, read-optimized path, while creation can have more complex coordination.

2. Latency — percentile targets with tail latency reasoning:

Operationp50p99p999
Redirect<5ms<50ms<200ms
Create<50ms<200ms<500ms

Why p999 for redirects? At 20K QPS, p999 means 20 requests/second hit that latency. If p999 is 2 seconds, that's 20 frustrated users every second, 1.7M bad experiences daily. Tail latency matters at scale.

What drives these latencies? p50 assumes cache hit (Redis: <1ms) + minimal processing. p99 assumes cache miss, single DB read (~10–20ms) + network. p999 assumes retry scenario or degraded path.

3. Data Durability: RPO = 0, RTO < 5 minutes

RPO (Recovery Point Objective) = 0: Once we acknowledge a URL creation, that mapping must never be lost. Requires synchronous replication to at least 2 nodes before acknowledging. RTO (Recovery Time Objective) < 5 minutes: If a primary fails, we must failover within 5 minutes.

Why is durability more important than availability? An unavailable system frustrates users temporarily. A system that loses data — where a short URL suddenly 404s — destroys trust permanently and breaks the internet's link graph.

Derived Capacity Requirements with Monte Carlo thinking:

Estimation 
Writes (URL Creation):
  Average: 500M / 2.6M seconds/month ≈ 190 QPS
  Daily pattern: 2x during business hours (US + Europe overlap)
  Peak: 5x average for viral events = ~1,000 QPS
  Design target: 2,000 QPS (2x headroom)

Reads (Redirects):
  Average: 10B / 2.6M seconds ≈ 3,850 QPS
  Peak (viral link): 10x = ~40,000 QPS
  Design target: 100,000 QPS (2.5x headroom for growth)

Storage per URL record:
  short_code:  7  bytes
  long_url:    200 bytes average
  created_at:  8  bytes
  expires_at:  8  bytes (nullable)
  user_id:     8  bytes (nullable)
  click_count: 8  bytes
  metadata:    50 bytes
  ─────────────────────
  Total:       ~300 bytes per URL
  With indexes and overhead: ~500 bytes per URL

Storage growth:
  Year 1: 500M × 12 × 500B = 3 TB
  5 years: 15 TB raw, ~45 TB with 3x replication

Bandwidth:
  Redirect response: ~500 bytes (headers + 301 response)
  At 100K QPS peak: 50 MB/s — trivial

Cache Sizing:
  Working set: ~10% of URLs are "hot" (power law)
  500M × 10% × 300 bytes = 15 GB
  Fits in single Redis instance with room to spare

Consistency Model with Explicit Trade-offs

DataConsistencyWhy
URL mappingsStrong (read-your-writes)User creates URL, immediately shares it — must work
Custom alias claimsLinearizableCan't have two users claiming same alias
Click countsEventual (seconds delay OK)Async increment, no user-facing impact
Analytics aggregatesEventual (minutes delay OK)Dashboard refresh, not real-time

Explicit Non-Goals: Link safety scanning — assume external service, we just call it. SEO/preview metadata — separate system. User authentication — anonymous creation OK (rate limited by IP). Multi-tenancy — single service, no tenant isolation. Geographic compliance — no data residency requirements. Billing/metering — assume separate service.

SLO Monitoring — how we'll know we're meeting requirements: Redirect latency p99 measured per region, alerted at >40ms (10ms before SLO). Availability measured as success rate, alerted at <99.95% over 5 minutes. Error budget: 0.01% errors/month = 100 errors per million requests — alerts when 50% consumed.

I'll use these requirements to drive my API design and architecture. The high read:write ratio (20:1+), strong consistency for writes, and p99 < 50ms redirect requirement will push me toward aggressive caching and read replicas. Ready to proceed?"

  • RPO/RTO explicitly defined with justification
  • Field-level storage breakdown
  • Cache sizing calculated with working set assumption
  • Bandwidth estimated
  • Consistency model as a table with rationale per data type
  • SLO monitoring approach mentioned

  • Clear architectural implications stated
Interviewer's Notes "Exceptional requirements definition. Tiered SLOs show mature operational thinking — this person has run systems in production. RPO/RTO framing, error budgets, and tail latency awareness all signal genuine experience. Strong hire signal."

The "Wow" Moments

"At 20K QPS, p999 means 20 requests/second hit that latency" — concrete math showing why tail latency matters
Tiered availability SLOs — most candidates give one number; this candidate recognized different operations have different criticality
"RPO = 0, RTO < 5 minutes" — used the exact terms a production team would use
Error budget concept — proactively introduced SRE-style thinking
What Separates This From L6

L6 gives good numbers with justification. L7 thinks in distributions and operational reality — they mention p999, error budgets, tiered SLOs, and RPO/RTO. They show they've operated systems at scale, not just designed them on a whiteboard.

The Non-Obvious Insight

Tiered availability SLOs by operation is something most candidates miss. Setting 99.99% for everything is expensive and unnecessary. By recognizing that redirects are more critical than URL creation, which is more critical than analytics dashboards, the candidate can optimize cost and complexity. This is genuine architectural wisdom.

🎯 Key Lesson — Phase 2

L5 engineers state requirements. L6 engineers justify requirements with business reasoning and specific numbers. L7 engineers think in operational reality — tiered SLOs, error budgets, tail latencies, and monitoring — because they've learned that requirements without measurement are just wishes.
