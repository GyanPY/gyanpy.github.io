Section 2.3

Detailed Example Walkthroughs

The same scenario across all 4 approaches: 5 servers, 1000 requests/sec total (200/server), one server crashes at t=10s. Watch how each strategy handles it differently.

Scenario Setup
  • 5 servers, 1000 requests/sec total (200/server average)
  • Server 3 crashes at t=10s (was handling ~200 req/sec)
  • Load balancer detects failure at t=11s and redistributes
  • Server 3 restarts at t=15s
Walkthrough 1
DB Auto-Increment
Single Bottleneck, Simple Recovery
▾
Timeline
t=0s:  All 5 servers connect to single database
       Database handles 1,000 inserts/sec
       IDs generated: 1, 2, 3, ... 1,000

t=5s:  State is healthy
       IDs: 1-5,000 assigned
       All servers functioning

t=10s: Server 3 crashes (was handling ~200 req/sec)
       Database: STILL HEALTHY, continues incrementing
       Other 4 servers: Still connected, working
       Impact: 200 req/sec fail until LB redirects

t=11s: Load balancer detects Server 3 down
       Traffic redistributed to 4 servers (250 req/sec each)
       Database: Still single point, now 800 req/sec
       No ID gaps or issues

t=15s: New steady state
       4 servers handling 1,000 req/sec via database
       IDs: Continuous sequence, no gaps
       Full availability restored
Key Observation

The database is the single point — it never noticed Server 3 died. No ID gaps, no wasted namespace, clean recovery. The bottleneck is the DB's insert throughput, which increases slightly from 1K to 800/sec after recovery.

Walkthrough 2
Pre-Allocated Ranges
~1M IDs Orphaned on Crash
▾
Timeline
t=0s:  Each server has range of 1M IDs
       Server 1: [0, 1M),   current=0
       Server 2: [1M, 2M),  current=1M
       Server 3: [2M, 3M),  current=2M
       Server 4: [3M, 4M),  current=3M
       Server 5: [4M, 5M),  current=4M

t=5s:  Each server advanced ~1,000 in counter
       Server 3: current=2,001,000

t=10s: Server 3 crashes at current=2,002,000
       Range [2M, 3M) still "allocated" to Server 3
       IDs 2,002,000 to 2,999,999 are ORPHANED

t=11s: Server 3 restarts
       Requests NEW range from Range Service
       Gets [5M, 6M), starts at 5,000,000
       Old range [2M, 3M) PERMANENTLY LOST (~998,000 IDs wasted)

t=15s: New steady state
       Server 3 working with new range [5M, 6M)
       ~998K IDs permanently unused (acceptable wastage)
Key Observation

~998,000 IDs permanently wasted. In a Kubernetes environment with frequent pod restarts, this wastage accumulates fast. At 100 pods each restarting once a day with 1M ranges, that's ~50M IDs/day wasted. Plan your namespace accordingly.

Walkthrough 3
Snowflake IDs
No Coordination, Instant Resume
▾
Timeline
t=0s:  Each server has unique machine ID (1-5)
       All clocks synchronized

t=5s:  Server 3 generating IDs:
       1718294405000 | 3 | 0
       1718294405000 | 3 | 1
       ...

t=10s: Server 3 crashes
       In-flight requests fail
       NO coordination needed for other servers
       Servers 1,2,4,5 continue generating INDEPENDENTLY

t=11s: Server 3 restarts
       Still has machine_id=3 (persistent assignment)
       Clock is synchronized (NTP running)
       Immediately resumes:
       1718294411000 | 3 | 0  ← new timestamp, fresh sequence
       NO range service call needed!

t=15s: New steady state
       All 5 servers generating independently
       No wasted IDs, no coordination required
       Only "lost" time: the seconds Server 3 was down
Key Observation

Zero wasted IDs, zero coordination needed at restart. The machine ID persists, the clock self-corrects, and generation resumes instantly. This is why Snowflake is the choice for extreme scale. The only "waste" is the gap in timestamps while Server 3 was down — but timestamps are ephemeral, not scarce.

Walkthrough 4
Hash-Based
Idempotent Recovery — Failed URLs Retry Same Code
▾
Timeline
t=0s:  All servers hash-and-check independently
       Each request: hash(URL) → check DB → insert

t=5s:  System has ~5,000 shortcodes
       Collision rate: ~0% (namespace nearly empty)
       Per request: hash → DB check → insert (3 operations)

t=10s: Server 3 crashes
       DB connections close
       In-flight hash-and-inserts: ROLLED BACK
       URLs being shortened: get error response

t=11s: Server 3 restarts
       Reconnects to DB
       No special recovery needed

t=15s: New steady state
       Any URLs that failed during crash:
       - If re-submitted: get SAME shortcode (idempotency!)
         Because hash("https://example.com/path") always = "a1b2c3d"
       - If not re-submitted: user must retry
       
       This is hash-based's SUPERPOWER:
       Retried requests = no duplicates, no confusion
Key Observation — Idempotency Shines Here

A failed URL creation that gets retried produces the exact same shortcode. No duplicate entries, no user confusion. This is the feature no other approach can match. The counter: if the user never retries, that URL just doesn't get shortened — but that's the user's choice, not a system bug.

§2.5 Decision Framework with Reasoning

Decision Tree · Full Annotated
Q1: Do you need idempotency (same URL → same shortcode)?
├── YES → Consider Hash-Based
│   └── Q2: Will you exceed 10M unique URLs?
│       ├── YES → Hash-Based won't scale; use Snowflake + dedup layer
│       └── NO  → Hash-Based is perfect for you
│
└── NO → Continue to Q3

Q3: What's your write throughput requirement?
├── < 10K writes/sec → DB Auto-Increment (simplest)
│   └── Q4: Do you need unpredictable codes?
│       ├── YES → Counter + Random Suffix
│       └── NO  → Pure Auto-Increment
│
├── 10K - 1M writes/sec → Pre-Allocated Ranges or Snowflake
│   └── Q5: Is time-ordering valuable?
│       ├── YES → Snowflake (IDs sort chronologically)
│       └── NO  → Pre-Allocated Ranges (simpler)
│
└── > 1M writes/sec → Snowflake
    └── Q6: Do you have clock synchronization infrastructure?
        ├── YES → Snowflake is your answer
        └── NO  → Build clock sync first, then Snowflake

The Questions to Ask (or Your Interviewer)

6 Questions That Reveal Which Strategy Is Right
  • "What's our expected write throughput at scale?" — determines if distributed generation is needed
  • "Do we need the same URL to always get the same shortcode?" — determines if hash-based is in play
  • "Are predictable/sequential codes a security concern?" — determines if obfuscation is needed
  • "Do we need time-sorted IDs for database performance?" — Snowflake gives this for free
  • "What's our availability target during region failures?" — determines coordination tolerance
  • "How long can shortcodes be?" — 6 chars vs 11 chars changes your options

§2.6 Migration and Hybrid Approaches

Can Variants Be Combined? (Yes — Sophisticated Systems Do This)

3 Effective Hybrid Patterns
  • Snowflake + Hash for lookup: Generate with Snowflake (unique, time-sorted), store hash of URL for deduplication check. Best of both: unique IDs + idempotent lookups.
  • Range-Based + Random Suffix: Get range from service (uniqueness guarantee), add random bits (unpredictability). Simple coordination + security.
  • Pool + Snowflake overflow: Primary = Pre-generated pool (predictable latency). Fallback = Snowflake when pool depletes. Resilience + consistent latency.

Migration Path: Auto-Increment → Snowflake (Zero Downtime)

Phase 1: Preparation (Week 1-2)
  • Deploy Snowflake ID generation service
  • Set Snowflake epoch and machine ID allocation
  • Ensure Snowflake IDs don't collide with existing IDs (e.g., existing IDs <1B, Snowflake starts >10B)
Phase 2: Dual-Write (Week 3-4)
  • New URLs get Snowflake IDs
  • Store generation_method column: 'auto_inc' or 'snowflake'
  • Old URLs still work with old IDs
  • Monitor Snowflake system stability
Phase 3: Cutover (Week 5)
  • All new writes use Snowflake
  • Auto-increment system in maintenance mode
  • Old IDs remain valid indefinitely
Phase 4: Cleanup (Month 2+)
  • Remove dual-write logic
  • Simplify codebase
  • Old ID format becomes "legacy" but still works

What Sophisticated Systems Actually Do

Real-World Examples
  • Twitter (Original Snowflake Paper): Snowflake for tweet IDs. Custom machine ID allocation via Zookeeper. GPS + NTP time synchronization.
  • Instagram: Similar to Snowflake but with PostgreSQL sharding. Uses PostgreSQL schema for machine ID ranges. Each shard generates within its ID space.
  • Distributed URL Shorteners: Often range-based for simplicity. Pool of pre-generated codes per region. Fallback to real-time generation.
🔑 Sound Bite — Comparing Variants

"Auto-increment is your starting point — don't over-engineer. When you hit 10K writes/sec, move to range-based or Snowflake. Range-based is simpler but wastes IDs; Snowflake is elegant but needs clock synchronization. Hash-based is the right answer only when you need idempotency. At Google scale, you're almost certainly running Snowflake-like with GPS/atomic clock time sources."

§3.2 The Distributed Systems Twist

Auto-Increment: Single-node → Distributed

The Concrete Example

"On a single node, auto-increment guarantees no duplicates because there's one counter. Across 10 nodes, you face the split-brain problem: if the network partitions and two nodes both think they're the leader, both might issue the same ID. The common solution is Raft-based consensus, but this trades off availability for consistency — during leader election, no IDs can be generated."

The Concrete Example

"On a single node, Snowflake works perfectly — local clock + sequence is unique. Across 10 nodes, you face clock skew: if Server A's clock is 100ms ahead of Server B, Server A might generate IDs that 'belong to the future' from B's perspective. At Google scale, this is solved with TrueTime (GPS + atomic clocks). At startup scale, you tolerate the small risk or add clock monitoring."

Fail-Open vs. Fail-Closed Analysis

Fail-Open: When Something Goes Wrong, Keep Operating (Weaken Guarantees)

Example: If clock sync fails, generate IDs anyway (risk: possible duplicates)

Fail-Closed: When Something Goes Wrong, Stop (Preserve Guarantees)

Example: If clock sync fails, refuse to generate IDs until clock is fixed

For Shortcode Generation: Fail-Closed Is Usually Correct
  • Duplicates are catastrophic — one shortcode pointing to two different URLs = security incident, data corruption
  • Users can tolerate "try again later" better than broken links
  • Switch to fail-open when: You have duplicate detection downstream, the cost of unavailability exceeds cost of rare duplicates, or you can detect and repair duplicates quickly

§3.4 Edge Cases That Impress Interviewers

Why Edge Cases Matter

These are questions that separate candidates who've operated these systems from those who've only designed them. Mentioning them proactively is a Strong Hire signal.

Edge Case 1: Snowflake Sequence Exhaustion L7 Signal
▾

Scenario: 4,096 IDs per millisecond per machine. What if you need more?

What happens: Snowflake blocks until the next millisecond (wastes CPU time waiting). At 4M requests/second per machine, you'll hit this. That's rare for normal traffic, but entirely possible for batch processing.

Sophisticated answer:

  • Monitor sequence exhaustion events (alert on any occurrence)
  • If frequent: move to microsecond resolution (need 10 fewer sequence bits but 10 more timestamp bits)
  • Or: Use multiple machine IDs per physical machine
  • Twitter's note: They've never hit this on real-user traffic at Twitter scale
Why This Impresses

Most candidates just say "Snowflake gives 4,096 IDs/ms" and move on. Knowing the failure mode when that ceiling is hit — and having a mitigation — signals you've actually thought about operating this at extreme scale.

Edge Case 2: Hash-Based Determinism Attack Security Signal
▾

Scenario: Attacker wants to find all URLs you've shortened.

What happens: They generate millions of URLs, hash them, try each as a shortcode. Since your hash is deterministic, they can check "does https://competitor.com/secret-product exist?" by computing its hash and checking.

Sophisticated answers:

  • Add user-specific salt: hash(URL + user_id) — different users get different codes for same URL
  • Add time-based salt: hash(URL + creation_date) — but now you lose idempotency!
  • Or: Rate limit shortcode resolution endpoint to prevent enumeration
The Trade-off

Adding salt defeats the determinism attack but destroys idempotency — the main reason to use hash-based in the first place. This is a real trade-off that must be explicitly acknowledged.

Edge Case 3: Range Service Thundering Herd Operational Signal
▾

Scenario: All your servers restart simultaneously (deployment, region failure recovery).

What happens: 100 servers all request ranges at the same instant → range service gets 100× normal traffic spike → range service itself becomes a bottleneck or fails.

Sophisticated answers:

  • Add startup jitter: sleep(random(0, 5s)) before requesting a range — spreads the thundering herd across 5 seconds
  • Use larger ranges: Servers don't exhaust simultaneously if ranges are large (pre-fetching helps too)
  • Pre-provision ranges during deployment: Assign ranges before traffic cuts over, so servers start with full ranges already allocated
Why This Impresses

Thundering herd on range service startup is a real operational problem in deployment. Mentioning jitter unprompted signals you've seen this in production or thought deeply about operational patterns.

🎯 Complete Deep Dive Series — Final Lesson

The fundamental trade-off in shortcode generation: coordination overhead versus uniqueness guarantee strength. Pure coordination (single counter) = perfect uniqueness but terrible availability. Pure independence (hash without checking) = great availability but statistical uniqueness. Every real system picks a point on this spectrum.

The art is knowing which point is right for your constraints — and being able to explain why you eliminated each alternative. That explanation, with specific numbers and failure modes, is what separates L6 from L7.
