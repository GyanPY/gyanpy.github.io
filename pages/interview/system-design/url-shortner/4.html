The Key Insight (L7 Differentiator)

The redirect path must be as close to a static file serving as possible. Every millisecond of processing is a millisecond of latency. The create path can be more complex because it's 50x less frequent and has 4x more latency budget.

The Key L7 Differentiation: Service Split by Latency SLA

Most candidates create one monolithic "API service" handling everything. By splitting Redirect Service from URL Service, you can optimize each hot path independently, scale independently, and reason about failure modes independently. What you DON'T do is as important as what you do — the Redirect Service's "NO:" list is as significant as its "ONLY does:" list.

⚡ Redirect Path — Obsessive Optimization

Latency Budget: 50ms p99 · Target: 20ms p50

This is 99% of our traffic. Every microsecond matters. Each step below has a strict budget. We treat this like a static file server.

Go Implementation

Go · Redirect Handler (Production)
func Redirect(w http.ResponseWriter, r *http.Request) {
    code := chi.URLParam(r, "code")

    // Redis lookup — 99% of requests end here
    url, err := redis.Get(ctx, "url:"+code).Result()
    if err == redis.Nil {
        // Cache miss — hit database (covering index, no heap access)
        url, expiresAt, err = db.GetURL(ctx, code)
        if err == sql.ErrNoRows {
            w.WriteHeader(404)
            return
        }
        if expiresAt.Before(time.Now()) {
            w.WriteHeader(410)
            return
        }
        // Populate cache for next request
        redis.SetEX(ctx, "url:"+code, url, 24*time.Hour)
    }

    // Fire-and-forget — NEVER block the redirect
    go emitClickEvent(code, r)

    w.Header().Set("Location", url)
    w.Header().Set("Cache-Control", "public, max-age=86400")
    w.WriteHeader(301)
}
Why This Code Matters in an Interview

Showing actual code (even pseudocode) signals you've built this before — not just designed it. Key highlights: goroutine for async click events (never blocks redirect), cache population on miss (so the next request is fast), and explicit 410 vs 404 handling (expired vs never existed).

✏️ Write Path — Correctness First

Latency Budget: 200ms p99 · Target: 50ms p50

Write path has 4× more budget than redirect. We invest that budget in correctness guarantees: idempotency, durability, deduplication, and validation. Complexity here is acceptable because creates are 50× less frequent than redirects.

Write Path · Step 1
Rate Limiting

Why Lua script? The check-and-decrement operation must be atomic to prevent race conditions under high concurrency. A Lua script in Redis runs atomically without needing a WATCH/MULTI/EXEC transaction.

Latency: 1ms

Returns 200 OK (not 201) on replay to signal to clients that this is a cached response, not a new resource. The full response body is replayed — not just the short code — so the client receives identical payloads on retry.

Validation + Blocklist Check
Diagram · Validation Step
┌─────────────────────────────────────────────────────────────────────────────┐
│ Step 3: Validation                                                          │
│                                                                              │
│   • URL format validation (regex + URL parsing)                             │
│   • URL length check (≤ 2048 chars)                                         │
│   • URL reachability check (optional, async)                                │
│   • Blocklist check (known phishing/malware domains)                        │

│                                                                              │
│   Blocklist: Bloom filter in memory (fast) + Redis set (complete)           │

Bloom filter for blocklist: Keeps known-bad domains in memory for O(1) lookup with no false negatives on the hot path. If the Bloom filter says "maybe blocked," we check the full Redis set. This two-tier approach minimizes latency for the 99.9% of legitimate URLs.

Latency: 1-5ms

Why SHA256 the URL? Can't index VARCHAR(2048) efficiently. Hashing normalizes URL length to 64 chars, which indexes beautifully. The (user_id, url_hash) composite index makes this lookup O(1).

Latency: 5ms
Write Path · Step 5
Code Generation
Diagram · Code Generation Step
┌─────────────────────────────────────────────────────────────────────────────┐
│ Step 5: Code Generation                                                     │
│                                                                              │
│   Each API server has a pre-allocated range from Range Allocator.           │
│                                                                              │
│   Local operation:                                                          │
│   • counter := atomic.AddInt64(&localCounter, 1)                            │
│   • plainCode := base62Encode(counter)                                      │
│   • shortCode := fpeEncrypt(plainCode)                                      │
│                                                                              │
│   Range pre-fetching:                                                       │
│   • When localCounter > 80% of range, async fetch next range                │
│   • This ensures zero blocking on hot path                                  │
│                                                                              │
│   Custom alias handling:                                                    │
│   • If custom alias requested, skip code generation                         │
│   • Check alias availability in DB (unique constraint)                      │
│   • If taken: return 409 Conflict                                           │
│                                                                              │
│   Latency: 0.1ms (local), 50ms (range fetch, rare)                          │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘

Why FPE (Format-Preserving Encryption)? We want unpredictable codes (security) but zero collisions (correctness). FPE applies a secret permutation to sequential counter values, giving us shuffled output with the same cardinality. Result: no collisions possible, no coordination needed, unpredictable codes.

Latency: 0.1ms (local) · 50ms on rare range fetch
Write Path · Step 6
Database Write (Synchronous Replication)
Diagram · Database Write Step
┌─────────────────────────────────────────────────────────────────────────────┐
│ Step 6: Database Write                                                      │
│                                                                              │
│   INSERT INTO urls (id, short_code, long_url, url_hash, user_id, ...)      │
│   VALUES (...)                                                              │
│                                                                              │
│   Synchronous replication:                                                  │
│   • PostgreSQL configured with synchronous_commit = on                      │

│   • This guarantees durability before we return success                     │
│                                                                              │
│   Why synchronous? "RPO = 0" requirement. If we acknowledge a URL           │
│   and then lose the primary, we must not lose that URL.                     │
│                                                                              │
│   Trade-off: ~5-10ms additional latency vs async replication                │
│   Acceptable: We have 200ms budget, this fits easily                        │
│                                                                              │
│   Latency: 15-25ms (network + disk fsync + replication ack)                 │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
Why Synchronous Replication is Worth the Latency Cost

RPO = 0 is a hard requirement. If we acknowledge a URL creation and then the primary crashes before async replication completes, we've told the user their short URL works — but it doesn't. That's a trust violation. The 5-10ms latency cost of synchronous replication is completely acceptable against our 200ms budget.

Latency: 15-25ms (network + disk fsync + replication ack)
Write Path · Step 7
Cache Population (Write-Through Strategy)
Diagram · Cache Population Step
┌─────────────────────────────────────────────────────────────────────────────┐
│ Step 7: Cache Population (Write-Through)                                    │
│                                                                              │
│   redis.SetEX("url:"+shortCode, longURL, 24*time.Hour)                      │
│                                                                              │
│   Why write-through?                                                        │
│   • Guarantees read-your-writes consistency                                 │
│   • User shares URL immediately after creation — it must work              │
│   • CDN will populate on first access anyway                                │
│                                                                              │
│   Why not write-behind?                                                     │
│   • If cache write fails, we have a window where redirects fail             │
│   • Our durability contract is "it works immediately"                       │
│                                                                              │
│   Failure handling:                                                         │
│   • If Redis write fails: log error, continue (DB is source of truth)       │
│   • Next redirect will miss cache, hit DB, and populate cache               │
│                                                                              │
│   Latency: 1ms                                                              │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
Write-Through vs Write-Behind — The Decision Matrix
  • Write-through (our choice): DB write first → cache write second. User gets URL that works immediately. Cache failure is non-fatal (DB is source of truth). Slightly slower writes.
  • Write-behind: Cache write first → async DB write. Lower write latency. But: if the API server crashes before the async DB write, we've told the user their URL works — and it doesn't. Violates our durability contract.
Latency: 1ms
