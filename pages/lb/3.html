The most common interview mistake: conflating API Gateways with L7 load balancers. They overlap, but they're not the same — an API Gateway manages API lifecycle while an L7 LB manages traffic. Knowing when to use an API Gateway, when to use just an L7 LB, and when to build gateway features into the application layer is what separates senior from principal-level thinking.

What IS an API Gateway?

Simple Definition

A smart front door for your APIs that handles common concerns so your services don't have to.

An L7 load balancer routes HTTP traffic intelligently. An API Gateway does that plus handles authentication, rate limiting, request validation, response transformation, caching, circuit breaking, developer portals, API versioning, analytics, and monetization — all as first-class features, not bolt-ons.

Without API Gateway — No Concierge

Every guest calls directly

Guest: "I need a restaurant reservation."
→ Calls restaurant directly.
Guest: "I need a taxi."
→ Calls taxi company directly.
Guest: "I need theater tickets."
→ Calls theater directly.

Each service handles its own coordination. No unified experience.

With API Gateway — Hotel Concierge

One point of contact, everything handled

Guest: "I need a restaurant, taxi, and theater tickets."
Concierge: "I'll handle everything. Here's your confirmation."
Guest: "That was easy!"

Auth, rate limiting, routing, logging — all handled at one place. Services just do their business logic.

The Practical Reality

The overlap zone is large enough that some teams use a well-configured NGINX or Envoy as a "lightweight API gateway" without the full product. This works for internal microservices where you don't need a developer portal, API key self-service, or usage billing. The choice becomes clearer at the extremes: external developer API (need API Gateway) vs. high-throughput internal service (need L4 + custom code).

When You Need an API Gateway

Scenario A Public API for External Developers e.g. Stripe, Twilio, any API-first business ▾
Without API Gateway

Developer: "How do I get an API key?"
You: "Uh… ask our backend team to manually create one?"

Developer: "How do I see my usage?"
You: "Check the logs… somewhere…"

Developer: Uses competitor's API instead.

With API Gateway (Stripe-style)
  • Developer signs up → gets sk_live_abc123 instantly
  • Usage dashboard: 5,432 / 10,000 requests used
  • Rate limit enforced per pricing plan automatically
  • Automatic billing triggers at usage thresholds
  • OpenAPI docs generated and hosted by gateway
Stripe-Style Gateway Request Flow
Developer sends:
  POST /v1/charges
  Authorization: Bearer sk_live_abc123

API Gateway pipeline:
  1. ✅ API key valid         → Identify developer: acme-corp
  2. ✅ Plan: "Growth"        → Limit: 10,000 req/month
  3. ✅ Current usage: 5,432  → Under limit, allow
  4. ✅ Forward to Billing Service
  5. 📊 Log: {dev: acme, endpoint: /charges, lat: 45ms, status: 200}
  6. 📈 Increment usage counter for acme's billing period

What gateway provides automatically:
  - Self-service key generation (no engineering ticket)
  - Per-developer usage dashboards
  - Automatic billing based on usage tier
  - 429 Too Many Requests when quota exceeded
  - API documentation (Swagger/OpenAPI auto-generated)
L6/L7 Insight — External Developers Require API Lifecycle Management

"An L7 LB can handle routing and even basic auth — but it cannot provide self-service API key generation, per-developer usage tracking, and usage-based billing. These require stateful, business-logic-aware features that are fundamentally API lifecycle management, not traffic management. The moment you have external developers you cannot control, you need an API Gateway."

The DRY Principle at Infrastructure Level

The gateway pattern applies Don't Repeat Yourself to infrastructure concerns. When you have 50 services all implementing JWT validation, each becomes a potential security vulnerability if one team uses an outdated library. Centralizing auth at the gateway means one implementation, one security review, one update path.

Scenario C Legacy System Migration — The Strangler Pattern Bank migrating from monolith to microservices over 3 years ▾
The Problem — Can't Rewrite Everything at Once

A 20-year-old monolith with millions of lines of code cannot be rewritten in a big-bang migration. Any attempt to do so fails — scope creep, missed functionality, parallel maintenance burden. The strangler pattern migrates one feature at a time, with the API Gateway routing to the right backend transparently.

Year 1First service extracted — Cards migrated to new service
/accounts/*→ Monolith (legacy) /transfers/*→ Monolith (legacy) /loans/*→ Monolith (legacy) /cards/*→ NEW Card Service ✓ migrated
Year 2Accounts extracted — two services now running in parallel
/accounts/*→ NEW Account Service ✓ migrated /transfers/*→ Monolith (legacy) /loans/*→ Monolith (legacy) /cards/*→ Card Service
Year 3Migration complete — Monolith retired 🎉
/accounts/*→ Account Service /transfers/*→ Transfer Service /loans/*→ Loan Service /cards/*→ Card Service
L6/L7 Insight — The Gateway as Migration Control Plane

"The strangler pattern works because clients see the same API URL throughout the migration. The gateway is the only component that knows which backend handles which path. Clients — mobile apps, web frontends, partner integrations — never update their code. This makes the migration zero-downtime and fully reversible: if the new Card Service has a bug, change one routing rule to point back to the monolith. No client change needed."

When You DON'T Need an API Gateway

Over-engineering is as dangerous as under-engineering. An API Gateway adds 5–20ms of latency, operational complexity, and cost. In these scenarios, it's overkill.

Scenario X Simple Internal Services — Same Team, No External Access Gateway = Overkill ▾
The Setup
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Frontend │────►│ Backend  │────►│ Database │
└──────────┘     └──────────┘     └──────────┘
3 services. Same team owns all of them. No external developers.

What an API Gateway would add:
  - 5-20ms latency on every request (unnecessary)
  - Another service to operate, monitor, upgrade
  - $200-2000/month in additional infrastructure cost
  - No developer portal needed (no external devs)
  - No per-user rate limiting needed (it's one frontend)
  - No API key management needed (same team = shared auth)

Verdict: Use L4 or L7 LB. An API Gateway solves none of your problems.
Scenario Y High-Performance Internal Data Path — Our KV Store Built Into App Layer ▾
Our Solution — API Layer IS the Gateway
Why NOT a traditional API Gateway between L4 LB and API Layer:

  API Gateway adds:
  - 5–20ms latency per request
  - Extra network hop (client → LB → Gateway → API → Storage)
  - Complexity (another service to deploy, scale, monitor)

  Our requirement:
  - P99 < 20ms for reads
  - 300K requests/second
  - Every millisecond matters

Instead, we built gateway features INTO the API Layer:
┌──────────────────────────────────────────────────────────────┐
│                    API Layer (40 nodes)                      │
│                                                              │
│   Built-in (not a separate gateway process):                 │
│   - Auth: JWT validation in-process (0.1ms, not 5ms)        │
│   - Rate limiting: Redis-backed token bucket per user        │
│   - Routing: consistent hash to storage partition            │
│   - Metrics: Prometheus counters incremented in-process      │
│   - Tracing: Jaeger spans created in the same goroutine      │
│                                                              │
│   No extra network hop. No separate process. No added lat.  │
└──────────────────────────────────────────────────────────────┘
The Pattern: Embed Gateway Features, Don't Deploy a Gateway

For latency-critical internal services, the correct pattern is to implement gateway capabilities — auth, rate limiting, metrics, routing — as middleware in the API server process. Languages like Go, Java, and Python all have middleware frameworks for this. The features are identical; the difference is zero network hops and near-zero overhead versus the 5–20ms added by an external gateway process.

Strong Hire Signal Answer

🔑 When the Interviewer Asks "Do You Need an API Gateway Here?"

"It depends on the use case — let me think through this for our KV store specifically."

For the KV store, I'd say no to a traditional API Gateway. Three reasons:

First, no external developers. We don't need a developer portal, self-service API keys, or usage billing. Those are the core reasons API Gateways exist — without external developers, you lose most of the value.

Second, latency-critical path. An API Gateway adds 5–20ms. Our P99 target is 20ms. That's the entire budget gone before the request reaches a storage node.

Third, homogeneous routing. All requests go through the same API layer. We don't need URL-based routing to 50 different microservices.

Instead, I've embedded gateway features into the API layer — JWT validation, Redis-backed rate limiting, partition routing, Prometheus metrics, Jaeger tracing — all as in-process middleware. Same capabilities, zero network hops, no added latency.

"However, if this were an external-facing API product like Stripe — with external developers, usage-based billing, self-service keys, and API docs — I'd absolutely deploy Kong or AWS API Gateway. The 10–20ms latency overhead is worth the operational leverage. The framework choice follows the problem, not the other way around."

🎯 The Three-Part Mental Model

L4 load balancer: fast, dumb, cheap. Routes packets by IP and port. Use it when you don't need to inspect HTTP and your application layer handles routing intelligence.

L7 load balancer: smart, medium cost. Routes HTTP requests by URL, headers, cookies. Use it when you need intelligent traffic distribution without full API lifecycle management.

API Gateway: comprehensive, expensive, complex. Manages the entire API lifecycle — authentication, rate limiting, developer experience, analytics. Use it when you have external developers, 10+ microservices with shared concerns, or a monolith-to-microservices migration.

The principal engineer signal: knowing that these three are on a spectrum, not in discrete buckets — and that you can build a custom point on that spectrum by embedding gateway features in your application layer when none of the off-the-shelf options fit your latency and throughput constraints.
