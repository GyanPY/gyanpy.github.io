The question separates experienced engineers from beginners. Foundations · OSI Networking Model

What Do "L4" and "L7" Mean?

These numbers refer to the OSI (Open Systems Interconnection) model — think of it as layers of an onion, each handling a different aspect of network communication. L4 is the Transport Layer; L7 is the Application Layer. A load balancer at each layer has a fundamentally different view of the traffic it handles.

L1 Physical L2 Data Link L3 Network (IP) L4 Transport (TCP/UDP) ← LB here L5 Session L6 Presentation L7 Application (HTTP) ← LB here

The Postal Service Analogy

L4 — Transport Layer

The Postal Truck

Only cares about: "Where is this package going?" (IP address + port). Doesn't open the package. Just delivers based on the address label. Very fast — no need to inspect contents.

L7 — Application Layer

The Mailroom Clerk

Opens the package (reads the HTTP request). Reads what's inside: "Oh, this is a request for /api/users/123." Can make smart decisions based on content. Slower — has to understand what's inside.

The Restaurant Analogy — Why Speed Is the Difference

L4 Host (fast, simple): "Table for how many? 4? Follow me." Time: 2 seconds. Done.

L7 Host (slow, smart): "Let me check your reservation… Oh, you're a VIP! Let me read your dietary preferences… You like window seats… Let me check which server handles VIP vegetarians near windows…" Time: 30 seconds.

At 300,000 customers per second, those extra milliseconds destroy your latency budget.

Reason 1: Raw Speed at 300K QPS

Load BalancerAdded LatencyAt 300K QPSEats P99 Budget
L4 (AWS NLB)0.1–0.5msNegligible<3% of 20ms budget
L7 (AWS ALB)1–5msAdds to every requestUp to 25% of 20ms budget
The P99 Budget Problem

Our P99 target is 20ms. Adding 5ms at the load balancer eats 25% of our entire latency budget before the request even reaches the API server. This is budget spent on packet routing — work the L4 load balancer can do 10× cheaper. The 5ms is not free; it's CPU cycles spent on TLS decryption, HTTP header parsing, and routing table lookup that L7 must do on every single request.

L6/L7 Insight — Why L7 Is Slower

"L7 overhead is not arbitrary — it's the cost of actually understanding what's in the packet. To read the URL and headers, the load balancer must: (1) complete the TLS handshake and decrypt the payload, (2) parse HTTP/1.1 or HTTP/2 framing, (3) read headers and match routing rules, (4) re-encrypt the payload for the backend connection. Each step is CPU work that L4 entirely avoids. At 300K QPS, this CPU work requires 50+ L7 LB instances vs 3–6 L4 instances."

Reason 2: TLS Termination Location

L7 Approach — TLS Terminated at Load Balancer

                     Decrypt TLS   Re-encrypt
                     Parse HTTP    to backend
                     Route req.
                     
Cost: CPU intensive (RSA/ECDH handshake per new connection)
      50+ LB instances needed at 300K QPS
      Trust boundary problem: LB sees plaintext, backend gets plaintext
L4 Approach — TLS Pass-Through (Our Choice)
Client ──[Encrypted]────────────────────────────► API Server
              │                                         │
         L4 just routes                           Decryption
         based on IP:Port                         happens here
         Never decrypts                           (in our control)
         Never re-encrypts

Cost: Minimal CPU (just packet routing)

      Trust boundary clear: plaintext only inside API server process
L7 TLS Termination
  • LB handles TLS handshake (CPU: RSA/ECDH operations)
  • Plaintext flows from LB to backend (internal network only)
  • Backend must trust LB completely
  • 50+ LB instances needed to handle decryption at 300K QPS
  • TLS certificates managed at LB (centralized but coupled)
L4 TLS Pass-Through
  • LB forwards encrypted packets without touching TLS
  • Decryption at API server (our code, our control)
  • Trust boundary is clear: plaintext only inside API process
  • 3–6 LB instances sufficient at 300K QPS
  • API server manages its own TLS certs (decoupled)

Reason 3: We Don't Need L7 Smarts at This Layer

Architecture — Intelligence Already in the API Layer
┌──────────┐    ┌──────────┐    ┌──────────────────────────────────┐
│  Client  │───►│   L4 LB  │───►│           API Layer              │
└──────────┘    │ (dumb,   │    │  ✓ Auth (reads JWT)              │
                │  fast)   │    │  ✓ Rate limiting                 │
                └──────────┘    │  ✓ Partition routing             │
                                │  ✓ Tenant isolation              │
                                │  ✓ Request validation            │
                                └──────────────────────────────────┘

L7 would give us URL-based routing, header inspection, sticky sessions.
But the API layer already does all of this — with full application context.

Having L7 do routing = redundant work + added latency + no benefit.
The Principle: Push Intelligence Up the Stack

L7 routing decisions at the load balancer are made with limited context — just URL and headers. The API layer can make the same decisions with full context: decoded JWT claims, database state, tenant configuration, real-time rate limit counters. Doing routing at L4 + application logic in the API layer gives you better decisions at lower cost. Only move routing down to L7 when you genuinely need it before the request reaches any application code.

Reason 4: Connection Handling

L4 — Pass-Through
Connection Model
Client ←── Single TCP connection ──► API Server

Client opens connection to L4 LB
L4 LB connects to API server
All packets flow same path
Connection "stuck" to one server
gRPC streams work natively

For long-lived connections (gRPC, connection pooling): ideal. One connection setup, thousands of requests flow through. No re-routing overhead per request.

L7 — Two Connections
Connection Model
Client ←─ Connection 1 ─► L7 LB ←─ Connection 2 ─► API Server

L7 maintains TWO connections
Can redirect requests to different servers
More flexible but: 2× connection overhead
Connection pooling complexity at LB layer

For request-based routing: L7's two-connection model enables re-routing per request. Useful for canary deployments where different requests go to different backends.

Why L4 Connection Pass-Through Wins for gRPC/KV Stores

A KV store client using gRPC opens a connection once and reuses it for thousands of requests. L4's pass-through model lets this connection go directly to an API server — no LB involvement after routing the initial connection. L7's two-connection model means the LB is in the data path for every single request, maintaining connection pools to backends, managing backpressure, and adding per-request overhead even for long-lived streams.

When Would We Use L7 Instead?

L7 Makes Sense When You Need These
  • URL-based routing to microservices: /users/* → User Service, /orders/* → Order Service — essential for a single API gateway in front of multiple backends
  • Canary / A/B deployment: Route 10% of requests to new version by reading a header or cookie
  • Web Application Firewall (WAF): Inspect requests for SQL injection, block suspicious IPs, validate structure before any backend sees it
  • Sticky sessions: Route user A to Server 1 based on session cookie (stateful applications)
  • Header injection: Add request-id, trace headers, or auth context before forwarding
L4 Is Correct When
  • Homogeneous backends: All servers handle all requests equally — no URL-based differentiation needed
  • Latency is critical: Sub-10ms P99 target where 1–5ms L7 overhead is unacceptable
  • High TLS overhead: TLS termination at LB requires 50+ instances vs 3–6 for L4 pass-through
  • Long-lived connections: gRPC, WebSockets, or connection pooling that shouldn't be interrupted
  • Application layer handles routing: API nodes do their own intelligent routing — no need to duplicate at LB

The Cost Multiplier Is Compounding

The 4× cost difference isn't just "L7 charges more per hour." It compounds: L7 requires more instances (CPU for TLS), each instance costs more per hour, and data transfer costs are higher because L7 LBs maintain larger connection state. At 300K QPS sustained for a year, choosing L7 when L4 suffices adds ~$18,000/year in avoidable infrastructure cost — for worse latency.

Strong Hire Signal Answer

🔑 When the Interviewer Asks "Why L4 vs L7?"

"I chose L4 at the edge for three reasons:"

  1. Latency. At 300K QPS with a 20ms P99 target, I can't afford 3–5ms of L7 overhead at the load balancer. That's 25% of my entire latency budget spent before the request even reaches an API server. L4 adds sub-millisecond latency and stays out of the critical path.
  2. TLS termination. I want TLS terminated at my API layer, not the load balancer. This keeps the trust boundary clear — plaintext only exists inside my API server process. It also avoids making the LB a CPU bottleneck for RSA/ECDH operations. At 300K QPS, L7 TLS termination would require 50+ LB instances versus 3–6 for L4 pass-through.
  3. We don't need L7 features here. All the smart routing — auth, rate limiting, partition routing, tenant isolation — happens in my API layer with full application context. The load balancer's only job is distributing packets to healthy nodes. L4 does exactly that with minimal overhead.

"If we needed URL-based routing to different services, or a WAF at the edge, or canary deployments without deploying new API code, I'd reconsider L7. But for a KV store with homogeneous API nodes where the application layer already handles routing, L4 is the right choice."

🎯 The Key Insight

The L4 vs L7 choice is not about which is "better" — it's about where intelligence should live in your system and what it costs at each layer.

L7 intelligence is powerful but has a non-zero cost: TLS decryption CPU, HTTP parsing, connection doubling, and higher latency. If your application layer already has that intelligence with better context (full JWT decoding, database state, tenant config), then paying L7's overhead gets you nothing new.

The principal engineer insight: push intelligence up the stack to where it has the most context, and make the layers below it as fast and simple as possible. L4 at the edge is fast and simple. L7 in the API application code is smart and context-rich. This combination beats L7 at the edge + dumb API nodes.
