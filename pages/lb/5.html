The Complete Decision Framework

Question 1: Who Are Your API Consumers?

External Developers (Stripe / Twilio model)

→ L7 LB + Full API Gateway + Developer Portal

API keys, per-plan rate limits, usage billing, documentation. This is non-negotiable — you cannot build a developer platform without it.

Performance-Critical Internal System

→ Skip to Question 3 (Latency Budget)

When P99 < 10ms is the constraint, the decision is driven by latency, not features. Gateway overhead is evaluated against the budget.

Internal Services Only

→ Continue to Question 2 (Service Count)

No external developers means no developer portal required. Service count determines whether cross-cutting concern centralization is worth the complexity.

Question 2: How Many Microservices? (internal services branch)

1–3 Services (small team)

→ L7 Load Balancer only. Simple, low overhead. Put auth in each service — for 3 services this is manageable and avoids the gateway as an extra point of failure. The code duplication cost is low; the gateway complexity cost is high.

4–10 Services (medium team)

→ L7 LB + Lightweight API Gateway (or service mesh). Cross-cutting concerns (auth, logging, rate limiting) start being painful to duplicate. A lightweight gateway (Envoy, Traefik) centralizes these without the full product overhead of Kong or Apigee.

10+ Services (large org)

→ L7 LB + Full API Gateway. At this scale, cross-cutting concerns must be centralized. An auth library update touching 50 PRs simultaneously is a security and operational disaster. Service mesh (Istio) may also be needed for east-west traffic between services.

Question 3: What's Your Latency Budget? (performance-critical branch)

Relaxed — P99 > 100ms is acceptable

→ L7 LB + API Gateway. Gateway overhead (5–10ms) is under 10% of budget. Acceptable trade-off for the operational leverage of an off-the-shelf gateway.

Tight — P99 target: 10–50ms

→ L4 LB + Lightweight Gateway (Envoy, custom). L4 eliminates HTTP parsing overhead at the LB. Choose a fast gateway implementation that minimizes middleware chain. Disable gateway features you don't need.

Ultra-tight — P99 < 10ms

→ L4 LB + Custom API Layer. Build gateway features (auth, rate limiting, routing, metrics) as in-process middleware. Zero extra network hops. ~0.6ms overhead versus ~5–10ms for an external gateway. This is the KV store pattern.

L6/L7 Insight — Three Questions, Every System

"These three questions cascade: Who consumes it? → How many services? → How tight is latency? The answers produce the architecture, not the other way around. An interviewer can't stump you with a novel system if you can extract the answers to these three questions from the requirements. The architecture follows deterministically."

Common Interview Questions — Strong Hire Answers

Q1 "Should I always include an API Gateway?" ▾
The Strong Hire Opening

"No, it depends on system requirements. Let me walk through my decision framework."

Complete Answer
YES to API Gateway when:
  ✓ External developers consuming your API
    → They need API keys, rate limits, docs — non-negotiable
  ✓ 5+ microservices with shared cross-cutting concerns
    → Centralize auth/logging/rate-limiting or duplicate it everywhere
  ✓ Complex auth requirements (OAuth, JWT validation across services)
    → One gateway handles it; 10 services each doing it = 10x failure surface
  ✓ API monetization needed (usage tracking, billing)
    → Only gateways have this built in

NO to API Gateway when:
  ✗ Simple app with one backend — gateway adds latency, solves nothing
  ✗ Performance-critical system where 5-10ms overhead is unacceptable
    → Our KV store: P99 = 20ms, gateway would eat 25-50% of that budget
  ✗ Internal-only services with simple auth
    → Small team can maintain auth in each service without coordination cost
  ✗ Small team (1-3 services) — maintenance overhead exceeds duplication cost

For our KV store: Deliberately NO gateway.
  P99 target: 20ms. Gateway adds 5-10ms. Math says no.
  Instead: auth and rate limiting built into the API layer directly.
Q2 "What's the difference between API Gateway and L7 Load Balancer?" ▾
Crisp Distinction
L7 Load Balancer (NGINX, AWS ALB):
  Primary job: Distribute traffic across servers
  Focus:       Infrastructure — where does the packet go?
  Features:    URL-based routing, SSL termination, health checks
  Auth:        None (or basic SSL client certs)
  Rate limit:  Simple (per IP, per connection)
  Analytics:   Basic infrastructure metrics
  Latency:     ~1-5ms added

API Gateway (Kong, Apigee, AWS API Gateway):
  Primary job: Manage API lifecycle
  Focus:       Business logic — what is this API call doing?
  Features:    Everything L7 LB does PLUS:
  Auth:        Rich — JWT, OAuth 2.0, API keys, SAML
  Rate limit:  Complex — per user, per plan, per endpoint
  Analytics:   API-specific: which users, which endpoints, error rates
  Developer portal: Self-service keys, docs, usage dashboards
  Latency:     ~5-20ms added

The Analogy:
  L7 LB is a traffic cop directing cars.
  API Gateway is a hotel concierge who checks IDs,
  enforces guest limits, logs visits, and provides information.

In Practice:
  Many systems use BOTH:
  L7 LB at the edge for traffic distribution and scale
  API Gateway behind it for smart API management
Q3 "You drew both a Load Balancer and API Gateway. Isn't that redundant?" ▾
This Is a Trap Question — The Interviewer Wants Depth

"Great question! They serve different purposes — they're complementary, not redundant. Here's the exact reasoning."

The Scale Argument
Load Balancer's job:
  - Make the API Gateway itself highly available
  - Distribute traffic across multiple Gateway instances
  - Handle connection pooling at scale
  - DDoS protection before requests reach application logic

API Gateway's job:
  - Authenticate each request
  - Enforce per-user rate limits
  - Route to the correct microservice
  - Transform requests/responses
  - Log and meter for billing

The Scale Picture:
  1M requests/sec arrive
        │
        ▼
  [L4/L7 Load Balancer] → Splits across 10 Gateway instances
        │
        ▼
  [GW-1] [GW-2] [GW-3] ... [GW-10]   ← Each handles 100K req/sec
        │
        ▼
  [Microservices]

Without the Load Balancer: A single Gateway instance would be overwhelmed
  at 1M req/sec — it would become the bottleneck.

Without the Gateway: The Load Balancer handles traffic distribution
  but can't authenticate, rate-limit, or route intelligently.

They work together: LB handles scale, Gateway handles smarts.
Neither can do the other's job.
Q4 "In my design, where exactly should auth live?" ▾

Auth placement is one of the most nuanced decisions in distributed systems design. There are three valid options — each correct in different contexts.

Option 1 Auth in API Gateway — Recommended for Microservices
Client → L7 LB → Gateway (validates JWT) → Service (trusts Gateway) │ JWT valid? → forward with user context header JWT invalid? → 401 Unauthorized (never reaches service)
    Pros
  • Single place to update auth logic
  • Services don't need auth code at all
  • Consistent security policy across all services
  • Auth library update = 1 PR, not 50
    Cons
  • Gateway is a critical path — if it fails, nothing authenticates
  • Services must trust the gateway completely (internal network assumption)
Option 2 Auth in Each Service — Small Teams
Client → L7 LB → Service A (validates JWT) → Service B (validates JWT) → Service C (validates JWT)
    Pros
  • No single point of failure for auth
  • Services are fully self-contained
  • Simpler architecture (fewer components)
    Cons
  • Duplicate auth code in each service
  • Hard to maintain consistency across services
  • Security update requires deploying every service
Option 3 Auth in Custom API Layer — Performance-Critical
Client → L4 LB → API Layer (validates JWT, custom-optimized) → Storage │ ~0.1ms JWT validation (in-process, no network hop) vs ~1-2ms for gateway auth (network call + parsing)
    Pros
  • Lowest possible latency (~0.1ms in-process)
  • Full control over implementation
  • Optimized for specific use case
    Cons
  • More code to write and maintain
  • Requires expertise in auth protocols
  • Update cycle is tied to app deployments
The Recommendation Matrix
  • Most microservices: Option 1 (Gateway handles auth). Services stay clean, auth is centralized.
  • Small team, 1-3 services: Option 2 (in each service). Simplicity beats premature abstraction.
  • Performance-critical (P99 < 10ms): Option 3 (custom API layer). KV store, trading systems, real-time infrastructure.

The Ultimate Interview Answer Template

🏆 The Universal Structure — Works for Any System

"The choice between Load Balancer, API Gateway, or both depends on the specific requirements. For [system name], I'm using [your architecture choice] because:"

Traffic pattern: [e.g. "300K QPS with long-lived gRPC connections — L4 handles this natively without connection overhead"]

Auth requirements: [e.g. "Internal service only, JWT validation built into the API layer — no gateway needed" OR "External developers need API keys and per-plan rate limits — gateway is required"]

Latency constraints: [e.g. "P99 target is 20ms — a gateway adds 5-10ms (25-50% of budget) which is unacceptable"]

Team/scale context: [e.g. "10 microservices — cross-cutting concerns must be centralized or we duplicate auth in each service"]

Key trade-off: [e.g. "Latency vs features — I'm trading off operational convenience for the 10ms we can't afford to lose" OR "Simplicity vs control — a separate gateway is simpler to operate but less optimized for our workload"]

Flexibility statement: [e.g. "If we added external developers, I'd add an API Gateway immediately — that's the trigger. Right now that's not the case."]

What the Template Does for You

This template forces you to justify every choice with a concrete system property (not "I like L4" but "L4 because gRPC connections and 20ms P99"). It demonstrates that you've thought through the trade-offs, not just picked a default. And the flexibility statement shows the interviewer you know the decision would change under different constraints — which is the L7 signal.

🎯 The Definitive Insight — Across Five Parts

Load Balancers and API Gateways are not competing choices. They're tools on a spectrum, each adding capability at a cost. L4 is the cheapest (0.1ms, minimal CPU), L7 adds HTTP intelligence (1–5ms), API Gateway adds API lifecycle management (5–20ms), Custom API Layer gives you the gateway pattern at near-zero overhead cost.

The question is always: which problems do I have at which layer, and what's the cheapest component that solves each one? External developers? You need gateway features that only an API Gateway provides. P99 of 10ms? You can't afford a gateway's overhead. 50 microservices? You can't afford to duplicate auth across all of them.

The principal engineer signal is deriving the answer from the requirements rather than memorizing a lookup table. When you can sit in front of any system design problem and extract three properties — who consumes it, how many services, what's the latency budget — and derive the correct architecture from those properties alone, you've internalized the framework. That's what these five parts were building toward.
