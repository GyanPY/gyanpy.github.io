Parts 1–6: LB Fundamentals Part 7: L7 vs L4 Decision Guide

The most common architectural question in load balancer design: when do you need L7 (ALB) and when is L4 (NLB) the right choice? This is not about which is "better" — they optimize for different things. L4 is faster and simpler because it routes without reading the request content. L7 is smarter because it understands HTTP — headers, paths, cookies, query parameters. Ten concrete scenarios show exactly when L7 intelligence is worth the trade-off.

The Fundamental Difference — The Postal Analogy

L4 (NLB) — The Address Reader

Like a mail sorter who only reads the address on the envelope (IP + port). They don't open the envelope. They just route it to the right building. Fast, because they never touch the contents. But they can't route to the right department — they don't know what's inside.

L7 (ALB) — The Content Reader

Like a mail sorter who opens the envelope, reads the letter, understands what it's about, and routes it to the right department, the right person, even the right desk. Smarter because it understands the content — but slower because opening every envelope takes time.

⚡ L4 is faster — because it doesn't open the envelope | 🧠 L7 is smarter — because it understands the content

10 Scenarios Where L7 Is the Right Call

Scenario 1 Multiple Services Behind One Domain Use ALB ▾

The most common reason to use L7. You have one domain (api.mycompany.com) but multiple backend services.

ALB Path-Based Routing
One domain — four services:

  api.mycompany.com/auth/*      → Target Group: Auth Service pods
  api.mycompany.com/users/*     → Target Group: User Service pods
  api.mycompany.com/payments/*  → Target Group: Payment Service pods
  api.mycompany.com/otp/*       → Target Group: OTP Service pods

Why L4 can't do this:
  NLB only sees IP addresses and port numbers.
  It has NO idea what the URL path is.
  It can't tell /auth/login from /payments/charge.
  Both would go to the same place.

Rule of thumb: Multiple services behind one domain → almost certainly need L7.
Scenario 2 Host-Based Routing — Multi-Tenant SaaS Use ALB ▾

Each customer has their own subdomain: acme.myapp.com, bigcorp.myapp.com, startup.myapp.com. They need to route to different backends.

ALB Host Header Routing
ALB inspects the HTTP Host header:

  Host: acme.myapp.com        → Target Group: Acme dedicated cluster
  Host: bigcorp.myapp.com     → Target Group: BigCorp dedicated cluster
  Host: *.myapp.com           → Target Group: Shared cluster (default)

NLB can't do this — it doesn't read HTTP headers.
It only knows the destination IP, not the hostname the client requested.
Scenario 3 WAF / DDoS Protection — Public-Facing Services Use ALB ▾

Your service is public-facing and needs protection against SQL injection, XSS, known bad IPs, bot traffic, and application-layer DDoS (slowloris attacks).

AWS WAF + ALB Integration
Internet → CloudFront (optional) → WAF → ALB → Your Service

WAF rules (applied before request reaches your code):
  ✗ Block requests with SQL injection patterns in query params
  ✗ Block requests from known bad IPs (threat intelligence feeds)
  ✓ Rate limit: max 1,000 requests/sec per IP address
  ✗ Block requests with suspicious User-Agent strings
  ✗ Geo-block: only allow traffic from specific countries
NLB + WAF — The Complexity Problem

NLB doesn't support AWS WAF directly. You'd need to put CloudFront with WAF in front of NLB — adding latency, complexity, and another failure point. If WAF is a core requirement, ALB is the simpler architecture.

Scenario 4 WebSocket or gRPC Mixed Routing Use ALB ▾

Your app uses WebSockets for real-time features AND HTTP for regular API calls, and they need to go to different backends.

ALB Protocol-Aware Routing
WebSocket routing (ALB understands HTTP Upgrade headers):
  Path: /ws/*  + Header: Upgrade=websocket  → WebSocket Target Group
  Path: /api/* (no Upgrade header)           → REST API Target Group

gRPC routing (ALB understands Content-Type):
  Content-Type: application/grpc            → gRPC Target Group
  Everything else                           → HTTP Target Group

Note: NLB CAN handle WebSockets and gRPC (they're just TCP connections),
but it CANNOT route them to different backends based on protocol.
Everything goes to the same destination regardless of upgrade headers.
Scenario 5 Sticky Sessions — Stateful Applications Use ALB ▾

A stateful application stores session data in memory on a specific server (legacy apps, game servers). Alice must always go to Pod 3; Bob must always go to Pod 7.

Cookie-Based vs IP-Based Sticky Sessions
ALB Cookie-Based Sticky Sessions:
  First request from Alice:
    ALB picks Pod 3 → sets cookie: AWSALB=pod3-hash
  
  All subsequent requests from Alice:
    ALB reads AWSALB cookie → routes to Pod 3 always ✅

NLB IP-Based Sticky Sessions:
  Routes based on source IP address

  Problem 1: Corporate NAT
    1,000 employees share one IP → ALL go to same pod
    Pod is overloaded; other pods are idle

  Problem 2: Mobile users
    IP changes when switching from WiFi to 4G
    Session is lost — user is routed to different pod

ALB cookie-based stickiness survives IP changes and works
correctly through corporate NAT.
Scenario 6 Request Transformation & Header Manipulation Use ALB ▾

You need to modify requests before they reach your backend: add trace headers, redirect HTTP→HTTPS, redirect www to apex domain, or return fixed health check responses.

ALB Rules — Request Transformation
ALB Rule 1: HTTP → HTTPS redirect
  If scheme = HTTP → Redirect to HTTPS (301 Permanent)
  Clients are automatically upgraded — no code needed

ALB Rule 2: www → apex redirect
  If Host = www.myapp.com → Redirect to myapp.com (301)

ALB Rule 3: Fixed health check response
  If path = /health → Return fixed 200 "OK"
  No backend needed — ALB responds directly
  Faster than hitting your app for every LB health probe

ALB Rule 4: Add tracing header
  For all other requests → Forward to backend
                         + Add X-Request-ID: {uuid} header

NLB limitation:
  NLB passes TCP packets through UNTOUCHED.
  It cannot read, modify, add, or remove HTTP headers.
  It has no concept of HTTP — it only sees bytes.
Scenario 7 Advanced Health Checks Use ALB ▾

You need to verify that backends are truly healthy — not just that the port is open. The app could be returning 500 errors while the port is accepting connections.

Shallow vs Deep Health Checks
L4 (NLB) Shallow Health Check:
  "Can I connect to port 8080?"
  Port open → ✅ Healthy
  Problem: App could be returning 500 errors!
           Database connection lost → 500 on every request
           NLB still routes traffic to the "healthy" port

L7 (ALB) Deep Health Check:
  "Does GET /health/ready return HTTP 200 with body containing 'OK'?"
  
  Your /health/ready endpoint can check:
    ✓ Database is connected and responding
    ✓ Redis is reachable
    ✓ External dependencies are up
    ✓ No out-of-memory conditions
  
  If any dependency fails → endpoint returns 503
  → ALB marks backend UNHEALTHY → stops routing
  → Only genuinely healthy backends receive traffic ✅

ALB health check options:
  - Hit any HTTP path (/health/ready, /ping, /_status)
  - Check for specific status codes (200, 201, not just "port open")
  - Match response body content ("OK" or {"status":"healthy"})
  - Different thresholds per target group (failures needed to mark unhealthy)
Scenario 8 Detailed Access Logs for Debugging Use ALB ▾

You need rich request-level logs to debug production issues — which endpoints are failing, which clients are causing errors, what response times look like per path.

ALB Log Entry vs NLB Log Entry
ALB Access Log (rich — HTTP-level detail):
  timestamp:     2024-01-15T03:22:15Z
  client_ip:     203.0.113.50
  request:       "POST /v1/otp/verify HTTP/1.1"
  status:        429
  response_time: 0.003s
  target_ip:     10.0.1.45
  user_agent:    "LoginService/2.1"
  ssl_cipher:    ECDHE-RSA-AES128-GCM-SHA256
  request_size:  256 bytes
  response_size: 89 bytes

NLB Log (sparse — connection-level only):
  timestamp:     2024-01-15T03:22:15Z
  client_ip:     203.0.113.50
  destination:   10.0.1.45:8080
  bytes_sent:    256
  bytes_received: 89
  (no HTTP method, no path, no status code, no headers)

"If you're debugging why specific API calls are failing at 3 AM,
 ALB logs are invaluable. NLB logs tell you traffic is flowing,
 but not what's happening inside the traffic."
Scenario 9 Canary / Blue-Green Deployments at LB Level Use ALB ▾

You want to send 10% of traffic to a new version and 90% to the old version — without changing your application code or Kubernetes configuration.

ALB Weighted Target Groups
ALB Weighted Target Groups:
  Target Group v1 (weight: 90) → Old version pods
  Target Group v2 (weight: 10) → New version pods (canary)

Gradual rollout:
  Week 1:  90/10  → Monitor error rate, latency for v2
  Week 2:  70/30  → Still looks good → expand
  Week 3:  50/50  → Validate at scale
  Week 4:   0/100 → Full migration complete

Instant rollback:
  If v2 shows elevated errors → change weights back to 100/0
  No deployment required — just update the LB rule
  Rollback takes seconds

NLB limitation:
  NLB does NOT support weighted routing.
  You'd need to manage this at the Kubernetes level
  via Istio, service mesh, or Argo Rollouts instead —
  which adds significant complexity to the deployment pipeline.
Scenario 10 Authentication Offloading — Cognito / OIDC Use ALB ▾

You want the load balancer to handle authentication before traffic reaches your service — useful for internal dashboards, admin panels, or simple web apps where you don't want to implement auth in your application.

ALB + Cognito / OIDC Auth Offloading
ALB + Cognito Integration:

  1. User hits ALB → ALB checks for valid session token
  2. No token / expired token → ALB redirects to Cognito login page
  3. User authenticates in Cognito (Google SSO, username/password)
  4. Cognito issues JWT token → sends user back to ALB
  5. ALB validates token → forwards request to backend
     WITH authenticated user info in headers:
       X-Amzn-Oidc-Identity: user@example.com
       X-Amzn-Oidc-Data: {jwt with user claims}

  6. Your service reads headers — zero auth logic needed in your code

Use cases:
  → Internal dashboards: /admin requires employee login
  → Partner portals: external partners authenticate via SAML
  → Simple web apps: don't want to build auth from scratch

NLB can't do this — it doesn't understand HTTP,
let alone OAuth 2.0 / OIDC token flows.

Why NLB Was Chosen for the OTP Service

✅

Single service (OTP service only) — no path routing needed to different backends

✅

Backend-to-backend clients (not public-facing) — no browser clients, no human users hitting this directly

✅

No path-based routing needed — all traffic goes to the same OTP service

✅

Low latency is critical (p99 < 50ms) — NLB's <1ms vs ALB's 1–3ms matters when the budget is tight

✅

Need static IPs for client whitelisting — internal services whitelist by IP; ALB IPs change dynamically

✅

Auth and rate limiting handled in middleware — these concerns are addressed in application code, not the LB

❌

No WAF needed — this is an internal service, not exposed to the public internet

❌

No WebSocket or mixed protocols — pure gRPC service, single protocol throughout

❌

No sticky sessions — the service is stateless; any pod can handle any request

L6/L7 Insight — Match the Tool to Current Requirements

"Every checkbox points to NLB for the OTP service. Using ALB would add 1–3ms of unnecessary latency on every request for features we don't use. But if tomorrow the OTP service becomes public-facing, or we add more services behind the same domain, or we need WAF — we switch to ALB. The important thing is matching the tool to the current requirements, not over-engineering for hypothetical futures. Architecture decisions should be reversible when requirements change."

🔑 The Interview One-Liner

"ALB is the right choice when you need L7 intelligence — path routing, WAF, header manipulation, auth offloading. But for a single internal service with backend clients where latency matters, NLB gives us lower latency, static IPs, and we handle HTTP concerns in middleware. We'd move to ALB if requirements change — like going public-facing or adding multiple services behind one domain."

🎯 Key Lesson — Part 7

The L4 vs L7 decision is not about which load balancer is "better" — it's about what intelligence the load balancer needs to have. L4 routes by IP and port. It's fast, universal, and handles any protocol. L7 routes by HTTP content: URL paths, host headers, cookies, query parameters, request body. It's smarter but adds parsing overhead.

The trigger for L7 is always the same: do you need the load balancer to understand what's inside the request? Multiple services behind one domain → L7 (path routing). WAF → L7 (content inspection). Auth offloading → L7 (JWT validation). Canary deployments → L7 (weighted routing). Sticky sessions beyond IP → L7 (cookie reading). If none of these apply — single service, internal clients, latency-critical — L4 is the right choice and L7 adds cost and latency for nothing.
