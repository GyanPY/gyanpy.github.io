The Key Insight Behind the Table

It's not "either/or" — it's about what problems you're solving at each layer. The right combination is determined by: Do you have external developers? How many services? What's your latency budget? What's your throughput? What does the application layer already handle? Every row in this table answers a different combination of those questions.

Decision Checklist — Why L7 LB Only
External API developers who need API keys and documentation?NO
10+ microservices that need centralized cross-cutting concerns?NO
Per-user rate limiting based on pricing plan?NO
API versioning for breaking changes?NO
Complex authentication (OAuth, JWT across services)?NO
URL routing, SSL termination, basic health checks?YES
✓ All NO on gateway questions → L7 LB is sufficient. All YES questions L7 handles natively.
🎤 Interview Answer

"For a simple web application, I'd use just an L7 load balancer — NGINX or AWS ALB."

The L7 LB gives me SSL termination for HTTPS, health checks for my web servers, and distribution across 2–3 server instances. I don't need an API Gateway because there's no external API, no complex auth requirements, and no microservices to coordinate. Adding a gateway would add 5–20ms of latency and operational complexity without solving any problem I actually have.

Why No Gateway for a Monolith

All modules (Product, Cart, Checkout, User) are in ONE deployable unit. There's no external API — customers use the website, not an API. Auth is simple session cookies handled inside the app. An API Gateway would add latency and complexity to solve routing problems that don't exist.

L6/L7 Insight — The Graduation Trigger

"The right time to add an API Gateway is when you decompose the monolith into microservices — not before. The trigger is the cross-cutting concern problem: if you have 5+ services each implementing the same auth logic, the cost of that duplication justifies a gateway. Before that point, a gateway is premature abstraction."

🎤 Interview Answer

"For the e-commerce site today, just an L7 load balancer — the app is still a monolith with modules inside one deployable unit."

But I'd design for the growth path: when the team grows and we decompose into microservices, I'd add an API Gateway at that point. The trigger is the cross-cutting concern problem — five services each implementing JWT validation means five copies of the same security-critical code. A gateway centralizes that to one implementation. Before decomposition, a gateway adds complexity without solving any real problem.

Why both L4 LB + API Gateway?
  L4 LB:   Makes API Gateway instances horizontally scalable
  Gateway: Handles all HTTP-level intelligence
  Separation: Each component does what it's best at
Why This Combination
Long-lived connections? (mobile gRPC streams, WebSockets)YES → L4
Need to scale the Gateway itself horizontally?YES → L4 in front
10+ microservices needing centralized auth?YES → API Gateway
Per-user rate limiting (free vs premium tiers)?YES → API Gateway
URL routing to different backend services?YES → API Gateway
🎤 Interview Answer

"For a social media platform, I'd use L4 Load Balancer at the edge feeding into an API Gateway, which routes to microservices."

L4 at the edge because: mobile apps maintain long-lived gRPC connections — L4's pass-through model handles these natively without the overhead of L7's two-connection model. More importantly, L4 makes the API Gateway itself horizontally scalable — we can run 20 Gateway instances and L4 distributes across them.

API Gateway behind it because: we have JWT authentication that shouldn't be duplicated across every service, per-user rate limiting (free: 100 req/min, premium: 1000), and URL-based routing to 10+ backend services. The gateway gives us unified logging and API versioning as a bonus.

Without API Gateway — The Developer Experience Collapses
  • Developer: "How do I get an API key?" → You: "Email us and we'll manually create one…"
  • Developer: "How do I see my usage?" → You: "Check our logs somehow…"

🎤 Interview Answer

"For a public developer API like Stripe, the API Gateway is the product — not just infrastructure. My architecture has three layers:"

L7 LB with WAF at the edge — SSL termination with certificates for our domains, Web Application Firewall to block SQL injection and DDoS before it reaches application code.

Full API Gateway (Kong or AWS API Gateway) — API key authentication and management, per-plan rate limiting (Free: 100/day, Startup: 10K/day, Enterprise: unlimited), usage tracking that feeds into billing, request/response logging, and API versioning so we can release v2 without breaking v1 integrations.

Developer Portal — self-service API key generation, usage dashboards, API documentation auto-generated from OpenAPI specs, code examples in 6 languages, sandbox environment. Without this, developers can't self-serve and we can't build a developer platform business.

The Latency Math — Why Gateway Is Disqualified
P99 latency budget: 20ms

Typical API Gateway overhead per request:
  Network hop (client → gateway):  +1–2ms
  TLS termination at gateway:       +1ms
  Request parsing + routing:        +1–2ms
  Auth check (external call):       +1–2ms
  Network hop (gateway → API):      +1–2ms
  Total gateway overhead:           +5–10ms

5–10ms = 25–50% of our ENTIRE P99 budget
Consumed before the request even reaches the storage layer.

Our in-process middleware overhead:
  JWT validation (in-process):      ~0.1ms
  Rate limit check (Redis):         ~0.5ms
  Partition routing (hash lookup):  ~0.01ms
  Total embedded overhead:          ~0.6ms  ← 10–15× less
ApproachDrawbacksBenefits
Separate API GatewayAdds 5–10ms latency; extra network hop; generic, not optimizedOff-the-shelf (Kong, Apigee); easy config; feature-rich out of box
Custom API Layer (our choice)More code to write; more to maintain; requires expertiseZero extra latency; optimized for our workload; full control
L6/L7 Insight — The Embedded Gateway Pattern

"The gateway is a pattern, not a product. The pattern is: centralize auth, rate limiting, routing, and observability at the entry point. The product implementation (Kong, Apigee, AWS API Gateway) adds a network hop and 5–10ms overhead. For latency-critical systems, implement the pattern in-process. For everything else, the product implementation is cheaper to operate."

🎤 Interview Answer

"For our high-performance KV store, I deliberately chose NOT to use a separate API Gateway. Here's the reasoning:"

L4 Load Balancer at the edge — adds 0.1–0.5ms, no HTTP parsing, handles long-lived gRPC connections natively, and makes the API layer horizontally scalable.

Custom API Layer with embedded gateway features — all the gateway functionality (JWT auth, Redis-backed rate limiting, partition routing, Prometheus metrics, Jaeger tracing) implemented as in-process middleware. Zero extra network hops. ~0.6ms overhead versus ~5–10ms for a separate gateway.

The math is unambiguous: our P99 target is 20ms. A typical gateway eats 5–10ms — that's 25–50% of the entire budget before we touch storage. The trade-off is more code to maintain, but for a performance-critical internal system with no external developers, it's the right call.

The Decision Process — Apply This to Any System

Universal Decision Process
Given a system design question, ask these questions in order:

1. Do I have external API developers?
   YES → API Gateway is almost certainly required
   NO  → continue

2. Do I have 10+ microservices with shared cross-cutting concerns?
   YES → API Gateway to avoid code duplication across services
   NO  → continue

3. Am I migrating from monolith to microservices?
   YES → API Gateway as the migration control plane (strangler pattern)
   NO  → continue

4. Do I need URL or header-based routing between services?
   YES → L7 Load Balancer is sufficient
   NO  → continue

5. Is P99 latency < 10ms? Or is 1–5ms L7 overhead unacceptable?
   YES → L4 Load Balancer + custom API layer (embed gateway features)
   NO  → L7 Load Balancer with some app-level logic

6. Is this a simple single-backend app?
   YES → L7 Load Balancer only (NGINX or AWS ALB)

Remember: These are additive. If the answer to #1 is YES and #4 is YES,
          you use: L7 LB + API Gateway + Developer Portal.
🎯 The Series Key Insight

Across four parts, one theme: every layer in the stack has a cost and a purpose. L4 is fast and dumb — use it when speed matters more than intelligence. L7 is smart and medium-cost — use it when you need HTTP-level routing without the full API lifecycle. API Gateway is comprehensive and expensive — use it when you have external developers, cross-cutting concerns across many services, or need API lifecycle management.

The principal engineer pattern is knowing that these aren't three competing choices — they're three different tools on a spectrum. You can use all three (social media platform), two (e-commerce microservices), or one (simple blog). You can also build a custom point on the spectrum by embedding gateway features in your application layer when none of the off-the-shelf implementations fit your latency and throughput constraints.

The question in any system design interview isn't "which one do I use?" — it's "what problems am I solving at each layer, and what's the cheapest component that solves each one?" Answer that question for each layer in the stack and the right architecture emerges naturally.
