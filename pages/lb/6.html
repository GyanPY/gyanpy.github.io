Parts 1-5: LB & API Gateway Part 6: Kubernetes Deep Dive

Parts 1–5 covered the theory. Part 6 applies it to the most common real-world context: a large Kubernetes cluster with 100 microservices and 1,000+ pods. How does a single HTTP request navigate five layers of routing to reach the correct pod? What happens at the kernel level with iptables and IPVS? How do services talk to each other without going through the ingress? This is the complete picture.

The Five-Layer Architecture

Layer 1 Cloud Load Balancer (AWS ALB / GCP LB / Azure LB) Always present
Entry point from internet. SSL termination. Routes to Ingress Controller pods. Health-checks the Ingress instances.
▼
Layer 2 Kubernetes Ingress Controller (NGINX / Traefik / Istio Gateway) Always present
URL-based routing — /users/* → User Service, /rides/* → Ride Service. Acts as an L7 LB inside Kubernetes. Defined in Ingress YAML resources.
▼
Layer 3 API Gateway (Kong / Ambassador / Custom) Optional
Authentication, rate limiting, request transformation. Only add if the system needs centralized auth across services — adds latency overhead.
▼
Layer 4 Kubernetes Service (ClusterIP) Always present
Virtual IP representing the service. Distributes traffic to pods using iptables/IPVS. This is where actual pod-level load balancing happens.
▼
Layer 5 Pods (Containers running your application code) Always present
The actual destination. user-service-pod-1, ride-service-pod-7, etc. Receive the request, do business logic, call other services via east-west traffic.

Layer 1: Cloud Load Balancer

Layer 1 Cloud Load Balancer — The Building's Front Door ▾
The Analogy

Think of a massive office building (Kubernetes cluster). The Cloud LB is the main entrance with security guards. Multiple doors (multiple LB instances). Guards don't know which office you need — they just let you into the building and direct you to the lobby.

AWS ALB → Ingress Controller Distribution
User's phone: https://api.uber.com/v1/rides/request

Cloud LB (AWS ALB):
  1. Terminates SSL (decrypts HTTPS → HTTP)
  2. Health-checks Ingress Controller pods every 30 seconds
  3. Picks a healthy Ingress pod (round-robin or least connections)
  4. Forwards the HTTP request to the selected Ingress pod

Available Ingress Controller pods:
  ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐
  │ Ingress-1 │ │ Ingress-2 │ │ Ingress-3 │ │ Ingress-4 │
  │    ✓      │ │    ✓      │ │    ✓      │ │    ✓      │
  └───────────┘ └───────────┘ └───────────┘ └───────────┘

  → Cloud LB picks Ingress-2 (round-robin)
  → HTTP request forwarded to 10.0.1.20:80 (Ingress-2 IP)
Created Automatically by Kubernetes

When you create a Kubernetes resource of type LoadBalancer or an Ingress with the right annotations, the cloud provider automatically provisions the ALB/GCP LB/Azure LB. You don't configure the cloud LB manually — Kubernetes does it via the cloud controller manager.

Layer 2: Kubernetes Ingress Controller

Layer 2 Ingress Controller — The Building Directory ▾
The Analogy

You're inside the building now. The Ingress Controller is the directory board in the lobby: "Looking for User Services? Floor 3. Ride Services? Floor 7. Payments? Floor 12." It reads the URL and tells the request where to go — it's an L7 load balancer inside Kubernetes.

Ingress Rule Matching — 100 Services
Request arrives: GET /v1/rides/request

Ingress rules (defined in Ingress YAML):
  Rule  1: /v1/users/*     → user-service:8080
  Rule  2: /v1/drivers/*   → driver-service:8080
  Rule  3: /v1/rides/*     → ride-service:8080        ← MATCH ✓
  Rule  4: /v1/payments/*  → payment-service:8080
  Rule  5: /v1/locations/* → location-service:8080
  ...     95 more rules

Decision: This request goes to ride-service!

Ingress forwards to:
  ride-service.default.svc.cluster.local:8080

Key Point — Ingress Controller IS an L7 Load Balancer Inside Kubernetes

"NGINX Ingress is essentially NGINX running inside your cluster, configured dynamically as you add or update Ingress resources. Every Ingress resource creates NGINX upstream configuration at runtime. The Ingress controller watches Kubernetes API for Ingress changes and reloads NGINX config automatically. No manual NGINX configuration files."

Layer 3: API Gateway (Optional)

Layer 3 — Optional API Gateway — Auth, Rate Limiting, Transformation ▾
Decision — Do We Need API Gateway?
Q1: Need auth BEFORE reaching services?
    YES → API Gateway validates JWT before forwarding
    NO  → Each service validates its own auth

Q2: Need per-user rate limiting?
    YES → API Gateway tracks user X made N requests in last minute
    NO  → Simple IP-based rate limiting in Ingress is enough

Q3: Need request transformation?
    YES → Convert external API format to internal format
    NO  → Same format throughout

For Uber-like app: YES — need API Gateway
  - Must validate user JWT before any service sees the request
  - Must rate-limit per user (prevent abuse)
  - Must log all API calls for billing/analytics
Kong API Gateway Pipeline — Four Steps
Request: GET /v1/rides/request
Header:  Authorization: Bearer eyJhbGciOiJSUzI1NiIs...

Step 1: Validate JWT
        ✓ Token is valid
        ✓ User ID: user-12345
        ✓ Not expired
        → Extract user identity

Step 2: Check Rate Limit
        ✓ user-12345 has made 45 requests in last minute
        ✓ Limit is 100/minute
        ✓ Under limit → ALLOW
        (If over limit: return 429 Too Many Requests immediately)

Step 3: Add Internal Headers
        X-User-ID: user-12345      ← Services read this, no JWT parsing needed
        X-Request-ID: req-abc-123  ← For distributed tracing

Step 4: Forward to ride-service
        ride-service.default.svc.cluster.local:8080
Pattern A — Ingress IS the Gateway (Simpler)

Cloud LB → NGINX Ingress + auth plugin → Services

  • Basic auth needs (simple JWT)
  • Simple rate limiting (per IP)
  • Less than 50 services
  • Lower operational overhead
Pattern B — Separate Gateway (More Powerful)

Cloud LB → Ingress → Kong → Services

  • Complex auth (OAuth, multiple IdPs)
  • Per-user rate limiting
  • Request body transformation
  • API analytics and usage billing
  • 50+ services (our case)

Layer 4: Kubernetes Service (ClusterIP)

Layer 4 Kubernetes Service — The Floor Receptionist ▾
The Analogy — Where Pod-Level Load Balancing Actually Happens

You're on Floor 7 (ride-service). There are 20 people (pods) who can help you. The Kubernetes Service is the receptionist on that floor: "Oh, you need help with rides? Let me connect you to one of our 20 available agents. Agent 7 seems less busy right now."

ClusterIP → Pod Selection
Service: ride-service
ClusterIP: 10.96.45.123  (virtual IP — doesn't exist on any machine!)
Port: 8080

                    (DNS resolves → 10.96.45.123:8080)

ride-service endpoints (actual pod IPs):
  10.244.1.15:8080  (ride-service-pod-1) — Node 1  ✓ healthy
  10.244.2.22:8080  (ride-service-pod-2) — Node 2  ✓ healthy
  10.244.3.18:8080  (ride-service-pod-3) — Node 3  ✓ healthy
  10.244.1.44:8080  (ride-service-pod-4) — Node 1  ✗ unhealthy (readiness probe failing)
  10.244.2.55:8080  (ride-service-pod-5) — Node 2  ✓ healthy

kube-proxy selects one using round-robin or IPVS algorithm

metadata:
  name: ride-service
spec:
  type: ClusterIP         # Internal only — no external access
  selector:
    app: ride-service     # Find pods with this label
  ports:
    - port: 8080          # Service port (ClusterIP listens here)
      targetPort: 8080    # Pod port (where your app runs)
      protocol: TCP

  replicas: 20            # 20 pods!

      app: ride-service

        app: ride-service # This label matches Service selector above

            - containerPort: 8080

  2. Call location-service to get user's current location
     → GET http://location-service.default.svc.cluster.local:8080/location
  3. Call matching-service to find nearby available drivers

  4. Call pricing-service to calculate fare estimate

Under the Hood: iptables vs IPVS

Kubernetes Service load balancing happens at the Linux kernel level via kube-proxy, which runs on every node. It has two modes — and for 100 services with 1,000 pods, the choice matters significantly.

iptables Mode — Default (Simpler)

How it works: kube-proxy writes Linux iptables rules for every Service/Pod combination. Packets are redirected at the kernel netfilter level.

  • Random load balancing (probabilistic)
  • Linear rule scan: O(n) per packet for n rules
  • 1000 pods × avg 5 rules = 5,000 iptables rules
  • Performance degrades as cluster grows
  • No health-aware routing (relies on endpoint removal)
IPVS Mode — Recommended for Large Clusters

How it works: Uses Linux IPVS (IP Virtual Server), a dedicated kernel load balancer with a hash table. O(1) lookup regardless of cluster size.

  • O(1) lookup — hash table, not linear scan
  • Multiple load balancing algorithms
  • Handles 10,000+ pods without degradation
  • Built-in connection tracking
  • Enable: kube-proxy --proxy-mode=ipvs
iptables Rules — What kube-proxy Actually Creates (Simplified)
# When ride-service is created, kube-proxy adds these iptables rules:

# Match traffic destined for ClusterIP
-A KUBE-SERVICES -d 10.96.45.123/32 -p tcp --dport 8080 -j KUBE-SVC-RIDE

# Probabilistic distribution across 4 pods (simplified for 4-pod example)
-A KUBE-SVC-RIDE -m statistic --mode random --probability 0.25 -j KUBE-SEP-POD1
-A KUBE-SVC-RIDE -m statistic --mode random --probability 0.33 -j KUBE-SEP-POD2
-A KUBE-SVC-RIDE -m statistic --mode random --probability 0.50 -j KUBE-SEP-POD3
-A KUBE-SVC-RIDE                                                -j KUBE-SEP-POD4

# DNAT: rewrite destination to actual pod IP
-A KUBE-SEP-POD1 -j DNAT --to-destination 10.244.1.15:8080
-A KUBE-SEP-POD2 -j DNAT --to-destination 10.244.2.22:8080
-A KUBE-SEP-POD3 -j DNAT --to-destination 10.244.3.18:8080
-A KUBE-SEP-POD4 -j DNAT --to-destination 10.244.1.44:8080

# For 100 services × 10 pods = 1000 pods → ~5000 iptables rules
# Every packet scans ALL rules → O(n) degradation at scale

IPVS Load Balancing Algorithms

rr
Round Robin — requests distribute sequentially. Request 1→Pod1, 2→Pod2, 3→Pod3...
lc
Least Connections — send to pod with fewest active connections. Best for uneven processing times.
dh
Destination Hash — same destination IP → same pod. Useful for response caching.
sh
Source Hash — same source IP → same pod. Soft session affinity without cookies.
sed
Shortest Expected Delay — estimate queue delay per pod and pick smallest.
nq
Never Queue — always pick an idle pod if available; fall back to SED otherwise.
L6/L7 Insight — Use IPVS for Production Clusters

"At 100 services × 10 pods = 1,000 pods, iptables generates ~5,000 rules. Every packet entering the cluster scans all 5,000 rules linearly — O(n) per packet. At 300K packets/sec, this becomes a measurable CPU bottleneck. IPVS uses a hash table: O(1) lookup regardless of how many pods exist. Enable it with kube-proxy --proxy-mode=ipvs --ipvs-scheduler=lc. For any cluster with more than 50 services, IPVS is the right choice."

Security Implication of Bypassing the Gateway

Since east-west traffic bypasses the API Gateway, services trust each other implicitly on the internal network. This is fine when all traffic originates from trusted pods in the same cluster. But if a pod is compromised, it can call other services without auth. This is why Service Mesh (mTLS between all pods) is important for large clusters with sensitive data — it adds auth at the pod level, not just at the gateway.

Why Service Mesh with 100 Services?
  • Automatic mTLS between ALL services — every call encrypted, zero code changes. Solves the east-west auth problem.
  • Automatic retries and timeouts — if service-A → service-B fails, Envoy retries. Your application doesn't handle this.
  • Circuit breakers — if service-B is consistently failing, Envoy stops calling it. Prevents cascade failures across 100 services.
  • Distributed tracing for EVERY call — see exactly which service is slow across a chain of 15 service calls. Essential for debugging at this scale.
  • Traffic management — canary deployments, A/B testing, traffic mirroring at the service mesh level without touching app code.

Strong Hire Interview Answer

🎤 "Design Load Balancing for 100 Microservices in Kubernetes"

"For 100 microservices in Kubernetes, I'd design a multi-layer architecture. Let me walk through each layer and the reasoning behind it."

Layer 1 — Cloud Load Balancer (L4 or L7 depending on WAF needs): Entry point from the internet. SSL termination. Routes to Ingress Controller pods. I'd use L4 for lowest latency unless I need a WAF at the cloud LB level, in which case L7 ALB.

Layer 2 — Ingress Controller (NGINX or Istio Gateway): URL-based routing to all 100 services. Defined in Ingress YAML resources — one rule per service path. Ingress controller watches Kubernetes API and reconfigures dynamically as services are added. This is the L7 load balancer inside the cluster.

Layer 3 — API Gateway (Kong or Ambassador) — for external-facing APIs: JWT validation, per-user rate limiting, request transformation, API analytics. Not needed for purely internal services — centralizes auth so none of the 100 services duplicate it. For 100 services, code duplication cost easily justifies the gateway.

Layer 4 — Kubernetes Services (ClusterIP) with IPVS: Each service gets a ClusterIP virtual IP. kube-proxy creates the routing rules. I'd use IPVS, not iptables — at 100 services × 10 pods = 1,000 pods, iptables generates ~5,000 rules scanned linearly per packet. IPVS uses a hash table: O(1) lookup. Enable with --proxy-mode=ipvs --ipvs-scheduler=lc (least-connections for services with uneven processing time).

Layer 5 — Service Mesh (Istio) — recommended at this scale: With 100 services, observability is critical — distributed tracing shows which service is slow in a chain of 15 calls. Automatic mTLS secures east-west traffic that bypasses the API Gateway. Circuit breakers prevent cascade failures. Envoy sidecars add ~1ms overhead but the operational value at 100 services is worth it.

Critical insight on traffic paths: External traffic follows the full stack (Cloud LB → Ingress → Gateway → Service → Pod). Internal service-to-service traffic (east-west) bypasses Ingress and API Gateway entirely — it goes directly through ClusterIP services. This means the API Gateway never sees internal traffic, which is why Service Mesh mTLS is important for securing that traffic.

🎯 The Kubernetes Load Balancing Key Insights

There are five distinct layers of load balancing in a Kubernetes cluster, and they each solve a different problem: the Cloud LB handles internet ingress and SSL, the Ingress Controller handles URL routing between services, the API Gateway handles auth and rate limiting, the ClusterIP handles pod-level distribution, and the Service Mesh handles east-west security and observability.

Pod-level load balancing (ClusterIP) is entirely different from the Load Balancers in Parts 1-5 — it happens at the Linux kernel level via iptables or IPVS DNAT rules, not via any application-layer proxy. Every packet gets its destination IP rewritten before it even leaves the node — sub-millisecond, no userspace involved.

East-west traffic bypasses Ingress and API Gateway — services call each other via ClusterIP DNS names directly. This is intentional (no latency overhead for internal calls) but has security implications (implicit trust). Service Mesh resolves this by adding mTLS at the sidecar level without touching application code.
