Scenario 1: Microservices with a Single Domain

Scenario 1 URL-Based Routing to 200+ Microservices e.g. Amazon / large e-commerce ▾
The Problem Without L7
  • One domain: api.amazon.com
  • Behind it: 200+ different services
  • L4 can only see: 10.0.0.50:443
  • L4 cannot see: /orders/12345 (inside TLS)
  • Result: L4 randomly forwards to any server — order requests hit product servers, payment requests hit recommendation servers
The Solution With L7
  • L7 decrypts and reads the URL path
  • /orders/* → Order Service (50 servers)
  • /products/* → Product Service (100 servers)
  • /payments/* → Payment Service (20 servers)
  • Each service scales independently
  • Client sees one domain, gets the right service
L4 — Dumb Food Court Host

Random seating

Customer: "I want sushi."
Host: "I don't know what sushi is. Here's a random restaurant."
Customer: "This is a pizza place!"

L7 — Smart Food Court Host

Intelligent routing

Customer: "I want sushi."
Host: "Sushi? Japanese food. Follow me to Sushi Palace."
Customer: "Perfect!"

NGINX · L7 URL-Based Routing Config
http {
    upstream order_service {
        server order-1.internal:8080;
        server order-2.internal:8080;
        server order-3.internal:8080;
    }
    upstream product_service {
        server product-1.internal:8080;
        server product-2.internal:8080;
    }
    upstream payment_service {
        server payment-1.internal:8080;
    }

    server {
        listen 443 ssl;
        server_name api.amazon.com;

        location /orders/ {
            proxy_pass http://order_service;
        }
        location /products/ {
            proxy_pass http://product_service;
        }
        location /payments/ {
            proxy_pass http://payment_service;
        }
    }
}
Why L4 Physically Cannot Do This

L4 only sees: Packet to api.amazon.com:443. The URL /orders/12345 is inside the TLS-encrypted payload — L4 never decrypts it. L4 would randomly distribute requests across all backend servers, sending order queries to product servers and vice versa. The only L4-compatible alternative would be separate domains per service (orders.amazon.com, products.amazon.com) — which destroys the unified API surface.

Scenario 2: Canary Deployments / A-B Testing

Scenario 2 Safe Traffic Splitting for New Versions e.g. Netflix rolling out new algorithm ▾
The Risk Without L7
  • New recommendation algorithm: 20% faster but might have bugs
  • Option A: Deploy to everyone → risky (100% blast radius)
  • Option B: beta.netflix.com → no real traffic
  • No way to test with real users at partial traffic
The Solution With L7
  • Route 10% of users to new version
  • If new version errors spike → auto-rollback
  • Gradually ramp: 10% → 25% → 50% → 100%
  • Blast radius on failure: only 10% affected
  • Compare error rates and latency per version
Canary Rollout — Progressive Traffic Shift
Day 1 — Canary (10% new, 90% stable)
v1 90%
v2 10%
Day 3 — Ramp up (50% / 50%)
v1 50%
v2 50%
Day 7 — Full rollout (0% / 100%)
v2 100% ✓
Envoy · Weighted Cluster Canary Config
routes:
  - match:
      prefix: "/recommendations"
    route:
      weighted_clusters:
        clusters:
          - name: recommendations_v1
            weight: 90         # Stable version
          - name: recommendations_v2_canary
            weight: 10         # New version under test
        total_weight: 100

# Automatic rollback: if v2 error rate > 1%, shift weight back
# Envoy outlier detection handles this automatically
L6/L7 Insight — Why Canary Requires L7

"Canary deployments require reading request-level attributes (user_id hash, random percentage, header flag) to decide which version to send to. L4 routes at the connection level — once a TCP connection is established to a server, all requests on that connection go there. L7 can inspect each individual HTTP request and make a per-request routing decision. That per-request granularity is exactly what canary deployments need."

Why This Matters Operationally
  • A bug in the new version only affects 10% of users at initial rollout
  • Error rate monitoring catches regressions before full exposure
  • Rollback is instant — change the weight from 10/90 to 0/100
  • No redeployment needed to roll back — just a config change at the LB

Scenario 3: Header-Based Routing (Multi-Tenancy)

Scenario 3 Tier-Based Priority Routing via HTTP Headers e.g. Slack — B2B SaaS multi-tenancy ▾
The Problem
  • Free customers: millions, low revenue
  • Enterprise customers: thousands, $$$$ revenue
  • System overloaded: who should get resources?
  • L4 can't distinguish — all packets look the same
The Solution
  • Enterprise request: X-Customer-Tier: enterprise
  • L7 reads the header and routes to premium servers
  • Under load: free tier gracefully degrades
  • Enterprise customers never notice an outage
⭐ Enterprise
X-Tier: enterprise
→ Premium Servers — dedicated, fast, always available. SLA: 99.99%. Reserved capacity even at peak load.
◆ Business
X-Tier: business
→ Standard Servers — shared but well-provisioned. SLA: 99.9%. May queue briefly under extreme load.
○ Free
X-Tier: free
→ Shared Servers — best effort. No SLA. Degraded gracefully during overload. Rate-limited at LB.
AWS ALB · Header-Based Routing Rules
Rule 1: Priority 1
  IF   Header "X-Customer-Tier" = "enterprise"
  THEN Forward to target-group-premium
       (20 instances, c5.4xlarge, dedicated)

Rule 2: Priority 2
  IF   Header "X-Customer-Tier" = "business"
  THEN Forward to target-group-standard
       (10 instances, c5.2xlarge, shared)

Rule 3: Default (catches free tier and untagged requests)
  Forward to target-group-shared
  (5 instances, t3.large, overcommitted during peak)
L6/L7 Insight — Header Routing Enables QoS Without Application Changes

"Header-based routing at L7 implements Quality of Service (QoS) without touching application code. The API server doesn't need to know its tier — the load balancer handles routing before the request reaches any application logic. This separation of concerns means you can add new tiers, adjust server pools, or modify routing rules with zero application deploys. Infrastructure change, not code change."

Scenario 4: Web Application Firewall (WAF)

Scenario 4 Security Inspection Before Requests Reach Application Code Any company handling sensitive data (banks, healthcare) ▾
Attack Vectors L4 Cannot Block
  • SQL injection: /users?id=1; DROP TABLE users;--
  • XSS: /search?q=<script>steal(cookies)</script>
  • Path traversal: /files/../../../etc/passwd
  • L4 sees: Encrypted packet to 10.0.0.50:443
  • L4 says: "Looks fine! *forwards malicious payload*"
WAF at L7 Intercepts Before Application
  • L7 decrypts and inspects every request
  • Pattern matching against OWASP attack library
  • SQL injection detected → BLOCK + LOG
  • XSS payload detected → BLOCK + ALERT
  • Application code never sees malicious request
WAF Request Inspection Pipeline
Client sends:
  GET /users?id=1; DROP TABLE users;--

L7 WAF Pipeline:
  Step 1: TLS Decrypt    → Plaintext HTTP request now visible
  Step 2: Parse URL      → query param: id = "1; DROP TABLE users;--"
  Step 3: WAF Rules      → Match against SQL injection patterns
  Step 4: MATCH FOUND!   → Rule: detect_sqli (OWASP CRS rule 942100)
  Step 5: Action: BLOCK  → Return 403 Forbidden
  Step 6: Log event      → {attack_type: "sqli", source_ip: "...", timestamp}

Application server:  Never receives this request ✓
Security team alert: Triggered immediately ✓
Attacker:            Gets 403, no information about what was detected
L4 — Airport Ticket Check

Checks ticket only

"Your ticket says Flight 123? Go ahead."
Doesn't check if passenger has weapons in bag. Malicious payload sails through to the application.

L7 WAF — Full Security Screening

Inspects contents

"Let me scan your bag… What's this suspicious liquid? BLOCKED. Security to Gate 7." Malicious payload stopped before reaching any backend.

WAF Is Defense-in-Depth, Not the Only Layer

A WAF at L7 is not a substitute for secure application code. It's a defense-in-depth layer that catches known attack patterns before they reach your code. Known zero-days, novel attacks, and application-logic vulnerabilities (like IDOR — accessing another user's data with a valid request) won't be caught by pattern-matching WAFs. WAF is Layer 1 of security; secure coding is Layer 2.

Scenario 5: Sticky Sessions (Stateful Applications)

Scenario 5 Session Affinity for In-Memory Stateful Apps Legacy applications storing session in server memory ▾
The Problem with Stateful Servers + L4
  • User logs in → Server A stores session["user123"] in RAM
  • Next request → L4 randomly routes to Server B
  • Server B: "Who is user123? I don't know this person!"
  • User: "Why am I logged out?!"
  • Shopping cart, multi-step forms, file uploads all break
The Solution with L7 Session Affinity
  • L7 reads cookie: SERVERID=server-a
  • "This user was on server-a before → send back to server-a"
  • Same server, same in-memory session, user stays logged in
  • Cookie inserted on first request, read on every subsequent one
HAProxy · Sticky Sessions via Cookie
backend web_servers
    balance roundrobin
    
    # Insert SERVERID cookie on first response, read it on subsequent requests
    cookie SERVERID insert indirect nocache
    
    server web1 10.0.0.1:80 check cookie web1
    server web2 10.0.0.2:80 check cookie web2
    server web3 10.0.0.3:80 check cookie web3

# First request: user hits web2 → response includes Set-Cookie: SERVERID=web2
# All future requests: HAProxy reads SERVERID=web2 → routes to web2
# If web2 goes down: HAProxy routes to another server (session loss, but graceful)
The Modern Alternative — Avoid Sticky Sessions If Possible

Modern applications avoid sticky sessions entirely by storing session state externally:

  • Redis: session["user123"] = {...} in shared Redis → any server can read it
  • JWT tokens: Session state encoded in the token itself → stateless servers, no affinity needed
  • Database sessions: Session row in PostgreSQL → any server reads the same row

Sticky sessions are a workaround for architectural debt. Use them when you can't modify the application; migrate away from them when you can.

Scenario 6: SSL/TLS Termination with Centralized Certificate Management

Scenario 6 Managing TLS for 100+ Domains at One Place Any company with many subdomains or multi-tenant domains ▾
The Problem at Scale
  • 100+ domains: shop.co, api.co, admin.co, partner.co…
  • Each needs its own TLS certificate
  • Certs expire every 90 days (Let's Encrypt) or 1 year
  • Manual renewal on each of 100 servers → expired cert incidents
  • Certificate key rotation requires coordinated deploy across all servers
The Solution — L7 Centralized TLS
  • All 100 certs stored and managed at L7 LB
  • Auto-renewal via Let's Encrypt (ACME protocol)
  • L7 uses SNI to pick correct cert per domain
  • Backend servers serve HTTP only (no certs, simpler)
  • Cert rotation: one config update, zero backend deploys
Centralized TLS Termination Architecture
┌──────────────────────────────────────────────────────────────┐
│                 L7 Load Balancer (TLS Hub)                   │
│                                                              │
│  Certificates managed here (auto-renewed):                   │
│    shop.company.com     → cert_shop.pem    (expires: 89 days)│
│    api.company.com      → cert_api.pem     (expires: 62 days)│
│    admin.company.com    → cert_admin.pem   (expires: 14 days)│
│    ... 97 more certs, all auto-renewed                       │
│                                                              │
│  Client connects to shop.company.com with TLS:               │
│    1. TLS ClientHello includes SNI: "shop.company.com"       │
│    2. L7 selects cert_shop.pem (matching domain)             │
│    3. TLS handshake completes                                │
│    4. L7 decrypts → inspects HTTP request                    │
│    5. Forwards as plain HTTP to backend                      │
└──────────────────────────────────────────────────────────────┘
         │ HTTP (no TLS)                │ HTTP (no TLS)
         ▼                              ▼
   ┌──────────────┐             ┌──────────────┐
   │ Backend      │             │ Backend      │
   │ Server 1     │             │ Server 2     │
   │ (no certs)   │             │ (no certs)   │
   └──────────────┘             └──────────────┘

Key: SNI (Server Name Indication) = TLS extension where the client tells
     the server which hostname it's connecting to, before TLS completes.
     This lets one IP address serve TLS for hundreds of different domains.
The Security Trade-off — Internal HTTP

When L7 terminates TLS, traffic from LB to backends is plain HTTP. This is acceptable only if the backend network is private and trusted (VPC, private subnets, no external access). If your internal network could be compromised, you should use mTLS between LB and backends — or use L4 pass-through and have backends terminate TLS themselves (as in Part 1's KV store design).

L6/L7 Insight — TLS Termination Is Operationally Compelling at Scale

"The KV store in Part 1 chose L4 TLS pass-through for latency and trust boundary reasons. But at 100 domains with 90-day cert rotation, the operational cost of distributing certs to every backend server is non-trivial. The right choice depends on your operational context: homogeneous API nodes handling their own certs (Part 1: L4 wins) vs. heterogeneous backends that shouldn't deal with cert management (Part 2: L7 wins). Neither is universally correct."

🎯 The Key Lesson — L7 Is Not "Better" Than L4

Every scenario in this guide shows L7 solving a problem L4 physically cannot. But that's because these are scenarios selected to show L7 at its best. In Part 1, we showed L4 at its best for a KV store at 300K QPS with 20ms P99 targets.

The correct mental model: L4 is fast and simple; L7 is smart and costly. Use L7 when you genuinely need its intelligence — URL routing, header inspection, canary deployment, WAF, sticky sessions, or centralized cert management. Use L4 when you don't need any of that, and when latency, throughput, or TLS trust boundaries make L7's overhead unjustifiable.

The L7 interview signal is not memorizing these scenarios — it's being able to look at a system's access pattern and immediately know which scenarios apply. "We have homogeneous API nodes, the application layer handles routing, and our P99 target is 20ms" → L4. "We have 15 microservices behind one domain and need a WAF" → L7. That instinctive mapping from system requirements to infrastructure choice is the principal engineer pattern.
