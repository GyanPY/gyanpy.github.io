Pillar 01

🧠 Core Concepts & Mental Models

First principles, core abstractions, guarantees, and invariants

The World Before Elasticsearch

Before Elasticsearch, full-text search in production systems was a patchwork of painful compromises. Traditional relational databases like MySQL and PostgreSQL supported full-text search through LIKE '%keyword%' queries or rudimentary FULLTEXT indexes. The fundamental problem with these approaches was architectural: relational databases were designed around the principle of precise, structured retrieval — finding rows where a column exactly matches a value. Full-text search is a fundamentally different paradigm where relevance ranking, linguistic analysis, approximate matching, and sub-second latency at scale are all first-class requirements.

Before Elasticsearch's rise (it was first released in 2010 by Shay Banon), teams would deploy Apache Solr (which itself wraps Apache Lucene) directly, operating it as a standalone search server. Solr required significant operational expertise: managing ZooKeeper for distributed coordination, manually configuring schema.xml files, and dealing with a Java ecosystem that was brittle and opaque. Lucene itself — the underlying inverted index library — had to be embedded directly in application code, meaning every application team was responsible for shard management, replication, query routing, and cluster health themselves. There was no REST API, no JSON-native interface, and no auto-scaling story. Horizontal scaling meant writing custom routing logic and shard management code.

Logging and observability were equally painful. Teams would write millions of log lines into flat files or simple relational tables, and any analytical query that touched large volumes of event data would time out or require expensive pre-aggregations. There was no system that could simultaneously handle unstructured text search, structured filtering, and real-time aggregations on billions of documents. Elasticsearch unified these capabilities into a single, operationally coherent system accessible over a simple HTTP/JSON interface.

The Core Abstraction & Why It Is Powerful

The core abstraction that Elasticsearch provides is the distributed inverted index exposed through a document-oriented JSON API. Let's unpack each component of this phrase deeply because each word carries significant engineering weight.

An inverted index is the fundamental data structure that makes full-text search possible at scale. In a traditional forward index, you map from a document identifier to its contents (document 1 contains "the quick brown fox"). In an inverted index, you flip this: you map from every term that appears in any document to the list of document IDs that contain that term. When a user searches for "brown fox," the engine performs a lookup on two terms simultaneously in the inverted index, retrieves the posting lists for each term, intersects them, scores the results by relevance using algorithms like BM25 (Best Match 25), and returns a ranked list of documents. This lookup is O(1) for the dictionary lookup plus O(k) for intersecting posting lists where k is proportional to the number of matching documents — dramatically faster than scanning every document for the presence of a keyword.

The distributed dimension means Elasticsearch automatically partitions this inverted index across multiple machines called shards. A shard is a fully self-contained Lucene index that can independently serve read and write requests. By distributing shards across nodes in a cluster, Elasticsearch achieves horizontal scalability: you can spread a trillion-document index across hundreds of nodes, parallelizing both indexing throughput and query execution. During search, a coordinating node fans out the query to all relevant shards simultaneously, collects partial results, merges them using a distributed merge sort, and returns the global top-k results — all transparently to the client.

The document-oriented JSON API means that Elasticsearch treats documents as the unit of storage and retrieval, and exposes all operations through a RESTful HTTP interface that speaks JSON. This is architecturally significant because it means Elasticsearch is schema-flexible: documents can have arbitrary nested fields, arrays, geo-coordinates, and date ranges — all handled natively. The JSON API means integration is trivial from any language or toolchain, and the dynamic mapping feature means Elasticsearch can auto-detect field types and configure appropriate analyzers without explicit schema definition.

Fundamental Guarantees (and Non-Guarantees)

✅ What ES Guarantees
  • Near-Real-Time (NRT) Search: Documents become searchable within ~1 second of indexing by default (configurable refresh interval)
  • Eventual Consistency: A document written to the primary shard will eventually be replicated to all replica shards, with no hard guarantee on replication lag
  • Durability via WAL: Writes are appended to a translog (write-ahead log) before being acknowledged, ensuring durability across node restarts
  • Shard-Level Atomicity: Individual document writes within a single shard are atomic — a document is either fully indexed or not at all
  • Cross-Cluster High Availability: With replica shards, cluster can tolerate node failures without data loss (provided replication factor > 1)
❌ What ES Does NOT Guarantee
  • Strong Consistency / Read-Your-Writes: A write confirmed to the client may not be immediately visible on all nodes — replica reads can return stale data
  • Cross-Document ACID Transactions: There is no native distributed transaction spanning multiple documents or indices
  • Zero Data Loss on Split-Brain: Under certain network partition scenarios without careful quorum configuration, writes to both sides of a split may be irreconcilable
  • Exactly-Once Delivery: The same document may be indexed multiple times if the client retries; idempotency requires using explicit document IDs
  • Strict Schema Enforcement: Dynamic mapping can silently create fields with incorrect types (e.g., treating a numeric string as a keyword)
🔥 Common Interview Mistake Candidates frequently say "Elasticsearch is strongly consistent." It is NOT. ES is an AP system (Available + Partition-tolerant) in CAP terms. Write operations are acknowledged once the primary shard has written — replicas are updated asynchronously. A stale read from a replica is entirely normal and expected. Confusing this with strong consistency is an immediate signal that the candidate lacks production experience.

Critical Invariants for Correctness

Several invariants must always hold for Elasticsearch to operate correctly. Understanding these is what separates an operator from an architect:

  • Primary Term & Sequence Number Ordering: Every write to a shard carries a monotonically increasing sequence number and a primary term (an epoch that increments when a new primary is elected). These two values together form a logical clock that prevents stale writes from overwriting newer data during node failures and re-elections. A shard replica will reject writes that arrive out of order.
  • Shard Count Immutability: The number of primary shards in an index is fixed at creation time and cannot be changed without reindexing. This is because document routing is determined by shard = hash(document_id) % num_primary_shards. Changing the shard count would break the routing formula and make all previously indexed documents invisible to queries. This invariant has major operational consequences for capacity planning.
  • Minimum Master Nodes (Quorum): Prior to ES 7.x, operators had to configure minimum_master_nodes = (N/2 + 1) to prevent split-brain. In ES 7+, this is automated via the voting configuration mechanism, but the quorum invariant still holds: a master can only be elected if a majority of master-eligible nodes agree. Violating this invariant results in two simultaneous active masters, causing data corruption.
  • Lucene Segment Immutability: Once a Lucene segment is written to disk, it is never modified in place. Updates and deletes are implemented by marking documents as deleted in a .del file and writing new versions as fresh documents. This immutability is what makes Lucene segments safe to cache and replicate — their content never changes once written. Garbage collection happens through segment merging, which reads old segments and writes new consolidated segments.

🏗️ Architecture Deep Dive

Internal components, write/read paths, data layout, and flow control

Internal Components & Their Responsibilities

Cluster State & Master Node

The master node is the brain of the cluster — it is the only node authorized to make global cluster state changes. The cluster state is a comprehensive data structure stored in memory on all nodes (replicated from master) that contains: the index-level metadata (settings, mappings, aliases), the shard allocation table (which shard lives on which node), the list of live nodes, and index templates. The master is elected through a Raft-based consensus process (ES 7+ replaced the older Zen Discovery with Raft for provably correct leader election). Any change to cluster state — creating an index, reallocating a shard, removing a node — must go through the master, which publishes the new state to all nodes synchronously (or fails the operation).

In ES 7+, dedicated master-eligible nodes are strongly recommended at scale. These nodes are configured with node.roles: [ master ] and perform no data serving, which prevents GC pauses from data operations from causing master election timeouts — a significant operational improvement over earlier versions where any node could be master and heavy indexing could cause false master failures.

Data Nodes & Shard Management

Data nodes are responsible for storing shard data and executing search/aggregation queries on those shards. Each shard is a fully functional Lucene index running in-process on the JVM. A data node hosts multiple shards (both primaries and replicas), managing their lifecycle, memory pressure, and I/O scheduling. Within a data node, each shard runs in its own Lucene directory on disk, isolated from other shards.

The Index Shard is the fundamental unit of allocation and failure recovery. When a node fails, the master detects the failure (via missed heartbeats), marks all shards on that node as unavailable, and initiates recovery by promoting replica shards of the failed primaries to primary status on other nodes. The recovery process involves copying shard data (either from a peer recovery or from snapshot storage) and replaying the translog of any writes that occurred after the last Lucene commit — ensuring no data loss.

Coordinating Nodes

Any node in an Elasticsearch cluster can act as a coordinating node for any request — it's a role that every node fills for requests that arrive at it, not a dedicated node type (though you can have dedicated coordinating-only nodes). When a search request arrives at a coordinating node, it is responsible for: parsing and validating the query, determining which shards need to be consulted (using the routing table from cluster state), fanning out sub-queries to those shards in parallel, collecting partial results, performing the final merge/sort/scoring, and returning the final response. This scatter-gather pattern means that complex aggregation queries may require significant heap memory on the coordinating node to hold intermediate results.

The Translog (Write-Ahead Log)

The translog is Elasticsearch's durability mechanism and is one of the most commonly misunderstood components. When a document is indexed, it is first written to the translog (a sequential append-only log file on disk), then added to an in-memory buffer (the Lucene index buffer). The client receives an acknowledgment at this point — the document is durable (it will survive a JVM crash because it's in the translog), but it is not yet searchable (it's still in the Lucene buffer, not in a committed segment). Periodically (every 1 second by default), a refresh operation creates a new in-memory Lucene segment from the buffer and makes it visible to search. This is the source of ES's "near-real-time" (not real-time) search characteristic — there is up to 1 second of visibility lag by default.

The translog continues accumulating until either a flush is triggered (a full Lucene commit + translog rotation) or it grows beyond its configured size. A flush is more expensive than a refresh: it performs an fsync on all Lucene segments and replaces the translog with a new empty one. If a node crashes, on recovery it replays the translog from the last committed Lucene checkpoint — exactly like PostgreSQL's WAL replay. In ES 8.x, translog operations can also be asynchronously acknowledged for higher throughput workloads (index.translog.durability: async) at the cost of up to the async interval of data loss on crash.

⚡ L7/L8 Differentiator The two-phase query-then-fetch model is critical to understand. In the query phase, each shard scores its local top-N documents and returns only document IDs + scores. The coordinating node merges these into a global top-N. Only then does it fetch the full source documents (in the fetch phase) for just those top-N documents. This minimizes network transfer: for a query returning 10 results from 100 shards, you score 100×N documents but only transfer 10 full documents over the network. Candidates who describe ES as doing a single scatter-gather miss this critical two-phase optimization.

On-Disk and In-Memory Data Layout

Lucene Segment Files

A Lucene segment is composed of multiple files that together constitute a complete, searchable, immutable inverted index for the subset of documents assigned to it. Understanding these files is crucial for diagnosing disk usage and I/O patterns:

  • .tim / .tip (Term Dictionary): The sorted list of all terms that appear in the segment, along with a pointer to the posting list for each term. This is the "index into the index" — you do a binary search or trie lookup here to find your term, then follow the pointer to the postings. The .tip file is a term index that maps term prefixes to positions in the .tim file for faster navigation.
  • .doc (Postings Lists): For each term, the ordered list of document IDs that contain that term, plus optional position data (for phrase queries) and payload data. This is compressed using frame-of-reference (FOR) and variable byte encoding, making it extremely compact. A posting list for a common term like "the" might contain millions of document IDs, compressed to a few megabytes.
  • .fdt / .fdx (Stored Fields): The original field values for each document (the _source). The .fdt file contains the raw data, and .fdx is an index into it for O(1) random access. These files are used during the fetch phase to retrieve full document sources. By default, the entire original JSON document is stored here (compressed with LZ4).
  • .dvd / .dvm (Doc Values): Column-oriented data storage for numeric fields, keyword fields, and dates. Doc values store field values in a column-store format rather than row-store, enabling extremely efficient sorting, aggregations, and script operations. Without doc values, sorting requires loading all field values into heap memory (field data cache), which is catastrophically expensive at scale. Doc values are memory-mapped and accessed in O(1) time per document.
  • .si (Segment Info): Metadata about the segment: number of documents, which files belong to it, the codec used, etc.
  • segments_N: The segment commit point — a file that lists all segments currently part of the Lucene index. An atomic rename of this file is how Lucene makes segment additions/merges atomic and crash-safe.

Back-Pressure and Flow Control

Elasticsearch manages back-pressure through several mechanisms that operate at different layers. Understanding these is critical for operating ES at scale without triggering cascading failures:

The indexing circuit breaker monitors the estimated heap memory required for indexing operations. If adding new documents to the Lucene buffer would cause estimated heap usage to exceed a threshold (default: 10% of JVM heap), the circuit breaker rejects new indexing requests with a 429 Too Many Requests response. This prevents out-of-memory errors that would cause a full JVM crash, which is far more expensive to recover from than a brief period of write rejection.

The bulk queue is a bounded queue on each node that buffers incoming bulk indexing requests before they are processed by the indexing thread pool. The thread pool itself is bounded (typically configured to the number of available CPUs). When both the thread pool and the queue are full, new requests are rejected. Operators should monitor thread_pool.bulk.rejected as a key signal of sustained overload.

The merge throttling mechanism limits the I/O bandwidth consumed by segment merge operations. If unconstrained, segment merges (which are CPU and I/O intensive) can starve the indexing path, causing indexing latency to spike. Elasticsearch uses an adaptive merge throttle (indices.store.throttle.max_bytes_per_sec) to limit merge bandwidth, trading off slightly higher segment count (and therefore slightly slower search) for stable indexing throughput.

Elasticsearch's write path is a multi-stage pipeline (translog → Lucene buffer → refresh → segment → merge → flush) and its read path is a two-phase scatter-gather (query phase for scores, fetch phase for source documents) — understanding both at the component level is mandatory for diagnosing production issues and making architectural decisions.

Pillar 03

⚖️ Trade-offs & Design Decisions

CAP theorem, PACELC analysis, consistency models, and operational trade-offs

CAP Theorem Analysis

Elasticsearch is unambiguously an AP system (Available + Partition-tolerant). During a network partition, Elasticsearch will continue to serve both reads and writes on the side of the partition that contains the active primary shards and the elected master, even if that means replica shards on the other side of the partition become stale. The cluster state will reflect unavailable replicas (yellow cluster health) rather than refusing service. This is the correct choice for Elasticsearch's primary use cases (search, logging, analytics) where availability is more valuable than strict consistency.

The specific tradeoff in ES's design: writes are acknowledged from the primary shard and replicated asynchronously to replicas. A client that writes to Node A and then immediately reads from Node B (which serves a replica) may see the old value or no value at all, because replication hasn't completed. ES provides a wait_for_active_shards parameter that can increase the number of copies that must acknowledge before the write is confirmed — at the extreme, setting it to all approximates strong consistency at the cost of availability (any replica failure now causes write failures).

🔥 Common Interview Mistake Saying "Elasticsearch is CP because writes go to the primary first" is wrong. The primary shard acknowledgment provides primary-level consistency for a single shard write, but the cluster as a whole operates with eventual consistency between primary and replicas. The wait_for_active_shards default is 1 (just the primary), not ALL — meaning replicas are updated asynchronously after the client has received a success response.

PACELC Analysis

PACELC (Partition: Available/Consistent; Else: Latency/Consistency) gives us a richer picture of Elasticsearch's tradeoffs even when the network is healthy:

  • Under Partition (P): ES chooses Availability (A). The cluster continues serving requests even if some replicas are unreachable, accepting the risk of stale reads.
  • Else (no partition, E): ES chooses Latency (L). In normal operation, reads are served immediately from any replica without any coordination overhead. There is no Paxos/Raft round-trip for reads — any replica can serve reads independently, achieving the lowest possible read latency but allowing momentarily stale reads. This is a deliberate latency-over-consistency choice.

The implication is that Elasticsearch optimizes for low-latency, high-throughput reads in the common case and accepts eventual consistency as the cost. This makes it an excellent choice for use cases where slight staleness is acceptable (search, analytics, dashboards) and a poor choice for use cases where read-after-write consistency is required (financial ledgers, inventory management).

Consistency Models in Elasticsearch

Read-your-writes: Not guaranteed by default. If your application requires that after writing a document you can immediately read it back, you have two options: (1) use the document's _id to do a GET request by ID (which routes to the primary shard, ensuring you get the latest version), or (2) pass ?preference=_local consistently routing reads to the same shard copy. Alternatively, using ?refresh=wait_for on the indexing request causes ES to wait for the document to become searchable before acknowledging — at the cost of 1 refresh interval (up to 1 second) of added write latency.

Causal consistency: Achievable within a single index with explicit use of if_seq_no and if_primary_term parameters for optimistic concurrency control. These allow you to implement compare-and-swap semantics: "update this document only if it hasn't changed since version X." This is how you prevent lost updates in concurrent modification scenarios without distributed transactions.

Strong consistency: Not natively supported across multiple documents. For use cases requiring strong consistency, Elasticsearch is the wrong database. Within a single shard, writes are serialized at the primary shard level, providing shard-level sequential consistency — but this doesn't compose across shards.

⚡ L7/L8 Differentiator Discussing how ES 8.x changed the default shard count from 5 to 1 and what this means operationally shows current knowledge. The old default of 5 shards was designed for clusters that commonly had 5+ nodes — but it caused massive over-sharding for small indices. The new default of 1 is correct for most small-to-medium indices; scaling shard count should be data-volume-driven, not chosen arbitrarily. Additionally, ES 8.x introduced the concept of "optional" replicas — replicas that the cluster will try to allocate but won't mark the cluster yellow if they can't be placed. This is important for cost-optimized deployments on spot instances.

Pillar 04

🛡️ Failure Modes & Resilience

Detection, automatic recovery, data loss risks, and manual intervention

Failure Scenario Analysis

💥 Single Node Crash — Follower/Replica Node

Master detects missed heartbeats from the failed node within cluster.fault_detection.follower_check.retry_count × interval (default ~30 seconds). The node is removed from the cluster state.

All primary shards on the failed node had been running on this replica node — primary shards are already on other nodes and remain active. Replica shards on the dead node are marked "unassigned." Master initiates peer recovery: copies shard data from the existing primary to a surviving node, then replays any translog operations since the last checkpoint.

Zero data loss — primary shards are on other nodes and remain healthy. Queries can still be served (potentially with reduced replica read scaling). Cluster health transitions to YELLOW (all primaries active, not all replicas).

Monitor recovery progress with GET /_cat/recovery. Consider temporarily increasing indices.recovery.max_bytes_per_sec to speed recovery if network is not saturated. Replace the failed node to restore GREEN health.

💥 Single Node Crash — Primary Shard Host

Same heartbeat mechanism. Master immediately detects primary shard loss and marks affected shards as "initializing" on surviving replica nodes.

Master promotes the best available replica to become the new primary. "Best" is determined by the most up-to-date replica (highest sequence number from the translog). The promoted replica undergoes a "primary terms" increment and begins accepting new writes. Other replicas sync from the new primary.

Potential for very small data loss: writes that were in-flight to the primary and had not yet been replicated to any replica may be lost. With wait_for_active_shards=1 (default), a write is confirmed once the primary acknowledges it — if the primary crashes before replicating, that write is lost. With wait_for_active_shards=all, loss is prevented but availability is reduced.

Verify cluster health transitions from RED to YELLOW to GREEN as recovery progresses. Check GET /_cluster/health?wait_for_status=green&timeout=30s. If an index goes RED, check shard allocation explanations with GET /_cluster/allocation/explain.

💥 Simultaneous Minority Node Failures

Each failed node detected individually via heartbeat timeouts. Master remains elected because majority of master-eligible nodes are still alive (e.g., 2 of 5 nodes fail: 3 surviving nodes include quorum of 3 for a 5-node cluster with quorum of 3).

Master redistributes unallocated shards from failed nodes across surviving nodes. Shard recovery runs in parallel across multiple destinations. This is the most common production failure scenario and ES handles it well — provided the surviving nodes have sufficient disk and heap to absorb the additional shards.

Depends on replica count. With 1 replica: if primary and its single replica happen to be on the two failed nodes simultaneously, that shard is RED (data temporarily unavailable until one node recovers). True data loss only occurs if ALL copies of a shard are on failed nodes AND the nodes cannot be recovered.

If surviving nodes lack resources to accept redistributed shards, temporarily disable rebalancing (cluster.routing.rebalance.enable: none) to prevent additional churn. Bring failed nodes back online to restore shard placement without full recovery.

💥 Network Partition — Quorum Side Has Majority

Nodes on the minority side lose contact with the master. After fault detection timeout, they attempt to elect a new master — but cannot form quorum. They remain master-less and stop accepting writes, serving reads from existing segments only.

The majority side continues operating normally with the existing master (or elects a new one from its members). When the partition heals, minority-side nodes rejoin the cluster and receive the authoritative cluster state from the master. Any segment data they have is validated against the primary term and sequence number — stale data is discarded and re-synced from primary shards.

Minimal on the majority side — it continues accepting writes normally. The minority side serves stale reads during the partition. On recovery, minority-side replicas must re-sync from primaries, which may involve discarding some locally-indexed but non-replicated documents.

Monitor with GET /_cluster/health from both sides. Once partition heals, watch for shard recovery activity and ensure recovering nodes have sufficient bandwidth.

💥 Network Partition — Even Split, No Quorum Possible

Both sides lose contact with enough master-eligible nodes to form quorum. Both sides attempt master election and both fail — or if they were previously split, both lose their master after the configured timeout.

In ES 7+ with the new Raft-based election, this is handled gracefully: neither side elects a master, so neither side can make cluster-state changes (though existing writes to existing primaries on each side may still be accepted temporarily). When partition heals, the side with the higher primary term wins and the other side must re-sync. This is a significant improvement over ES 6.x where this could cause split-brain.

Moderate. Writes accepted by primaries on both sides during the split period may conflict. The resolution uses the highest primary term — the side that made fewer changes (lost the election race) must discard writes. For critical data, ES 7+ offers better protection than earlier versions, but this scenario still warrants careful monitoring.

If the partition is prolonged, consider stopping one side of the cluster manually to avoid data divergence. On recovery, inspect _cluster/state and check for any uncommitted operations before re-enabling writes.

💥 Split-Brain (Legacy ES <7) / Two Active Masters

In legacy deployments with incorrect minimum_master_nodes configuration (or misconfigured Zen Discovery), both sides of a partition may elect a master. Each master starts accepting cluster-state changes independently, causing cluster state divergence.

None — split-brain is catastrophic in ES <7. Both masters accept writes, allocate shards, and make index changes. When the partition heals, there is no safe automatic resolution: both cluster states claim to be authoritative.

High. Data written during the split to indices managed by both masters is irreconcilable without manual intervention. Index mappings may have diverged, making documents unsearchable.

Stop both clusters. Identify the "correct" side (usually the one that had the original master). Restore the other side from snapshot. Re-import any data written to the sacrificed side if available. ES 7+ eliminates this scenario with Raft-based consensus — this is a primary reason to upgrade.

🐌 Slow / Degraded Node (Hot Replica / GC Pressure)

Slow nodes don't trigger normal failure detection — they appear "alive" to heartbeats but cause query timeouts. Visible via slow query logs (index.search.slowlog), elevated p99 latencies, and the "hot threads" API: GET /_nodes/hot_threads. Circuit breakers may start tripping on the slow node.

ES has limited automatic remediation for slow nodes. The adaptive replica selection (ARS) feature in ES 6.1+ dynamically routes searches to faster replicas, effectively deprioritizing slow nodes for read traffic. This is done using response time exponential moving averages to select the "best" shard copy for each request.

No data loss — the node is alive and durable. However, if the node is lagging on replication, it may be serving stale reads for longer than expected. If the node is experiencing extreme GC pauses, it may fail heartbeats and get ejected from the cluster, triggering shard redistribution.

Investigate GC pauses with GET /_nodes/stats/jvm. Check for field data cache OOM (indices.fielddata.breaker.tripped). For hot nodes, use shard exclusion to drain traffic: PUT /_cluster/settings { "transient": { "cluster.routing.allocation.exclude._ip": "1.2.3.4" } }.

💾 Disk Corruption or Partial Write

Lucene validates segment files using checksums (CRC-32) when loading. A corrupt segment file is detected at shard load time (node restart or recovery) and causes the shard to fail to load. Visible in logs as CorruptIndexException. Elasticsearch 6.x+ also added elasticsearch-shard verify-index CLI tool for manual validation.

If a replica of the corrupted shard is available on another node, the master will promote the uncorrupted replica to primary and initiate peer recovery from it to a new location. The corrupt shard on the failed node is abandoned.

No data loss if at least one non-corrupt copy exists. Data loss if all copies of a shard are corrupt — possible if disk corruption affects multiple nodes (e.g., a SAN/RAID issue). This underscores the importance of regular snapshots to a separate storage system (S3, GCS) as a second line of defense.

Use POST /_cluster/reroute with "allocate_stale_primary" action as a last resort to force-allocate a corrupt shard (accepting data loss). Always restore from snapshot first if available. Never run on a 0-replica, no-snapshot configuration in production.

⏱️ Clock Skew Between Nodes

Elasticsearch checks time synchronization between nodes using the master node's clock as reference. ES 7+ actively warns when detected clock skew exceeds 200ms via cluster health warnings. Severe skew can cause master elections to behave unexpectedly because Raft uses timeouts that are wall-clock dependent.

Limited. ES will log warnings but not automatically remedy clock drift. NTP/chrony on the OS level must be configured correctly. In Kubernetes, the host clock is shared with containers — ensuring the host NTP sync is critical.

No direct data loss from clock skew. However, time-range queries (@timestamp >= now-1h) that span nodes with different clocks may produce inconsistent results — documents near the boundary of a time window may be included or excluded inconsistently. Also, Kibana dashboards using relative time ranges may behave incorrectly.

Enforce NTP synchronization across all ES nodes. Use GET /_nodes/stats to check timestamp values and manually compare across nodes. Add clock skew monitoring as a separate infrastructure alert — don't rely on ES's own warnings.

🌩️ Thundering Herd After Recovery

When a previously unavailable node or index comes back online, many clients that had been retrying or queuing requests may simultaneously flood the recovering node. This is observable as a sudden spike in CPU, heap usage, and query latency immediately after recovery completion.

ES has limited built-in thundering herd protection. The circuit breakers will activate if memory pressure becomes critical, rejecting some requests with 429s. The thread pool queue provides some buffering but can be exhausted quickly.

No data loss. However, the recovering node may be re-crashed by the load surge, causing another recovery cycle — a failure amplification loop. This is particularly dangerous for hot data nodes that hold highly-queried shards.

Implement circuit breakers in client libraries (exponential backoff with jitter). Add a warm-up period in deployment pipelines: after bringing a node online, gradually re-enable it in the load balancer. Use cluster.routing.allocation.node_concurrent_recoveries to throttle concurrent shard recoveries.

🌊 Cascading Failures Prevention

One node fails → shards redistribute to surviving nodes → surviving nodes become overloaded (more shards = more heap, more I/O) → circuit breakers trip or GC pressure increases → nodes become slow → master election timeouts → another node is evicted → cascade accelerates. This is the most dangerous failure pattern in ES operations.

1. Capacity headroom: Run cluster at <70% disk usage and <60% heap usage to absorb one node failure
2. Balanced shard distribution: Use automatic shard balancing with cluster.routing.allocation.balance.shard to prevent hot nodes
3. Freeze unneeded indices: Cold/frozen tier removes searchable-snapshot shards from heap pressure
4. Shard-level circuit breakers: Prevent single runaway queries from collapsing an entire node
5. Delayed rebalancing: index.unassigned.node_left.delayed_timeout: 5m prevents immediate shard storm when a node briefly disconnects

📐 Back-of-Envelope — Shard Recovery Speed A 50GB shard recovering from a peer node over a 1Gbps network link takes: 50GB ÷ 125MB/s (theoretical max at 1Gbps) = ~400 seconds = ~6.7 minutes, assuming no other traffic. With ES's default recovery throttle of 40MB/s: 50GB ÷ 40MB/s = ~1,250 seconds = ~20 minutes. For a cluster with 20 failed shards of 50GB each recovering in parallel (limited to 2 concurrent recoveries per node by default): 10 sequential batches × 20 minutes = ~3.3 hours minimum. This explains why over-sharded clusters with large shard sizes take hours to recover — and why keeping shards at 30–50GB and having enough nodes for parallel recovery is operationally critical.
🎯
Key Takeaway — Pillar 4

ES 7+'s Raft-based master election eliminated the split-brain catastrophe of earlier versions, but cascading failure from node overload during shard redistribution remains the most dangerous operational scenario — requiring proactive capacity headroom, not reactive recovery.

Pillar 05

📈 Scaling Strategies

Vertical scaling, horizontal scaling, sharding, replication, and cross-region patterns

Vertical Scaling: Limits and Failure Points

Vertical scaling (adding RAM and CPU to individual nodes) in Elasticsearch runs into several hard walls that make it fundamentally limited compared to horizontal scaling. The most critical constraint is the JVM heap. Elasticsearch runs on the JVM, and heap sizes above 30–32GB trigger a critical performance cliff: below 32GB, the JVM uses "compressed ordinary object pointers" (CompressedOops), where object references are encoded in 32 bits instead of 64 bits. This nearly doubles the number of objects that fit in a given heap size and dramatically reduces garbage collection pressure. Above 32GB, full 64-bit pointers are required, which increases heap usage by 40–50% and causes GC pauses that can be severe enough to trigger cluster instability. The operational recommendation is to hard-limit JVM heap at 30GB per node, regardless of how much physical RAM the machine has.

The remaining RAM above the 30GB heap is invaluable — it's used by the OS file system cache for memory-mapping Lucene segment files (.tim, .doc, .dvd). Lucene is designed around the assumption that the OS will cache frequently-accessed pages, and this is the primary mechanism for "hot" data serving. A node with 120GB RAM allocated 30GB to JVM heap has 90GB for file system cache, which can hold substantial portions of index data in memory for zero-disk-seek retrieval. This is why high-memory instances are valuable in ES — not for the heap, but for the file cache.

CPU vertical scaling has better returns for compute-heavy operations like complex aggregations and expensive script queries, but for simple search operations, more CPUs past a certain point don't improve latency because the bottleneck is typically disk I/O for cache misses or the coordination overhead of merging results from many shards. Vertical scaling of disk (SSDs over HDDs, NVMe over SATA) provides significant latency improvements for any operation that causes a cache miss.

Horizontal Scaling: Architectural Changes

Horizontal scaling in Elasticsearch means adding more nodes and increasing shard count. However, the critical constraint — primary shard count is fixed at index creation — means horizontal write scaling requires architectural planning upfront, not reactive scaling. The common pattern is:

  • Time-series index rotation (ILM): For log data, metrics, and events, use Index Lifecycle Management to roll over to a new index when the current index reaches a size or document count threshold. Each new index can be created with the appropriate shard count for the expected data volume. Hot-warm-cold-frozen architecture allows efficient tiering of data across node types.
  • Datastreams: ES 7.9+ introduced datastreams as a first-class abstraction for append-only time-series data, wrapping the ILM rollover pattern with automated backing index management. Under the hood it's still rolling indices, but the API is simpler and more consistent.
  • Shrink API: An existing index with multiple primary shards can be reduced to fewer primary shards using the Shrink API (e.g., 10 shards → 5 shards). This is useful when you over-sharded at creation. Requires the index to be read-only and all primary shards to be on a single node. The target shard count must be a factor of the source count.
  • Split API: The inverse of Shrink — an existing index can be split into more primary shards. Again requires the index to be read-only. Target shard count must be a multiple of the source count. Neither operation is cheap — both require reading and re-indexing all data.

Replication Strategies

Synchronous vs. Asynchronous Replication

Elasticsearch uses a hybrid replication model. The write to the primary shard is always synchronous from the client's perspective (the client waits for the primary to acknowledge). Replication to replica shards happens concurrently and in-band with the primary acknowledgment: the primary forwards the write to all active replica shards and waits for them to acknowledge before sending the final ACK to the client (with the default wait_for_active_shards=1 configuration, only the primary's ACK is needed, making replica replication effectively asynchronous from the client's perspective).

This is important: the replica always receives the exact same operations as the primary, in the same order (using the same Lucene operations, not derived from the document), ensuring eventual consistency. There is no "log shipping" or "change data capture" — replicas execute the same indexing operations as the primary.

Sharding / Partitioning Deep Dive

Elasticsearch uses hash-based routing by default: shard = hash(routing_key) % num_primary_shards. The default routing key is the document's _id field. This distributes documents uniformly across shards (assuming a diverse set of IDs), preventing hotspots. However, hash-based routing has a critical limitation: it makes shards completely independent — you cannot do cross-shard range queries efficiently unless you scatter to all shards, which doesn't scale.

Custom Routing for Hotspot Prevention

Custom routing is a powerful feature that allows you to ensure related documents co-locate on the same shard. The pattern: PUT /index/_doc/1?routing=tenant_id. Now all documents for tenant_id land on the same shard, enabling shard-local queries (passing the same routing parameter to the search request makes ES query only the relevant shards) — dramatically reducing query fan-out for multi-tenant applications. The risk is shard imbalance: if one tenant has 10x the data of others, their shard becomes hot. This requires custom routing strategies with consistent hashing across multiple shard buckets rather than a single key.

📐 Back-of-Envelope — Shard Sizing Target: 30–50GB per shard, max 200M documents per shard. For a 10TB index: 10,000GB ÷ 40GB/shard = 250 primary shards. At 1 replica: 500 total shards. Across a 25-node cluster: 500 ÷ 25 = 20 shards per node. ES recommendation: max 20 shards per GB of heap. At 30GB heap per node: 600 shards per node max — we're at 20, safely within bounds. Query fan-out: a search touches all 250 primary shards (or 250 replica shards if reads are balanced) — coordinating node must merge 250 partial results. For most queries this is fine; for very expensive aggregations across 250 shards, query parallelism helps but coordination overhead grows. Consider rollup indices or summary aggregations for very high-cardinality aggregations across this many shards.

Resharding and Reindexing Challenges

The inability to change primary shard count in-place is Elasticsearch's most significant operational limitation. The standard approach for resharding is the Reindex API: create a new index with the desired shard count and copy all documents from the old index to the new one using the POST /_reindex API. This operation is expensive — it reads every document from the source index and re-indexes it into the destination, meaning it's a full I/O operation on the entire dataset. For a 10TB index, this can take hours or days, during which both old and new indices must exist simultaneously (2× storage cost). The common pattern:

  • Create new index with correct shard count
  • Point a write alias at the old index, a read alias at both (or use blue-green)
  • Start reindex in background
  • When complete, point write alias at new index, stop writes to old
  • Wait for all in-flight reads to drain from old index
  • Delete old index

ES 7.16+ introduced data stream aliases which simplify this process significantly for time-series data by providing a logical layer over rolling backing indices — you can reindex a single backing index without touching the others, and swap which backing index is "current" atomically via the datastream API.

Cross-Region Replication: Active-Active vs. Active-Passive

Cross-Cluster Replication (CCR) is the Elasticsearch feature for cross-region data distribution, introduced in ES 6.5. CCR operates in an active-passive model: a "follower" index in one cluster continuously replicates from a "leader" index in another cluster by replaying the leader's Lucene operations in sequence. The follower is read-only — it cannot accept writes. This makes CCR fundamentally active-passive for the replicated index.

For active-active (writes accepted in both regions), the standard approach at the application layer is to use a message queue (Kafka, Kinesis) as the source of truth, with ES clusters in each region consuming from the same queue. This ensures both ES clusters receive the same events and converge to the same state. Conflict resolution is event-ordering-based: the queue provides a total order. This sidesteps the need for distributed transactions between ES clusters at the cost of architectural complexity and the queue becoming a critical path dependency.

Read vs. Write Scaling — Can They Scale Independently?

Yes — and this is a key architectural strength of Elasticsearch. Read scaling is achieved by adding replica shards: each additional replica copy of a shard can serve read requests, so read throughput scales linearly with replica count (and with nodes). This is the easiest scaling lever — just increase number_of_replicas on an existing index (this can be done live with zero downtime). Write scaling requires primary shard count changes (harder, requires reindexing) or time-series index rotation (easier, plan this upfront). The combination of hot nodes (SSDs, high throughput for current writes) and warm/cold nodes (HDDs, lower cost for older read-heavy data) in a tiered architecture independently optimizes the write and read cost profiles.

🎯
Key Takeaway — Pillar 5

Elasticsearch's primary scaling limitation is the immutability of primary shard count per index — all large-scale ES architectures work around this through time-series index rotation (ILM/datastreams), careful pre-capacity-planning, or expensive reindexing operations; read scaling via replicas is trivially cheap by comparison.

🏭 Real-World Engineering Decisions

Production use cases, anti-patterns, decision matrices, and operational pitfalls

Where Elasticsearch Shines Above All Alternatives

Elasticsearch's core strengths map to three distinct but overlapping problem categories:

  • Full-text search with relevance ranking over large corpora: When documents are unstructured text and users need ranked results (not exact lookups), ES's Lucene-based BM25 scoring, language analysis pipeline, and rich query DSL (boolean, fuzzy, phrase, multi-match, function score) are unmatched in operational simplicity. Alternatives like Solr are architecturally equivalent but operationally more complex. Databases (even with full-text indexes) lack the rich relevance tuning and aggregation capabilities.
  • Real-time log analytics and observability at scale: When you need to ingest millions of events per second, make them searchable within seconds, and support arbitrary ad-hoc analytical queries (time-range, regex, term aggregations, histograms), ES's combination of NRT indexing, columnar doc values for aggregations, and distributed query execution is the dominant solution. The Elastic Stack (ES + Kibana + Logstash + Beats) provides an end-to-end observability platform that took years to match with alternatives.
  • Geospatial search and complex multi-dimensional filtering: ES natively supports geo-point and geo-shape fields with distance queries, bounding box searches, and polygon containment checks. Combined with BM25 text scoring and numeric range filters, this makes ES uniquely suited for location-aware search (find restaurants near me matching "vegan sushi" with minimum 4-star rating) — a combination no SQL database handles efficiently at scale.

Three Real-World Production Examples

🌐 GitHub — Code Search at Scale

GitHub uses Elasticsearch (and its own fork "Blackbird") to power its code search functionality across hundreds of millions of repositories and trillions of lines of code.

1
The Problem
Searching across billions of code files with syntax-aware queries (find all Python files where this function is called, with file path filters, language filters, and fuzzy matching on identifiers) at sub-second latency.
2
Why Elasticsearch
Custom analyzers for code tokenization (splitting on camelCase, underscores, dots for language-specific token patterns), geoshard routing by repository (all files of a repo co-locate on the same shard for repo-scoped queries), and the rich filter aggregation for faceted navigation (by language, repository, path) made ES the fit. No SQL database could handle the tokenization requirements and query complexity simultaneously at this scale.
3
Key Challenge
Index freshness: when code changes (a commit), the affected files must be reindexed within minutes. GitHub chose partial update strategies (updating individual file documents rather than whole repository bulk reindexes) to achieve low indexing latency. The scale challenge: maintaining <1s search latency across clusters handling petabytes of indexed code.

🛍️ Uber — Marketplace Search and Surge Pricing Visibility

Uber uses Elasticsearch for its driver and restaurant search systems, powering location-based discovery with real-time availability data.

1
The Problem
Find available drivers within a geofenced area, ranked by ETA and rating, with real-time availability status updates (a driver's availability changes every few seconds as they accept/complete trips).
2
Why Elasticsearch
Geo-distance queries with combined relevance scoring (distance × rating × availability) are ES's native strength. The partial update API (POST /_update with doc_as_upsert) allows efficiently updating just the availability field of a driver document without reindexing the entire document. Uber uses custom routing by geographic region to avoid cross-shard fan-out for location queries.
3
Key Challenge & Tradeoff
Update throughput: millions of driver position updates per minute create significant indexing load. Uber trades off search freshness vs. update throughput by setting a 2-second refresh interval (rather than the 1-second default) — drivers' positions are slightly older than 2 seconds in search results, which is acceptable for ETA calculations. The alternative (more frequent refresh) would dramatically increase segment count and degraded query performance.

🏦 Netflix — Log Analytics & Security Operations

Netflix uses the Elastic Stack at massive scale for their security operations center (SIEM), infrastructure monitoring, and service mesh analytics.

1
The Problem
Ingest and make searchable hundreds of billions of log events per day across thousands of microservices, with security analysts needing to perform forensic queries (find all HTTP 403 responses from service X in the last 4 hours where the source IP was in ASN Y) within seconds.
2
Why Elasticsearch
ILM-based hot-warm-cold architecture: logs land on NVMe SSD hot nodes with high write throughput, age to warm HDD nodes after 3 days (search still fast, much cheaper storage), then move to frozen tier (searchable snapshots on S3) after 14 days (pay only for query compute, not storage). This tiered approach reduces storage cost by 10x vs. keeping all data on SSDs while maintaining the ability to query 6+ months of data.
3
Key Challenge
Mapping explosion: when developers add new log fields dynamically, ES's dynamic mapping creates new field mappings. At Netflix's scale, thousands of unique field names per service × hundreds of services = cluster state explosion. Netflix enforces strict field naming conventions and uses dynamic: strict mappings with explicitly-allowed fields, with an approval workflow for adding new fields. Unknown fields go into a catch-all message keyword field rather than creating new mappings.

When NOT to Use Elasticsearch

🔥 Anti-Pattern: Using ES as a Primary OLTP Database Elasticsearch lacks multi-document ACID transactions, has eventual consistency, and does not support foreign key constraints or referential integrity. Using ES as the sole store for financial records, inventory counts, or user account data is architecturally dangerous. The correct pattern is to use a relational database or NewSQL system (PostgreSQL, CockroachDB) as the primary store and sync data into ES for search purposes.
🔥 Anti-Pattern: High-Cardinality Aggregations on Keyword Fields Running a terms aggregation on a field with millions of unique values (like user IDs or session tokens) requires ES to build a global ordinal mapping in memory. For a field with 10 million unique values, this consumes gigabytes of heap on each node involved in the query — potentially triggering OOM and node failure. For high-cardinality analytics, consider pre-aggregating into a summary index or using an analytics database like ClickHouse or BigQuery.
🔥 Anti-Pattern: Using Default Dynamic Mapping in Production Dynamic mapping auto-creates field mappings when new fields are first encountered. This is convenient for development but catastrophic in production: a typo in a field name creates a permanent mapping entry, a numeric field that sometimes contains strings causes a mapping conflict exception, and tens of thousands of unique field names from uncontrolled log sources can grow cluster state to multi-gigabyte sizes, causing master instability.

Most Common Operational Mistakes in Production

  • Over-sharding small indices: Creating indices with 5–10 shards when the data volume doesn't justify it. Each shard carries fixed overhead (file handles, heap, recovery cost). The ES 8.x default changed to 1 primary shard for this reason — most small indices should have 1 shard.
  • Ignoring the JVM heap 32GB ceiling: Setting -Xmx40g disables CompressedOops, causing 40–50% additional heap pressure and more frequent GC pauses that can destabilize the cluster.
  • Running data and master roles on the same nodes at scale: Heavy GC pauses from data operations cause master heartbeat timeouts, triggering unnecessary master elections. Dedicate master-eligible nodes at clusters with 10+ data nodes.
  • Not setting dynamic: strict on production indices: Dynamic mapping accumulates unbounded field names from uncontrolled data sources, leading to mapping explosion and cluster state growth.
  • Using field data (not doc values) for aggregations: Field data loads entire field into heap. For any high-cardinality field, this causes OOM. Doc values are the default and correct choice — never disable them on fields used for sorting or aggregations.

How Google-Scale Deployments Differ from Vanilla ES

At Google-equivalent scale, the assumptions that make vanilla Elasticsearch work break down in fundamental ways. Here is how hyperscale operators solve these problems differently:

Cluster state management: At vanilla ES, the cluster state is a monolithic in-memory object replicated to all nodes synchronously. At hyperscale (500+ nodes, 100K+ shards), publishing a cluster state update requires serializing a multi-gigabyte object and sending it to hundreds of nodes — this can take tens of seconds, blocking cluster state changes. Hyperscalers decompose this by running multiple coordinated ES clusters behind a routing middleware layer, keeping each cluster under the scalable cluster state size (typically <100 nodes, <20K shards per cluster). Elastic itself has been iteratively improving this with lazy cluster state publication (only sending diffs, not full state) in ES 8.x.

Ingest pipeline separation: At hyperscale, the indexing throughput requirements (millions of documents per second) and query requirements (thousands of complex queries per second) are incompatible on the same cluster. Hyperscalers run separate ingest clusters that accept writes and perform heavy analysis (Logstash/pipeline processing), with CCR or snapshot-based replication pushing data to separate query clusters optimized for low-latency reads. The ingest cluster can be tuned for write throughput (large translog, async durability, single replica), while the query cluster is tuned for read latency (more replicas, warmed segment caches, lower refresh interval).

Compute-storage separation: Vanilla ES couples compute (query execution) with storage (shard data on local disk). At hyperscale, this means every node must be both a high-memory machine (for query execution) and a high-storage machine (for data). The frozen tier (searchable snapshots) partially decouples this by storing data on object storage and caching locally. At hyperscale, this evolves to a full disaggregated architecture: shared object storage holds all shard data, and stateless compute nodes load and execute queries against segments they've cached locally — similar to the Snowflake architecture for databases. Elastic's "stateless" architecture project (announced 2023–2024) moves toward this model.

Multi-Tenancy and Noisy Neighbor Isolation

Multi-tenancy in Elasticsearch is a hard problem because there is no native resource isolation between indices on the same node. A single expensive query from Tenant A can consume all available search thread pool slots, causing Tenant B's queries to queue and time out. Strategies for isolation at scale:

  • Dedicated clusters per tenant: The strongest isolation — each tenant gets their own Elasticsearch cluster. Operationally expensive (many clusters to manage) but completely prevents noisy neighbor. Feasible for large enterprise customers; impractical for millions of small tenants.
  • Cluster-per-tier with routing middleware: Group tenants into clusters by resource tier (gold, silver, bronze). A middleware layer routes requests to the appropriate cluster. Tenants within the same tier compete for resources, but the blast radius of a noisy neighbor is limited to same-tier tenants. Tier assignment can be data-volume or QPS based.
  • Index-level isolation with thread pool prioritization: A single shared cluster with per-tenant indices, using custom thread pool routing to prioritize high-SLA tenant queries. ES doesn't natively support per-index thread pool assignment, but routing all tenant-specific requests through dedicated coordinating nodes with tenant-specific configurations approximates this.
  • Query-level circuit breakers per tenant: Inject a custom query context that limits per-tenant memory consumption. This requires custom ES plugin development or a middleware proxy that rewrites queries to include resource limits (e.g., terminate_after clause to limit documents scanned, timeout on all queries).

Regulatory Data Residency

For deployments subject to GDPR (EU), CCPA (California), or other data residency requirements, the challenge is ensuring that personal data indexed in Elasticsearch remains within specified geographic boundaries. This requires:

  • Region-isolated clusters: Each regulatory region (EU, US, APAC) gets a dedicated ES cluster. Application routing layer directs writes and queries to the appropriate regional cluster based on the user's jurisdiction. No data flows between regional clusters.
  • Cross-Cluster Search scoped by region: Federated search across regional clusters must be careful not to cross-cluster replicate or return data across boundaries. ES's CCS supports executing searches on a specific cluster alias — the middleware layer passes only the allowed cluster aliases for each user's jurisdiction.
  • Encryption and key management: ES 8.x supports encryption at rest natively. For GDPR's right-to-erasure requirement, "crypto-shredding" (deleting the encryption key for a specific user's data) can effectively erase their data without requiring a full reindex. This requires careful per-user key management design.

"Google Spanner for Elasticsearch" — What Innovations Would Be Needed?

⚡ L7/L8 Differentiator This is the kind of speculative design question L8 candidates are expected to engage with deeply. A strong answer proposes concrete technical mechanisms, not vague platitudes about "making it more scalable."

A hypothetical "Google-Spanner-equivalent for Elasticsearch" — call it "Elasticsearch Infinity" — would need to solve several fundamental architectural limitations of current ES:

  • True distributed transactions for search: Spanner uses TrueTime and two-phase commit for externally consistent transactions. An ES-equivalent would need a distributed transaction layer for atomic multi-document operations — perhaps using a distributed sequencer service (similar to Google's Percolator) that provides monotonically increasing commit timestamps and allows snapshot-isolated reads at any past timestamp.
  • Disaggregated compute and storage at full scale: All shard data lives in a distributed object store (like Colossus internally at Google), with stateless query compute nodes that fetch and cache shard segments on demand. Adds cold-start latency for infrequently queried data, but eliminates the storage-compute coupling that limits vanilla ES scalability. Elastic's stateless architecture is moving toward this.
  • Automatic shard count elasticity: Instead of fixed shard counts at index creation, use consistent hashing with virtual nodes (similar to Cassandra's vnodes) that allow transparent redistribution without the shard-count-immutability limitation. When adding nodes, virtual nodes migrate without full reindexing.
  • Native multi-tenancy with resource isolation: First-class tenant abstractions in the query engine, with per-tenant memory quotas, CPU scheduling weights, and dedicated I/O priorities enforced at the Lucene reader level — not just at the HTTP layer.
  • Global secondary indexes: Currently, sorting on a field requires scattering to all shards. With a distributed sorted index (similar to BigTable's sorted column families) maintained separately from the primary sharded index, certain aggregation and sort queries could be served without full scatter-gather.

Cost Efficiency at Scale — Optimizations Only Worth It at High Scale

  • Frozen tier / searchable snapshots (>5TB): Below 5TB, the operational complexity of managing frozen tier (snapshot lifecycle, cache sizing, query latency penalty) isn't worth the storage cost savings. Above 5TB, the 10x storage cost reduction (S3 vs. SSD) becomes economically compelling.
  • Rollup indices (>50B events): Pre-aggregating time-series data into summary indices (hourly, daily granularity) only makes sense when raw data query patterns are dominated by time-range aggregations (dashboards, reports) rather than document-level searches. Below this scale, query performance on raw data is acceptable.
  • Dedicated coordinating nodes (>20 data nodes): The overhead of running a coordinating-only node (pure memory + CPU, no disk) is only worthwhile when query coordination overhead (scatter-gather merging) is consuming meaningful resources on data nodes, which typically happens at 20+ data nodes with heavy analytical query loads.

Key Takeaway — Pillar 7

Google-scale Elasticsearch requires decomposing the monolithic cluster into a multi-cluster federation with separated write/read paths, compute-storage disaggregation, and tenant resource isolation mechanisms that do not exist in vanilla ES — and the architectural direction of ES 8.x (stateless nodes, frozen tier, Raft-based master) is incrementally closing this gap.

Step 4: Rapid-Fire Staff-Level Questions

Q1
Explain why Elasticsearch's default shard count changed from 5 to 1 in version 7.0, and what this means operationally.
The change was driven by the "oversharding" antipattern: with a 5-shard default, small indices (10MB of data) would be split into 5 shards, each containing 2MB — far below the recommended 30–50GB range. Each shard carries fixed overhead: file handles, JVM heap for segment metadata, and recovery cost. A cluster with 1,000 small indices under the old default would have 5,000 shards; under the new default, 1,000 shards — dramatically reducing cluster state size and GC pressure. Operationally, this means for new clusters on ES 7+, you should set shard count explicitly based on data volume calculations (target 30–50GB per shard) rather than accepting defaults, and legacy clusters migrating from ES 6 may benefit from consolidating small indices.
Q2
What is the difference between a "refresh" and a "flush" in Elasticsearch, and when would you change the defaults for each?
A refresh creates a new in-memory Lucene segment from the Lucene buffer, making recently-indexed documents searchable — it does NOT sync to disk. A flush performs a full Lucene commit (all segment files fsynced to disk) and rotates the translog (the write-ahead log is cleared since all data is now in committed Lucene segments). Change the refresh interval to -1 during bulk initial indexing to disable automatic refresh (massive throughput improvement — no segment creation overhead), then call POST /_refresh manually when done. Increase flush interval when durability concerns allow it (index.translog.durability: async), which reduces disk I/O during heavy indexing. Never change these for operational (non-bulk) indices without understanding the durability implications.
Q3
A query that runs in 200ms on a small index takes 8 seconds on a 10-billion-document index with the same cluster. Walk through your diagnostic approach.
Start with the Profile API ("profile": true in the query) to break down where time is spent: is it in the query phase (Lucene scoring) or the fetch phase (retrieving source documents)? If the query phase is slow, examine the query type — a wildcard or regex query degrades linearly with document count; switch to n-gram tokenization. If it's a terms aggregation, check segment count per shard (too many small segments from under-merging = slow aggregation): call Force Merge to reduce segment count. If it's the fetch phase, consider using _source filtering to return only needed fields. Check if doc values are enabled for sorted/aggregated fields — if not, field data loading is the bottleneck. Finally, consider whether the cluster is undersized for this query load (thread pool saturation visible in /_nodes/stats/thread_pool).
Q4
How does Elasticsearch handle a situation where you need to update a document 100 times per second for 1 million documents simultaneously?
Each update in ES is actually a delete + new insert at the Lucene level (documents are immutable in Lucene segments). At 100 updates/sec × 1M documents = 100K updates/sec total, the cluster must handle this as new indexing load — using the Bulk API with batching is essential (single bulk requests of 5–10MB). The old document versions accumulate as "deleted" documents in segments, increasing disk usage and slowing queries (deleted docs are skipped but still take space). Segment merges eventually clean up deleted documents — configure an aggressive merge policy (max_deleted_ratio: 0.2) to trigger merges sooner. For extremely high-frequency updates on specific fields (like price or inventory), consider using doc_values + scripted updates (partial updates) to avoid re-indexing the full source document. Ultimately, at this update rate, evaluating whether a caching layer in front of ES that batches updates before committing to ES would reduce write amplification.
Q5
Explain what "field data cache" vs. "doc values" are and why using field data on a high-cardinality field in production can bring down your cluster.
Doc values are a column-store, disk-backed, memory-mappable structure built at index time for every field marked with doc_values: true (the default for keyword and numeric fields). They allow O(1) per-document access for sorting and aggregations with minimal heap usage. Field data cache is the legacy alternative: when you run an aggregation on a field that was indexed as text (not keyword) without doc values, ES loads all values for that field across all matching documents into JVM heap as an uninverted index. For a field with 1 million unique values, this can consume gigabytes of heap per node simultaneously. At scale, this triggers GC storms → JVM pause → node fails heartbeat → master ejects it → shard redistribution → cascade failure. Prevention: never aggregate on text fields without enabling fielddata explicitly (which is disabled by default for good reason); use keyword sub-fields for aggregation on analyzed text; monitor indices.fielddata.memory_size_in_bytes and the fielddata circuit breaker tripped counter.
Q6
Your Elasticsearch cluster is at 85% heap usage and you cannot add new nodes for 48 hours. What do you do?
Immediately identify what's consuming heap: check field data cache (GET /_stats/fielddata), query cache, segment memory, and request cache. If field data is large: clear it (POST /_cache/clear?fielddata=true) and identify which queries are triggering it — add the field as keyword sub-field and migrate queries. Force-merge read-only indices to 1 segment per shard (POST /old-index/_forcemerge?max_num_segments=1) — merged segments use less heap than many small segments. Reduce the query cache size (indices.queries.cache.size: 5%). Cancel any heavy aggregation queries running in-flight that are holding large data structures. As a last resort, reduce the field data circuit breaker limit (indices.breaker.fielddata.limit: 30%) to force earlier rejection of field-data-intensive queries rather than allowing them to fill heap. Set an urgent alert at 75% to avoid this situation recurring.
Q7
How does cross-cluster search (CCS) work in Elasticsearch, and what are its latency implications?
CCS allows a coordinating node in one ES cluster to scatter queries to remote clusters and merge their results, treating the remote cluster's shards like local shards in the two-phase query-then-fetch model. The coordinating node establishes persistent TCP connections to the remote cluster's gateway nodes. The query phase fans out to remote shards in parallel with local shards — the latency overhead is determined by the network RTT to the remote cluster plus their query execution time. For cross-region CCS (e.g., US-East to EU-West, ~100ms RTT), the query phase adds at least 100ms to every search request — making CCS suitable only for use cases that can tolerate higher-latency search (analytics, forensic investigation) rather than real-time user-facing search. Minimize CCS cost by explicitly naming only the relevant remote clusters in the search request, avoiding wildcard cluster references that scatter to all remote clusters unnecessarily.
Q8
What is the "Global Ordinals" concept in Elasticsearch and when does it matter?
Global ordinals are a segment-level and cluster-wide mapping from keyword field values to sequential integer ordinals, built at query time before executing terms aggregations. Within a single segment, local ordinals map unique values to integers (e.g., "apple"→0, "google"→1) for efficient bucket computation. Across segments, global ordinals provide a unified mapping so that the same value in different segments maps to the same integer, enabling the coordinating node to merge partial aggregation results correctly. Building global ordinals is a significant one-time cost on the first aggregation after an index refresh or merge — it can take seconds for high-cardinality fields. In ES 7.2+, you can pre-build global ordinals eagerly at index refresh time (eager_global_ordinals: true on the field mapping) to avoid this latency spike on the first query after index warming. Critical for keyword fields used heavily in terms aggregations — matters most for high-cardinality fields and for use cases where consistent sub-second aggregation latency is required.
Q9
Describe how you would implement a multi-tenant SaaS product search where each tenant's data is isolated and their searches only return their own data.
Two main architectural choices: index-per-tenant (each tenant gets their own index) vs. shared index with tenant field filtering. Index-per-tenant provides true isolation (mapping changes per tenant, independent scaling, simple access control using index-level permissions) but becomes operationally expensive above ~1,000 tenants (cluster state bloat from thousands of indices). Shared index with tenant filtering is operationally simpler but requires every query to include a mandatory must filter clause (term: { tenant_id: "tenant_xyz" }) — enforce this at the search service layer (never expose ES directly to clients). Shard routing by tenant ID (?routing=tenant_id) ensures all of a tenant's documents co-locate on the same shard, making tenant-scoped queries single-shard and eliminating cross-shard fan-out. For large enterprises, consider a hybrid: shared indices for small tenants, dedicated indices for large tenants. Implement row-level security using ES's Document Level Security (DLS) feature (requires Elastic Security license) as an additional defense-in-depth layer.
Q10
How would you implement zero-downtime reindexing for a live production index with 2 billion documents?
Use the alias + reindex pattern. Step 1: ensure the current index is accessed via an alias (products-alias → products-v1). Step 2: create the new index (products-v2) with the correct mapping. Step 3: start the reindex API operation as a background task (POST /_reindex?wait_for_completion=false) which copies documents from v1 to v2. Step 4: during reindexing, continue writing new documents to both v1 (via the write alias) AND v2 (dual-write in the indexing service) to capture changes that occur while reindex runs. Step 5: when reindex completes, reconcile any documents that were updated/deleted in v1 after reindex started using a change-tracking mechanism (use a last-modified timestamp field, re-reindex all documents modified after reindex start time). Step 6: perform an atomic alias swap: POST /_aliases { "actions": [{"remove":{"index":"products-v1","alias":"products-alias"}}, {"add":{"index":"products-v2","alias":"products-alias"}}] }. Step 7: delete v1 after confirming v2 serves correctly. For 2B docs, expect the reindex to take 12–24+ hours — the write-to-both pattern ensures no writes are lost during this window.
