Pillar 1 — Core Concepts & Mental Models
First principles: what problem Aerospike solves, its core abstraction, fundamental guarantees, and invariants

🌍 What Problem Does Aerospike Solve?

To understand why Aerospike exists, you must understand the state of high-performance databases in the 2008–2012 era. The internet economy was producing a class of problems that no existing database could handle gracefully: sub-millisecond read/write operations at sustained millions of requests per second, with hundreds of terabytes of data, and strong consistency guarantees. These requirements were mutually exclusive in the existing tool landscape.

Relational databases (MySQL, PostgreSQL) provided strong consistency but collapsed under write-heavy load beyond a few thousand operations per second without heroic sharding complexity. Memcached provided microsecond reads at scale but had zero persistence, zero consistency, and zero durability — it was a cache, not a database. Redis solved the persistence problem partially but was fundamentally single-threaded and ran entirely in RAM, making data sizes above a few hundred gigabytes prohibitively expensive. Cassandra and HBase provided scale and persistence but sacrificed strong consistency (Cassandra) or required complex HDFS infrastructure (HBase) with latency in the tens-of-milliseconds range.

The specific pain point that motivated Aerospike's creation was real-time bidding (RTB) in digital advertising. An ad exchange must decide, in under 10 milliseconds total (including network round trips), whether to bid on an ad impression, at what price, and for which creative. The decision requires looking up a user profile (potentially hundreds of attributes: interest segments, prior ad exposures, frequency caps, demographic inferences) stored in a persistent database. At peak load, a major exchange processes 1–3 million bid requests per second. The data set (user profiles for hundreds of millions of users) is far too large to fit in RAM on commodity hardware, but the latency budget cannot tolerate disk I/O on spinning hard drives (8ms average seek time × 1M requests/second = 8 million ms/second of I/O, clearly impossible).

The insight that founded Aerospike: SSDs fundamentally change the latency calculus. Enterprise NVMe SSDs provide ~50–150 microsecond random read latency, compared to 8,000 microseconds for spinning disk — a 50–160x improvement. But in 2010, no database was designed specifically to exploit this property. Most databases treated SSDs as "fast disks" and used them with I/O patterns optimized for spinning disk (sequential I/O, large buffers), leaving the SSD's random-access latency advantage unexploited.

Aerospike's founding innovation was the Hybrid Memory Architecture (HMA): keep all index entries in RAM (where they provide nanosecond-latency lookups), but store all values on SSD (where enterprise NVMe provides ~100μs random reads). This allows a single commodity server with 64GB RAM to index and access multiple terabytes of SSD data with sub-millisecond total operation latency — impossible with any contemporary alternative.

🔮 The Core Abstraction

Aerospike's data model is a multi-dimensional namespace → set → record → bins hierarchy. Understanding each level deeply is essential:

  • Namespace: The top-level organizational unit, analogous to a database in RDBMS. Each namespace has its own storage configuration — some namespaces may be RAM-only (for hot data), others SSD-backed (for large datasets). A cluster can have multiple namespaces with completely different storage policies. Each namespace is independently replicated across the cluster.
  • Set: An optional grouping within a namespace, loosely analogous to a table. Sets are a logical concept — records in a set share the same physical storage as other records in the same namespace. Sets enable selective secondary indexing and scan operations. Unlike a table, sets do not define a schema — different records in the same set can have entirely different bin structures (schemaless).
  • Record: The fundamental unit of storage and atomicity. A record is identified by a key (which can be an integer, string, or byte array). The key is hashed using RIPEMD-160 to produce a 20-byte digest that is the actual storage key. Aerospike does not store the original key by default (it stores only the digest) unless send-key true is configured. This is a critically important detail for debugging and operational practices.
  • Bins: The columns within a record. A record can have up to 32,767 bins (per namespace). Bins are typed (integer, string, bytes, double, list, map, GeoJSON, HyperLogLog, blob). Unlike columns in a relational table, bins are defined per-record — you can add or remove bins on individual records without schema migrations. Bin values are stored together in a single contiguous serialized block on disk, meaning reading one bin reads all bins for that record (no column store capability). This is a significant design trade-off for records with large numbers of wide bins.

The abstraction is powerful because it provides the ergonomics of a key-value store with the expressive power of a document database, the operational simplicity of a managed database, and the raw performance of a purpose-built in-memory system — all simultaneously. The bin model means you can store complex hierarchical data structures (maps, lists, maps-within-lists) as native types with in-place partial update operations (e.g., increment a specific integer bin without reading the entire record, or append to a list bin without fetching the full list).

📋 Fundamental Guarantees

✅ What Aerospike GUARANTEES

  • Strong consistency (optional, per-namespace): With Strong Consistency (SC) mode enabled, all reads and writes to a record are serialized through the master partition for that record. Every read sees the last committed write — Aerospike provides read-your-writes and linearizability within a single record's operations. This requires a majority quorum of replica nodes to acknowledge a write before it is visible to reads.
  • ACID at the record level: Individual record operations (read, write, delete, operate) are atomic. A single operate() call can modify multiple bins in a record atomically — either all changes are committed or none are. There is no partial write visible to other clients.
  • Durable writes (configurable): With the default write policy (commit-master), data is written to persistent storage on the master before the write is acknowledged to the client. With commit-all, data is confirmed on all replica nodes before acknowledgment.
  • Sub-millisecond latency at scale: Under the Hybrid Memory Architecture, Aerospike guarantees consistent sub-millisecond operation latency for single-record operations — this is an architectural property (RAM index + SSD data), not just a marketing claim.
  • Automatic cluster management: Node failures, additions, and repartitioning are handled automatically. Data migration after topology changes happens in the background without operator intervention.

❌ What Aerospike Does NOT Guarantee

  • Multi-record transactions (in traditional versions): Prior to Aerospike 7.0, Aerospike did not support multi-record transactions. A read-modify-write across two records could see a torn state. Note: Aerospike 7.0 (released 2023) introduced multi-record transactions (MRT) with ACID guarantees — this is a major version change critical to know.
  • SQL query capability: Aerospike is not a relational database. Complex JOIN operations, GROUP BY aggregations across large datasets, and multi-table ACID transactions (pre-7.0) are either not supported or require Secondary Index queries with application-level logic.
  • Full secondary index performance at high cardinality: Secondary indexes in Aerospike are in-memory per-node. A secondary index query fans out to all nodes and merges results. At very high cardinality (millions of distinct values), secondary index queries approach full-scan performance — they do not provide the logarithmic query performance of a B-tree index in a relational database.
  • Query planner / optimizer: Aerospike has no query optimizer. There are no EXPLAIN plans, no statistics-based query planning, and no automatic index selection. Query paths are entirely programmer-determined.

⚖️ Invariants That Must Always Hold

  • Partition ownership is always deterministic: Aerospike uses a fixed 4,096-partition scheme. Every record key maps to exactly one partition via RIPEMD-160 hash. The cluster's partition map (which node owns which partitions as master) is always consistent — there is no ambiguity about where a record lives. Violating this (e.g., through misconfiguration of the cluster nodes list) causes split-brain where two groups of nodes independently claim partition ownership.
  • Generation counter monotonicity: Every record carries an internal generation counter that increments on every write. This counter is used for optimistic concurrency control. An application using check-and-set (CAS) operations relies on the generation counter never decreasing. A node crash and recovery (replaying from persistence) must restore the same or higher generation counter — never lower.
  • TTL correctness: Records with configured TTL (time-to-live) must expire on or after their deadline, never before. The eviction and expiration subsystem must process TTLs in order. A record that has not yet expired must be readable. Partial writes (writing data but not the TTL field) would violate this invariant.
  • Replication factor consistency: If a namespace is configured with replication factor N, every partition must have exactly N copies across different nodes (in a healthy cluster with sufficient nodes). During migration, replica counts may be temporarily below N, but the cluster must converge to exactly N replicas in finite time after any topology change.

⚡ L7/L8 Differentiator

Most candidates describe Aerospike as "a NoSQL database with SSD storage." The L7+ insight is that HMA is not just "storing on SSD" — it's the architectural decision to bypass the operating system page cache entirely via Direct I/O on raw devices, eliminating OS cache management overhead and achieving deterministic latency. Without Direct I/O, the OS might page out "cold" Aerospike data to make room for other processes, creating unpredictable latency spikes. With Direct I/O on raw NVMe, Aerospike controls its own I/O scheduling and achieves consistent p99 latency rather than excellent p50 with terrible p99.

📌 Aerospike 7.x Major Changes (2023–2026)

Aerospike 7.0 (2023): Multi-Record Transactions (MRT) — the single most significant change in Aerospike's history. ACID transactions spanning multiple records are now supported natively, using a two-phase commit protocol with a transaction coordinator role. This eliminates the most common objection to using Aerospike for financial applications. Aerospike 7.1+: Enhanced XDR (Cross-Datacenter Replication) — improved conflict resolution with Last-Write-Wins based on hybrid logical clocks, and topology-aware routing that reduces cross-region bandwidth. Aerospike 7.x: Improved Secondary Index architecture — SI now supports range queries on numeric bins and geo-spatial queries with improved performance at higher cardinality.

🔑

Aerospike's core insight — keep the index in RAM and the data on NVMe SSD with Direct I/O, bypassing all OS page cache management — is what enables consistent sub-millisecond latency at terabyte scale that is architecturally impossible for any database treating SSD as "fast spinning disk."

🔩 Internal Component Breakdown

1. Transaction Subsystem (TS)

The Transaction Subsystem is the heart of Aerospike's request processing pipeline. Each incoming client request (read, write, delete, operate, batch) is assigned to a transaction thread from a configurable pool (transaction-threads-per-queue, default 4 per queue, with multiple queues). The TS performs key routing (mapping digest to partition to node), executes the operation against the storage engine, manages replication to replica nodes, and sends the response back to the client.

A critical design choice: Aerospike uses a shared-nothing, lockless design within the TS for single-record operations. Each partition's data is owned by a single TS queue, and the hash of the partition determines which queue processes that record's transactions. This eliminates inter-thread lock contention for single-record operations — a record's entire lifecycle within a transaction is processed by a single thread from dispatch to completion. Contention only occurs at the per-record level (when two concurrent writes target the same record simultaneously), which is handled with a fine-grained record-level lock (not a partition-level lock).

2. Storage Engine

The Storage Engine manages persistent data on SSD devices. Key implementation details:

  • Append-only write model: New records and updates are always written to the current write block at the tail position. Aerospike never overwrites existing data in place on SSD. This decision has profound consequences: (1) It completely eliminates write amplification from random-write I/O patterns that destroy SSD endurance. (2) It makes the write path extremely fast — a write is always sequential, which maximizes SSD write bandwidth. (3) It creates the need for a garbage collection / defragmentation process (Defragmenter) to reclaim space from dead (overwritten or expired) records.
  • Write Buffer (WBLOCK): All incoming writes are first staged in RAM in a write buffer (configurable size, typically 128KB). When the buffer fills, it is flushed to SSD as a single write I/O. This converts application-level random writes (individual records of variable size) into large sequential SSD writes, maximizing SSD throughput. The cost: data in the write buffer is in RAM, so a node crash without a write buffer flush causes loss of the in-flight buffer. The write-block-size parameter controls this.
  • Defragmenter: A background process that continuously scans existing SSD extents (typically 128MB chunks). When the live record density in an extent falls below a threshold (default: 50% live records), the defragmenter reads all live records from the extent, rewrites them to new extents, and marks the old extent as free. This process runs continuously at a controlled rate to avoid I/O saturation. The defrag-sleep parameter controls how aggressively defragmentation runs. If defragmentation falls behind (write rate exceeds defrag rate), disk space fills up and the node stops accepting writes.
  • Persisted Index (PI) — Aerospike 5.2+: Prior to 5.2, the primary index was entirely in DRAM and had to be rebuilt on every node restart by scanning the SSD — at terabyte scale, this cold start could take 30–60 minutes. Persisted Index stores the primary index to a separate SSD (or flash) device, allowing cold start in under a minute even for multi-terabyte datasets. This is a significant operational improvement for disaster recovery scenarios.

3. Cluster Protocol (Paxos-like Consensus)

Aerospike uses a proprietary cluster formation and membership protocol based on a heartbeat mechanism and a Paxos-like consensus for partition ownership decisions. Every node broadcasts heartbeats to all other nodes (configurable via mesh or multicast topology). If a node misses a configurable number of consecutive heartbeats from a peer, it is considered failed, and the surviving cluster initiates a cluster reformation event.

During cluster reformation, a new succession list is established via a distributed agreement protocol — all surviving nodes must agree on the same set of live nodes. This succession list then determines a new partition map: partitions are redistributed among surviving nodes, and data migration begins to restore the configured replication factor.

The partition map at any moment has three possible states for each node-partition combination: Master (read/write authoritative), Replica (receives write replications from master, serves as failover), or Absent (does not hold this partition). The cluster always maintains exactly one master per partition — there is never a case where two nodes simultaneously believe they are master for the same partition in a correctly configured cluster.

4. Cross-Datacenter Replication (XDR)

XDR is Aerospike's asynchronous geo-replication subsystem. It maintains an internal change log (the XDR digest log — or in 5.x+, the XDR change notification stream derived from the storage engine's write activity) and ships changes to remote Aerospike clusters over TCP. XDR provides eventual consistency across datacenters, not strong consistency. The lag depends on network bandwidth and cross-DC write rate, typically ranging from milliseconds to seconds under normal conditions.

In Aerospike 7.x, XDR supports topology-aware routing: XDR can be configured to route writes only to specific namespaces or sets in the remote cluster, enabling fine-grained replication policies (e.g., replicate only user profile data to EU, not ad targeting data, for GDPR compliance).

5. Scan and Query Engine

The Scan Engine enables full-namespace or full-set scans, streaming all records through a configurable predicate filter. Scans are partition-parallel: the cluster divides partitions among all nodes, each node scans its local partition data, and results are streamed back to the client. A scan in Aerospike can read from the master partitions (consistent but higher load on masters) or from replica partitions (lower load on masters, but may return slightly stale data if recent writes have not replicated yet).

Secondary Index (SI) queries use in-memory B-tree indexes maintained per node. An SI query is sent to all nodes (fan-out), each node queries its local SI and returns matching records, and the client merges results. This fan-out model means SI query latency is approximately equal to the slowest node's response time — a single slow node (due to a GC pause or I/O saturation) creates tail latency for SI queries. This is a critical architectural weakness compared to range queries in a B-tree RDBMS.

📊 Full Data Flow — Write Path

Client Key Hashing & Routing

The Aerospike client library hashes the key using RIPEMD-160, takes the first 12 bits of the digest to determine the partition ID (0–4095), and consults its local copy of the cluster partition map to determine which node is the current master for that partition. The client opens a direct TCP connection to the master node (no coordinator proxy — this is direct routing). Latency contribution: ~1μs for in-process hash + map lookup; ~100μs network RTT.

Master Node: Transaction Dispatch

The master node receives the write request on its fabric listener. The request is placed in the appropriate transaction queue (determined by the partition ID modulo the number of queues). A transaction thread picks it up. The thread acquires the per-record lock (a fine-grained lock on the specific 20-byte digest). Latency: ~1μs lock acquisition if uncontended; can be tens of μs if the same record is being concurrently updated.

Write to Storage Engine

The transaction thread serializes the record (key, bins, metadata: generation, TTL, last-modified timestamp) into a flat byte array. The serialized record is written to the in-memory write buffer (WBLOCK). If the WBLOCK is full, it is flushed to SSD synchronously (blocking write) and a new WBLOCK is started. The Primary Index entry for this record is updated in DRAM: the 64-byte index entry is updated with the new {device_id, byte_offset, size} pointing to the freshly written data. The old index entry (pointing to the previous version of the record on SSD) now points to dead data — the Defragmenter will reclaim it later. Latency: ~1μs for WBLOCK write (RAM); +100μs if a WBLOCK flush to SSD is triggered.

Replication to Replica Nodes

After writing to the master's storage engine, the transaction thread sends the write to all replica nodes in parallel over the intracluster fabric (separate TCP connections from client-facing connections). The policy is configurable: commit-master (default) — acknowledge to client after master write completes, replica write is in-flight asynchronously. commit-all — wait for all replicas to confirm before acknowledging. With commit-master, replica writes complete within ~1ms on a local network. Latency: 0 added for commit-master (parallel with client response); ~200μs additional for commit-all.

Client Acknowledgment

The master node sends the write acknowledgment to the client. The TCP response contains: result code (OK, KEY_EXISTS, GENERATION, etc.), and optionally the post-write record (if the client requested it). The client library delivers the result to the application. Total write latency: ~300–600μs end-to-end for commit-master on LAN; ~500μs–1ms for commit-all.

📖 Full Data Flow — Read Path

Client Routing (Same as Write)

RIPEMD-160 hash → partition ID → partition map lookup → direct TCP to master node. In Strong Consistency mode, reads MUST go to the master. In availability mode (eventual), reads can be served by any replica, distributing read load across the cluster.

Primary Index Lookup (RAM)

The master node receives the read request. The transaction thread performs a hash table lookup in the Primary Index (entirely in DRAM) using the 20-byte digest as the key. If the key is found, the index entry returns: {device_id, byte_offset, size_in_bytes}. If not found, the node returns KEY_NOT_FOUND immediately without any SSD I/O. Index lookup latency: ~50–100ns (RAM access + hash table probe).

SSD Read via Direct I/O

If the record exists, the transaction thread calls pread64() with the device file descriptor (opened with O_DIRECT to bypass OS page cache), the byte offset from the index entry, and the size. The Linux kernel submits this as a raw I/O request to the NVMe device. The NVMe controller services the request directly from flash cells. SSD read latency: 50–150μs on enterprise NVMe (e.g., Samsung PM9A3, Micron 9400).

Deserialization and Response

The raw bytes from SSD are deserialized into Aerospike's wire protocol format. If the client requested specific bins (projection), only those bins are serialized into the response — the rest are discarded. The response is sent back over TCP. Deserialization latency: ~1–5μs depending on record size.

Total read latency: ~200–300μs typical on LAN with NVMe storage. Sub-100μs with data-in-memory=true.

🔄 Backpressure and Flow Control

Aerospike implements backpressure at several levels:

  • Transaction queue depth limit: The transaction-queues × transaction-threads-per-queue configuration bounds the maximum concurrent in-flight transactions. When the transaction queue is full (queue depth exceeds the limit), new incoming requests are rejected immediately with an AEROSPIKE_ERR_SERVER error. The client library receives this and can implement its own backpressure (delay + retry). This is a hard backpressure mechanism — the server refuses further work rather than queuing indefinitely.
  • Write quota (storage space): When the SSD fill ratio exceeds the stop-writes-pct threshold (default 90%), the node stops accepting new writes. This prevents the defragmenter from being unable to keep up, which would otherwise lead to full disks and data corruption. The eviction feature can be configured to evict the oldest TTL records before this limit is hit.
  • Migration throttle: Data migration after cluster topology changes is throttled by the migrate-threads and migrate-sleep configuration parameters. This prevents migration from consuming all I/O bandwidth and causing performance degradation for live traffic.
🔥 Common Interview Mistake

Candidates say "Aerospike reads are fast because it uses SSDs." This misses the critical mechanism. The speed comes from: (1) the index is 100% in RAM — the SSD is never touched for a key-not-found case, (2) Direct I/O bypasses OS page cache — no Linux kernel page management latency, (3) the write path is always append-only sequential — no random-write SSD wear pattern that would cause latency spikes from SSD garbage collection. The combination of these three design choices, not just "using SSD," is the source of Aerospike's performance characteristics.

🗺️ CAP Theorem Positioning

Aerospike's CAP positioning is uniquely configurable — it is one of very few databases that can operate in both CP and AP modes, selectable at the namespace level. This is the critical differentiator to articulate at L7+:

CP Mode (Strong Consistency — SC Mode)

Enabled with strong-consistency true in the namespace configuration. In SC mode, partitions that lose quorum (fewer than replication-factor/2 + 1 nodes accessible) become unavailable rather than serving potentially stale data. Writes require acknowledgment from a majority of roster nodes. This provides linearizability — any read will see the most recent committed write. The cost: during a network partition, the minority side of the split stops accepting writes (and optionally reads), sacrificing availability for consistency.

Use for: financial balances, inventory counts, session state, anything where stale reads cause business harm.

AP Mode (Availability — Default)

Default configuration. During a network partition, both sides of the split continue serving reads and writes from their local partitions. This means both sides may independently accept writes to the same record, creating conflicts. Conflicts are resolved with Last-Write-Wins (based on wall clock timestamp, with generation counter as tiebreaker). The more recent write (by timestamp) survives; the other is silently discarded.

Use for: session cookies, ad frequency caps, recommendation scores, any data where brief staleness or loss of concurrent conflicting writes is acceptable.

Pillar 4 — Failure Modes & Resilience
Detection, automatic recovery, data loss risk, and manual intervention for every failure class

🔴 Failure Scenario Analysis

💀 Single Node Crash (Master)

Detection
Peers miss heartbeats after heartbeat-timeout (default 10 missed intervals, ~3-5 seconds). Cluster initiates reformation without the failed node.
Auto-Recovery
Replica nodes that held replicas for the failed master's partitions are immediately promoted to master for those partitions. Client receives updated partition map on next connection or via info protocol push. Data migration begins to restore the replication factor by creating new replicas on surviving nodes.
Data Loss Risk
With commit-master: writes acknowledged between the last successful replica replication and the crash are lost (typically <100ms of writes). With commit-all: zero data loss. In-progress writes at crash time are lost regardless.
Manual Action
Restart the failed node (it rejoins automatically). If permanent replacement: provision new hardware, install Aerospike, add to cluster via mesh config. Monitor migration progress via AMC or asadm.

👥 Minority Node Failures (e.g., 1 of 5 nodes)

Detection
Same heartbeat mechanism. Each failed node's partitions are identified by surviving majority.
Auto-Recovery
All failed nodes' master partitions are promoted from replicas on surviving nodes. The cluster operates at reduced replica coverage until migration completes. If RF=2 and 2 nodes fail simultaneously, some partitions may have only 1 copy on surviving nodes — still available but no longer protected against another failure.
Data Loss Risk
No data loss if the failed nodes held replicas (not the only copy). If a partition had both its master and all replicas on the failed nodes (extremely unlikely with good placement), that partition's data is lost.
Manual Action
Restore failed nodes as quickly as possible to restore full replication factor. Use asadm --enable -e "show stat namespace" to monitor migration-remaining counts.

🌐 Network Partition (Asymmetric — One Side Has Quorum)

Detection
Each partition side independently detects missed heartbeats from the other side. Both sides attempt cluster reformation with only the nodes they can see. The majority side (quorum) succeeds in forming a new cluster. The minority side also reforms as a smaller cluster (in AP mode) or blocks writes (in SC mode).
Auto-Recovery
AP mode: both sides continue serving reads/writes independently, creating a split-brain scenario. SC mode: majority side operates normally; minority side rejects writes and optionally reads. When the partition heals, the cluster re-merges: SC mode performs a careful merge ensuring no conflicting writes; AP mode uses LWW to resolve conflicts created during the partition.
Data Loss Risk
AP mode: concurrent writes to the same record on both partition sides → the losing write (by timestamp) is silently discarded. SC mode: no data loss but service interruption on the minority side.
Manual Action
Fix the network partition. In AP mode, review application logs for any business-critical operations that may have been affected by LWW resolution after the merge. Consider implementing application-level conflict detection for high-value records.

⚡ Network Partition (Even Split — No Quorum)

Detection
In a 4-node cluster split 2-2: both sides see 2 nodes, and 2 < floor(4/2)+1 = 3 (majority). Neither side has quorum.
Auto-Recovery
SC mode: both sides become unavailable for writes (no quorum on either side). AP mode: both sides operate independently as separate clusters, creating the most dangerous split-brain scenario where both sides accept writes with equal authority.
Data Loss Risk
SC mode: no data loss but complete unavailability. AP mode: both sides accumulate potentially conflicting writes; LWW resolution on re-merge will silently discard one side's writes for any record written on both sides. For a 10-minute partition at 1M ops/sec, this could mean millions of silently lost writes.
Manual Action
For SC deployments: this is expected and correct behavior — restore network and monitor re-merge. For AP deployments: implement a "fencing" mechanism at the application layer — client-side awareness of partition state, or use an external coordination service (ZooKeeper) to determine which side should continue serving writes.

🧠 Split-Brain (Stale Partition Map on Client)

Detection
After a cluster reformation, clients may still have the old partition map cached and route requests to nodes that are no longer masters for those partitions. The node receiving a misrouted request detects it is not the master, returns a NOT_MASTER error to the client.
Auto-Recovery
The Aerospike client library (Go, Java, Python, C) automatically detects NOT_MASTER responses, fetches the updated partition map from the cluster, and retries the request to the correct new master. This is transparent to the application — the application sees a brief latency increase (one additional RTT for the retry) but not an error.
Data Loss Risk
None — NOT_MASTER is a routing error, not a data error. The request is retried successfully once the client updates its map.
Manual Action
No action needed if using official Aerospike client libraries. If using a custom client, implement NOT_MASTER handling with partition map refresh + retry.

🐌 Slow/Degraded Node (Not Dead, But Lagging)

Detection
Heartbeats continue (node is alive from cluster perspective) but transaction latency on the node spikes. Application clients see high p99 latency for requests routed to that node. Aerospike does not automatically detect or reroute based on latency — it only detects hard failures.
Auto-Recovery
None automatic. The cluster continues routing master partition requests to the slow node. Clients will experience high tail latency for any record whose master partition is on the degraded node.
Data Loss Risk
None — the node is still alive and persisting data. Risk is performance degradation, not data loss.
Manual Action
Identify root cause: asadm -e "show latency" per node. Common causes: SSD defragmentation backlog (increase defrag-sleep), OS memory pressure causing swap (ensure adequate RAM), hardware failure (check SSD health via smartctl). In extreme cases, manually remove the node from the cluster to force partition migration to healthy nodes.

💾 Disk Corruption / Partial Write

Detection
Aerospike stores a magic byte and CRC checksum with each record. During startup SSD scan (or Defragmenter scan), corrupted records are detected by CRC mismatch and the 0xAE magic byte check. Aerospike logs corruption events to the system log at ERROR level.
Auto-Recovery
Corrupted records are marked as dead and not served to clients. On restart, the primary index is not populated for corrupted records — they are effectively treated as if they never existed. The Defragmenter will eventually reclaim their space.
Data Loss Risk
Records at corrupted SSD locations are lost. With RF=2+, the replica on another node serves as the authoritative copy — corrupted data loss is limited to the single corrupted physical block. With RF=1, data at corrupted locations is permanently lost.
Manual Action
Replace the failing SSD. After replacement and node restart, data migration restores replicas to the new device. Monitor SSD health proactively with NVMe SMART attributes: nvme smart-log /dev/nvme0 — specifically available spare percentage and percentage used.

⏱️ Clock Skew Between Nodes

Detection
Aerospike uses wall clock timestamps for Last-Write-Wins conflict resolution and TTL expiration. Clock skew is not actively monitored by Aerospike itself — it relies on the OS NTP/chrony synchronization. Aerospike 7.x's XDR with HLC (Hybrid Logical Clocks) is partially mitigated but not eliminated.
Auto-Recovery
Aerospike does not self-detect or compensate for clock skew. If Node A's clock is 5 seconds ahead of Node B's clock, writes to Node A will always "win" LWW conflicts against Node B writes, even if Node B's write was logically more recent.
Data Loss Risk
With significant clock skew (>1 second), LWW conflict resolution produces incorrect results — the semantically "newer" write may be discarded if it came from a clock-behind node. TTL expiration may be premature (clock ahead) or late (clock behind).
Manual Action
Mandatory: run chrony or ntpd on all nodes. Alert on chrony tracking showing offset >100ms. In SC mode, clock skew matters less for conflict resolution (serialized through master) but still affects TTL accuracy. Configure Aerospike AMC to alert on inter-node clock differences.

🌩️ Thundering Herd After Recovery

Detection
After a node recovery, data migration from peers to the recovering node causes a sustained high-read-I/O load on all nodes (they must read their data to send to the recovering node). Simultaneously, the recovering node experiences high write I/O as it receives migration data. This I/O surge can cause temporary latency spikes across the entire cluster.
Auto-Recovery
Aerospike's migration throttling (migrate-threads, migrate-sleep) bounds the migration I/O rate to prevent catastrophic cluster-wide performance degradation. The cluster remains fully operational for client traffic during migration — migration runs at lower priority than foreground transactions.
Data Loss Risk
None from thundering herd itself — it is a performance, not a data integrity issue.
Manual Action
Tune migrate-sleep higher (e.g., 5ms) during peak traffic hours to throttle migration more aggressively. Monitor migration progress with asadm -e "show stat namespace" — specifically migrate_rx_partitions_remaining. Accept longer migration time in exchange for maintaining client SLAs.

🔗 Cascading Failures

Detection
A node failure triggers migration, which increases I/O load on surviving nodes. If surviving nodes are already near capacity, the migration I/O pushes them over the edge, causing transaction timeouts and potentially further node failures — a death spiral. Detected by monitoring cluster-wide latency spike concurrent with a node failure event.
Auto-Recovery
Aerospike has limited automatic protection. The stop-writes-pct threshold prevents disk fill, but does not prevent CPU/memory cascade failures.
Data Loss Risk
If cascade leads to multiple simultaneous node failures, partitions whose all replicas are on failed nodes experience data loss.
Manual Action
Prevention: provision 30–40% headroom on each node (CPU, I/O, memory) so that losing any single node does not push survivors above their safe operating point. During an active cascade, manually freeze migration (asinfo -v "set-config:context=service;migrate-max-num-incoming=0") to stop migration I/O, restore node stability, then re-enable migration gradually.
📐 Back-of-Envelope: Data Loss Window for commit-master

Replication lag to replicas under commit-master: ~500μs on LAN. At 1M writes/second × 500μs replication lag = ~500 in-flight unacknowledged replica writes at any moment. If a master crashes, these 500 writes are lost. For financial applications: 500 potential lost writes at 1M writes/sec is a 0.05% data loss rate per node crash. For ad serving (where each write is an ad impression or frequency cap update): 500 lost writes per crash is operationally acceptable. This quantifies the actual risk of commit-master, allowing an informed decision about whether commit-all's latency cost is justified.

🔑

Aerospike's partition map-based direct routing means clients automatically recover from master failures within one retry (NOT_MASTER → refresh map → re-route), but the thundering herd risk during mass recovery is the most dangerous production failure pattern — pre-provisioning 30% headroom is the only reliable prevention.

Pillar 5 — Scaling Strategies
Vertical limits, horizontal architecture, replication, partitioning, cross-region, and what breaks at scale

📏 Vertical Scaling: How Far and What Breaks First

Aerospike is designed to scale vertically to very large single-server configurations. Its shared-nothing internal architecture means larger hardware = near-linear performance increase up to very high scale:

  • RAM: linear scaling. Each additional GB of RAM extends the index capacity by ~15 million records (at 64 bytes/index entry). A server with 512GB RAM can hold the primary index for ~8 billion records. The upper limit is hardware RAM capacity — there is no software-imposed limit on primary index size relative to RAM.
  • NVMe capacity: near-linear. Additional SSD capacity directly increases data storage capacity. A server with 8 × 8TB NVMe drives = 64TB data capacity. Aerospike's transaction throughput is bounded by SSD IOPS, not just SSD capacity.
  • CPU: transaction thread parallelism. Aerospike scales transaction throughput nearly linearly with CPU core count up to ~48 cores (beyond which the overhead of managing many transaction queues reduces marginal gains). A 48-core server can sustain ~3–4M ops/second with typical workloads.
  • What breaks first: RAM for index. At multi-trillion-record scale, the primary index size exceeds available server DRAM. The Aerospike All-Flash (AAF) storage engine option (available in Aerospike 5.x+) stores the index on flash (NVMe) rather than DRAM, reducing the RAM requirement by ~100x at the cost of ~10x higher index lookup latency (from ~100ns DRAM to ~50μs NVMe). This enables a single server to index far more records than physically possible with DRAM alone.

↔️ Horizontal Scaling

Adding nodes to an Aerospike cluster is operationally simple and architecturally automatic:

Configure the new node

Set the node's mesh or multicast heartbeat configuration to include existing cluster nodes. Start the Aerospike daemon. The new node automatically discovers the cluster via heartbeat.

Cluster reformation

The existing cluster detects the new node. A new partition map is computed: the 4,096 partitions are redistributed across N+1 nodes. The new node receives a portion of each existing node's partitions.

Data migration

Existing nodes begin migrating their newly-reassigned partitions to the new node in the background. This transfers both data and index information. Migration rate is bounded by migrate-threads and migrate-sleep. For a large cluster adding one node, migration completes in minutes to hours depending on data volume.

Live traffic shift

As partitions migrate, the cluster begins routing client requests for those partitions to the new node (for newly received partitions where migration is complete). The transition is continuous and gradual — not a single cutover event.

Key insight: Because Aerospike uses a fixed 4,096-partition scheme with consistent hashing, adding a node does NOT require re-hashing all keys (unlike a naive consistent hashing scheme that re-hashes when the ring changes). Each partition migrates as a unit — the records within a partition don't move individually, they move as a block. This makes resharding operationally simple and deterministic.

📦 Partitioning: How Aerospike Avoids Hotspots

Aerospike uses consistent hash partitioning with RIPEMD-160. The key-to-partition mapping is uniform as long as the key distribution is not heavily skewed. Hotspot scenarios:

  • Hotspot problem: If 80% of traffic accesses the same 100 records (e.g., a global configuration record or a viral trending item), those records' partition masters handle disproportionate load regardless of how many nodes are in the cluster. Hash partitioning distributes records, not access frequency.
  • Hotspot mitigation: (1) Application-layer caching (separate cache layer) for hot records. (2) Record-level read replication: in AP mode, reads can be served by any replica, distributing read load. (3) Bins-as-shards: for a single extremely hot record, split it into N records (e.g., config:global:shard_0 through config:global:shard_N) and aggregate client-side. (4) Distributed rate limiting: use Aerospike's operate() with add() on separate counter bins, distributing counter increments across multiple records.

📐 Back-of-Envelope: Cluster Sizing for RTB

RTB workload: 2M bid requests/second, each requiring a user profile read (500 bytes average record size), 1.5B unique user profiles, 90-day retention. Storage: 1.5B × 500B = 750GB data. Index: 1.5B × 64B = 96GB RAM just for index. Throughput: 2M reads/sec. Per-node capacity (2-socket Intel with 8 NVMe): ~500K reads/sec, 128GB RAM. Nodes needed for throughput: 2M/500K = 4 nodes. Nodes for RAM (with RF=2): 2 × 96GB / 128GB = 1.5 → 2 nodes. Nodes for data (with RF=2): 2 × 750GB / (8 × 8TB) ≈ 0.02 nodes. Throughput is the binding constraint: 4–6 nodes (with headroom). This calculation pattern is exactly what interviewers expect.

🔑

Aerospike's scaling limit is almost always DRAM for the primary index, not SSD capacity or CPU — the 64-byte-per-record index overhead means RAM, not disk or processing power, is typically what forces horizontal scaling, and the All-Flash (AAF) option is the architectural escape valve when DRAM cost becomes prohibitive.

Pillar 6 — Real-World Engineering Decisions
Where Aerospike shines, production examples, anti-patterns, decision matrix, and Day 1 monitoring

✨ Where Aerospike Shines

  • Real-Time Bidding (RTB): The foundational use case. Sub-millisecond user profile lookups at millions of QPS with terabytes of data. No other database matches this combination.
  • Session Management at Scale: Billions of active sessions, each requiring sub-ms read/write, with automatic TTL expiration. Aerospike's native TTL handling (no background reaper needed) and HMA efficiency make it ideal.
  • Fraud Detection: Real-time lookups against hundreds of millions of known-fraudulent entities (IPs, payment methods, device fingerprints) during transaction authorization (<10ms total budget). The combination of sub-ms latency, strong consistency mode (to prevent double-spending through concurrent fraud writes), and binary data type support for compact representations is the winning combination.
  • Rate Limiting at Global Scale: Aerospike's operate() command with add/increment bins provides atomic counter operations. A distributed rate limiter (N requests per second per user) can be implemented with a single Aerospike record per user with TTL set to the rate window — no external lock or coordination needed.

🏭 3 Real-World Production Examples

Example 1 — The Trade Desk

RTB at 10M+ bid requests/second

The Trade Desk, one of the largest demand-side platforms (DSPs), uses Aerospike as its primary user profile store serving its RTB bidding engine. The system must perform profile lookups for every bid request received from ad exchanges worldwide — over 10 million bid requests per second during peak US evening hours. Each profile lookup retrieves behavioral segment data, purchase history indicators, and frequency cap counts for each user.

Why Aerospike? At 10M RPS with a 50ms total bid response budget (including network latency), the database can consume at most 1–2ms of that budget. Only Aerospike's HMA architecture (RAM index + NVMe data) delivers consistent sub-millisecond profile reads at this scale. Redis would require hundreds of terabytes of RAM (cost-prohibitive). Cassandra's p99 latency (typically 5–20ms) is too high. DynamoDB latency is typically 2–5ms even with caching. The architectural uniqueness is irreplaceable here.

Example 2 — Criteo

Personalized Ad Targeting for 1B+ Users

Criteo (French ad-tech company, ~$2B revenue) uses Aerospike to store and serve personalized user interest profiles for over 1 billion users. Their retargeting product requires that every ad impression trigger a lookup of the user's recent browsing history (up to 500KB per user profile), scoring against thousands of advertiser catalogs, within the RTB deadline.

Why Aerospike? Profile sizes at Criteo reach 50KB–500KB per user — far too large to fit in a RAM-only system for 1B users. At 500KB average: 1B users × 500KB = 500TB data. Aerospike's HMA stores this on SSD with only 64GB RAM for the index (1B × 64B), enabling the entire dataset to live on a manageable number of SSD-equipped servers rather than requiring a Redis cluster with 500TB of DRAM. The XDR cross-DC replication maintains a synchronized copy in EU (for GDPR data locality) and US simultaneously.

Example 3 — Airtel (Telecom)

Subscriber Session Management for 300M+ Mobile Users

Bharti Airtel, India's second-largest mobile carrier, uses Aerospike to manage session state for over 300 million active mobile subscribers. Each data session (4G/5G internet connection) generates multiple per-second lookups to validate the subscriber's plan, check data quota remaining, and update usage counters for billing. The system processes ~5 million state update operations per second during peak hours.

Why Aerospike? Telecom session management requires strong consistency (you cannot let a subscriber use data beyond their quota due to a stale read), sub-millisecond response (session validation is in the per-packet data path), and extreme durability (losing billing data = revenue loss and regulatory violation). Aerospike's SC mode with commit-all provides the consistency and durability guarantees. The Persisted Index feature (5.2+) reduces cold start time from 45 minutes to under 2 minutes — critical for disaster recovery scenarios where billing must resume quickly after a regional failure.

🚫 When NOT to Use Aerospike

Anti-Pattern 1: Complex Relational Queries

If your workload requires JOINs across multiple record types, GROUP BY aggregations, or analytical queries across billions of records, Aerospike is the wrong tool. It has no query optimizer, no JOIN capability, and its secondary index at high cardinality performs like a full scan. Use PostgreSQL (OLTP), BigQuery/Snowflake (OLAP), or a hybrid architecture where Aerospike stores hot state and a warehouse stores historical analytics.

Anti-Pattern 2: Rich Text Search

Aerospike has no full-text search capability. If you need to query records by substring matching, relevance scoring, or natural language search, use Elasticsearch or OpenSearch. Aerospike can coexist with a search layer — Aerospike stores the canonical record, Elasticsearch indexes it for search, and application code writes to both.

Anti-Pattern 3: Event Sourcing / Append-Only Logs

Aerospike stores one record per key — it is not a log database. If your model requires immutable append-only writes (event sourcing, audit logs, time-series of raw events), Aerospike's LWW model will overwrite older events. Use Kafka (streaming), Cassandra (time-series with compound keys), or ClickHouse (columnar analytics) for these patterns.

Anti-Pattern 4: Small Teams without Ops Expertise

Aerospike requires careful hardware selection (NVMe RAID, memory sizing), cluster configuration (namespace tuning, defrag parameters), and operational discipline (SSD health monitoring, migration management). For a 3-engineer startup that needs "a database that just works," DynamoDB or MongoDB Atlas provides significantly lower operational overhead at the cost of higher latency and lower peak throughput.

🚨 Day 1 Monitoring Setup

Node Availability Monitor

Alert on any node leaving the cluster partition map. Use asinfo -v "cluster-key" or AMC: if the cluster key changes unexpectedly outside of a planned maintenance window, a node has joined or left. Page on-call immediately.

Disk Utilization Monitor

Alert at disk_pct_avail < 20% (warning) and <10% (critical). Aerospike stops writes at stop-writes-pct (default 90% full = 10% available). A disk full event causes immediate service degradation — monitor before it hits.

Transaction Latency Histogram

Monitor read_tps, write_tps, and per-histogram-bucket latency from asinfo -v "latency:". Alert if p99 latency exceeds your SLA. Configure thresholds: if p99 read > 2ms for NVMe-backed namespaces, something is wrong (defrag pressure, SSD degradation, memory pressure).

Migration Status Monitor

After any cluster topology change, monitor migrate_rx_partitions_remaining and migrate_tx_partitions_remaining. Alert if migration does not complete within 2× the expected migration time (based on data volume and migrate-threads config). Stuck migration indicates a node-to-node connectivity issue.

SSD Health Monitor

External to Aerospike: monitor NVMe SMART attributes via nvme smart-log. Alert if percentage_used > 80% or available_spare < 20%. SSD wear-out is the most common cause of unexpected node failures in production Aerospike deployments — monitor it proactively.

The most common production failure mode for Aerospike is SSD wear-out causing I/O latency spikes that are invisible at the application layer until they become catastrophic — monitoring NVMe SMART health independently from Aerospike's own metrics is mandatory in production deployments.

🔬 How Google Would Approach Aerospike's Problems Differently

Google's infrastructure-scale challenges create requirements that push beyond even Aerospike's capabilities:

  • Bigtable's approach vs Aerospike's approach: Both are built for high-performance key-value access. 🏠 Multi-Tenancy: Noisy Neighbor Isolation

Aerospike's namespace model provides a limited form of multi-tenancy isolation:

  • Namespace-level storage isolation: Different namespaces can be assigned to different SSD devices. Namespace A (ad serving) can use NVMe drives for ultra-low latency; Namespace B (analytics) can use SATA SSDs for higher capacity at lower cost. This prevents storage I/O from different workloads competing on the same device.
  • Rate limiting via transaction queues: Aerospike does not natively provide per-namespace or per-client transaction rate limiting. A noisy tenant that sends 1M ops/sec will consume proportional server resources, affecting all other tenants on the same cluster. Mitigation: application-level rate limiting at the API gateway; or physical cluster separation for critical workloads.
  • The hard truth: For strict multi-tenancy with SLA isolation guarantees between tenants, physical cluster separation is the only reliable approach in Aerospike. Namespace isolation provides storage isolation but not compute or network isolation. Google-scale multi-tenancy on Aerospike would require a resource governor layer above the Aerospike API that throttles per-tenant request rates and provides per-tenant latency SLA enforcement.

🌐 Data Residency and Regulatory Compliance

  • XDR topology-aware replication for GDPR: Configure XDR to replicate EU user data only to EU DC (set-filter in XDR configuration): records with region:EU tag are replicated to EU cluster only; US records to US cluster only. This requires application-level tagging of every record with a region attribute.
  • Namespace-level data residency: Create separate namespaces for EU and non-EU data. EU namespace exists only in EU clusters (no XDR to US). Non-EU namespace replicates globally. Application routing logic must direct EU user writes to the EU namespace.
  • Key-based namespace routing challenge: Implementing data residency in Aerospike requires the application to know, at write time, which namespace (and therefore which DC) should store a given record. This couples the application to data residency logic — a potential maintenance burden as regulations evolve.

⚡ L7/L8 Differentiator

The hyperscale insight is that Aerospike's per-node storage architecture has a fundamental ceiling: the 4,096-partition scheme means that beyond ~200 nodes, the partition-per-node ratio falls so low that any single large migration event triggers a major data movement percentage across the cluster. This is the architectural limit that drives organizations like large telcos to run multiple independent Aerospike clusters with application-layer routing, rather than a single massive cluster.

⚡ Step 4: 10 Rapid-Fire Questions

1
What happens to data in Aerospike's write buffer (WBLOCK) if the node crashes before it is flushed to SSD?

Data in the in-memory WBLOCK at crash time is permanently lost — Aerospike's append-only write model stages writes in a RAM buffer before flushing to SSD, and a crash clears that RAM. With RF=2 and commit-master policy, the replica already has a copy of all previously acknowledged writes, so data loss is limited to the current WBLOCK contents (typically <100ms of writes). With commit-all policy, all acknowledged writes are confirmed on all replicas before acknowledgment, so a master crash loses nothing that was acknowledged to the client — only writes that were in-flight at crash time (never acknowledged) are lost. The risk is quantifiable: WBLOCK_size / write_bandwidth = data loss window. At 128KB WBLOCK and 100MB/s write rate, the data loss window is ~1.28ms of writes.

2
Explain Aerospike's partition map and why having 4096 partitions is a specific design choice, not an arbitrary number.

4096 is chosen as a power-of-two value large enough to ensure fine-grained, even distribution across a reasonable cluster size (up to ~200 nodes before granularity becomes limiting) while small enough that the partition map (4096 entries × a few bytes each) fits in a single network packet for gossip distribution. Using 4096 means the first 12 bits of the 20-byte RIPEMD-160 digest determine the partition (2^12 = 4096), making the partition assignment a simple bit-shift — no modulo operation. At 200 nodes, 4096/200 = ~20 partitions per node, still providing meaningful load distribution granularity. Below ~4 nodes, some nodes hold more than 1000 partitions, which is fine. The choice is a deliberate balance between distribution granularity and gossip overhead. Changing this number requires a full cluster rebuild — it is not runtime-configurable.

3
You have a high-value record that is read 10 million times per second across your cluster but only written 10 times per second. How do you prevent this record's master node from becoming a bottleneck?

This is the hot record / hot partition problem. The partition containing this record's master will handle all 10M reads/sec regardless of cluster size — adding more nodes doesn't help because you can't distribute a single record's reads across nodes in normal Aerospike operation. Solutions: (1) Application-layer caching (memcached/local process cache): cache the record in the application for ~100ms; at 10M reads/sec with 10 writes/sec, the record changes less than once per million reads — stale cache tolerance is near-perfect here. (2) Enable AP read from replica: in AP mode, configure reads to be served by any replica — this distributes 10M reads across RF nodes instead of concentrating them on the master. (3) Record sharding: split the record into N copies (e.g., config:global:replica_0 through config:global:replica_7), randomly select a copy to read — write to all N copies. This distributes reads across N different partitions/nodes. Technique 3 is the most robust for true hyperscale but requires application complexity.

4
What is the Aerospike Defragmenter, why is it necessary, and what happens if it falls behind?

The Defragmenter is a background process required by Aerospike's append-only write model. Because Aerospike never overwrites data in place (always appends new versions to the end of current write blocks), old/deleted/expired record versions accumulate as dead space on SSD. The Defragmenter continuously scans SSD extents, identifies those with low live-record density (below defrag-lwm-pct, default 50%), reads the live records out, rewrites them to new extents, and marks the old extent as free. If the Defragmenter falls behind — which happens when write/delete rate exceeds defragmentation throughput — disk utilization climbs toward the stop-writes-pct threshold (default 90% full). At that threshold, Aerospike stops accepting writes entirely, causing immediate service outage for write traffic. The recovery is to wait for defragmentation to reclaim space, which can take minutes to hours depending on the backlog. Prevention: tune defrag-sleep lower (more aggressive defragmentation), ensure sufficient SSD I/O bandwidth headroom, and alert at 70% disk utilization before hitting the 90% stop-writes threshold.

5
How does Aerospike's strong consistency mode (SC mode) work, and what is the "regime" concept?

SC mode uses a concept of "regimes" — a cluster generation counter that increments on each cluster topology change. Every write in SC mode is tagged with the current regime number. When a network partition occurs and the cluster splits, both halves enter different regimes (they see different topology). The majority side's regime is authoritative; the minority side's regime is invalid. When the partition heals, nodes in the lower (minority) regime must catch up: they replay the difference between their regime's writes and the majority side's writes using the regime log. This ensures that any write made on the minority side during the partition (in a regime that was later invalidated) is either reconciled or rejected. The key insight: in SC mode, Aerospike does NOT use Last-Write-Wins for conflict resolution — it uses regime ordering to determine which writes are authoritative. A write from the minority regime is not "newer" just because its timestamp is later; it is invalid because its regime was invalidated by the majority cluster's continued operation. This is fundamentally different from AP mode's LWW behavior and is the mechanism that provides linearizability.

6
Why does Aerospike use RIPEMD-160 for key hashing instead of SHA-256 or MD5?

RIPEMD-160 produces a 20-byte (160-bit) digest, which is smaller than SHA-256 (32 bytes) while providing essentially the same distribution uniformity for a non-cryptographic use case (uniform hash distribution, not collision resistance for adversarial inputs). Compared to MD5 (16 bytes), RIPEMD-160 provides better collision resistance and is still considered cryptographically sound for non-adversarial hashing. The critical constraint: the Primary Index entry is exactly 64 bytes (one CPU cache line). Using RIPEMD-160's 20-byte digest leaves room for 44 bytes of metadata (generation, TTL, set ID, device offset, etc.) within the 64-byte budget. If SHA-256 (32 bytes) were used, only 32 bytes would remain for metadata — insufficient for Aerospike's per-record metadata needs. The 64-byte cache-line alignment is not accidental: it ensures every primary index lookup is a single L1 cache line fetch, minimizing CPU cache misses for index operations and contributing to the sub-100ns index lookup performance.

7
Describe Aerospike's approach to multi-record transactions in v7.0+ and what trade-offs it introduces.

Aerospike 7.0 introduced Multi-Record Transactions (MRT) using a two-phase commit protocol. In Phase 1 (prepare), the transaction coordinator (a special transaction object in the client library) sends lock requests to all partitions involved in the transaction; each partition master acquires a provisional lock on the targeted records and returns a "prepared" acknowledgment. In Phase 2 (commit), the coordinator sends commit messages to all partition masters, which atomically apply their changes and release locks. If any participant fails to prepare in Phase 1, the coordinator sends abort messages and all provisional changes are rolled back. Trade-offs: (1) Latency: MRT requires 2 network round trips (prepare + commit) instead of 1, roughly doubling transaction latency compared to single-record operations. (2) Lock contention: records under provisional lock in Phase 1 are not accessible to other transactions, creating potential contention on hot records. (3) Distributed transaction coordination is vulnerable to coordinator failure — if the client crashes mid-transaction, the locks must time out before other transactions can proceed (default lock timeout: several seconds). (4) Cross-partition transactions do not benefit from Aerospike's shared-nothing partition model — they inherently require coordination, limiting parallelism.

8
Your Aerospike cluster's p99 latency suddenly spikes from 500μs to 50ms. Walk through your diagnosis process.

Step 1: Check if the spike is cluster-wide or single-node using asadm -e "show latency" per node. If single-node: that node is the problem. If cluster-wide: a systematic issue (network, load spike, cluster reformation). Step 2 (single-node): Check the node's SSD health — is defragmentation running aggressively? (asinfo -v "stats" -h node, look at defrag_reads rate). Is the SSD itself degraded? (nvme smart-log /dev/nvme0 — check media_errors and percentage_used). Is memory pressure causing swap? (free -m — any swap usage at all is catastrophic for Aerospike latency). Step 3: Check transaction queue depth (asinfo -v "queues:") — deep queues indicate CPU saturation. Step 4: Check disk fill percentage — are we above 80%? Heavy defragmentation at high fill causes I/O stall cycles. Common root causes in order of frequency: (1) SSD wear causing elevated read latency, (2) memory pressure/swap, (3) defragmentation I/O saturation, (4) network congestion between nodes causing replication delays (commit-all path). The resolution depends on root cause: SSD replacement, memory expansion, defrag parameter tuning, or network remediation respectively.

9
Compare Aerospike's approach to eviction vs. TTL. When would you use each, and what are the failure modes?

TTL (Time-To-Live) is a per-record expiration time set at write time — the record expires and becomes invisible to reads after a specified number of seconds. TTL expiration is lazy (the record remains on disk until the eviction or defragmentation process removes it) but the Primary Index entry is invalidated at TTL expiry, making the record unreadable. TTL is deterministic: you know exactly when a record will expire. Eviction is a namespace-level policy that removes the oldest records (by TTL) when the namespace reaches its configured memory or storage threshold (evict-used-pct). Eviction is reactive — records are removed when resources are pressured, not on a fixed schedule. Failure modes: (1) TTL expiration without eviction enabled: on disk, expired records accumulate as dead space that the Defragmenter must reclaim. If record creation rate >> defragmentation rate, disk fills up. (2) Eviction with no TTL: eviction has no "oldest" concept without TTL — records without TTL are treated as having infinite TTL and will never be evicted (they are "zero TTL" records, immune to eviction by default, configurable via evict-zero-ttl). (3) TTL + eviction combined: if eviction fires before TTL expiration, it evicts data that the application expected to be available. Applications must be designed for this possibility.

10
What is Aerospike's "rack-aware" feature, why does it matter for high availability, and how does it work?

Rack awareness is a placement policy that ensures replica partitions are always placed on nodes in different physical failure domains (racks, availability zones, or power domains). Without rack awareness, in a 6-node cluster across 2 racks, the default placement might put both the master and replica for a partition on nodes in the same rack — a single rack failure (power loss, top-of-rack switch failure) would then take both copies offline simultaneously, causing data unavailability. With rack awareness enabled, Aerospike's partition map calculation is constrained: for a partition with master on Rack A, the replica must be placed on a node in Rack B (or a different configured rack). This is configured via the rack-id parameter on each node — nodes with the same rack-id are considered to be in the same failure domain. The trade-off: rack awareness reduces the number of eligible nodes for replica placement, which can slow migration (replica must migrate to a specific rack) and may limit the effectiveness of load balancing if racks have unequal node counts. For truly high-availability production deployments with Aerospike, rack awareness is not optional — it transforms a "high availability unless your rack fails" system into a "high availability unless two racks fail simultaneously" system.

# asinfo -v "latency:" output example reads: # Columns: time, ops/sec, pct >1ms, pct >8ms, pct >64ms 15:30:00, 245123 ops/sec, 0.34%, 0.01%, 0.00% ← HEALTHY 15:30:10, 244876 ops/sec, 0.35%, 0.02%, 0.00% ← HEALTHY 15:30:20, 98234 ops/sec, 12.45%, 4.31%, 0.87% ← PROBLEM: 4.31% >8ms # 15:30:20: TPS dropped by 60%, 4.31% of reads took >8ms # Diagnosis: SSD defrag burst (common at top of hour if hourly cleanup runs) # OR: migration started competing with reads for I/O bandwidth writes_master: 15:30:00, 245100 ops/sec, 0.22%, 0.00%, 0.00% 15:30:20, 98200 ops/sec, 10.11%, 3.20%, 0.15% # Both reads AND writes degrade simultaneously: # → SSD I/O is the bottleneck (not network, not CPU alone) # If only writes degrade with reads fine: # → Replication network congestion (commit-all path slow)

How to read the latency histogram:

  • >1ms column: The fraction of operations exceeding 1ms. For NVMe-backed namespaces, >2% consistently indicates something is wrong. For RAM-backed namespaces, >0.1% is concerning.
  • >8ms column: Any non-trivial percentage here (>0.1%) is alarming for production Aerospike — 8ms is 16× the expected 500μs latency. This indicates severe I/O pressure or a node-level problem.
  • >64ms column: Any non-zero value here is a critical alert — these are transactions experiencing 64× their expected latency. Likely causes: disk fault, swap usage, or OS-level I/O starvation.

🔔 Part 5: End-to-End Monitoring Simulation
Incident Simulation: Silent Data Loss via Disk Fill

How Aerospike silently stops accepting writes when disk fills, and how monitoring catches it

T-60 minutes: Disk utilization at 78%. Normal. No alerts firing. Engineers unaware.

T-30 minutes: A new data pipeline goes live, doubling the write rate for the entity_features namespace. Disk utilization rising at 0.5%/minute. The Tier 2 alert fires at 80% (Slack notification only): "Aerospike namespace entity_features disk utilization at 80.3% — rising. Defrag rate insufficient."

T-15 minutes: Disk utilization hits 87%. The Tier 1 critical alert fires: "CRITICAL: Aerospike entity_features approaching stop-writes threshold (87%). Immediate action required." On-call engineer is paged. The engineer checks asadm -e "show stat namespace": defrag_reads rate is 50K/sec but disk is filling at 200K records/sec of new writes. Defrag is losing the race.

T-0: Disk hits 90%. Aerospike's stop-writes-pct threshold is crossed. The node stops accepting writes for the entity_features namespace. Clients receive AEROSPIKE_ERR_DEVICE_OVERLOAD (or AEROSPIKE_ERR_FULL in older versions). Application error rate spikes. Fraud detection model feature updates start failing — the system continues to score fraud based on increasingly stale features.

Diagnosis using Grafana: Panel 3 (disk utilization) shows the hockey-stick rise starting at T-60. Panel 2 (write TPS) shows a step-change at T-30 (new pipeline). Panel 7 (defrag rate) shows defrag at ~25% of the write rate — completely unable to keep up. The root cause is the new pipeline, not a defrag issue. Fix: Rate-limit the new pipeline, or scale horizontal by adding a new node (which triggers migration and distributes the disk load). Panel 6 (migration status) would show new migration in progress after adding the node.

Key lesson: The 80% threshold alert at T-30 gave 30 minutes of warning. Without it, the on-call would have seen the issue only at T-0 when writes started failing — at that point, recovery is reactive. The proactive alert enables preventive action. This is why the Day 1 monitoring setup is non-optional.

🔑

The most critical monitoring insight for Aerospike: disk fill is a silent, predictable, preventable failure mode that causes hard write failures with no graceful degradation — alerting at 80% disk utilization (not 90%) is the only way to have response time before the system stops accepting writes entirely.
